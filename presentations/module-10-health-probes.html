💓 Why Health Checks Matter

From Process Status to Application Health

Let's start with the fundamental question: why do we need probes at all? Can't Kubernetes just check if the container process is running? The answer reveals a critical gap in container orchestration.

🔄 Pod Lifecycle & Restart Policies

Pod Phase Progression

Pending → Running → Succeeded / Failed

  • Always: Restart container regardless of exit code (default for Deployments)
  • OnFailure: Restart only on non-zero exit (common for Jobs)
  • Never: Never restart — let the pod enter Failed state
Key Insight: Restart policies only respond to process exits — they cannot detect a hung process, a deadlock, or a broken database connection.
The pod lifecycle is straightforward — pods move through phases and restart policies determine what happens on container exit. But notice the gap: restart policies only trigger when the process actually dies. If your app is stuck in an infinite loop or can't reach its database, the process is still "running" from Kubernetes' perspective.

💥 What Happens Without Probes

🔴 Cascading Failure Scenario

  1. App loses database connection
  2. Process stays running, returns 500s
  3. Service keeps routing traffic to unhealthy pod
  4. Users experience errors across the cluster
  5. Load balancer spreads failure further

✅ With Probes Configured

  1. App loses database connection
  2. Readiness probe fails → pod removed from Service
  3. Liveness probe fails → container restarted
  4. Traffic routed only to healthy pods
  5. Self-healing with zero manual intervention
This side-by-side comparison shows the real impact. Without probes, a single unhealthy pod can poison your entire service. With probes, Kubernetes automatically routes traffic away from unhealthy pods and restarts them — true self-healing infrastructure.

🔍 Probe Types

Liveness, Readiness, and Startup

Now let's look at the three probe types Kubernetes provides. Each serves a distinct purpose, and understanding when to use which is critical for production reliability.

💚 Liveness Probes

Purpose: Detect when a container is stuck and needs to be restarted. Answers: "Is the application still alive?" 
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  containers:
  - name: app
    image: myapp:1.0
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 15
      periodSeconds: 10
      failureThreshold: 3
  • Failure action: kubelet restarts the container
  • Use for deadlocks, infinite loops, unrecoverable states
  • Should check internal health only — not external dependencies
Liveness probes are your safety net against zombie processes. When the probe fails, kubelet kills and restarts the container. The key design principle: liveness probes should only check the application's own health, never external dependencies. If you check a database in your liveness probe and the DB goes down, every pod gets restarted — making things worse.

💚 Liveness: TCP & Exec Variants

TCP Socket Probe

livenessProbe:
  tcpSocket:
    port: 3306
  initialDelaySeconds: 30
  periodSeconds: 10

Succeeds if TCP connection opens. Ideal for databases, message brokers.

Exec Command Probe

livenessProbe:
  exec:
    command:
    - /bin/sh
    - -c
    - pg_isready -U postgres
  initialDelaySeconds: 30
  periodSeconds: 15

Runs command inside container. Exit code 0 = success.

TCP probes simply try to open a socket — useful when HTTP endpoints aren't available, such as databases or Redis instances. Exec probes run a command inside the container, giving you maximum flexibility. Here we use pg_isready to check PostgreSQL health from within the container.

🔵 Readiness Probes

Purpose: Control when a pod receives traffic from a Service. Answers: "Is the application ready to accept requests?" 
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  containers:
  - name: app
    image: myapp:1.0
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      successThreshold: 1
      failureThreshold: 3
  • Failure action: pod removed from Service endpoints (not restarted)
  • Can check downstream dependencies (caches warmed, connections established)
  • Pod stays running — can recover and rejoin the Service
Readiness probes are fundamentally different from liveness probes. When a readiness probe fails, the pod is removed from the Service's endpoint list — no traffic gets routed to it. But the pod stays running and can recover. This is perfect for temporary issues like cache warming or brief dependency outages. Unlike liveness probes, readiness probes CAN check external dependencies.

🔷 Startup Probes

Purpose: Protect slow-starting containers from being killed by liveness probes before they're ready. 
apiVersion: v1
kind: Pod
metadata: { name: legacy-app }
spec:
  containers:
  - name: app
    image: legacy-app:2.0
    startupProbe:
      httpGet: { path: /healthz, port: 8080 }
      failureThreshold: 30
      periodSeconds: 10
    livenessProbe:
      httpGet: { path: /healthz, port: 8080 }
      periodSeconds: 10
  • Liveness and readiness probes disabled until startup probe succeeds
  • Max startup time = failureThreshold × periodSeconds (300s above)
  • Ideal for legacy apps, Java apps with long JVM warmup, ML model loading
Startup probes solve the classic problem of slow-starting applications. Before startup probes existed, you had to set a very large initialDelaySeconds on your liveness probe, which meant slow detection of actual failures. With startup probes, you get a generous window for startup but fast failure detection once the app is running. In this example, the app gets up to 300 seconds to start, but once running, liveness checks happen every 10 seconds.

📈 Probe Comparison

Aspect Liveness Readiness Startup
Question Is it alive? Can it serve traffic? Has it finished starting?
On Failure Restart container Remove from Service Restart container
Check Dependencies? No Yes No
When Active After startup succeeds After startup succeeds Container start only
Default if Omitted Assumed alive Assumed ready Assumed started
Production Recommendation: Use all three probes together for maximum resilience. Startup handles initialization, liveness catches deadlocks, readiness controls traffic flow.
This table is your reference guide. The critical difference is the failure action: liveness and startup kill the container, while readiness just removes it from traffic rotation. Notice that liveness and readiness probes don't even activate until the startup probe has succeeded — this is the coordination mechanism that makes all three work together.

⚙️ Probe Configuration

Parameters, Protocols, and Tuning

Now that we understand the three probe types, let's dive into the configuration parameters that control their behavior. Getting these right is the difference between a resilient application and one that flaps between healthy and unhealthy states.

⚙️ Probe Parameters

Parameter Default Description
initialDelaySeconds 0 Seconds to wait before first probe
periodSeconds 10 Interval between consecutive probes
timeoutSeconds 1 Max seconds to wait for probe response
successThreshold 1 Consecutive successes to be considered healthy
failureThreshold 3 Consecutive failures before taking action
Time to failure: initialDelaySeconds + (periodSeconds × failureThreshold) — e.g., 15 + (10 × 3) = 45 seconds before restart.
These five parameters control all probe timing. The most important formula to understand is the total time before action is taken. With defaults, it takes 30 seconds of failed probes before a restart. For liveness probes, successThreshold must be 1 — Kubernetes enforces this. For readiness probes, you can require multiple consecutive successes before adding the pod back to the Service.

🌐 HTTP Probe Deep Dive

livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    scheme: HTTPS
    httpHeaders:
    - name: Authorization
      value: Bearer abc123
    - name: X-Custom-Header
      value: probe-check
  initialDelaySeconds: 10
  periodSeconds: 10
  timeoutSeconds: 3
  • Success: HTTP status code ≥ 200 and < 400
  • port: Container port number or named port reference
  • scheme: HTTP (default) or HTTPS — does not verify certificates
  • httpHeaders: Custom headers for auth or routing
  • path: Dedicated health endpoint, not a business endpoint
HTTP probes are the most common type for web applications. The kubelet makes an HTTP GET request to the specified path and port. Any 2xx or 3xx response is considered success; 4xx and 5xx mean failure. When using HTTPS, certificate validation is skipped, so self-signed certs work fine. Custom headers are useful when your app requires authentication even for health endpoints.

🔌 TCP Probe Deep Dive

spec:
  containers:
  - name: redis
    image: redis:7
    ports: [{ containerPort: 6379 }]
    readinessProbe:
      tcpSocket: { port: 6379 }
      initialDelaySeconds: 5
      periodSeconds: 10
    livenessProbe:
      tcpSocket: { port: 6379 }
      initialDelaySeconds: 15
      periodSeconds: 20
Best For: Databases, caches, message brokers — any service that doesn't speak HTTP.
Limitation: Only confirms the port is open — does not verify the application is functioning correctly.
TCP probes attempt to open a TCP connection to the specified port. If the connection succeeds, the probe passes. This is the simplest probe type — useful when HTTP isn't available. The trade-off is that it only tells you the process is listening on a port, not that it's actually processing requests correctly. A database might accept connections but be in a read-only state, for instance.

💻 Exec Probe Deep Dive

spec:
  containers:
  - name: app
    image: myapp:1.0
    livenessProbe:
      exec:
        command:
        - /bin/sh
        - -c
        - |
          test -f /tmp/healthy &&
          find /tmp/healthy -mmin -1 | grep -q healthy
      initialDelaySeconds: 10
      periodSeconds: 15
      timeoutSeconds: 5
  • Exit code 0 = success, any other code = failure
  • Runs inside the container — must have required binaries
  • Useful for custom checks: file existence, process state, script validation
  • Each exec probe creates a process — watch for PID pressure in high-frequency probes
Exec probes give you complete flexibility — you can run any command or script inside the container. The example checks both that a file exists and that it was modified within the last minute, which is a common pattern where the application touches a file periodically to signal health. Be aware that exec probes spawn a process each time, so setting periodSeconds too low can cause PID exhaustion in containers with low PID limits.

🛰 gRPC Probes

Native gRPC Health Checking (Kubernetes 1.24+)

Kubernetes can natively call the gRPC Health Checking Protocol without needing a sidecar or exec workaround.

  • service: Optional — specifies which gRPC service to check (empty = overall status)
  • App must implement the grpc.health.v1.Health service
  • Returns SERVING, NOT_SERVING, or UNKNOWN
gRPC probes are a relatively recent addition to Kubernetes. Before this feature, you had to use exec probes with grpc_health_probe or a sidecar container. Now Kubernetes natively speaks the gRPC health protocol. Your application just needs to implement the standard grpc.health.v1.Health service interface.

🛰 gRPC Probe Configuration

spec:
  containers:
  - name: grpc-service
    image: mygrpc:1.0
    ports:
    - containerPort: 50051
    livenessProbe:
      grpc:
        port: 50051
        service: "myservice"
      initialDelaySeconds: 10
      periodSeconds: 10
    readinessProbe:
      grpc:
        port: 50051
      initialDelaySeconds: 5
      periodSeconds: 5
Tip: The optional service field lets you check specific services within a multi-service gRPC server.
This example shows both liveness and readiness gRPC probes. The liveness probe checks a specific service named "myservice", while the readiness probe checks the overall server status by omitting the service field.

🎯 Probe Best Practices

Patterns, Anti-Patterns, and Production Wisdom

Configuring probes correctly is surprisingly nuanced. Let's look at the patterns that work, the anti-patterns to avoid, and real-world examples from production environments.

⚠️ Common Anti-Patterns

🔴 Too Aggressive

  • periodSeconds: 1
  • failureThreshold: 1
  • Restart loops on transient issues

🔴 Checking Dependencies

  • Liveness checks DB connection
  • DB down → all pods restart
  • Cascading failure

🔴 Expensive Operations

  • Heavy database queries
  • Probe exceeds timeoutSeconds
  • Causes false failures
These are the three most common mistakes teams make with probes. Aggressive probes cause restart storms — a brief network hiccup shouldn't kill every pod. Checking dependencies in liveness probes creates a domino effect where an external outage becomes a self-inflicted cluster-wide restart. And expensive probes that exceed the timeout cause false positives — the app is healthy but the probe thinks it's dead because the check itself is too slow.

🛠️ Probe Design Guidelines

💚 Liveness Probe Rules

  • Check only internal application state
  • Keep it lightweight (< 100ms)
  • Use generous thresholds (failureThreshold: 3+)
  • Never call external services
  • Consider: "Would a restart fix this?"

🔵 Readiness Probe Rules

  • Check dependencies if needed
  • Verify cache is warmed
  • Can be more thorough than liveness
  • Consider: "Can I serve a good response?"
  • Use successThreshold: 2+ to avoid flapping
Golden Rule: Liveness answers "should I restart?" — Readiness answers "should I receive traffic?"
The fundamental distinction: liveness probes determine if a restart will help, readiness probes determine if the pod can serve valid responses right now. Use different endpoints for each — /healthz for liveness that checks internal state, and /ready for readiness that can verify downstream connectivity. A common pattern is having liveness return 200 as long as the event loop is responsive, while readiness checks the connection pool and cache state.

🌎 Real-World: Web Application

spec:
  containers:
  - name: web
    image: web-app:3.2
    ports: [{ containerPort: 8080 }]
    startupProbe:
      httpGet: { path: /healthz, port: 8080 }
      failureThreshold: 30
      periodSeconds: 5
    livenessProbe:
      httpGet: { path: /healthz, port: 8080 }
      periodSeconds: 10
      timeoutSeconds: 3
      failureThreshold: 3
    readinessProbe:
      httpGet: { path: /ready, port: 8080 }
      periodSeconds: 5
      successThreshold: 2
      failureThreshold: 3
Startup: 150s max • Liveness: restart after 30s • Readiness: remove from LB within 15s
This is a production-ready configuration for a typical web application. The startup probe gives 150 seconds for initialization — enough for dependency injection and cache warming. Liveness checks a lightweight internal endpoint every 10 seconds. Readiness uses a separate endpoint that verifies downstream services, and requires 2 consecutive successes before adding the pod back to the Service, preventing flapping.

🚀 Real-World: Microservice Probes

gRPC Microservice

containers:
- name: order-svc
  image: order-service:1.5
  livenessProbe:
    grpc:
      port: 50051
    periodSeconds: 10
  readinessProbe:
    grpc:
      port: 50051
      service: "orders"
    periodSeconds: 5
  • Liveness: Checks the gRPC server is responsive (overall status)
  • Readiness: Checks the specific "orders" service is ready to handle requests
The gRPC microservice uses native gRPC probes — liveness checks the server is responsive, readiness checks the specific "orders" service. This is a common pattern for microservice architectures using gRPC.

🚀 Real-World: Sidecar Probes

Database Sidecar (PgBouncer)

containers:
- name: pgbouncer
  image: pgbouncer:1.20
  livenessProbe:
    tcpSocket:
      port: 6432
    periodSeconds: 15
  readinessProbe:
    exec:
      command:
      - /bin/sh
      - -c
      - psql -h 127.0.0.1
        -p 6432 -U probe
        -c "SELECT 1"
    periodSeconds: 10
  • Liveness (TCP): Is PgBouncer listening on port 6432?
  • Readiness (Exec): Can it actually execute a query end-to-end?
  • This layered approach catches both process-level and application-level failures
The database sidecar pattern shows a common approach: TCP for liveness (is PgBouncer listening?) and exec for readiness (can it actually execute a query?). This layered approach catches both process-level and application-level failures.

🔎 Debugging Probe Failures

Step 1: Check Pod Events

# See probe failure events
kubectl describe pod my-app | grep -A 5 "Events"
# Look for Warning events
kubectl get events --field-selector reason=Unhealthy \
  --sort-by='.lastTimestamp'

Step 2: Check Logs & Test Manually

# Current and previous container logs
kubectl logs my-app -c app --tail=50
kubectl logs my-app -c app --previous

# Exec into pod and test the health endpoint
kubectl exec my-app -- curl -s http://localhost:8080/healthz
kubectl exec my-app -- wget -qO- http://localhost:8080/ready
When probes fail, follow this three-step debugging process. First, describe the pod to see events — Kubernetes logs every probe failure as a Warning event with details about the failure. Second, check container logs, especially previous logs if the container was restarted. Third, exec into the pod and manually test the health endpoints to see what they return. Often the issue is a misconfigured port, wrong path, or the endpoint returning a 503 during initialization.

🛑 Graceful Shutdown & preStop Hooks

Pod Termination Sequence

1. Pod set to Terminating → 2. preStop hook runs → 3. SIGTERM sent → 4. Grace period countdown → 5. SIGKILL if still running

  • preStop + SIGTERM: Happen concurrently with endpoint removal from Services
  • Race condition: Add a sleep 5 in preStop to let endpoints propagate
  • terminationGracePeriodSeconds: Must exceed preStop + drain time
Graceful shutdown is tightly coupled with probes. When Kubernetes terminates a pod, it removes the pod from Service endpoints and runs the preStop hook simultaneously. This creates a race condition — the kube-proxy might still route traffic for a few seconds after termination starts.

🛑 preStop Hook Configuration

spec:
  terminationGracePeriodSeconds: 60
  containers:
  - name: app
    image: myapp:1.0
    lifecycle:
      preStop:
        exec:
          command:
          - /bin/sh
          - -c
          - |
            # Signal app to stop accepting connections
            curl -s http://localhost:8080/shutdown
            # Wait for in-flight requests to drain
            sleep 15
Key rule: Always set terminationGracePeriodSeconds high enough to cover your preStop hook duration plus your application's drain time.
The sleep in preStop ensures existing connections drain before the app shuts down. Always set terminationGracePeriodSeconds high enough to cover your preStop hook plus your application's drain time.

🛡️ Pod Disruption Budgets (PDBs)

PDBs limit how many pods in a Deployment can be unavailable during voluntary disruptions (node drains, cluster upgrades, autoscaling).

Min Available

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: web

Max Unavailable

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
spec:
  maxUnavailable: "25%"
  selector:
    matchLabels:
      app: api
PDBs are the bridge between health probes and operational safety. They ensure that when you're draining nodes or upgrading your cluster, you don't accidentally take down too many pods at once. Use minAvailable for small replica counts and maxUnavailable with percentages for larger deployments.

🛡️ PDB Behavior & Interactions

  • Works with kubectl drain, cluster autoscaler, and upgrade controllers
  • Does not protect against involuntary disruptions (node crashes, OOM kills)
  • Combine with readiness probes — unready pods count as unavailable
Probe interaction: A pod that fails its readiness probe counts as unavailable, which can block further voluntary disruptions. Always consider PDB + readiness probe behavior together.
Note the interaction with readiness probes: a pod that fails its readiness probe counts as unavailable, which can block further voluntary disruptions.

🔍 Debugging Your Application

Common Failure States & Essential Troubleshooting Tools

Health probes help Kubernetes detect problems automatically, but when something goes wrong you need to diagnose the root cause yourself. This section covers the most common failure states you'll encounter in EKS and the kubectl tools to investigate them.

🚨 Common Application Failure States

🔴 CrashLoopBackOff

  • OOMKilled — memory limit exceeded
  • Missing dependencies or binaries
  • Configuration errors at startup

🔴 ImagePullBackOff

  • Wrong image name or tag
  • ECR auth failure or expired token
  • Registry unreachable from EKS nodes

🔴 Pending Pods

  • Insufficient CPU/memory on nodes
  • Node affinity or taint mismatch
  • PVC stuck in pending (EBS not bound)
CreateContainerConfigError: Missing ConfigMap or Secret referenced by the pod spec — container never starts.
OOMKilled: Container exceeded its memory limit. Increase resources.limits.memory or fix the memory leak.
These are the five failure states you'll encounter most often in EKS. CrashLoopBackOff means the container starts and immediately exits — Kubernetes backs off exponentially between restarts. ImagePullBackOff is usually an ECR authentication issue or a typo in the image reference. Pending pods indicate a scheduling problem — the cluster can't find a node that meets the pod's requirements. CreateContainerConfigError is a silent killer — the pod never starts because a ConfigMap or Secret it references doesn't exist. And OOMKilled is the kernel's out-of-memory killer terminating your process — you'll see exit code 137.

🛠️ Essential Troubleshooting Tools

# Inspect pod status, events, and conditions
kubectl describe pod <pod-name>
kubectl describe node <node-name>

# View logs — current, previous crash, specific container
kubectl logs <pod> --tail=100
kubectl logs <pod> --previous
kubectl logs <pod> -c <container> --follow

# Interactive shell into a running container
kubectl exec -it <pod> -- /bin/sh

# Ephemeral debug container (no shell in distroless images)
kubectl debug -it <pod> --image=busybox --target=<container>

# Resource usage (requires metrics-server)
kubectl top pod     # CPU/memory per pod
kubectl top node    # CPU/memory per node

# Cluster events sorted by time
kubectl get events --sort-by=.lastTimestamp -n <namespace>
Workflow: Events ➔ Describe ➔ Logs ➔ Exec/Debug ➔ Top
Follow this workflow when debugging: start with events to see what Kubernetes is telling you, then describe the pod for detailed status and conditions. Check logs next — use --previous to see logs from a crashed container. If the container is running but misbehaving, exec into it to inspect the filesystem, test network connectivity, or check environment variables. For distroless or minimal images that lack a shell, use kubectl debug to attach an ephemeral container with debugging tools. Finally, kubectl top shows real-time resource consumption, which is essential for diagnosing OOMKilled and CPU throttling. On EKS, ensure the metrics-server add-on is deployed for kubectl top to work.

🪦 Debugging Decision Tree

SymptomFirst CommandWhat to Look For
Pod stuck Pendingkubectl describe podEvents: FailedScheduling, resource requests, node selectors
Pod in CrashLoopBackOffkubectl logs --previousStack traces, missing env vars, exit code (137=OOM, 1=app error)

This decision tree gives you a fast path from symptom to diagnosis. The exit code in CrashLoopBackOff is particularly telling — 137 means OOMKilled by the kernel, 1 is a generic application error, and 143 is SIGTERM which usually means a failed liveness probe. For EKS-specific issues, also check that your node IAM role has the right ECR permissions and that your VPC networking allows pod-to-service communication.

🎯 Key Takeaways

  • Liveness probes detect stuck containers and trigger restarts — never check external dependencies
  • Readiness probes control traffic routing — can and should check if the app can serve good responses
  • Startup probes protect slow-starting apps from premature liveness kills
  • Probe protocols include HTTP, TCP, gRPC, and exec for different workload types
  • Use all three together for comprehensive health management
  • Tune parameters carefully — too aggressive causes flapping, too relaxed delays recovery
  • Graceful shutdown requires coordination between preStop hooks, SIGTERM handling, and PDBs
  • Recognize failure states: CrashLoopBackOff, ImagePullBackOff, Pending, OOMKilled
  • Systematic debugging: Events → Describe → Logs → Exec/Debug → Top
Let's recap the core lessons. The three probe types serve distinct purposes and work best together. The most common mistake is checking dependencies in liveness probes. Tune your parameters based on your application's actual behavior, and always implement graceful shutdown to prevent dropped connections during termination. When things go wrong, follow the debugging workflow: check events first, then describe, then logs, then exec or debug into the container, and use top for resource issues.
