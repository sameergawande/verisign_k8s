🏗️ FluxCD Architecture

A Toolkit Approach to GitOps

Flux v2 was redesigned as a set of composable controllers rather than a monolithic operator. Each controller owns a specific domain, and they communicate through Kubernetes CRDs and status conditions.

⚙️ Flux Controllers Overview

Source

Fetches artifacts from Git, Helm, OCI, and S3-compatible sources

Kustomize

Reconciles Kustomization manifests and plain YAML to the cluster

Helm

Manages HelmRelease lifecycle — install, upgrade, test, rollback

Notification

Dispatches alerts to external systems and receives webhooks

Image Automation

Scans registries, applies policies, and commits image updates to Git

Each controller is independently deployable and can be installed separately via the Flux CLI or Terraform. The source controller is foundational — every other controller depends on it to fetch artifacts. The notification controller is optional but critical for production observability.

🔄 Reconciliation Loop

The Core GitOps Principle

Flux continuously compares desired state (in Git) with actual state (in cluster) and converges any drift.

  • Fetch — source controller pulls latest artifact revision
  • Detect — compute diff between desired and actual state
  • Apply — server-side apply changed resources
  • Validate — run health checks against applied resources
  • Report — update status conditions and emit events
Key: Reconciliation is idempotent and runs on a configurable interval. No manual kubectl apply needed.
The reconciliation loop is the heartbeat of Flux. Each controller runs its own loop at a configurable interval. The source controller fetches and caches artifacts, then downstream controllers react to new revisions. Server-side apply ensures proper field ownership and conflict detection.

📦 Source Management

Fetching Artifacts from External Systems

The source controller is the foundation of Flux. It provides a common interface for fetching artifacts and exposing them to other controllers. It handles authentication, revision tracking, and caching.

📂 GitRepository CRD

apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata: { name: app-manifests, namespace: flux-system }
spec:
  interval: 5m
  url: https://github.com/verisign/k8s-manifests
  ref: { branch: main }
  secretRef: { name: git-credentials }
  ignore: |
    /docs/
    /tests/
    /**/*.md
  • interval — how often to poll for new commits
  • ref — branch, tag, semver, or commit SHA
  • secretRef — basic-auth, SSH key, or token credentials
  • ignore — gitignore-style patterns to exclude paths
The GitRepository CRD is the most commonly used source. The interval controls polling frequency — shorter intervals mean faster delivery but more API calls. The secretRef points to a Kubernetes Secret with auth credentials. For SSH, the secret contains identity and known_hosts fields. For HTTPS, it uses username and password fields.

⚖️ HelmRepository CRD

apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata: { name: bitnami, namespace: flux-system }
spec:
  interval: 30m
  url: https://charts.bitnami.com/bitnami
  type: default  # 'default' for HTTP, 'oci' for OCI
---
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata: { name: ecr-charts, namespace: flux-system }
spec:
  type: oci
  url: oci://123456789.dkr.ecr.us-east-1.amazonaws.com
  interval: 60m
  provider: aws
  • type: default — classic Helm HTTP/S chart repositories
  • type: oci — OCI-compliant registries (ECR, ACR, GCR, GHCR)
  • provider — aws, azure, or gcp for cloud-native auth
HelmRepository supports both classic index-based repos and modern OCI registries. For AWS ECR, the provider field enables automatic token rotation using IRSA. The interval for Helm repos can be longer since chart updates are less frequent than Git commits. OCI support is the recommended approach for private chart distribution.

📦 OCIRepository CRD

apiVersion: source.toolkit.fluxcd.io/v1beta2
kind: OCIRepository
metadata: { name: app-manifests-oci, namespace: flux-system }
spec:
  interval: 10m
  url: oci://ghcr.io/verisign/k8s-manifests
  ref: { tag: latest }
  provider: generic
  verify:
    provider: cosign
    secretRef: { name: cosign-public-key }
Use Cases: Push manifests as OCI artifacts via flux push artifact
Verification: Built-in Cosign and Notation support
OCIRepository is distinct from the OCI type on HelmRepository. It fetches arbitrary OCI artifacts — not Helm charts specifically. This is useful when you want to publish Kustomize bases or plain YAML as versioned OCI artifacts. The verify field enables supply chain security by validating signatures before accepting artifacts.

🪧 Bucket Source (S3/GCS)

apiVersion: source.toolkit.fluxcd.io/v1beta2
kind: Bucket
metadata: { name: terraform-outputs, namespace: flux-system }
spec:
  interval: 15m
  provider: aws
  bucketName: verisign-k8s-config
  endpoint: s3.amazonaws.com
  region: us-east-1
  prefix: clusters/production/
  • provider — aws, gcp, or generic (MinIO)
  • prefix — filter objects by path prefix
  • Ideal for consuming Terraform outputs or generated configs
  • Authenticates via IRSA (AWS) or Workload Identity (GCP)
Tip: Bucket sources are eventually consistent. Use Git for resources requiring strict ordering and audit trails.
Bucket sources bridge the gap between infrastructure provisioning and Kubernetes configuration. A common pattern is Terraform writing output manifests to S3, then Flux picking them up. This decouples the Terraform and Kubernetes deployment pipelines while maintaining automation. IRSA-based auth means no static credentials.

🛠️ Kustomize Controller

Reconciling Manifests to the Cluster

The kustomize controller is the workhorse of Flux. Despite the name, it handles both Kustomize overlays and plain YAML directories. It is responsible for applying manifests, pruning removed resources, and running health checks.

📑 Kustomization CRD

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata: { name: app-stack, namespace: flux-system }
spec:
  interval: 10m
  retryInterval: 2m
  timeout: 5m
  sourceRef: { kind: GitRepository, name: app-manifests }
  path: ./clusters/production
  prune: true
  force: false
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: api-server
      namespace: production
  • prune: true — garbage-collect resources removed from Git
  • healthChecks — wait for rollout before marking as Ready
  • retryInterval — back-off interval on reconciliation failure
The Kustomization CRD is not the same as kustomize.config.k8s.io — it is a Flux-specific resource. The prune flag is critical for production: when you delete a manifest from Git, Flux will delete the corresponding cluster resource. The healthChecks field blocks the Ready condition until all specified resources are healthy, which is essential for dependency chains.

🔠 Variable Substitution (postBuild)

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: app-staging
  namespace: flux-system
spec:
  sourceRef:
    kind: GitRepository
    name: app-manifests
  path: ./apps/base
  postBuild:
    substitute:
      CLUSTER_ENV: staging
      REPLICAS: "3"
      DOMAIN: staging.verisign.com
    substituteFrom:
      - kind: ConfigMap
        name: cluster-vars
      - kind: Secret
        name: cluster-secrets
Pattern: Use ${VARIABLE} placeholders in your manifests. Flux replaces them at apply time — no templating engine required.
PostBuild substitution is one of the most powerful Flux features for multi-environment management. You define a single base manifest with variable placeholders, then each Kustomization provides environment-specific values. substituteFrom pulls values from existing ConfigMaps and Secrets, keeping sensitive data out of Git. This is simpler than Helm templating for many use cases.

🔗 Dependency Ordering

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata: { name: infrastructure, namespace: flux-system }
spec:
  sourceRef: { kind: GitRepository, name: app-manifests }
  path: ./infrastructure
  prune: true
---
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata: { name: applications, namespace: flux-system }
spec:
  dependsOn: [{ name: infrastructure }]
  sourceRef: { kind: GitRepository, name: app-manifests }
  path: ./applications
  prune: true
Ordering chain: CRDs → Infrastructure → Platform → Applications
DependsOn creates an explicit ordering between Kustomizations. The dependent Kustomization will not reconcile until all its dependencies report a Ready status. This is essential for bootstrapping — you need CRDs installed before controllers, and controllers running before application resources that use those CRDs. A typical production setup has 3-4 layers in the dependency chain.

⚖️ Helm Controller

GitOps-Driven Helm Release Management

The Helm controller bridges Flux's GitOps model with the Helm ecosystem. It manages HelmRelease resources declaratively, handling the full lifecycle of install, upgrade, test, and rollback without direct Helm CLI usage.

📜 HelmRelease CRD

apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata: { name: nginx-ingress, namespace: infrastructure }
spec:
  interval: 15m
  chart:
    spec:
      chart: ingress-nginx
      version: "4.x"  # Semver range
      sourceRef:
        kind: HelmRepository
        name: ingress-nginx
  values:
    controller:
      replicaCount: 3
      service: { type: LoadBalancer }
  upgrade: { remediation: { retries: 3 } }
  rollback: { cleanupOnFail: true }
  • version — semver ranges for auto minor/patch upgrades
  • upgrade.remediation — auto-retry or rollback on failure
  • rollback.cleanupOnFail — delete new resources on failed rollback
The HelmRelease CRD replaces imperative helm install/upgrade commands with a declarative spec. Semver ranges let you automatically pick up patch releases while staying within a major version. The remediation settings are critical for production — they define what happens when an upgrade fails, including automatic rollback after a configurable number of retries.

🔑 Values from ConfigMaps & Secrets

apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata: { name: app-release, namespace: production }
spec:
  chart:
    spec:
      chart: my-app
      sourceRef: { kind: HelmRepository, name: internal }
  valuesFrom:
    - kind: ConfigMap
      name: app-common-values
      valuesKey: values.yaml
    - kind: Secret
      name: app-secret-values
      valuesKey: secrets.yaml
      targetPath: database.password
  values:
    replicaCount: 3
    image: { tag: v2.1.0 }
Tip: Merge order — valuesFrom entries are merged first (in order), then inline values override. Use targetPath for surgical injection.
ValuesFrom is the recommended way to handle sensitive Helm values. Instead of storing database passwords or API keys in Git, reference them from Secrets that are managed by an external secrets operator or sealed secrets. The targetPath field allows injecting a single secret value at a specific path in the values hierarchy without exposing the full values structure.

🔄 HelmRelease Lifecycle

State Machine: Install → Upgrade → Test → Rollback

Phase Trigger Configuration
Install First reconciliation install.remediation.retries
Upgrade Values or chart version change upgrade.remediation.retries
Test After install/upgrade test.enable: true
Rollback Failed upgrade or test rollback.timeout, cleanupOnFail
Uninstall HelmRelease deleted Automatic via garbage collection
Tip: Set upgrade.remediation.remediateLastFailure: true to automatically retry the last failed release on the next reconciliation.
Understanding the lifecycle is critical for production operations. When Flux detects a change (new chart version, updated values, or interval expiry), it runs through the appropriate phase. If an upgrade fails, Flux can automatically rollback to the last successful release. Helm tests run after install or upgrade and can trigger rollback on failure. Each phase has its own timeout and remediation settings.

🚀 Advanced Flux

Image Automation, Notifications & Multi-Tenancy

These advanced features transform Flux from a deployment tool into a full platform for continuous delivery. Image automation closes the loop from container build to deployment, notifications provide observability, and multi-tenancy enables platform teams to safely share clusters.

📷 Image Automation

# 1. Scan registry for new tags
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageRepository
metadata: { name: app-image }
spec:
  image: ghcr.io/verisign/my-app
  interval: 5m
---
# 2. Select latest matching tag
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata: { name: app-policy }
spec:
  imageRepositoryRef: { name: app-image }
  policy: { semver: { range: ">=1.0.0 <2.0.0" } }
---
# 3. Write selected tag back to Git
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageUpdateAutomation
metadata: { name: app-update }
spec:
  sourceRef: { kind: GitRepository, name: app-manifests }
  interval: 10m
  update:
    path: ./clusters/production
    strategy: Setters
Image automation is a three-step pipeline. ImageRepository scans a container registry for tags. ImagePolicy selects the latest tag matching a semver range, alphabetical, or numerical filter. ImageUpdateAutomation writes the selected tag back to Git using setter markers in your YAML. This closes the CI/CD loop — when your CI builds a new image, Flux detects it and updates the deployment manifest automatically.

🔔 Notification Controller

Provider (outbound)

apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Provider
metadata: { name: slack-alerts, namespace: flux-system }
spec:
  type: slack
  channel: k8s-deployments
  secretRef: { name: slack-webhook-url }

Alert (filter & route)

apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Alert
metadata: { name: critical-alerts, namespace: flux-system }
spec:
  providerRef: { name: slack-alerts }
  eventSeverity: error
  eventSources:
    - { kind: Kustomization, name: "*" }
    - { kind: HelmRelease, name: "*" }
Tip: Supported providers include Slack, Microsoft Teams, PagerDuty, Opsgenie, Discord, GitHub, GitLab, Grafana, and generic webhook.
The notification controller has two sides: outbound alerts (Provider + Alert) and inbound webhooks (Receiver). Alerts filter Flux events by severity and source type, then route them to external systems. In production, you typically set up error-level alerts to PagerDuty and info-level alerts to Slack. The wildcard name pattern catches all resources of that kind.

🏢 Multi-Tenancy with Flux

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata: { name: team-alpha-apps, namespace: flux-system }
spec:
  serviceAccountName: team-alpha-sa
  sourceRef: { kind: GitRepository, name: team-alpha-repo }
  path: ./deploy
  prune: true
  targetNamespace: team-alpha
  patches:
    - patch: |
        - op: add
          path: /metadata/labels/team
          value: alpha
      target:
        labelSelector: "app.kubernetes.io/managed-by"
  • serviceAccountName — impersonate a scoped SA for RBAC isolation
  • targetNamespace — force all resources into a specific namespace
  • Combine with NetworkPolicies and ResourceQuotas per tenant
Multi-tenancy in Flux is achieved by scoping each team's Kustomization to a specific service account with limited RBAC permissions. The targetNamespace override prevents teams from deploying into other namespaces. Combined with the cross-namespace reference restrictions in Flux, this provides strong isolation. The platform team manages the flux-system namespace, while each tenant gets their own namespace and Git repository.

📈 Flux Monitoring with Prometheus

Key Metrics

  • gotk_reconcile_duration_seconds — reconciliation latency
  • gotk_reconcile_condition — Ready/Stalled/Suspended status
  • gotk_suspend_status — suspended resource count
  • controller_runtime_reconcile_errors_total — error rate

Grafana Dashboard

Flux provides an official Grafana dashboard (ID: 16714) covering:

  • Reconciliation status per resource
  • Error rates and durations
  • Source revision tracking
  • Controller resource usage
Alert rule: Fire when gotk_reconcile_condition{type="Ready",status="False"} persists for > 10 minutes.
Flux exposes Prometheus metrics on each controller's /metrics endpoint. The gotk prefix stands for GitOps Toolkit. The reconcile_condition metric is the most important — it tells you whether each resource is Ready, Stalled, or Suspended. Create Prometheus alerting rules for sustained non-Ready conditions. The official Grafana dashboard gives a complete overview of cluster GitOps health.

🔧 Operational Patterns

Day-Two Management with Flux

Now that we understand all the controllers, let's cover the operational patterns you'll use daily when running Flux in production. These patterns cover common scenarios like maintenance windows, drift correction, and troubleshooting.

⏸️ Suspend & Resume Reconciliation

# Suspend a specific Kustomization
flux suspend kustomization app-stack

# Suspend all HelmReleases in a namespace
flux suspend helmrelease --all -n production

# Resume reconciliation
flux resume kustomization app-stack

# Force an immediate reconciliation
flux reconcile kustomization app-stack --with-source
Maintenance: Suspend before manual cluster changes to prevent Flux from reverting them
Emergency: Suspend, apply hotfix manually, then update Git and resume
  • Suspension persists across controller restarts (stored in CRD spec)
  • --with-source forces the source controller to re-fetch immediately
Suspend and resume are essential operational tools. During maintenance windows or incident response, you may need to stop Flux from reconciling while you make manual changes. Always remember to update Git to match your manual changes before resuming, otherwise Flux will revert them. The reconcile command with --with-source is useful when you've pushed a fix to Git and don't want to wait for the next interval.

🔍 Drift Detection & Correction

How Flux Detects Drift

Flux uses server-side apply with field ownership tracking. Any field managed by Flux that is modified externally triggers a drift correction on the next reconciliation.

# Check for drift manually
flux diff kustomization app-stack

# View which fields Flux manages
kubectl get deployment api-server -o yaml | grep "manager: flux"

# Force drift correction now
flux reconcile kustomization app-stack
Warning: With force: true in the Kustomization spec, Flux will replace resources instead of patching — use with caution as it recreates objects.
Drift detection is automatic and continuous. Flux relies on server-side apply's field management to know which fields it owns. If someone runs kubectl edit and changes a Flux-managed field, Flux will revert it on the next reconciliation. Fields not managed by Flux (like manually added annotations) are left untouched. The flux diff command is useful for previewing what changes Flux would make without actually applying them.

🔎 Troubleshooting Flux

# Check overall Flux health
flux check

# Get status of all Flux resources
flux get all -A

# Inspect a specific resource with conditions
flux get kustomizations -A
flux get helmreleases -A

# View Flux controller logs
flux logs --kind=Kustomization --name=app-stack
kubectl logs -n flux-system deploy/kustomize-controller

# View events for a specific resource
flux events --for Kustomization/app-stack
Status conditions to watch: Ready (healthy), Reconciling (in progress), Stalled (permanent error — needs manual intervention)
Troubleshooting follows a top-down approach. Start with flux check to verify all controllers are running. Then flux get all shows the status of every Flux resource. Stalled conditions indicate a permanent error that won't resolve with retries — like a missing Secret reference or invalid YAML. The flux logs command filters controller logs to a specific resource, which is much easier than searching raw pod logs. Events provide a timeline of reconciliation attempts and their outcomes.

🚀 Flux Bootstrap & Cluster Management

# Bootstrap Flux on a new cluster
flux bootstrap github \
  --owner=verisign \
  --repository=fleet-infra \
  --branch=main \
  --path=clusters/production \
  --personal=false

# Bootstrap with existing repo
flux bootstrap git \
  --url=ssh://git@github.com/verisign/fleet-infra \
  --branch=main \
  --path=clusters/staging

Recommended Repository Structure

fleet-infra/
├── clusters/
│   ├── production/
│   │   ├── flux-system/     # Auto-generated
│   │   ├── infrastructure.yaml
│   │   └── applications.yaml
│   └── staging/
├── infrastructure/    # Shared infra components
└── applications/      # Application manifests
Bootstrap is a one-time operation that installs Flux controllers and commits their manifests to Git. This makes Flux self-managing — it reconciles its own installation from Git. The recommended repo structure separates cluster-specific configuration from shared components. Each cluster directory contains Kustomization resources that reference shared infrastructure and application directories. This pattern scales well to dozens of clusters.
