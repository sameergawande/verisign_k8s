🤔 Why Scaling Matters

The cost of getting it wrong

Before diving into the mechanics, let's establish why automatic scaling is a critical capability, not just a nice-to-have.

⚡ The Scaling Challenge

🔴 Over-provisioned

  • Wasted compute and higher cloud costs
  • Resources unavailable to other teams
  • Quota exhaustion on the cluster

🔴 Under-provisioned

  • Degraded response times and timeouts
  • OOMKilled pods
  • Customer-facing impact
The goal: Right-size your application automatically based on real demand — not guesswork, not capacity planning meetings.
Both extremes are bad. Over-provisioning is throwing money away. Under-provisioning causes outages. Manual scaling requires human attention and is always reactive. Autoscaling responds to actual demand, finding the sweet spot between cost and performance.

💰 Resource Requests & Limits

The foundation everything else depends on

Before we can talk about autoscaling, we need to understand resource requests and limits. These are the foundation that the scheduler and autoscaler both depend on.

💰 Requests vs. Limits

Requests (Guaranteed)

  • What the scheduler uses for placement
  • Guaranteed minimum resources
  • Used by HPA for percentage calculations
  • Must be set for HPA to work

Limits (Maximum)

  • Enforced ceiling the container cannot exceed
  • CPU: throttled when exceeded
  • Memory: OOMKilled when exceeded
  • Protects neighbors on the same node
resources:
  requests:
    cpu: "250m"       # 0.25 CPU cores guaranteed
    memory: "256Mi"   # 256 MiB guaranteed
  limits:
    cpu: "500m"       # Can burst to 0.5 cores, then throttled
    memory: "512Mi"   # Killed if exceeds 512 MiB
The distinction between requests and limits is critical. Requests are a scheduling guarantee — the scheduler will only place your pod on a node that has at least that much capacity available. Limits are enforcement — the kubelet enforces these at runtime. CPU is compressible (throttled), memory is not (OOMKilled).

🔢 Understanding Resource Units

CPU Units

ValueMeaning
11 vCPU / 1 core
500mHalf a core (500 millicores)
250mQuarter core
100m1/10th of a core

1000m = 1 CPU. Always compressible.

Memory Units

ValueMeaning
128Mi128 mebibytes (2^20)
256Mi256 MiB
1Gi1 gibibyte (2^30)
512M512 megabytes (10^6)

Use Mi/Gi (binary). Not compressible.

Tip: Common pitfall — 128M (megabytes) is not equal to 128Mi (mebibytes). Always use Mi/Gi for consistency with how Kubernetes actually allocates memory. Millicores are the standard CPU unit — think of them as thousandths of a CPU core.

🎯 Quality of Service (QoS) Classes

QoS Class Criteria Eviction Priority
Guaranteed requests == limits for all containers Last to be evicted
Burstable requests < limits (or only requests set) Evicted after BestEffort
BestEffort No requests or limits set First to be evicted 
# Guaranteed QoS — best for production critical workloads
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "500m"      # Same as request
    memory: "512Mi"  # Same as request
Tip: Production workloads should be Guaranteed or Burstable — never deploy BestEffort to production. QoS class determines what happens under memory pressure. The kubelet evicts BestEffort first, then Burstable, and Guaranteed last.

📊 Metrics Server

The data pipeline for autoscaling

Before the HPA can make scaling decisions, it needs data. The Metrics Server is the component that provides this data. Let's understand how it works.

📊 Metrics Server Architecture

How metrics flow through the system

  1. kubelet collects resource usage from the container runtime (containerd)
  2. Metrics Server scrapes kubelet's /metrics/resource endpoint every 15 seconds
  3. Metrics Server aggregates and serves data via the metrics.k8s.io API
  4. HPA controller queries metrics.k8s.io every 15 seconds (default)
  5. HPA controller calculates desired replicas and patches the Deployment
# Verify Metrics Server is running on your cluster
kubectl get deployment metrics-server -n kube-system

# Check the metrics API is available
kubectl get apiservice v1beta1.metrics.k8s.io

# View node-level metrics
kubectl top nodes

# View pod-level metrics
kubectl top pods -n my-namespace
The Metrics Server is a prerequisite for HPA. On EKS, it's typically installed as an add-on. It's an in-memory store — it only keeps the latest metrics, not historical data. For historical metrics and dashboards, you need Prometheus. The 15-second scrape interval means there's always some lag in autoscaling decisions.

⚡ Horizontal Pod Autoscaler

The core scaling mechanism

Now we get to the main event — the Horizontal Pod Autoscaler. This is the primary autoscaling mechanism most developers will use. We'll cover the algorithm, the API, and the tuning options.

🧮 The HPA Algorithm

Core Formula

desiredReplicas = ⌈ currentReplicas × ( currentMetric / desiredMetric ) ⌉

Example Calculation

ParameterValue
Current replicas3
CPU request per pod250m
Current average CPU usage200m per pod (80%)
Target CPU utilization50%
Calculation⌈ 3 × (80 / 50) ⌉ = ⌈ 4.8 ⌉ = 5 replicas
Tip: The HPA uses the ratio of current to desired metric values — it's percentage-based against requests, not limits. The ceiling function means Kubernetes always rounds up. This is why setting appropriate requests is critical for HPA to work correctly.

🔢 HPA Algorithm — Edge Cases

  • Tolerance zone (±10%): HPA won't scale if the ratio is within 0.9–1.1 of the target. This prevents flapping.
  • Missing metrics: Pods with missing metrics are treated conservatively — scale-down assumes 100% utilization, scale-up assumes 0%.
  • Unready pods: Excluded from the metric average during scale-up (included for scale-down).
  • Multiple metrics: HPA evaluates each metric independently and takes the maximum desired replica count.
The 10% tolerance band is critical for stability. If your target is 50% CPU and current usage is between 45%–55%, the HPA takes no action. This prevents constant scaling oscillation.
These edge cases are what make HPA production-ready. Without the tolerance zone, you'd get constant scaling up and down by 1 replica. Without the conservative treatment of missing metrics, a pod that's temporarily unreachable could cause inappropriate scale-down. These details matter when troubleshooting why HPA isn't behaving as expected.

📝 HPA with autoscaling/v2 API

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target: { type: Utilization, averageUtilization: 60 }
  - type: Resource
    resource:
      name: memory
      target: { type: Utilization, averageUtilization: 75 }
Tip: autoscaling/v2 replaced v2beta2. It supports multiple metric types, custom metrics, and behavior policies. Always use v2 for new HPAs. minReplicas prevents scaling to zero (use KEDA for that), maxReplicas is your safety cap, and you can specify multiple metrics. With multiple metrics, HPA evaluates each independently and uses the highest result. The 60% CPU target leaves headroom for traffic spikes before scaling kicks in.

📊 HPA with Custom Metrics

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: myapp-hpa }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: myapp }
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource          # Standard CPU metric
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  - type: Pods              # Custom: requests/sec per pod
    pods:
      metric: { name: http_requests_per_second }
      target: { type: AverageValue, averageValue: "1000" }
  - type: External          # SQS queue depth
    external:
      metric: { name: sqs_queue_length }
      target: { type: Value, value: "100" }
Custom metrics are where HPA gets really powerful. The Pods type targets a metric exposed by the pods themselves (via Prometheus adapter). The External type targets metrics from outside the cluster — like an SQS queue depth. This requires a custom metrics adapter (like prometheus-adapter or KEDA) to be installed on the cluster. The highest desired replica count across all metrics wins.

⚙️ HPA Behavior Policies

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: myapp-hpa }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: myapp }
  minReplicas: 3
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0     # Scale up immediately
      policies:
      - { type: Percent, value: 100, periodSeconds: 60 }  # Double
      - { type: Pods, value: 10, periodSeconds: 60 }
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300   # Wait 5 min
      policies:
      - { type: Percent, value: 10, periodSeconds: 60 }
      selectPolicy: Min
Behavior policies are the fine-tuning knobs for HPA. The key pattern: scale UP aggressively (respond to spikes quickly) and scale DOWN conservatively (prevent premature scale-down). The stabilizationWindowSeconds for scale-down means HPA looks at the highest recommended replica count over the last 5 minutes — this prevents flapping on bursty workloads.

🕒 Stabilization Windows Explained

How Stabilization Works

The stabilization window defines a time period during which the HPA examines all computed replica counts and selects the most appropriate one.

Scale-Up Window

  • Default: 0 seconds (immediate)
  • Selects highest recommendation in window
  • Set higher to smooth transient spikes

Scale-Down Window

  • Default: 300 seconds (5 min)
  • Selects highest recommendation in window
  • Increase for slow-ramp-down workloads
Tip: For DNS-related services, consider a longer scale-down window (600s+) because DNS caching can redirect traffic back to recently removed endpoints. The stabilization window is a sliding window — the HPA picks the highest recommendation in the window to prevent premature scale-down.

📈 Vertical Pod Autoscaler

Right-sizing pod resources

While HPA scales the number of pods, VPA scales the resources allocated to each pod. Let's explore this complementary approach.

📈 VPA — Vertical Pod Autoscaler

What VPA does

VPA automatically adjusts CPU and memory requests and limits for pods based on observed usage patterns.

Recommender

Watches resource usage and computes optimal requests/limits

Updater

Evicts pods that need resource updates (triggers restart with new values)

Admission Controller

Mutates new pods to use recommended resources on creation

VPA has three components that work together. The Recommender watches metrics and computes suggestions. The Updater evicts pods that are significantly over or under-resourced. The Admission Controller intercepts pod creation and injects the recommended values. Note that VPA requires pod restarts to apply changes — it cannot change resources on a running pod.

⚙️ VPA Operating Modes

Mode Recommends Applies on Create Evicts to Update Use Case
Off ✅ ❌ ❌ Dry run — observe recommendations
Initial ✅ ✅ ❌ Apply on new pods only
Auto ✅ ✅ ✅ Fully automatic (restarts pods) 
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata: { name: myapp-vpa }
spec:
  targetRef: { apiVersion: apps/v1, kind: Deployment, name: myapp }
  updatePolicy:
    updateMode: "Off"   # Start with Off to observe
  resourcePolicy:
    containerPolicies:
    - containerName: myapp
      minAllowed: { cpu: "100m", memory: "128Mi" }
      maxAllowed: { cpu: "2", memory: "4Gi" }
Start with Off mode to observe VPA's recommendations before letting it make changes. This is especially important in production. The minAllowed and maxAllowed provide guardrails to prevent VPA from setting unreasonable values. Important caveat: do NOT use VPA in Auto mode with HPA on the same CPU/memory metrics — they will conflict. You can combine them if HPA uses custom metrics and VPA manages CPU/memory.

⚠️ VPA + HPA — Compatibility

Do NOT use VPA (Auto mode) and HPA on the same metrics (CPU/memory) simultaneously. They will fight each other.

❌ Conflicting Setup

  • HPA scales on CPU utilization
  • VPA adjusts CPU requests
  • VPA changes requests → HPA sees different utilization → scales differently → VPA readjusts...

✅ Compatible Setup

  • HPA scales on custom metrics (RPS, queue depth)
  • VPA manages CPU/memory requests
  • Each autoscaler owns different dimensions
  • Or: VPA in Off mode for recommendations only
This is one of the most common mistakes teams make. If HPA targets CPU and VPA adjusts CPU requests, they create a feedback loop. The safe patterns are: use VPA in Off mode alongside HPA (just for recommendations), or have HPA scale on custom metrics while VPA handles resource sizing.

⚡ KEDA

Kubernetes Event-Driven Autoscaling

KEDA extends Kubernetes autoscaling to handle event-driven workloads and scale-to-zero scenarios. Let's understand where it fits.

⚡ KEDA — Event-Driven Autoscaling

What KEDA adds beyond HPA

KEDA is a lightweight component that can scale Kubernetes workloads based on external event sources — and can scale to zero.

KEDA Capabilities

  • Scale to/from zero replicas
  • 60+ built-in scalers (SQS, Kafka, Prometheus, Cron)
  • Works alongside native HPA via ScaledObject/ScaledJob

When to use KEDA

  • Queue-based workers (SQS, Kafka)
  • Schedule-based or scale-to-zero scenarios
  • Complex multi-metric scaling logic
KEDA is not a replacement for HPA — it extends it. Under the hood, KEDA creates and manages HPA objects. Its superpower is the ability to scale to zero and the rich library of event source scalers. For workloads like queue consumers that should only exist when there's work to do, KEDA is the right choice.

📝 KEDA ScaledObject Example

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: sqs-consumer-scaler
spec:
  scaleTargetRef:
    name: sqs-consumer           # Deployment to scale
  pollingInterval: 15
  cooldownPeriod: 60             # Wait 60s before scale-to-zero
  minReplicaCount: 0             # Can scale to zero!
  maxReplicaCount: 50
  triggers:
  - type: aws-sqs-queue
    metadata:
      queueURL: https://sqs.us-east-1.amazonaws.com/123456789/queue
      queueLength: "5"           # Target 5 messages per pod
    authenticationRef: { name: aws-credentials }
Scale-to-zero: When the SQS queue is empty, KEDA scales the Deployment to 0 replicas. When messages arrive, it scales back up. This can significantly reduce costs for bursty workloads.
This example shows a common pattern: a worker that processes messages from an SQS queue. When the queue is empty, why pay for running pods? KEDA scales to zero. When messages arrive, KEDA scales up based on queue depth. The queueLength of 5 means KEDA targets 5 messages per pod — so 50 messages in the queue would result in 10 pods.

🎯 Scaling Best Practices

✅ Do

  • Always set resource requests
  • Start with CPU-based HPA at 60-70% target
  • Use conservative scale-down (300s+ window)
  • Monitor HPA events and load test to validate
  • Use VPA in Off mode for right-sizing data

❌ Don't

  • Skip resource requests (HPA won't work)
  • Set target too high (90%+) or too low (20%)
  • Use VPA Auto + HPA on same metric
  • Set maxReplicas too low for actual demand
  • Ignore readiness probes or cluster capacity
These best practices come from real-world production experience. The 60-70% CPU target is a common starting point — it leaves enough headroom for the scaling lag we discussed. Load testing is essential to validate that your scaling configuration can actually handle your traffic patterns. And always remember: pod autoscaling is useless if the cluster doesn't have node capacity — that's where the Cluster Autoscaler comes in.

📝 Production-Ready HPA Configuration

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: myapp-hpa, labels: { app: myapp } }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: myapp }
  minReplicas: 3          # HA across AZs
  maxReplicas: 30         # Safety cap
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 65
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - { type: Percent, value: 50, periodSeconds: 60 }
      - { type: Pods, value: 5, periodSeconds: 60 }
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - { type: Percent, value: 10, periodSeconds: 120 }
      selectPolicy: Min
This is a production-ready HPA you can use as a starting point. The min of 3 ensures at least one pod per AZ for high availability. The scale-up is aggressive (50% increase or 5 pods, whichever is more), while scale-down is gentle (10% every 2 minutes with a 5-minute stabilization window). Adjust these values based on load testing results for your specific application.

📚 Module 2 — Key Takeaways

  • Resource requests are the foundation — HPA can't work without them
  • Metrics Server provides real-time CPU/memory data for the HPA algorithm
  • HPA algorithm: desiredReplicas = ⌈currentReplicas × (current/desired)⌉ with ±10% tolerance
  • autoscaling/v2 supports multiple metrics, custom metrics, and behavior policies
  • Scale up fast, scale down slow — the golden rule of behavior tuning
  • VPA right-sizes pods (Off mode with HPA); KEDA enables scale-to-zero
  • ~55+ seconds from spike to new pod serving traffic — plan headroom accordingly
These eight points cover the essential scaling knowledge. The most important for day-to-day work: always set resource requests, use autoscaling/v2, tune behavior policies for your workload, and monitor HPA status. In the next module, we'll look at managing application data with persistent storage.
