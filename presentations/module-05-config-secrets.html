🏗️ The Twelve-Factor App

Factor III: Store Config in the Environment

The 12-factor app methodology, originally from Heroku, provides foundational guidance for building cloud-native applications. Factor III specifically addresses configuration management.

📚 Why Externalize Configuration?

❌ Anti-Pattern: Embedded Config

  • Config hardcoded in application code
  • Requires rebuilding images for each environment
  • Secrets committed to source control
  • No separation of concerns

✅ Best Practice: External Config

  • Config injected at runtime
  • Same image across dev, staging, prod
  • Secrets managed independently
  • Config changes without redeployment
The core principle is strict separation of config from code. Configuration varies between deploys — code does not. At Verisign, this means the same container image runs in dev, staging, and production EKS clusters with different ConfigMaps and Secrets.

🚀 Kubernetes Config Primitives

Kubernetes provides two native resources for external configuration:

  • ConfigMaps — Non-sensitive configuration data (key-value pairs, files)
  • Secrets — Sensitive data (credentials, tokens, certificates)
Both can be consumed as: Environment variables, command-line arguments, or files mounted into volumes.
ConfigMaps and Secrets are the two primary mechanisms Kubernetes provides. They share a similar API but have important behavioral differences, especially around security. We'll cover both in depth.

📃 ConfigMaps

Externalizing Non-Sensitive Configuration

Let's start with ConfigMaps — the primary mechanism for injecting non-sensitive configuration into your pods.

🔧 Creating ConfigMaps — From Literals

Use --from-literal for individual key-value pairs:

# Create ConfigMap from literal values
kubectl create configmap app-config \
  --from-literal=DATABASE_HOST=postgres.db.svc.cluster.local \
  --from-literal=DATABASE_PORT=5432 \
  --from-literal=LOG_LEVEL=info \
  --from-literal=CACHE_TTL=300
# Verify the ConfigMap
kubectl get configmap app-config -o yaml
Tip: Literal creation is great for simple key-value pairs, but becomes unwieldy for more than a handful of values.
The simplest way to create a ConfigMap is from literal values on the command line. Each --from-literal flag adds one key-value pair. This is useful for quick tests but for production, declarative YAML is preferred.

📄 Creating ConfigMaps — From Files

Load entire configuration files into a ConfigMap:

# Create from a single file (key = filename)
kubectl create configmap nginx-conf \
  --from-file=nginx.conf

# Create from a file with a custom key name
kubectl create configmap nginx-conf \
  --from-file=main-config=nginx.conf

# Create from a directory (each file becomes a key)
kubectl create configmap app-configs \
  --from-file=./config-dir/
Key Insight: When using --from-file, the filename becomes the key and the file contents become the value. With a directory, each file in the directory becomes a separate key.
File-based ConfigMaps are extremely useful for injecting entire configuration files — things like nginx.conf, application.properties, or JSON config files. The entire file content is stored as the value.

📝 Creating ConfigMaps — Declarative YAML

The preferred approach for production workloads:

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: verisign-apps
  labels:
    app: dns-service
    environment: production
data:
  # Simple key-value pairs
  DATABASE_HOST: "postgres.db.svc.cluster.local"
  DATABASE_PORT: "5432"
  LOG_LEVEL: "info"
  CACHE_TTL: "300"
  # Multi-line configuration file
  application.yaml: |
    server:
      port: 8080
      shutdown: graceful
    logging:
      level:
        root: INFO
        com.verisign: DEBUG
    cache:
      type: redis
      ttl: 300
Declarative YAML is the recommended approach for Verisign. It can be version-controlled, reviewed in PRs, and applied through GitOps pipelines. Notice how you can mix simple key-value pairs with multi-line file content in the same ConfigMap.

📦 Consuming ConfigMaps as Environment Variables

Individual Keys with valueFrom

spec:
  containers:
  - name: app
    image: verisign/dns-service:1.4
    env:
    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: DATABASE_HOST
    - name: DB_PORT
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: DATABASE_PORT

All Keys with envFrom

spec:
  containers:
  - name: app
    image: verisign/dns-service:1.4
    envFrom:
    - configMapRef:
        name: app-config
      prefix: CFG_  # optional
Caution: envFrom skips keys that are not valid environment variable names.
There are two patterns for consuming ConfigMaps as env vars. valueFrom gives you control over individual key mapping and renaming. envFrom loads everything at once — convenient but less explicit. The optional prefix helps avoid naming collisions.

📁 Consuming ConfigMaps as Volumes

apiVersion: v1
kind: Pod
metadata:
  name: dns-service
spec:
  containers:
  - name: app
    image: verisign/dns-service:1.4
    volumeMounts:
    - name: config-volume
      mountPath: /etc/app/config
      readOnly: true
  volumes:
  - name: config-volume
    configMap:
      name: app-config
      # Optional: select specific keys
      items:
      - key: application.yaml
        path: application.yaml
      - key: LOG_LEVEL
        path: log-level.txt
Hot Reload: ConfigMap volume mounts are updated automatically by the kubelet — typically within ~60 seconds (synced on the kubelet sync period + cache propagation delay). Your application must watch for file changes to pick up updates.
Volume mounts are the most powerful consumption pattern. Each key becomes a file in the mount directory. The huge advantage is hot reload — the kubelet automatically updates the mounted files when the ConfigMap changes, typically within about 60 seconds. But your application must be designed to detect and reload configuration from disk.

🔄 Hot Reload Behavior Deep Dive

How Volume-Mounted ConfigMap Updates Work

  • Kubelet periodically syncs ConfigMap data (default: every 60s)
  • Updates use atomic symlink swap: ..data → ..data_tmp → rename
  • All files update atomically together
  • Total propagation delay: kubelet sync + API server cache TTL
Important Exceptions:
  • subPath mounts do NOT receive updates — they are a one-time copy
  • Environment variables from ConfigMaps are never updated — require pod restart
Understanding the update mechanism is crucial. The kubelet uses symlinks for atomic updates, which means all files in a ConfigMap mount update together. But there are two critical exceptions: subPath mounts and environment variables never receive updates. If you need live config updates, use volume mounts without subPath.

🔐 Kubernetes Secrets

Managing Sensitive Configuration Data

Now let's move to Secrets. While similar to ConfigMaps in API structure, Secrets have fundamentally different security characteristics that you must understand.

🔑 Secret Types in Kubernetes

TypeDescriptionUse Case
OpaqueGeneric, arbitrary user dataPasswords, API keys, tokens
kubernetes.io/tlsTLS cert + private keyIngress TLS termination
kubernetes.io/dockerconfigjsonDocker registry credentialsPulling from private registries
kubernetes.io/basic-authBasic auth credentialsUsername/password pairs
kubernetes.io/ssh-authSSH private keyGit clone, SSH access
kubernetes.io/service-account-tokenService account tokenAPI authentication (legacy)
Kubernetes supports several built-in Secret types. Opaque is the most common — it's the default for any generic secret data. TLS and docker-registry types have specific validation requirements that help catch configuration errors early.

🔧 Creating Secrets

# Opaque secret from literals
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password='S3cur3P@ssw0rd!'

# TLS secret from certificate files
kubectl create secret tls app-tls-cert \
  --cert=tls.crt \
  --key=tls.key

# Docker registry secret
kubectl create secret docker-registry ecr-creds \
  --docker-server=123456789.dkr.ecr.us-east-1.amazonaws.com \
  --docker-username=AWS \
  --docker-password=$(aws ecr get-login-password)
Shell History Risk: Creating secrets from the command line leaves credentials in your shell history. Use --from-file or declarative YAML with sealed/external secrets for production.
Here are the three most common imperative creation patterns. Note the shell history warning — this is a real security concern. For Verisign production workloads, secrets should be managed through External Secrets Operator syncing from Vault, not created imperatively.

📝 Declarative Secret YAML

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: verisign-apps
type: Opaque
# data values must be base64-encoded
data:
  username: YWRtaW4=          # echo -n 'admin' | base64
  password: UzNjdXIzUEBzc3cwcmQh  # echo -n 'S3cur3P@ssw0rd!' | base64
---
# Alternative: use stringData for plain-text input
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: verisign-apps
type: Opaque
stringData:
  username: admin
  password: "S3cur3P@ssw0rd!"
⚠️ Critical Understanding: Base64 encoding is NOT encryption. Anyone with kubectl get secret -o yaml access can decode your secrets instantly. Base64 is merely a data encoding format.
This is one of the most important points in this module. Base64 is NOT security — it's just encoding. The stringData field is a convenience that lets you write plain text, but the API server converts it to base64 data on storage. Either way, the values are trivially decodable.

📦 Consuming Secrets in Pods

apiVersion: v1
kind: Pod
metadata:
  name: dns-service
spec:
  containers:
  - name: app
    image: verisign/dns-service:1.4
    env:
    # Individual secret keys as env vars
    - name: DB_USERNAME
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: username
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: password
    # Mount secrets as files
    volumeMounts:
    - name: tls-certs
      mountPath: /etc/tls
      readOnly: true
  volumes:
  - name: tls-certs
    secret:
      secretName: app-tls-cert
      defaultMode: 0400  # Read-only for owner
Consuming secrets follows the same patterns as ConfigMaps — env vars or volumes. Notice the defaultMode of 0400 for the TLS cert volume — restricting file permissions is an important security practice. Secret files should only be readable by the application process.

📊 Environment Variable Precedence

Understanding Override Order

When you have multiple sources setting the same environment variable, understanding the precedence rules is essential to avoid surprises.

📋 Environment Variable Precedence Rules

Order of Precedence (highest to lowest):

  1. env[].value — Inline value defined directly in the Pod spec
  2. env[].valueFrom — Single key reference (configMapKeyRef / secretKeyRef)
  3. envFrom — Bulk import from ConfigMap or Secret
spec:
  containers:
  - name: app
    envFrom:
    - configMapRef:
        name: app-defaults     # LOG_LEVEL=info (lowest priority)
    env:
    - name: LOG_LEVEL
      value: "debug"           # This WINS (highest priority)
    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: app-overrides
          key: DATABASE_HOST   # Overrides envFrom value
The precedence rule is simple: explicit env entries override envFrom entries. Among envFrom entries, later entries in the list override earlier ones. This pattern lets you set defaults in a shared ConfigMap via envFrom, then override specific values with targeted env entries.

📁 Advanced Volume Patterns

Projected Volumes and subPath Mounts

Let's look at two advanced volume patterns that solve real-world configuration challenges.

📂 Projected Volumes

Combine multiple sources into a single volume mount:

apiVersion: v1
kind: Pod
metadata:
  name: dns-service
spec:
  containers:
  - name: app
    volumeMounts:
    - name: all-config
      mountPath: /etc/app
      readOnly: true
  volumes:
  - name: all-config
    projected:
      sources:
      - configMap:
          name: app-config
          items:
          - key: application.yaml
            path: application.yaml
      - secret:
          name: db-credentials
          items:
          - key: password
            path: secrets/db-password
      - downwardAPI:
          items:
          - path: labels
            fieldRef:
              fieldPath: metadata.labels
      - serviceAccountToken:
          path: token
          expirationSeconds: 3600
          audience: vault
Projected volumes are incredibly powerful. You can combine ConfigMaps, Secrets, the Downward API, and service account tokens into a single mount point. This is especially useful for applications that expect all their configuration in one directory. The service account token projection with audience and expiration is the modern approach, replacing the legacy auto-mounted tokens.

📄 subPath Mounts

Mount a single key without overwriting the entire directory:

❌ Without subPath

# This REPLACES everything
# in /etc/nginx/
volumeMounts:
- name: config
  mountPath: /etc/nginx/
  # All existing files in
  # /etc/nginx/ are hidden!

✅ With subPath

# Only adds nginx.conf,
# preserving other files
volumeMounts:
- name: config
  mountPath: /etc/nginx/nginx.conf
  subPath: nginx.conf
  # Other files in /etc/nginx/
  # remain visible!
⚠️ Trade-off: subPath mounts do NOT receive automatic updates when the ConfigMap changes. If you need hot reload, use a regular volume mount to a different directory and symlink to it.
subPath is essential when you need to inject a single config file into a directory that already has other files — like adding a custom nginx.conf without hiding the rest of /etc/nginx. But remember the critical trade-off: subPath mounts never receive updates. This is a common source of confusion in production.

🔒 Immutable ConfigMaps and Secrets

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v3
  namespace: verisign-apps
data:
  DATABASE_HOST: "postgres.db.svc.cluster.local"
  LOG_LEVEL: "info"
immutable: true  # Cannot be changed after creation

✅ Benefits

  • Protects against accidental changes
  • Reduces API server load (no watch needed)
  • Improves cluster performance at scale
  • Enables versioned config rollout

📋 Update Strategy

  • Create new ConfigMap with version suffix
  • Update Deployment to reference new name
  • Rolling update deploys new config
  • Delete old ConfigMap after rollout
Immutable ConfigMaps and Secrets are a production best practice, especially at scale. Once marked immutable, the resource cannot be modified — only deleted and recreated. This eliminates the kubelet watch overhead and prevents accidental configuration changes. Use a versioned naming convention like app-config-v3.

🛡️ Secret Best Practices

Security Beyond Base64

Now let's talk about how to actually secure your secrets in a Kubernetes environment — because by default, they're not very secure at all.

🛡️ RBAC for Secrets

Restrict who can read secrets with fine-grained RBAC:

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: verisign-apps
rules:
- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["db-credentials", "api-keys"]  # Specific secrets only
  verbs: ["get"]
  # Never grant: list, watch (can enumerate all secrets)
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-secret-access
  namespace: verisign-apps
subjects:
- kind: ServiceAccount
  name: dns-service-sa
  namespace: verisign-apps
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
RBAC is your first line of defense. Restrict secret access to specific resource names — never grant blanket list or watch on all secrets. Also consider that anyone with exec access to a pod can read environment variables from /proc, so secrets as files with restricted permissions may be safer.

🔐 Encryption at Rest

By default, Secrets are stored unencrypted in etcd!

AWS EKS Encryption

# EKS cluster config
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: verisign-prod
secretsEncryption:
  keyARN: arn:aws:kms:us-east-1:
    123456789:key/abc-123

Uses AWS KMS envelope encryption

What Encryption at Rest Does

  • Encrypts Secret data before writing to etcd
  • Decrypts transparently on read via API server
  • Protects against etcd backup exposure
  • Does NOT protect against API access
Remember: Encryption at rest protects etcd data. It does not prevent users with kubectl get secret permissions from reading secrets through the API.
Encryption at rest is essential for compliance. On EKS, Verisign uses AWS KMS envelope encryption. The API server encrypts each Secret with a data encryption key (DEK), and that DEK is encrypted with your KMS key. This protects against etcd snapshot exposure but not against API-level access.

🛡️ External Secrets — The Production Pattern

External Secrets Operator (ESO) Architecture

How It Works

  1. Secrets stored in external vault (HashiCorp Vault, AWS Secrets Manager)
  2. ESO watches ExternalSecret resources
  3. ESO fetches values from vault
  4. ESO creates/updates native K8s Secrets
  5. Pods consume normal Secret references

Verisign Pattern

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: db-credentials
  data:
  - secretKey: username
    remoteRef:
      key: secret/data/verisign/db
      property: username
  - secretKey: password
    remoteRef:
      key: secret/data/verisign/db
      property: password
Covered in Depth: Module 16 covers ESO + Vault integration in detail for Verisign's production secrets workflow.
For Verisign's production environments, External Secrets Operator is the standard pattern. Secrets are authored in HashiCorp Vault, and ESO synchronizes them into Kubernetes Secrets. This provides centralized secret management, audit logging, rotation, and keeps secrets out of Git. We'll cover this extensively in Module 16.

📋 Secret Best Practices Summary

🔒 Access Control

  • Fine-grained RBAC on secrets
  • Limit list/watch permissions
  • Use specific resourceNames
  • Audit secret access

🔐 Storage

  • Enable encryption at rest
  • Use external secret stores
  • Never commit to Git
  • Rotate credentials regularly

🔧 Runtime

  • Prefer volume mounts over env vars
  • Set restrictive file permissions
  • Use readOnly: true
  • Limit secret scope per pod
Here's the complete checklist for secret security. The three pillars are access control through RBAC, secure storage through encryption and external vaults, and runtime protection through volume mounts with proper permissions. Volume mounts are preferred over environment variables because env vars are visible in /proc and often leak into logs and crash dumps.

🛠️ Complete Working Example

Putting It All Together

Let's look at a complete example that combines ConfigMaps and Secrets using multiple consumption patterns.

🛠️ Full Example — ConfigMap + Secret

apiVersion: v1
kind: ConfigMap
metadata:
  name: web-app-config
data:
  APP_ENV: "production"
  APP_PORT: "8080"
  nginx.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://localhost:8080;
      }
    }
---
apiVersion: v1
kind: Secret
metadata:
  name: web-app-secrets
type: Opaque
stringData:
  DB_PASSWORD: "pr0d-s3cret!"
  API_KEY: "sk-verisign-abc123"
Here we define both the ConfigMap with application settings and an nginx config file, plus a Secret with database credentials and an API key. Next slide shows how the Deployment consumes them both.

🛠️ Full Example — Deployment

apiVersion: apps/v1
kind: Deployment
metadata: { name: web-app }
spec:
  replicas: 3
  selector: { matchLabels: { app: web-app } }
  template:
    spec:
      containers:
      - name: app
        image: verisign/web-app:2.1
        envFrom: [{ configMapRef: { name: web-app-config } }]   # APP_ENV, APP_PORT
        env:
        - name: DB_PASSWORD
          valueFrom: { secretKeyRef: { name: web-app-secrets, key: DB_PASSWORD } }
        volumeMounts:
        - { name: nginx-config, mountPath: /etc/nginx/conf.d/default.conf, subPath: nginx.conf }
        - { name: secret-files, mountPath: /etc/app/secrets, readOnly: true }
      volumes:
      - { name: nginx-config, configMap: { name: web-app-config } }
      - { name: secret-files, secret: { secretName: web-app-secrets, defaultMode: 0400 } }
This Deployment demonstrates the recommended hybrid approach: bulk environment variables from ConfigMap via envFrom, selective secret injection via individual secretKeyRef, a subPath mount for the nginx config file, and a volume mount for secret files with restricted permissions. This is a real-world pattern you'll use frequently.

📚 Module 5 Summary

⚙️ ConfigMaps

  • Externalize config from code (12-factor)
  • Create from literals, files, or YAML
  • Consume as env vars or volume mounts
  • Volume mounts support hot reload (~60s)
  • subPath mounts do NOT auto-update
  • Use immutable for production stability

🔐 Secrets

  • Base64 encoding is NOT encryption
  • Types: Opaque, TLS, docker-registry
  • Enable encryption at rest (AWS KMS)
  • Apply fine-grained RBAC
  • Prefer volume mounts over env vars
  • Use ESO + Vault in production (Module 16)
Let's recap the key takeaways. ConfigMaps handle non-sensitive config with hot reload support via volumes. Secrets handle sensitive data but require additional security measures — RBAC, encryption at rest, and external secret management. Both resources should be immutable in production for stability and performance.

🎯 Key Takeaways

1. Separate config from code — Use ConfigMaps and Secrets to make your images environment-agnostic.
2. Volume mounts > env vars — Volumes support hot reload and are more secure for secrets.
3. Base64 ≠ encryption — Always enable encryption at rest and use external secret stores.
4. Immutable for production — Prevent accidental changes and reduce API server load.
Four key takeaways to remember. Separate config from code. Prefer volumes over environment variables. Never rely on base64 for security. And use immutable resources in production.
