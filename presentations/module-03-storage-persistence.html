📦 The Kubernetes Storage Model

Ephemeral by default — persistent by design

Kubernetes was designed with stateless workloads as the primary use case. Understanding why requires understanding how containers and pods handle filesystem state.

💣 Containers Are Ephemeral

  • Every container starts with a fresh filesystem from its image layers
  • Writes go to a writable container layer (copy-on-write)
  • When the container restarts, all writes are lost
  • Pod rescheduling to a different node = completely new filesystem
This means: Logs written to disk, temp files, cached data, database files — all gone when a pod restarts. This is by design, not a bug.
This is the most important mental model shift for developers coming from traditional infrastructure. On a VM, your disk persists across reboots. On Kubernetes, your pod can be killed and rescheduled at any time, and the filesystem resets to the image baseline. This makes stateless apps trivial but stateful apps require explicit storage configuration.

🔒 When You Need Persistence

Databases

  • PostgreSQL
  • MySQL
  • MongoDB
  • Redis (AOF mode)

Message Queues

  • Kafka
  • RabbitMQ
  • NATS JetStream
  • Apache Pulsar

File Storage

  • User uploads
  • ML model artifacts
  • Shared config files
  • Build caches

The Rule of Thumb

If your workload writes data that must survive a pod restart, you need persistent storage. If it only reads data or writes ephemeral data, the default model works fine.

Not every workload needs persistence. Stateless API servers, web frontends, and data processors that read from queues and write to external stores don't need persistent volumes. Reserve persistent storage for workloads that own their data — databases, message brokers, and file stores.

📍 Volumes vs PersistentVolumes vs PVCs

Three Layers of Storage Abstraction

  • Volume — a directory accessible to containers in a pod; lifecycle tied to the pod
  • PersistentVolume (PV) — a cluster-level storage resource with its own lifecycle, independent of any pod
  • PersistentVolumeClaim (PVC) — a developer's request for storage; binds to a PV
Analogy: PV = the actual hard drive provisioned by the platform team. PVC = a ticket requesting "I need 10Gi of fast storage." Kubernetes matches claims to volumes automatically.
This three-layer model separates concerns cleanly. Platform teams manage PVs and StorageClasses. Developers create PVCs to request what they need. The Kubernetes control plane handles the binding. This is the same producer-consumer pattern we see throughout Kubernetes.

📚 Volume Types Overview

Type Persists? Use Case
emptyDir Pod lifetime only Scratch space, shared data between containers in a pod
hostPath Node lifetime Node-level logs, DaemonSet workloads (avoid in production)
configMap Cluster lifetime Configuration files mounted as volumes
secret Cluster lifetime Certificates, credentials mounted as files
persistentVolumeClaim Until PV deleted Databases, stateful apps, durable file storage
Production note: hostPath is a security risk and ties pods to specific nodes. Use it only for system-level DaemonSets, never for application workloads.
emptyDir is the simplest volume type — it's a directory on the node that gets created when the pod is scheduled and deleted when the pod is removed. It's useful for sidecar patterns where containers need to share files. configMap and secret volumes are how we inject configuration. PVC is the focus of this module — it's the path to real persistence.

📝 emptyDir & hostPath in Practice

# emptyDir — shared scratch space between containers
apiVersion: v1
kind: Pod
metadata:
  name: log-processor
spec:
  containers:
  - name: app
    image: myapp:v1
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/app
  - name: log-shipper
    image: fluentbit:latest
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/app
      readOnly: true
  volumes:
  - name: shared-logs
    emptyDir:
      sizeLimit: 500Mi
Pattern: The app writes logs to /var/log/app and the sidecar reads and ships them. Both containers share the same emptyDir volume.
This sidecar pattern is extremely common. The emptyDir volume acts as a communication channel between the main application container and the log shipping sidecar. The sizeLimit prevents a runaway log from consuming the entire node's disk. emptyDir lives on the node's filesystem or optionally in memory using emptyDir.medium: Memory.

🗃 PersistentVolume (PV) Deep Dive

apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-ebs-01
spec:
  capacity:
    storage: 50Gi
  volumeMode: Filesystem
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: gp3
  csi:
    driver: ebs.csi.aws.com
    volumeHandle: vol-0abc123def456789
    fsType: ext4
  • capacity — how much storage the volume provides
  • accessModes — who can mount and how (RWO, ROX, RWX)
  • reclaimPolicy — what happens when the PVC is deleted
  • storageClassName — links to a StorageClass for dynamic provisioning
PVs are cluster-scoped resources — they don't belong to any namespace. This YAML shows a manually-created PV backed by an AWS EBS volume. In practice, you'll rarely create PVs manually because dynamic provisioning via StorageClasses handles it for you. But understanding the PV spec is essential for troubleshooting.

📄 PersistentVolumeClaim (PVC) Deep Dive

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
  namespace: my-team
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: gp3
  resources:
    requests:
      storage: 50Gi
# Reference in a Pod spec
spec:
  containers:
  - name: postgres
    image: postgres:16
    volumeMounts:
    - name: data
      mountPath: /var/lib/postgresql/data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: postgres-data
The PVC is the developer-facing resource. You specify what you need — access mode, storage class, and size — and Kubernetes finds or creates a matching PV. The PVC is namespace-scoped, so it lives alongside your application resources. The pod references the PVC by name in its volumes section.

🔄 PVC Binding Lifecycle

How PVCs Find Their PVs

  1. Pending — PVC created, waiting for a matching PV
  2. Bound — PVC matched to a PV (by size, access mode, StorageClass)
  3. In Use — Pod mounts the PVC; volume is attached to the node
  4. Released — Pod deleted, PVC deleted; PV enters Released state
  5. Reclaimed — PV is deleted, retained, or recycled per reclaim policy
# Inspect PVC status
kubectl get pvc postgres-data
# NAME            STATUS   VOLUME     CAPACITY   ACCESS MODES   STORAGECLASS
# postgres-data   Bound    pvc-a1b2   50Gi       RWO            gp3

kubectl describe pvc postgres-data | grep -A2 Events
The most common issue developers hit is a PVC stuck in Pending. This usually means no PV matches the request — wrong StorageClass, insufficient capacity, or the provisioner is not running. Always check kubectl describe pvc for events when troubleshooting.

🔓 Access Modes Explained

Mode Abbreviation Description Typical Backend
ReadWriteOnce RWO Single node can mount read-write EBS, local SSD
ReadOnlyMany ROX Many nodes can mount read-only EFS, NFS
ReadWriteMany RWX Many nodes can mount read-write EFS, NFS, CephFS
EBS limitation: AWS EBS volumes are ReadWriteOnce only — they attach to a single EC2 instance. If you need shared storage across pods on multiple nodes, use EFS (NFS-based).
Note: ReadWriteOncePod (RWOP) is a newer mode that restricts access to a single pod, even on the same node. Useful for sensitive data workloads.
Access modes are about node-level access, not pod-level — a subtle but important distinction. RWO means the volume can be mounted on one node, but multiple pods on that same node can mount it. RWOP, introduced in Kubernetes 1.27 as stable, tightens this to a single pod. For EKS, EBS is RWO and EFS supports RWX.

🗑 Reclaim Policies

Retain

  • PV preserved after PVC deletion
  • Data stays intact
  • Manual cleanup required
  • Best for production databases

Delete

  • PV and underlying storage deleted
  • Data permanently removed
  • Default for dynamic provisioning
  • Suitable for ephemeral workloads

Recycle (Deprecated)

  • Runs rm -rf on volume
  • Makes PV available again
  • Deprecated in favor of dynamic provisioning
  • Do not use
Critical: The default reclaim policy for dynamically provisioned PVs is Delete. For any stateful workload, override this to Retain to prevent accidental data loss.
This is one of the most important operational decisions for stateful workloads. If someone accidentally deletes a PVC with a Delete reclaim policy, the underlying EBS volume is destroyed and the data is gone. Always set Retain for databases and important stateful workloads. You can change an existing PV's reclaim policy with kubectl patch.

⚙️ StorageClasses & Dynamic Provisioning

Automate volume creation with declarative storage tiers

StorageClasses are the key to making storage self-service for developers. Instead of asking a platform engineer to create an EBS volume, developers declare what they need and the StorageClass provisioner handles the rest.

🛠️ StorageClass Anatomy

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
  fsType: ext4
reclaimPolicy: Retain
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer
  • provisioner — which CSI driver creates the volumes
  • parameters — passed to the provisioner (EBS type, encryption, etc.)
  • volumeBindingMode — WaitForFirstConsumer delays binding until a pod is scheduled (AZ-aware)
  • allowVolumeExpansion — enables online resizing of PVCs
WaitForFirstConsumer is essential on EKS. EBS volumes are AZ-specific — if you provision a volume in us-east-1a but the pod lands on a node in us-east-1b, it can't attach. WaitForFirstConsumer ensures the volume is created in the same AZ as the node where the pod is scheduled.

💾 AWS EBS CSI Driver on EKS

gp3 — General Purpose

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
  • 3,000 baseline IOPS
  • 125 MB/s throughput
  • Most workloads

io2 — High Performance

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: io2-high-perf
provisioner: ebs.csi.aws.com
parameters:
  type: io2
  iopsPerGB: "50"
  encrypted: "true"
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
  • Up to 64,000 IOPS
  • 99.999% durability
  • Latency-sensitive DBs
The EBS CSI driver is an EKS add-on that must be installed on the cluster. It replaces the legacy in-tree EBS provisioner. gp3 is the recommended default — it's cheaper than gp2 and provides better baseline performance. io2 is for workloads like production databases that need guaranteed IOPS. Always encrypt volumes in production.

📁 AWS EFS CSI Driver for Shared Storage

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: efs-sc
provisioner: efs.csi.aws.com
parameters:
  provisioningMode: efs-ap
  fileSystemId: fs-0abc123def456789
  directoryPerms: "700"
  gidRangeStart: "1000"
  gidRangeEnd: "2000"
mountOptions:
  - tls
EFS Strengths: ReadWriteMany, auto-scaling capacity, cross-AZ, shared access across pods and nodes
EFS Trade-offs: Higher latency than EBS, higher cost per GB, not suitable for database workloads
EFS is NFS-based and supports ReadWriteMany, making it the go-to for workloads that need shared filesystem access — think content management systems, shared configuration, or ML training jobs reading shared datasets. EFS uses Access Points for dynamic provisioning, creating isolated directories within a single filesystem. Always use the tls mount option for encryption in transit.

🔄 Dynamic Provisioning Workflow

What Happens When You Create a PVC

  1. Developer creates a PVC referencing a StorageClass
  2. PVC stays Pending (WaitForFirstConsumer mode)
  3. Pod is created referencing the PVC; scheduler places the pod on a node
  4. StorageClass provisioner creates the volume in the correct AZ
  5. A PV is automatically created and bound to the PVC
  6. The volume is attached to the node and mounted into the pod
# Watch the provisioning happen in real time
kubectl get pvc -w
kubectl get pv
kubectl describe pvc my-claim  # Check events for provisioning status
This workflow is entirely automated. The developer writes a PVC and a pod spec, and the platform handles everything else. The key insight is that with WaitForFirstConsumer, steps 3-6 happen together — the volume isn't created until a pod actually needs it. This avoids AZ mismatch issues and prevents unused volumes from accumulating.

🧮 EBS vs EFS — Choosing the Right Backend

Criteria EBS (gp3/io2) EFS
Access Mode ReadWriteOnce ReadWriteMany
Performance Low latency, high IOPS Higher latency, throughput-oriented
Availability Single AZ Multi-AZ
Use Cases Databases, caches, single-writer Shared files, CMS, ML datasets
Cost Model Per GB provisioned Per GB stored (scales automatically)
Decision rule: If one pod writes, use EBS. If multiple pods need concurrent read-write access, use EFS.
This is the most common decision point for storage on EKS. EBS is the right default for most stateful workloads because of its performance characteristics. EFS fills the niche for shared storage. A common mistake is using EFS for databases — it doesn't have the IOPS or latency profile that databases need.

📊 StatefulSets

Managing workloads that need stable identity and persistent storage

StatefulSets are the Kubernetes primitive for running stateful workloads. They solve the problems that Deployments can't — stable network names, ordered operations, and per-replica persistent storage.

🔄 StatefulSets vs Deployments

Deployment

  • Pods are interchangeable
  • Random pod names (app-7b8f9c-x2k4)
  • Shared PVC (if any)
  • Parallel scaling
  • No startup/shutdown ordering
  • Use for: stateless APIs, web servers, workers

StatefulSet

  • Pods have stable identities
  • Predictable names (db-0, db-1, db-2)
  • Per-replica PVC via templates
  • Ordered scaling (0, 1, 2...)
  • Ordered startup and shutdown
  • Use for: databases, Kafka, ZooKeeper
Key distinction: Use a Deployment unless your workload requires stable identity, ordered operations, or per-replica persistent storage. Most workloads are Deployments.
The critical difference is identity. In a Deployment, pods are cattle — interchangeable and disposable. In a StatefulSet, pods are pets — each has a unique identity, a stable hostname, and its own persistent volume. Databases need this because replica-0 might be the primary and replica-1 and replica-2 are secondaries.

🛠️ StatefulSet YAML Anatomy

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres-headless  # Required: headless Service name
  replicas: 3
  selector:
    matchLabels: { app: postgres }
  template:
    metadata:
      labels: { app: postgres }
    spec:
      containers:
      - name: postgres
        image: postgres:16
        ports: [{ containerPort: 5432 }]
        volumeMounts:
        - { name: data, mountPath: /var/lib/postgresql/data }
  volumeClaimTemplates:           # Per-replica PVCs
  - metadata: { name: data }
    spec:
      accessModes: ["ReadWriteOnce"]
      storageClassName: gp3
      resources:
        requests: { storage: 50Gi }
Two things distinguish a StatefulSet spec from a Deployment: the serviceName field and the volumeClaimTemplates section. The serviceName must reference a headless Service that provides DNS entries for each pod. The volumeClaimTemplates create a unique PVC for each replica, named data-postgres-0, data-postgres-1, and data-postgres-2.

🌐 Stable Network Identities

# Headless Service — no ClusterIP, provides DNS for each pod
apiVersion: v1
kind: Service
metadata:
  name: postgres-headless
spec:
  clusterIP: None       # This makes it "headless"
  selector:
    app: postgres
  ports:
  - port: 5432

DNS Records Created

  • postgres-0.postgres-headless.my-ns.svc.cluster.local
  • postgres-1.postgres-headless.my-ns.svc.cluster.local
  • postgres-2.postgres-headless.my-ns.svc.cluster.local
Why this matters: Applications can connect to a specific replica by name. The primary can always be reached at postgres-0, regardless of which node it runs on.
The headless Service with clusterIP: None tells Kubernetes not to create a virtual IP. Instead, DNS queries return the individual pod IPs. This is how database replication works on Kubernetes — the primary is always postgres-0, and replicas can find the primary by its stable DNS name. Even if the pod is rescheduled to a different node, the DNS name stays the same.

🗃 VolumeClaimTemplates

How VolumeClaimTemplates Work

Each StatefulSet replica gets its own PVC, named using the pattern:

<volumeClaimTemplate name>-<statefulset name>-<ordinal>

# For a StatefulSet named "postgres" with 3 replicas:
kubectl get pvc
# NAME              STATUS   VOLUME     CAPACITY   STORAGECLASS
# data-postgres-0   Bound    pvc-aaa    50Gi       gp3
# data-postgres-1   Bound    pvc-bbb    50Gi       gp3
# data-postgres-2   Bound    pvc-ccc    50Gi       gp3
Persistence guarantee: When postgres-1 is rescheduled, it reattaches to data-postgres-1 — the same volume with the same data. Identity and storage stay bound together.
This is the core value of StatefulSets for data workloads. The PVCs created by volumeClaimTemplates are NOT deleted when the StatefulSet is scaled down or when individual pods restart. They persist until manually deleted. This means you can scale a StatefulSet to zero and scale back up, and each replica reconnects to its original data.

🔢 Ordered Deployment & Scaling

Scale Up (0 → 3)

  1. Create postgres-0, wait until Ready
  2. Create postgres-1, wait until Ready
  3. Create postgres-2, wait until Ready

Each pod must be Running and Ready before the next one starts.

Scale Down (3 → 1)

  1. Terminate postgres-2, wait until gone
  2. Terminate postgres-1, wait until gone
  3. postgres-0 remains

Highest ordinal terminated first. Reverse order.

Why ordering matters: Databases often initialize the primary (ordinal 0) first, then join replicas to the cluster one at a time. Unordered startup can cause split-brain or data corruption.
The ordering guarantee is critical for distributed systems. For example, a PostgreSQL cluster using Patroni needs pod-0 to initialize the primary before pod-1 can stream replication from it. If both started simultaneously, neither would know who the primary is. The reverse order for scale-down ensures replicas are drained before the primary is affected.

🔄 StatefulSet Update Strategies

RollingUpdate (Default)

spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 0
  • Updates pods in reverse ordinal order
  • Waits for each to be Ready
  • partition allows canary updates

Partition-Based Canary

spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 2  # Only update pods >= 2
  • Pods with ordinal ≥ partition get the new version
  • Pods below partition keep the old version
  • Gradually lower partition to roll out
Partition strategy: Set partition=2 on a 3-replica StatefulSet to update only pod-2. Validate, then set partition=1, then partition=0 for a controlled rollout.
The partition field is a powerful canary mechanism unique to StatefulSets. Unlike Deployment rollouts that are all-or-nothing per ReplicaSet, StatefulSet partitions let you update one replica at a time and validate before proceeding. For database upgrades, this is invaluable — update one replica, verify data integrity, then continue.

🎯 Real-World Storage Patterns

Production-tested approaches for stateful workloads on EKS

Now that we understand the primitives, let's look at how they come together in real production scenarios. These patterns are drawn from common workloads teams run on EKS.

🐘 PostgreSQL on Kubernetes

apiVersion: apps/v1
kind: StatefulSet
metadata: { name: postgresql }
spec:
  serviceName: postgresql-headless
  replicas: 3
  selector: { matchLabels: { app: postgresql } }
  template:
    metadata: { labels: { app: postgresql } }
    spec:
      containers:
      - name: postgresql
        image: postgres:16
        env:
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef: { name: pg-secret, key: password }
        - { name: PGDATA, value: /var/lib/postgresql/data/pgdata }
        ports: [{ containerPort: 5432 }]
        volumeMounts: [{ name: data, mountPath: /var/lib/postgresql/data }]
        resources:
          requests:
            cpu: "500m"
            memory: "1Gi"
          limits:
            cpu: "2"
            memory: "4Gi"
  # ... volumeClaimTemplates: gp3, 100Gi ReadWriteOnce per replica
This is a production-grade PostgreSQL StatefulSet. Key details: PGDATA is set to a subdirectory of the mount point because PostgreSQL requires the data directory to be empty on init. Resources are set with both requests and limits. In a real production setup, you'd use an operator like CloudNativePG or Zalando's postgres-operator to handle replication, failover, and backups.

🛠️ Database Operators — The Production Path

Why Use an Operator Instead of Raw StatefulSets?

A raw StatefulSet gives you storage and identity. An operator adds the operational logic that databases need to run reliably.

CloudNativePG

  • PostgreSQL-specific
  • Automated failover
  • Continuous backup to S3
  • Point-in-time recovery

Percona Operators

  • MySQL, MongoDB, PG
  • Multi-cluster replication
  • Automated backups
  • Version upgrades

Strimzi

  • Apache Kafka
  • Topic management
  • TLS automation
  • Rack-aware replication
Running a raw StatefulSet for a database is possible but operationally expensive. Operators encode the domain expertise of DBAs into software — they handle leader election, failover, backup scheduling, and version upgrades. For production databases on EKS, always evaluate an operator before building your own automation around raw StatefulSets.

📈 Volume Expansion — Resizing PVCs

# Check if the StorageClass allows expansion
kubectl get sc gp3 -o jsonpath='{.allowVolumeExpansion}'
# true

# Edit the PVC to request more storage
kubectl patch pvc data-postgres-0 -p \
  '{"spec": {"resources": {"requests": {"storage": "100Gi"}}}}'

# Monitor the resize operation
kubectl get pvc data-postgres-0 -w
kubectl describe pvc data-postgres-0 | grep Conditions
EBS volumes support online expansion: No pod restart required. The filesystem is resized while the volume is mounted. Expansion happens in two phases — the EBS volume resizes first, then the filesystem grows to fill it.
Volumes can only grow, never shrink. Plan your initial size carefully, but know you can always expand later.
Volume expansion is a common day-2 operation. A database that was provisioned with 50Gi will eventually need more space. With allowVolumeExpansion: true on the StorageClass and the EBS CSI driver, this is seamless. The resize happens in two stages: the cloud provider resizes the block device, then the kubelet resizes the filesystem. Both happen online with EBS.

📸 Volume Snapshots — Create

# VolumeSnapshotClass — defines the snapshot provisioner
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: ebs-snapclass
driver: ebs.csi.aws.com
deletionPolicy: Retain
# VolumeSnapshot — take a snapshot of a PVC
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: postgres-snap-20260313
spec:
  volumeSnapshotClassName: ebs-snapclass
  source:
    persistentVolumeClaimName: data-postgres-0
Volume snapshots are the Kubernetes-native backup mechanism. They map directly to EBS snapshots on AWS. VolumeSnapshotClass configures how snapshots are taken, and VolumeSnapshot triggers the actual snapshot.

📸 Volume Snapshots — Restore

# Restore from snapshot — create a new PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-restored
spec:
  accessModes: ["ReadWriteOnce"]
  storageClassName: gp3
  resources:
    requests: { storage: 100Gi }
  dataSource:
    name: postgres-snap-20260313
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
Snapshot workflow: SnapshotClass configures how, VolumeSnapshot triggers when, and a PVC with dataSource restores from it. Useful for pre-upgrade backups, disaster recovery, and cloning environments.
A PVC with a dataSource referencing a VolumeSnapshot restores from that snapshot. This is useful for pre-upgrade backups, disaster recovery, and cloning environments.

💡 Storage Best Practices for EKS

Configuration

  • Use gp3 as the default StorageClass
  • Always set WaitForFirstConsumer binding mode
  • Enable allowVolumeExpansion
  • Encrypt all volumes (encrypted: "true")
  • Set Retain reclaim policy for stateful workloads

Operations

  • Monitor PVC usage with Prometheus metrics
  • Automate volume snapshots for backups
  • Use operators for production databases
  • Test failover and recovery procedures
  • Tag EBS volumes for cost allocation
These best practices reflect the operational reality of running stateful workloads on EKS. Encryption should be non-negotiable. WaitForFirstConsumer prevents the most common provisioning failure. Monitoring PVC usage prevents the silent failure of a database filling its disk. And testing recovery before you need it is the difference between a minor incident and a major outage.

🔍 Storage Troubleshooting Guide

Symptom Likely Cause Fix
PVC stuck in Pending No matching StorageClass or CSI driver not installed kubectl describe pvc — check events
Pod stuck in ContainerCreating Volume attachment failure (AZ mismatch) Use WaitForFirstConsumer
Volume attach timeout EBS volume stuck attaching to a different node Force detach via AWS console or CLI
Filesystem read-only Disk full or filesystem errors Expand PVC or run fsck
StatefulSet pods not starting Previous pod not yet Ready (ordering) Check pod-0 health first
These are the most common storage issues you'll encounter on EKS. The number one issue is PVC Pending — always start with kubectl describe pvc to see the events. AZ mismatch is the second most common, which is why WaitForFirstConsumer is essential. Force-detaching an EBS volume should be a last resort — it can cause data corruption if the volume was being written to.

📚 Module 3 — Key Takeaways

  • Kubernetes storage is ephemeral by default — persistence requires explicit configuration via PVs and PVCs
  • PVC is the developer interface — declare what you need, let the platform provision it
  • StorageClasses enable self-service — dynamic provisioning eliminates manual volume management
  • EBS = single-writer, EFS = shared storage — choose based on access pattern
  • StatefulSets provide stable identity — ordered operations, persistent names, per-replica storage
  • Use operators for production databases — they encode operational expertise as code
  • Always set Retain reclaim policy — protect against accidental data loss
These seven points capture the essential knowledge for managing data on Kubernetes. The most important mental model is the separation between the developer interface (PVC) and the platform implementation (PV, StorageClass, CSI driver). Developers declare intent, and the platform fulfills it. This same pattern repeats throughout Kubernetes.

Budget extra time for the StatefulSet exercises as they involve waiting for ordered pod creation.

❓ Questions & Discussion

What stateful workloads does your team run today, and how do you currently manage their storage?

Open the floor for questions. Common topics: How do we migrate existing databases to Kubernetes? Should we run databases on K8s or use managed services like RDS? How do we handle backup and restore? What about multi-AZ database replication? These are all valid discussions that help gauge the audience's current state and readiness for stateful K8s workloads.
