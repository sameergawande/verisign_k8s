🤔 The Problem We're Solving

Why developers needed something better

Before we talk about Kubernetes, let's talk about the pain points that led to its creation. Every technology exists to solve a problem — understanding the problem makes the solution intuitive.

💔 Before Kubernetes — Developer Pain Points

Deployment Challenges

  • "Works on my machine" syndrome
  • Manual server provisioning tickets
  • Environment drift between dev/staging/prod
  • Multi-hour deployment windows

Operational Burden

  • Developers on-call for infrastructure issues
  • Scaling requires capacity planning meetings
  • Rollback = panic + SSH + hope
  • No self-service for infrastructure needs
Ask the room: how many of you have experienced these pain points? Most developers have lived through at least some of these. The key insight is that these aren't just inconveniences — they directly impact velocity and reliability.

✅ After Kubernetes — Developer Gains

Deployment Gains

  • Containers guarantee environment parity
  • Self-service deployments via kubectl apply
  • Same manifest works dev → staging → prod
  • Rolling updates with zero downtime

Operational Gains

  • Platform team owns infrastructure
  • Auto-scaling based on actual metrics
  • Rollback = kubectl rollout undo
  • Declarative config = version-controlled infra
The transformation is dramatic. Notice that every "after" item maps directly to a "before" pain point. Kubernetes doesn't just add features — it fundamentally changes the developer-to-infrastructure relationship.

📊 Before vs. After — Side by Side

Concern Before K8s With K8s
Deploy a new version SSH, scripts, prayer kubectl apply -f deployment.yaml
Scale to handle load File a ticket, wait days HPA scales automatically
Rollback a bad release Restore from backup, redeploy kubectl rollout undo
Service discovery Hardcoded IPs, load balancer configs DNS-based: svc.namespace.svc.cluster.local
Environment parity Snowflake servers Same container image everywhere
Health monitoring External monitoring + manual restart Liveness/readiness probes + auto-restart
This table is a great reference to come back to throughout the course. Each row represents a capability we'll explore in depth in later modules. The key takeaway: Kubernetes turns operational toil into declarative configuration.

🛠️ The Developer Workflow

From code to cluster

Now let's trace the actual workflow a developer follows to get code running in a Kubernetes cluster. Understanding this end-to-end flow is critical for everything that follows.

🚀 Code to Cluster — The Journey

The Developer Pipeline

  1. Write Code — Application logic, tests, Dockerfile
  2. Build Image — docker build -t myapp:v1.2.3 .
  3. Push to Registry — docker push ecr-repo/myapp:v1.2.3
  4. Declare Desired State — Write/update Kubernetes manifests
  5. Apply to Cluster — kubectl apply -f k8s/
  6. Kubernetes Reconciles — Controller loop makes it so
  7. Observe & Iterate — Logs, metrics, traces
This is the core loop every developer runs through. In practice, steps 2-5 are usually automated by a CI/CD pipeline, but developers need to understand each step. At Verisign, this pipeline typically uses ECR for the registry and EKS for the cluster.

📝 What Developers Own

Application Artifacts

  • Application source code and tests
  • Dockerfile / container definition
  • Application configuration

Kubernetes Manifests

  • Deployment and Service definitions
  • ConfigMaps and Secrets
  • Resource requests, limits, and health probes
Key Insight: Developers define what they need. The platform provides how it's delivered.
This is the critical boundary. Developers own the application and its Kubernetes resource definitions. They don't need to know which physical node their pod runs on, how networking is implemented, or how storage is provisioned — they just declare what they need.

🤝 The Developer-Platform Contract

Clear boundaries, clear responsibilities

One of the most powerful aspects of Kubernetes is how it creates a clear contract between developers and platform teams. Let's define that contract explicitly.

👤 Developer vs. Platform Responsibilities

Developer Responsibility Platform Responsibility
Container image (Dockerfile) Container runtime (containerd)
Resource requests & limits Node capacity & scheduling
Health check endpoints Health check execution & remediation
Deployment strategy choice Rolling update orchestration
Service port definitions Network routing & load balancing
PVC requests (size, access mode) Storage provisioning (EBS, EFS)
Scaling thresholds (HPA config) Cluster autoscaler, node pools
This table is the contract. When both sides honor it, the result is a highly productive system. Developers get self-service capabilities, and platform teams get consistent, manageable workloads. At Verisign with EKS, AWS manages the control plane, further simplifying the platform team's scope.

📜 The Kubernetes API — Your Contract Interface

The API Server is the single point of interaction

Every interaction flows through the API server — kubectl, CI/CD pipelines, controllers, and the scheduler all speak the same API.

# Every developer action is an API call
kubectl apply -f deployment.yaml    # POST/PUT to /apis/apps/v1/deployments
kubectl get pods                    # GET to /api/v1/pods
kubectl logs my-pod                 # GET to /api/v1/pods/my-pod/log
kubectl port-forward svc/myapp 8080 # Upgrade to SPDY/WebSocket
Why this matters: The API is versioned, documented, and consistent. Your manifests from today will work on future clusters because Kubernetes guarantees API compatibility.
The API server is the front door to everything in Kubernetes. Understanding that kubectl is just an HTTP client helps developers reason about what's happening under the hood. The versioned API is a stability guarantee — you're writing against a contract, not an implementation.

🆕 Self-Service Infrastructure

Kubernetes transforms infrastructure requests into API calls developers can make without filing tickets:

I need...

  • A load balancer
  • Persistent storage
  • A cron job
  • DNS entry and TLS termination
  • Horizontal scaling

I write...

  • Service type: LoadBalancer
  • PersistentVolumeClaim
  • CronJob
  • Service + Ingress with TLS
  • HorizontalPodAutoscaler
This is perhaps the most transformative aspect of Kubernetes for developer productivity. What used to take days of ticket-driven processes now takes a YAML file and a kubectl apply. The platform team pre-configures what's allowed (via RBAC, quotas, policies), and developers operate within those guardrails.

📦 Core Abstractions Review

Pods, Deployments, and Services — the developer lens

Let's review the three foundational abstractions. Even if you've seen these before, we're going to look at them through a developer-centric lens — focusing on why each exists, not just what it does.

🗺 Abstraction-to-Need Mapping

Developer Need K8s Abstraction What It Provides
Run my container Pod Smallest deployable unit, shared network/storage
Keep N replicas running Deployment / ReplicaSet Desired state, rolling updates, rollback
Stable network endpoint Service Load balancing, DNS, stable IP
External traffic routing Ingress / Gateway HTTP routing, TLS, path-based routing
Configuration management ConfigMap / Secret Externalized config, decoupled from image
Persistent data PVC / StorageClass Dynamic provisioning, lifecycle management
Run-to-completion tasks Job / CronJob Batch processing, scheduled tasks
This table is your Rosetta Stone. Every time you encounter a new Kubernetes resource type, ask: what developer need does this serve? If you can't answer that question, you probably don't need it yet.

📦 Pods — The Atomic Unit

  • Smallest deployable unit in K8s
  • One or more containers sharing network & storage
  • Every container in a pod shares localhost
  • Pods are ephemeral — they can be killed and rescheduled
  • You rarely create pods directly — use Deployments
apiVersion: v1
kind: Pod
metadata:
  name: myapp
  labels: { app: myapp, version: v1.2.3 }
spec:
  containers:
  - name: myapp
    image: 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3
    ports: [{ containerPort: 8080 }]
    resources:
      requests: { cpu: "250m", memory: "256Mi" }
      limits:   { cpu: "500m", memory: "512Mi" }
    livenessProbe:
      httpGet: { path: /healthz, port: 8080 }
    readinessProbe:
      httpGet: { path: /ready, port: 8080 }
Tip: Never depend on a specific pod existing. Design for ephemerality. The pod spec is the most important manifest a developer writes. Notice the key elements: container image with a specific tag (never use :latest in production), resource requests and limits, and health probes. Every one of these is a developer responsibility.

⚙️ Deployments — Desired State Management

apiVersion: apps/v1
kind: Deployment
metadata: { name: myapp, labels: { app: myapp } }
spec:
  replicas: 3
  selector: { matchLabels: { app: myapp } }
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }
  template:
    metadata: { labels: { app: myapp, version: v1.2.3 } }
    spec:
      containers:
      - name: myapp
        image: ecr-repo/myapp:v1.2.3
        ports: [{ containerPort: 8080 }]
        # ... resources, probes (same as Pod)
Developer value: You declare "I want 3 healthy copies of my app running at all times" and Kubernetes makes it happen — and keeps it that way. The Deployment is the workhorse resource for developers. Key things to note: replicas controls availability, the strategy section controls how updates happen (maxUnavailable: 0 means zero-downtime deploys), and the selector must match the pod template labels. The Deployment controller continuously reconciles actual state to desired state.

🔗 Services — Stable Networking

apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
    protocol: TCP
  type: ClusterIP

Why Services exist:

  • Pods have dynamic IPs — they change on restart
  • Services provide a stable virtual IP
  • Built-in load balancing across pod replicas
  • DNS-based discovery — no hardcoded IPs
  • Selector connects Service to Pods via labels
Think of it as: A stable address that always routes to healthy pods.
The Service is how pods find and talk to each other. The selector is the glue — it matches labels on pods. Kubernetes automatically updates the endpoints as pods come and go. This is the foundation of service discovery, which we'll cover in depth in Module 4.

📜 Declarative vs. Imperative

The most important mental shift

This is arguably the most important conceptual shift for developers coming to Kubernetes. Let's make it crystal clear.

🔄 Two Ways to Interact with Kubernetes

🚨 Imperative (Avoid in Production)

# Step-by-step commands
kubectl create deployment myapp \
  --image=myapp:v1
kubectl scale deployment myapp \
  --replicas=3
kubectl set image deployment/myapp \
  myapp=myapp:v2

Tells Kubernetes what to do step by step.

✅ Declarative (Best Practice)

# Declare desired state
kubectl apply -f deployment.yaml

# Update? Change the YAML, re-apply
kubectl apply -f deployment.yaml

# Everything is in version control
git diff deployment.yaml

Tells Kubernetes what you want — it figures out how.

The declarative approach is fundamental to Kubernetes' power. With imperative commands, you lose the ability to audit, diff, review, and version control your infrastructure. Declarative manifests are code — they go in Git, they get reviewed in PRs, and they serve as documentation of your system's desired state.

🏆 Why Declarative Wins

🔎 Auditability

  • Git history and PR reviews for every change
  • Blame shows who changed what

🔄 Reproducibility

  • Same YAML = same result everywhere
  • Disaster recovery is a kubectl apply

🤖 Automation

  • GitOps workflows (ArgoCD, Flux)
  • CI/CD pipelines with drift detection
Tip: The reconciliation loop is the superpower — Kubernetes continuously compares desired state (your YAML) to actual state (what's running) and takes action to converge. If a pod crashes, the controller sees the discrepancy and creates a new one. This is self-healing infrastructure, driven by your declarative manifests.

🌐 Environment Parity

Same image, same behavior, everywhere

Environment parity is one of the core promises of containerized workloads on Kubernetes. Let's explore how Kubernetes delivers on this promise.

📋 Environment Parity with Kubernetes

The Promise

The same container image runs identically in every environment. Configuration is injected, not baked in.

What stays the same

  • Container image (immutable artifact)
  • Application code, dependencies, and runtime
  • OS-level packages

What changes per environment

  • ConfigMaps and Secrets
  • Resource requests/limits and replica counts
  • Ingress rules / hostnames
This is a direct implementation of the 12-factor app principle of dev/prod parity. The image is your immutable artifact — it's the same bytes in every environment. Only the configuration changes. Kubernetes makes this natural with ConfigMaps and Secrets.

🛠️ Environment Parity in Practice

# Directory structure for multi-environment manifests
k8s/
├── base/
│   ├── deployment.yaml        # Common Deployment spec
│   ├── service.yaml           # Common Service spec
│   └── kustomization.yaml     # Base kustomization
├── overlays/
│   ├── dev/
│   │   ├── kustomization.yaml # Dev-specific patches
│   │   └── configmap.yaml
│   ├── staging/
│   │   └── ...                # Same structure as dev
│   └── prod/
│       └── ...                # Same structure as dev
# Deploy to each environment
kubectl apply -k k8s/overlays/dev/
kubectl apply -k k8s/overlays/staging/
kubectl apply -k k8s/overlays/prod/
Kustomize is built into kubectl and is the standard approach for managing environment-specific configurations. The base contains the common definition, and overlays patch only what differs. This ensures environments stay in sync while allowing necessary differences like replica counts and resource limits.

🎯 12-Factor App Principles

How Kubernetes embodies modern application design

The 12-factor app methodology predates Kubernetes, but Kubernetes was essentially designed to run 12-factor apps. Let's map the principles to Kubernetes features.

📚 12-Factor Principles & Kubernetes

Factor Principle Kubernetes Implementation
I Codebase One repo, one container image
II Dependencies Bundled in container image
III Config ConfigMaps, Secrets, env vars
IV Backing Services Services, ExternalName, endpoints
V Build, Release, Run Image build → tag → Deployment
VI Processes Stateless pods, external state
This is the first half of the 12 factors. Notice how each one maps cleanly to a Kubernetes concept. This isn't a coincidence — Kubernetes was designed by people who deeply understood these principles. If your application follows 12-factor design, it will thrive on Kubernetes.

📚 12-Factor Principles & Kubernetes (cont.)

Factor Principle Kubernetes Implementation
VII Port Binding containerPort + Service
VIII Concurrency Horizontal scaling (replicas, HPA)
IX Disposability Fast startup, graceful shutdown (preStop)
X Dev/Prod Parity Same image + Kustomize overlays
XI Logs Write to stdout/stderr, collected by platform
XII Admin Processes Jobs, CronJobs, kubectl exec
Design implication: Applications that follow these principles are "Kubernetes-native" — they leverage the platform fully and avoid fighting it.
Factor IX — Disposability — is particularly important. Kubernetes will kill and reschedule your pods. If your application can't handle graceful shutdown (SIGTERM), you'll have problems. Factor XI — Logs — means your app should never write to log files inside the container. Write to stdout/stderr and let the platform handle collection.

🚫 Factor IX Deep Dive — Disposability

spec:
  terminationGracePeriodSeconds: 30
  containers:
  - name: myapp
    image: myapp:v1.2.3
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 5"]
    # Pod termination sequence:
    # 1. Marked Terminating → removed from endpoints
    # 2. preStop hook runs → SIGTERM sent
    # 3. Wait terminationGracePeriodSeconds
    # 4. SIGKILL if still running
Tip: Not handling SIGTERM is a common mistake — in-flight requests will be dropped if your process doesn't gracefully shut down. The pod termination lifecycle is critical for zero-downtime deployments. The preStop sleep of 5 seconds gives kube-proxy and ingress controllers time to update their routing tables before the pod stops accepting traffic.

☁️ Kubernetes on EKS

Your platform at Verisign

Let's ground everything we've discussed in the specific platform you'll be working with — Amazon EKS.

☁️ Amazon EKS — What It Means for Developers

AWS Manages

  • Control plane (API server, etcd, scheduler, controllers)
  • HA and multi-AZ availability
  • Version upgrades and certificates

Your Team Manages

  • Worker node groups (EC2 or Fargate)
  • Application deployments and configuration
  • Namespaces, RBAC, and add-ons
EKS advantage: The control plane is someone else's problem. You focus on workloads.
EKS is a managed Kubernetes service, which means the hardest part of running Kubernetes — the control plane — is handled by AWS. This lets the platform team focus on worker node management and developer enablement, and lets developers focus entirely on their applications.

🛠️ EKS Developer Workflow

# 1. Authenticate to the cluster
aws eks update-kubeconfig --name verisign-cluster --region us-east-1

# 2. Verify connectivity and set namespace
kubectl cluster-info
kubectl config set-context --current --namespace=my-team

# 3. Deploy your application
kubectl apply -f k8s/

# 4. Monitor your deployment
kubectl rollout status deployment/myapp
kubectl get pods -w

# 5. Check logs and debug
kubectl logs -l app=myapp --tail=100 -f
kubectl describe pod myapp-abc123
This is the daily workflow for developers on EKS. The aws eks update-kubeconfig command configures your local kubectl to authenticate with the EKS cluster using IAM. From there, it's standard Kubernetes commands. Namespaces are typically used to isolate team workloads.
