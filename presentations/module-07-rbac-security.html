🛡️ Kubernetes Security Model

Authentication, authorization, and admission control

Every API request to Kubernetes passes through three gates: authentication (who are you?), authorization (what can you do?), and admission control (should this specific request be allowed?). Understanding this pipeline is essential.

🚪 API Request Flow — Three Security Gates

Every request to the API server passes through:

1. Authentication

  • Who are you? Validates identity
  • Returns user/group info

2. Authorization

  • What can you do?
  • RBAC evaluates; allow or deny

3. Admission

  • Should this be allowed?
  • Mutating + validating webhooks
Think of these as three security checkpoints. Authentication verifies identity. Authorization checks permissions against RBAC policies. Admission controllers can modify or reject requests even after authorization succeeds — for example, enforcing security policies or injecting sidecars.

👤 Authentication Methods

X.509 Client Certificates

  • CN = username, O = group; used by cluster components

Bearer Tokens

  • ServiceAccount tokens (JWT) and bootstrap tokens

OpenID Connect (OIDC)

  • Enterprise SSO via Azure AD, Okta, Dex

AWS IAM Authenticator (EKS)

  • Maps IAM roles to K8s users via aws-auth ConfigMap
Kubernetes itself does not store user accounts — it delegates authentication to external systems. In EKS, the AWS IAM Authenticator is the primary method: IAM roles and users are mapped to Kubernetes identities via the aws-auth ConfigMap. OIDC is increasingly popular for human users in enterprise environments.

🔒 RBAC Deep Dive

Role-Based Access Control in Kubernetes

RBAC is the standard authorization mode in Kubernetes. It lets you define fine-grained permissions using four API objects. Let's break each one down with real examples.

📦 RBAC API Objects

Define Permissions

  • Role — namespace-scoped permissions
  • ClusterRole — cluster-wide permissions

Grant Permissions

  • RoleBinding — binds Role/ClusterRole to subjects in a namespace
  • ClusterRoleBinding — binds ClusterRole to subjects cluster-wide
Key insight: Roles define what is allowed. Bindings define who gets those permissions. You always need both.
The separation of Roles and Bindings is intentional — it allows reuse. A single ClusterRole can be bound to different users in different namespaces via multiple RoleBindings. This is how most multi-tenant clusters are configured.

📄 Role — Namespace-Scoped Permissions

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: development
  name: pod-reader
rules:
- apiGroups: [""]          # core API group
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get"]
Tip: This Role only grants permissions within the development namespace — it cannot affect other namespaces. The empty string apiGroup represents the core API group. The resources field supports sub-resources like pods/log.

🌐 ClusterRole — Cluster-Wide Permissions

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-viewer
rules:
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get", "watch", "list"]
- apiGroups: ["metrics.k8s.io"]
  resources: ["nodes"]
  verbs: ["get", "list"]
- nonResourceURLs: ["/healthz", "/readyz"]
  verbs: ["get"]
Tip: ClusterRoles can reference nonResourceURLs like /healthz — something Roles cannot do. ClusterRoles also serve as reusable templates that can be bound per-namespace via RoleBindings, a common pattern in multi-tenant setups.

🔗 RoleBinding — Granting Namespace Access

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-dev
  namespace: development
subjects:
- { kind: User, name: jane, apiGroup: rbac.authorization.k8s.io }
- { kind: Group, name: dev-team, apiGroup: rbac.authorization.k8s.io }
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
Subjects can be Users, Groups, or ServiceAccounts. roleRef can point to a Role or ClusterRole.
A RoleBinding can reference a ClusterRole — this is a powerful pattern. It grants the ClusterRole's permissions but only within the RoleBinding's namespace. This lets you define permissions once as a ClusterRole and reuse them across multiple namespaces.

🌎 ClusterRoleBinding — Cluster-Wide Access

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: cluster-node-viewers
subjects:
- kind: Group
  name: ops-team
  apiGroup: rbac.authorization.k8s.io
- kind: ServiceAccount
  name: monitoring-sa
  namespace: monitoring
roleRef:
  kind: ClusterRole
  name: node-viewer
  apiGroup: rbac.authorization.k8s.io
Tip: ClusterRoleBindings grant permissions across ALL namespaces — use sparingly. A ClusterRoleBinding to cluster-admin gives full cluster access. Restrict who can create ClusterRoleBindings via admission control or OPA/Gatekeeper policies.

📈 Verbs and Resources Matrix

Verb HTTP Method Description
get GET (single) Retrieve a specific resource
list GET (collection) List resources
watch GET (watch) Stream changes to resources
create POST Create a new resource
update PUT Replace entire resource
patch PATCH Modify part of a resource
delete DELETE Remove a resource
Wildcard: Use "*" for all verbs or all resources — but avoid this in production.
Each RBAC verb maps to an HTTP method on the API server. The watch verb is unique to Kubernetes — it allows streaming updates via long-polling. When defining Roles, be specific about verbs. A common mistake is granting all verbs when only get/list are needed.

🛠️ Aggregated ClusterRoles

kind: ClusterRole
metadata:
  name: monitoring-endpoints
  labels:
    rbac.example.com/aggregate-to-monitoring: "true"
rules:
- apiGroups: [""]
  resources: ["services", "endpoints", "pods"]
  verbs: ["get", "list", "watch"]
kind: ClusterRole
metadata: { name: monitoring }
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      rbac.example.com/aggregate-to-monitoring: "true"
rules: []  # auto-filled by controller

How it works

The controller manager automatically combines rules from all ClusterRoles matching the label selector. Built-in roles like admin, edit, and view use aggregation.

Aggregated ClusterRoles are how Kubernetes extends its built-in roles. When you install a CRD, you can add a ClusterRole with the appropriate label, and it automatically gets merged into the admin/edit/view roles. This is a clean extension mechanism that avoids modifying built-in roles directly.

🧰 ServiceAccounts

Identity for workloads running in the cluster

While users authenticate externally, pods authenticate using ServiceAccounts. Understanding ServiceAccounts is critical because they define what your workloads can do within the cluster and with external services.

🔭 ServiceAccount Anatomy

apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: production
  annotations:
    eks.amazonaws.com/role-arn: >-
      arn:aws:iam::123456789012:role/app-role
automountServiceAccountToken: false

Key Properties

  • Namespace-scoped identity
  • Every namespace has a default SA
  • Tokens are projected as volumes
  • Annotations enable cloud integrations
  • Can disable auto-mount for security
Every pod runs as a ServiceAccount. If you don't specify one, it uses the default SA in the namespace. Best practice is to create dedicated ServiceAccounts per application and disable auto-mounting of tokens when API access isn't needed. The EKS annotation enables IAM Roles for Service Accounts (IRSA).

💲 ServiceAccount Token Projection

apiVersion: v1
kind: Pod
metadata: { name: my-app }
spec:
  serviceAccountName: app-sa
  containers:
  - name: app
    image: my-app:1.0
    volumeMounts:
    - { name: sa-token, mountPath: /var/run/secrets/tokens }
  volumes:
  - name: sa-token
    projected:
      sources:
      - serviceAccountToken:
          path: app-token
          expirationSeconds: 3600
          audience: api
Tip: Projected tokens are time-bound and audience-scoped — a major security improvement over legacy static tokens. Kubernetes 1.20+ uses bound service account tokens by default. These tokens are time-limited, audience-bound, and object-bound. They expire and get automatically rotated. Always prefer projected tokens over legacy long-lived tokens.

🔗 Binding Roles to ServiceAccounts

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-sa-pod-reader
  namespace: production
subjects:
- kind: ServiceAccount
  name: app-sa
  namespace: production
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
Pattern: Create a dedicated ServiceAccount per application, grant only the permissions it needs, and reference it in the pod spec with serviceAccountName.
This is the standard pattern for granting in-cluster permissions to workloads. The ServiceAccount subject includes a namespace field, which is important when creating cross-namespace bindings. Never grant permissions to the default ServiceAccount — always create dedicated ones.

☁️ Pod Identity — IRSA on EKS

IAM Roles for Service Accounts

  • Maps K8s ServiceAccount to IAM Role
  • Uses OIDC federation
  • No shared node-level credentials
  • Fine-grained per-pod access

How IRSA Works

  1. EKS cluster has an OIDC provider
  2. IAM role trusts the OIDC provider
  3. SA annotated with IAM role ARN
  4. Pod receives projected OIDC token
  5. AWS SDK exchanges token for creds
IRSA eliminates the need for node-level IAM roles or storing AWS credentials in secrets. Each pod gets its own identity with only the AWS permissions it needs. This is a major security improvement over the older kiam/kube2iam approaches. EKS Pod Identity is the newer alternative that simplifies the setup further.

🛡️ Security Context

Constraining what pods and containers can do

RBAC controls what you can do with the API. Security contexts control what your containers can do at the OS level — which user they run as, what Linux capabilities they have, and whether they can escalate privileges.

🗃️ Pod SecurityContext

apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: my-app:1.0
runAsUser/runAsGroup: Sets the UID/GID for all containers in the pod
fsGroup: Sets the group ownership of all mounted volumes
Pod-level security context applies to all containers in the pod. runAsNonRoot is a critical setting — it prevents containers from running as root even if the Dockerfile specifies USER root. The seccomp profile restricts available system calls. Always set these in production workloads.

📦 Container SecurityContext

containers:
- name: app
  image: my-app:1.0
  securityContext:
    allowPrivilegeEscalation: false
    readOnlyRootFilesystem: true
    capabilities:
      drop: ["ALL"]
      add: ["NET_BIND_SERVICE"]
    privileged: false

Container-level overrides pod-level settings

  • allowPrivilegeEscalation: Prevents gaining more privileges than parent
  • readOnlyRootFilesystem: Prevents writes to container filesystem
  • capabilities: Fine-grained Linux capability control
  • privileged: Never set to true in production
The recommended approach is drop ALL capabilities first, then add back only what you need. NET_BIND_SERVICE allows binding to ports below 1024. readOnlyRootFilesystem forces applications to write only to mounted volumes, which improves security and makes container behavior more predictable. Use emptyDir volumes for temp files.

📦 Building Secure Container Images

Minimal Images

  • Multi-stage builds — compile in one stage, copy only the binary to the final image
  • Distroless — Google's images with no shell or package manager
  • Alpine — ~5 MB base; good for interpreted languages
  • scratch — empty base for statically linked Go / Rust binaries
# Multi-stage build example
FROM golang:1.22 AS build
WORKDIR /src
COPY . .
RUN CGO_ENABLED=0 go build -o /app

FROM gcr.io/distroless/static:nonroot
COPY --from=build /app /app
USER 65532:65532
ENTRYPOINT ["/app"]
Result: Final image is ~15 MB with no shell, no root, no package manager
The number one container best practice is to minimize your attack surface. Fewer binaries means fewer CVEs. Multi-stage builds keep your final image lean — build tools, compilers, and test dependencies never ship to production. The distroless nonroot tag already sets a non-root user, which pairs well with the runAsNonRoot security context we just covered. On EKS, smaller images also reduce ECR storage costs and speed up node scaling.

🔎 Image Security Pipeline

🛡️ Run as Non-Root

  • Add USER directive in Dockerfile
  • Pair with runAsNonRoot: true in security context
  • Never store secrets in image layers — use K8s Secrets or AWS Secrets Manager

🔍 Image Scanning

  • Trivy — open-source, CI-friendly scanner
  • ECR scanning — automatic on push (Basic or Enhanced with Inspector)
  • Scan in CI and continuously in the registry

✏️ Image Signing

  • Cosign / Sigstore — keyless signing with OIDC identity
  • Verify signatures at admission with a policy engine
  • Pin images by @sha256: digest, not mutable tags
Supply-chain rule of thumb: Build minimal → scan for CVEs → sign the artifact → verify at deploy
These practices form a container supply-chain security pipeline. Trivy can run as a single binary in any CI system — pass --exit-code 1 to fail the build on HIGH/CRITICAL CVEs. ECR Enhanced Scanning uses Amazon Inspector and provides continuous monitoring, not just scan-on-push. Cosign with Sigstore lets you sign images without managing long-lived keys — it ties the signature to your CI system's OIDC identity. Remind students that docker history exposes every layer, so secrets baked into images are trivially extractable even if you delete them in a later layer.

📐 Pod Security Standards

Privileged

  • Unrestricted; all capabilities allowed
  • For CNI plugins, storage drivers only

Baseline

  • Prevents known escalations
  • Good starting point for migration

Restricted

  • Non-root, drop all capabilities
  • Production hardened target
Recommendation: Target Restricted for application workloads. Use Baseline as a transition step. Reserve Privileged for infrastructure components only.
Pod Security Standards replace the deprecated PodSecurityPolicy (PSP). They define three levels of restriction. Most application workloads should target Restricted. The Baseline level is useful during migration from unrestricted clusters. These standards are enforced by the Pod Security Admission controller, which we'll cover next.

🚨 Pod Security Admission Controller

apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
Tip: enforce rejects pods that violate the policy. audit/warn logs violations without rejecting. Migration strategy: start with warn, then audit, then enforce. Pod Security Admission is built into Kubernetes since 1.25 — configured via namespace labels, no installation needed.

✅ Security Best Practices

Hardening your cluster for production

Now that we understand the mechanisms, let's talk about best practices. These recommendations come from the Kubernetes security documentation, CIS benchmarks, and real-world production experience.

🔐 Principle of Least Privilege in Practice

RBAC Least Privilege

  • Use Roles over ClusterRoles when possible
  • Grant specific verbs — avoid *
  • Name specific resources
  • Use resourceNames for fine-grained control
  • Audit with kubectl auth can-i
# Fine-grained: specific resource names
kind: Role
metadata: { namespace: production, name: configmap-updater }
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  resourceNames: ["app-config"]
  verbs: ["get", "update"]
# Verify permissions
kubectl auth can-i get pods --as=jane -n dev
kubectl auth can-i --list \
  --as=system:serviceaccount:prod:app-sa
The resourceNames field is underused but powerful — it restricts access to specific named resources. The kubectl auth can-i command is essential for verifying RBAC policies. Use the --as flag to impersonate users or ServiceAccounts and test their effective permissions before deploying changes.

📝 Audit Logging

apiVersion: audit.k8s.io/v1
kind: Policy
rules:
- level: Metadata
  resources:
  - { group: "", resources: ["secrets"] }
- level: RequestResponse
  resources:
  - { group: "rbac.authorization.k8s.io",
      resources: ["clusterroles", "clusterrolebindings"] }
- level: None
  resources:
  - { group: "", resources: ["endpoints", "services/status"] }

Audit Levels

  • None: Skip logging
  • Metadata: Request metadata only
  • Request: Metadata + request body
  • RequestResponse: Log everything
Audit logs are essential for security monitoring and forensics. Always log RBAC changes at RequestResponse level — you need to know who modified permissions. Log secrets at Metadata level only to avoid exposing secret values in logs. In EKS, enable control plane logging and ship audit logs to CloudWatch for analysis and alerting.

📡 Image Security

Image Scanning

  • Scan in CI/CD pipeline (Trivy, Grype)
  • ECR image scanning (on push)
  • Block critical CVEs from deploying
  • Scan base images regularly
Always use: imagePullPolicy: Always with immutable tags or digests

Image Signing & Verification

  • Sign images with Cosign / Notation
  • Verify signatures at admission
  • Use allowlisted registries only
  • Pin images by SHA256 digest
# Pin image by digest
containers:
- name: app
  image: 123456789012.dkr.ecr.us-east-1
    .amazonaws.com/app@sha256:abc123...
Image security is a critical part of the supply chain. Never use the latest tag in production — always use immutable tags or SHA256 digests. ECR supports both automatic scanning and manual scanning. For signing, Cosign from the Sigstore project is becoming the industry standard. Admission controllers like Kyverno or OPA/Gatekeeper can enforce registry allowlists and signature verification.

⚡ EKS Security Best Practices

Cluster Configuration

  • Enable envelope encryption for secrets
  • Use private API endpoint
  • Enable control plane logging
  • Restrict CIDR for API access
  • Keep Kubernetes version current

Workload Configuration

  • Use IRSA or EKS Pod Identity
  • Enforce Pod Security Standards
  • Use NetworkPolicies (needs CNI support)
  • Rotate ServiceAccount tokens
  • Use external secrets (Secrets Manager)
aws-auth ConfigMap: Regularly audit this to ensure only authorized IAM principals have cluster access. Consider migrating to EKS access entries for more granular control.
EKS-specific security goes beyond generic Kubernetes security. The aws-auth ConfigMap is a common source of misconfiguration — audit it regularly. Envelope encryption ensures etcd secrets are encrypted with a KMS key you control. EKS access entries are the newer replacement for aws-auth that provide better integration with IAM and support condition keys.

🔑 IRSA: IAM Roles for Service Accounts

Least-privilege AWS access for pods

We just locked down in-cluster access with RBAC and Pod Security. But pods often need to reach AWS services — S3, SQS, Secrets Manager. Without IRSA, every pod on a node shares the EC2 instance profile, violating least privilege. IRSA extends the security boundary to AWS IAM. On Verisign's EKS platform, IRSA is the standard pattern for any workload accessing AWS services like S3, SQS, or Secrets Manager.

⚠️ Why Node-Level IAM Is Dangerous

Without IRSA: Every pod on a node inherits the EC2 instance profile — the same IAM role for all workloads.
  • Over-privileged pods — a logging sidecar gets the same S3/SQS access as your data pipeline
  • Blast radius — one compromised container can access every AWS resource the node role permits
  • No audit granularity — CloudTrail shows the node role, not which pod made the API call
IRSA solves this by mapping each Kubernetes ServiceAccount to its own IAM role with scoped permissions — true least privilege for AWS access.
The node IAM role is typically broad because it serves all workloads. Any pod can query the IMDS endpoint at 169.254.169.254 to retrieve those credentials. IRSA eliminates this by giving each ServiceAccount its own short-lived, scoped IAM credentials. This is the AWS equivalent of what we just did with RBAC inside the cluster.

🔀 How IRSA Works: OIDC → STS → Credentials

End-to-End Flow

  1. EKS cluster exposes an OIDC endpoint — registered as an IAM Identity Provider
  2. Pod starts — EKS webhook injects a projected JWT token + AWS_ROLE_ARN env var
  3. AWS SDK calls STS AssumeRoleWithWebIdentity with the JWT
  4. STS validates the token signature via the OIDC provider's JWKS
  5. STS returns temporary credentials scoped to that ServiceAccount's IAM role
Zero code changes: The AWS SDK credential chain detects IRSA environment variables automatically.
The projected token is short-lived (default 24 hours), audience-scoped to sts.amazonaws.com, and auto-rotated by the kubelet. The application never handles credentials directly — the SDK does everything transparently. This is the same OIDC federation pattern used by GitHub Actions and other CI systems.

🏷️ Configuring IRSA

1. Annotate the ServiceAccount

apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: my-namespace
  annotations:
    eks.amazonaws.com/role-arn: >-
      arn:aws:iam::111122223333:role/
      my-app-irsa-role

The annotation tells the EKS webhook which IAM role to inject into pods using this SA.

2. IAM Trust Policy

{
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Federated":
      "arn:aws:iam::111122223333:oidc-provider/oidc.eks..." },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": { "StringEquals": {
      "OIDC_ID:aud": "sts.amazonaws.com",
      "OIDC_ID:sub": "system:serviceaccount:my-namespace:my-app-sa"
    }}
  }]
}
Security boundary: The :sub condition scopes the role to an exact namespace + ServiceAccount. Without it, any SA in the cluster could assume this role.
The trust policy is the critical security control. The :sub condition must exactly match namespace:serviceaccount-name. For quick setup, eksctl create iamserviceaccount handles both the IAM role and the K8s ServiceAccount annotation in one command. For production, manage these through Terraform or CloudFormation for auditability and repeatability.

✅ IRSA Best Practices

Do

  • One IAM role per workload — never share roles across services
  • Scope IAM policies to specific resource ARNs — no "Resource": "*"
  • Always include the :sub condition in trust policies
  • Verify with kubectl exec -- aws sts get-caller-identity

Avoid

  • Sharing the node instance profile across workloads
  • Wildcard actions or resources in IRSA policies
  • Using StringLike with broad :sub patterns
  • Storing long-lived AWS keys in Secrets — IRSA replaces this entirely
EKS Pod Identity: AWS's newer alternative (GA 2023) simplifies IRSA by removing trust-policy management. IRSA remains the standard for existing clusters.
Think of IRSA as the AWS extension of the RBAC least-privilege principle we covered earlier. Inside the cluster, RBAC controls what a ServiceAccount can do with the Kubernetes API. Outside the cluster, IRSA controls what that same ServiceAccount can do with AWS APIs. Together they form a complete identity and access management story for EKS workloads.

🎯 Key Takeaways

Least Privilege: Grant only the minimum permissions needed. Start with no access and add permissions as required.
PSS Restricted Profile: Use the restricted Pod Security Standard for all production namespaces to enforce non-root, read-only FS, and no privilege escalation.
IRSA for Pod Identity: Use IAM Roles for Service Accounts to give pods scoped AWS access without sharing static credentials.
Audit Logging: Enable Kubernetes audit logging and forward to your SIEM to detect unauthorized access and policy violations.
These four takeaways represent the highest-impact security practices. Least privilege applies to both RBAC and IAM. PSS restricted is the new standard replacing PSPs. IRSA eliminates the need for long-lived AWS credentials. Audit logging is essential for compliance and incident response.
