⚠️ Kubernetes Secrets Limitations

Why native Secrets fall short in production

Before introducing external solutions, we need to understand exactly where Kubernetes Secrets break down. This motivates the entire module.

Why K8s Secrets Aren’t Enough

base64 encoding is NOT encryption — anyone with API access can decode secrets instantly.
  • etcd storage — secrets stored unencrypted by default; encryption at rest requires explicit configuration
  • RBAC gaps — get permission on Secrets grants access to all secrets in a namespace
  • No audit trail — no built-in tracking of who accessed which secret and when
  • No rotation — no native mechanism for automatic credential rotation
  • No dynamic secrets — cannot generate short-lived, on-demand credentials
  • GitOps conflict — secrets in manifests cannot be safely committed to version control
Walk through each limitation. The base64 issue is the most common misconception. etcd encryption at rest helps but doesn't solve access control or rotation. RBAC is namespace-scoped, so fine-grained per-secret access requires additional tooling. These gaps collectively drive the need for external secrets management.

External Secrets Management Landscape

🛡️ HashiCorp Vault

  • Self-hosted or HCP
  • Dynamic secrets
  • Policy-based access
  • Multi-cloud support

☁️ AWS Secrets Manager

  • Fully managed
  • Automatic rotation
  • IAM integration
  • Cross-account sharing

🔑 Azure Key Vault

  • HSM-backed keys
  • Managed identities
  • Certificate lifecycle
  • FIPS 140-2 compliant
Key insight: The External Secrets Operator provides a unified Kubernetes-native interface to all of these backends.
Each provider has strengths. Vault excels at dynamic secrets and multi-cloud. AWS Secrets Manager integrates tightly with AWS services. Azure Key Vault offers HSM-backed storage. ESO abstracts the backend, so your K8s manifests remain consistent regardless of the secrets provider.

🛡️ HashiCorp Vault

Architecture, secrets engines, and Kubernetes integration

Vault is the most feature-rich secrets management platform. We'll cover its architecture and the specific components that matter for Kubernetes integration.

Vault Architecture

Core Components

  • Storage Backend — Consul, Raft, DynamoDB, S3
  • Secrets Engines — KV, database, PKI, AWS, transit
  • Auth Methods — Kubernetes, OIDC, LDAP, AppRole
  • Audit Devices — file, syslog, socket

Security Model

  • Seal/Unseal — master key split via Shamir’s secret sharing
  • Policies — path-based ACLs in HCL
  • Tokens — identity-bound, TTL-limited
  • Leases — time-bound secret access with renewal
Vault's architecture is modular. The storage backend is pluggable - Raft is the recommended integrated storage for production. Secrets engines are mounted at paths and each handles a different type of secret. Auth methods validate identity and map it to policies. Everything is auditable.

KV v2 Secrets Engine

KV v2 adds versioning, metadata, and soft-delete on top of simple key-value storage. 
# Enable KV v2 secrets engine
vault secrets enable -version=2 -path=apps kv

# Write a secret
vault kv put apps/myapp/db \
  username="app_user" \
  password="s3cur3P@ss!"

# Read current version
vault kv get apps/myapp/db

# Read specific version
vault kv get -version=2 apps/myapp/db

# Soft-delete a version
vault kv delete -versions=3 apps/myapp/db

# Undelete (recover soft-deleted)
vault kv undelete -versions=3 apps/myapp/db
Versioning: KV v2 retains up to 10 versions by default. Configure with vault kv metadata put -max-versions=20.
KV v2 is the most commonly used secrets engine. Versioning means you can roll back a secret to a previous value without redeploying. Soft-delete allows recovery of accidentally deleted secrets. The metadata endpoint tracks creation time, version count, and custom metadata per secret.

Vault Policies

Path-based access control written in HCL or JSON:

# policy: app-readonly.hcl
# Grant read access to application secrets
path "apps/data/myapp/*" {
  capabilities = ["read", "list"]
}

# Allow listing available secrets
path "apps/metadata/myapp/*" {
  capabilities = ["read", "list"]
}

# Deny access to admin secrets
path "apps/data/admin/*" {
  capabilities = ["deny"]
}
Capabilities: create, read, update, delete, list, sudo, deny
Note: KV v2 paths require /data/ and /metadata/ prefixes in policies
Policies are the core authorization mechanism. They follow deny-by-default semantics. The KV v2 path distinction trips up many people - the API path includes /data/ for reading secrets and /metadata/ for version info, even though the CLI abstracts this away. Policies are attached to tokens via auth method role mappings.

Kubernetes Auth Method

How It Works

Pods authenticate to Vault using their ServiceAccount JWT token. Vault validates the token against the Kubernetes API server.

# Enable Kubernetes auth
vault auth enable kubernetes

# Configure with cluster details
vault write auth/kubernetes/config \
  kubernetes_host="https://kubernetes.default.svc" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# Create a role binding SA to policy
vault write auth/kubernetes/role/myapp-role \
  bound_service_account_names=myapp-sa \
  bound_service_account_namespaces=production \
  policies=app-readonly \
  ttl=1h
Flow: Pod SA token → Vault validates with K8s API → issues Vault token with bound policies → Pod reads secrets
The Kubernetes auth method is the natural choice for pods. The role binds a specific ServiceAccount in a specific namespace to a set of Vault policies. TTL controls how long the Vault token is valid. This means no static credentials need to be distributed to pods - they use their existing K8s identity.

🔄 External Secrets Operator

Bridging external secret stores with Kubernetes

The External Secrets Operator, or ESO, is the Kubernetes-native way to sync secrets from external providers into native K8s Secret objects. It decouples your workloads from any specific secrets backend.

ESO Architecture

Components

  • Operator — controller watching ESO CRDs
  • SecretStore — connection config to a provider
  • ExternalSecret — declares what to sync
  • ClusterSecretStore — cluster-wide store

Reconciliation Loop

  • ESO watches ExternalSecret resources
  • Connects to provider via SecretStore
  • Fetches secret data from external store
  • Creates/updates native K8s Secret
  • Re-syncs on configured interval
Result: Workloads consume standard K8s Secrets — no code changes needed. ESO handles the sync transparently.
ESO follows the standard Kubernetes operator pattern. It installs CRDs and runs a controller that watches for ExternalSecret resources. When it finds one, it uses the referenced SecretStore to connect to the external provider, fetch the data, and create a native Kubernetes Secret. Pods never know the difference.

SecretStore vs ClusterSecretStore

SecretStore (Namespaced)

  • Scoped to a single namespace
  • Teams manage their own stores
  • Credentials in same namespace
  • Isolation by default
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-store
  namespace: app-team

ClusterSecretStore (Global)

  • Available to all namespaces
  • Platform team manages centrally
  • Shared provider credentials
  • Reduces duplication
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: vault-cluster-store
Best practice: Use ClusterSecretStore for shared infrastructure secrets and namespaced SecretStore for team-specific backends.
The choice depends on your organizational model. ClusterSecretStore reduces operational overhead when all teams use the same Vault cluster. Namespaced SecretStore is appropriate when teams manage their own secret backends or need different authentication credentials. You can use both in the same cluster.

ExternalSecret CRD Anatomy

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: myapp-db-credentials
  namespace: production
spec:
  refreshInterval: 5m          # How often to sync
  secretStoreRef:
    name: vault-store           # Reference to SecretStore
    kind: SecretStore
  target:
    name: db-credentials        # K8s Secret to create
    creationPolicy: Owner       # ESO owns the Secret lifecycle
  data:
    - secretKey: username        # Key in K8s Secret
      remoteRef:
        key: apps/myapp/db      # Path in Vault
        property: username       # Field within the secret
    - secretKey: password
      remoteRef:
        key: apps/myapp/db
        property: password
Walk through each field. refreshInterval controls how often ESO polls the provider. secretStoreRef points to the connection config. target defines the resulting K8s Secret. The data array maps individual fields from the external store to keys in the K8s Secret. creationPolicy: Owner means ESO will delete the K8s Secret if the ExternalSecret is deleted.

Data Mapping and Templating

ESO supports advanced data transformations with Go templates:

spec:
  target:
    name: db-connection
    template:
      type: Opaque
      data:
        DSN: |
          postgresql://{{ .username }}:{{ .password }}@
          {{ .host }}:{{ .port }}/{{ .database }}?sslmode=require
  data:
    - secretKey: username
      remoteRef: { key: apps/myapp/db, property: username }
    - secretKey: password
      remoteRef: { key: apps/myapp/db, property: password }
    - secretKey: host
      remoteRef: { key: apps/myapp/db, property: host }
    - secretKey: port
      remoteRef: { key: apps/myapp/db, property: port }
    - secretKey: database
      remoteRef: { key: apps/myapp/db, property: database }
Templating is powerful for assembling connection strings, config files, or any derived format from raw secret fields. The template has access to all data keys. You can also set labels and annotations on the generated Secret via the template metadata. This eliminates the need for init containers that assemble config from individual env vars.

Refresh Intervals and Sync Status

Refresh Behavior

  • refreshInterval: 5m — poll every 5 minutes
  • refreshInterval: 0 — sync once, never refresh
  • refreshInterval: 1h — suitable for stable secrets
  • Trade-off: API cost vs freshness
# Check sync status
kubectl get externalsecret -n production

# NAME                  STORE        REFRESH   STATUS
# myapp-db-credentials  vault-store  5m        SecretSynced

# Detailed status
kubectl describe externalsecret \
  myapp-db-credentials -n production

# Conditions:
#   Type: Ready
#   Status: "True"
#   Reason: SecretSynced
Cost awareness: With 500 ExternalSecrets at 1m refresh, that’s 500 API calls per minute to your secrets provider. Size intervals appropriately.
Refresh interval is a key tuning parameter. Too frequent and you hit API rate limits and cost issues. Too infrequent and secret rotations take too long to propagate. For most production secrets, 5-15 minutes is a good starting point. Use 0 for secrets that never change, like static encryption keys. Monitor the sync status as part of your cluster health checks.

🔗 Integration Patterns

Connecting Vault and ESO in practice

Now we bring Vault and ESO together. These patterns represent real-world configurations you'll implement in production environments.

SecretStore with Vault Backend

apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-store
  namespace: production
spec:
  provider:
    vault:
      server: "https://vault.internal.verisign.com:8200"
      path: "apps"              # KV v2 mount path
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "myapp-role"
          serviceAccountRef: { name: "myapp-sa" }
      caProvider: { type: ConfigMap, name: vault-ca, key: ca.crt }
Authentication flow: ESO uses the referenced ServiceAccount token to authenticate with Vault’s Kubernetes auth method, receiving a Vault token bound to the configured policies.
This is the most common production setup. The SecretStore connects to Vault using Kubernetes auth. The ServiceAccount referenced must match what's configured in the Vault Kubernetes auth role. The caProvider field is important when Vault uses an internal CA. Always use HTTPS for the Vault server URL in production.

ExternalSecret: data vs dataFrom

data — Select Specific Keys

spec:
  data:
    - secretKey: db-user
      remoteRef:
        key: apps/myapp/db
        property: username
    - secretKey: db-pass
      remoteRef:
        key: apps/myapp/db
        property: password

Cherry-pick fields, rename keys

dataFrom — Sync All Keys

spec:
  dataFrom:
    - extract:
        key: apps/myapp/db
    - find:
        name:
          regexp: "apps/myapp/.*"
        tags:
          env: production

Import all fields, filter by pattern/tags

Tip: Use data when you need control over key names. Use dataFrom when you want all fields from a secret path without explicit mapping.
data gives you explicit control - you can rename keys and select specific properties. dataFrom is convenient when you want everything from a path. The find option with regexp is powerful for syncing multiple secrets matching a pattern. You can combine data and dataFrom in the same ExternalSecret.

Templated Secrets

Build application-ready configs from raw secret data:

spec:
  target:
    name: app-config
    template:
      engineVersion: v2
      data:
        # JDBC connection string
        spring.datasource.url: |
          jdbc:postgresql://{{ .db_host }}:5432/{{ .db_name }}
        spring.datasource.username: "{{ .db_user }}"
        spring.datasource.password: "{{ .db_pass }}"
        # JSON config file
        config.json: |
          { "redis": { "host": "{{ .redis_host }}",
              "port": {{ .redis_port }},
              "auth": "{{ .redis_token }}" } }
Use case: Generate Spring Boot properties, JSON configs, .env files, or any text format directly from Vault fields — no init containers required.
Templated secrets eliminate an entire class of config assembly logic from your deployments. Instead of mounting five separate env vars and building a connection string in application code, ESO does it at the secret level. This works especially well for legacy applications that expect specific config file formats.

Secret Rotation Workflow

End-to-End Rotation Flow

  1. Update in Vault — rotate credential via Vault CLI, API, or automated pipeline
  2. ESO detects change — next refresh interval polls Vault and finds new version
  3. K8s Secret updated — ESO patches the native Secret with new data
  4. Pod consumption — depends on mount type (volume vs env var)
Critical distinction: Volume-mounted secrets update automatically (kubelet sync ~60s). Environment variables require a pod restart to pick up changes.
Automation: Use Reloader or stakater/Reloader to trigger rolling restarts when Secrets change.
The rotation workflow is only as fast as the slowest link. If refresh is 5m and kubelet sync is 60s, worst case is about 6 minutes. For env vars, you need a restart mechanism. Stakater Reloader watches Secret objects and triggers rolling restarts on the associated Deployments. Plan rotation windows to account for this propagation delay.

Volume-Mounted vs Env Var Secrets

Aspect Volume Mount Environment Variable
Auto-update Yes (kubelet sync ~60s) No — requires pod restart
Access pattern File read from path Process environment
Rotation support Seamless if app watches files Needs restart controller
Visibility Only via filesystem Exposed in /proc, logs, crash dumps
Best for Certificates, config files, rotating creds Simple key-value pairs, legacy apps
Security note: Env vars can leak in error reports, docker inspect, and child processes. Prefer volume mounts for sensitive credentials.
This is an important architectural decision. Volume mounts are strictly better from both a security and rotation perspective. The only downside is that the application needs to read from a file path instead of an environment variable. Many frameworks support both. For credentials that rotate, volume mounts with file-watching application code is the recommended pattern.

☁️ AWS Secrets Manager Integration

Cloud-native secrets for EKS workloads

For teams running on EKS, AWS Secrets Manager and Parameter Store are natural choices. ESO integrates with both through IAM Roles for Service Accounts.

AWS Secrets Manager with ESO

apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata: { name: aws-secrets-manager }
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef: { name: eso-sa, namespace: external-secrets }
---
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata: { name: rds-credentials }
spec:
  refreshInterval: 10m
  secretStoreRef: { name: aws-secrets-manager, kind: ClusterSecretStore }
  target: { name: rds-creds }
  dataFrom:
    - extract: { key: production/rds/myapp }
IRSA: ESO uses IAM Roles for Service Accounts — no static AWS credentials needed in the cluster.
AWS Secrets Manager integration uses IRSA for authentication. The ESO service account is annotated with an IAM role ARN that has permissions to read from Secrets Manager. The secret in AWS SM is typically stored as JSON, and dataFrom extract will map each JSON key to a K8s Secret key. This is the recommended pattern for EKS environments.

AWS Parameter Store Integration

apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: aws-param-store
spec:
  provider:
    aws:
      service: ParameterStore
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef:
            name: eso-sa
            namespace: external-secrets
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: app-params
spec:
  refreshInterval: 15m
  secretStoreRef:
    name: aws-param-store
    kind: ClusterSecretStore
  target:
    name: app-parameters
  data:
    - secretKey: api-key
      remoteRef:
        key: /prod/myapp/api-key
    - secretKey: feature-flags
      remoteRef:
        key: /prod/myapp/features
Parameter Store vs Secrets Manager: Parameter Store is lower cost for simple key-value pairs. Secrets Manager adds automatic rotation and cross-account replication.
Parameter Store is a cost-effective option for non-rotating secrets and configuration values. It supports SecureString parameters encrypted with KMS. The main trade-off versus Secrets Manager is that Parameter Store lacks built-in rotation and has lower throughput limits. Many teams use both: Secrets Manager for credentials and Parameter Store for configuration.

🏭 Production Patterns

Hardening, monitoring, and multi-source strategies

Now we cover the patterns that distinguish a proof-of-concept from a production-ready secrets management setup. These are the considerations that matter at Verisign scale.

Multi-Source Secrets

Combine secrets from multiple backends in a single application:

# Source 1: Vault for database credentials
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata: { name: db-creds }
spec:
  secretStoreRef: { name: vault-store, kind: SecretStore }
  target: { name: app-secrets, creationPolicy: Merge }
  data:
    - secretKey: DB_PASSWORD
      remoteRef: { key: apps/myapp/db, property: password }
---
# Source 2: AWS SM for third-party API keys
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata: { name: api-keys }
spec:
  secretStoreRef: { name: aws-sm-store, kind: ClusterSecretStore }
  target: { name: app-secrets, creationPolicy: Merge }
  data:
    - secretKey: STRIPE_KEY
      remoteRef: { key: production/stripe, property: api_key }
creationPolicy: Merge allows multiple ExternalSecrets to contribute keys to a single K8s Secret.
Multi-source is common in practice. Database credentials might live in Vault for dynamic generation, while third-party API keys are in AWS Secrets Manager because they're managed by a different team. The Merge creation policy lets both ExternalSecrets write to the same target Secret. Be careful with key collisions - the last writer wins.

Secret Lifecycle Management

Creation Policies

  • Owner — ESO creates and deletes the Secret
  • Merge — ESO adds keys to existing Secret
  • Orphan — ESO creates but won’t delete
  • None — Secret must pre-exist

Deletion Policies

  • Delete — remove K8s Secret when ExternalSecret is deleted
  • Merge — remove only managed keys
  • Retain — keep K8s Secret on deletion
Production tip: Use deletionPolicy: Retain for critical secrets to prevent accidental data loss during ESO upgrades or ExternalSecret modifications.
Lifecycle policies are crucial for production. Owner is the simplest but can cause issues during ESO upgrades if ExternalSecrets are temporarily removed. Retain is safer for production because the K8s Secret persists even if the ExternalSecret is accidentally deleted. Merge is useful for multi-source patterns. Plan your lifecycle policies based on your deployment workflow and risk tolerance.

Monitoring Sync Failures

Key Metrics (Prometheus)

# ESO exposes metrics on :8080/metrics
- externalsecret_sync_calls_total
- externalsecret_sync_calls_error
- externalsecret_status_condition{type="Ready"}

# Alert rule example
- alert: ExternalSecretSyncFailed
  expr: |
    externalsecret_status_condition{type="Ready",status="False"} == 1
  for: 10m
  labels:
    severity: critical
  annotations:
    summary: "ExternalSecret {{ $labels.name }} sync failing"
  • Status conditions — kubectl get es -A shows Ready status per ExternalSecret
  • Events — ESO emits K8s events on sync success/failure
  • Dashboards — track sync latency, error rates, and provider API usage
Monitoring is non-negotiable in production. A failed sync means pods may be running with stale or missing credentials. Set up alerts for any ExternalSecret in a non-Ready state for more than a few minutes. Track provider API usage to stay within rate limits. Build a Grafana dashboard showing sync health across all namespaces.

Security Hardening

🔒 Network Policies

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: eso-egress
  namespace: external-secrets
spec:
  podSelector:
    matchLabels:
      app: external-secrets
  policyTypes: [Egress]
  egress:
    - to:
        - ipBlock:
            cidr: 10.0.1.0/24
      ports:
        - port: 8200
          protocol: TCP

🛡️ RBAC for ESO

  • Restrict ESO SA to minimum Secret permissions
  • Limit namespaces via ClusterSecretStore conditions
  • Audit ESO-created Secrets with labels
  • Enable Vault audit logging
  • Encrypt etcd at rest with KMS
  • Use PodSecurity restricted profile for ESO pods
Defense in depth: restrict ESO's network access to only the secrets providers it needs. Lock down RBAC so ESO can only create Secrets, not read arbitrary cluster resources. Label all ESO-managed secrets for audit trail. Enable etcd encryption at rest - on EKS this is handled by AWS, but verify it's enabled. Vault audit logs provide a complete record of every secret access.
