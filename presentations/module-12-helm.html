🔍 Why Helm?

The problem with raw YAML at scale

Before we dive into Helm mechanics, let's establish why it exists and what problems it solves for platform teams.

The YAML Sprawl Problem

A typical microservice requires 5–10 YAML manifests — Deployment, Service, ConfigMap, Secret, Ingress, HPA, PDB, ServiceAccount, NetworkPolicy...
  • Duplication — same patterns repeated across dozens of services
  • Environment drift — dev, staging, and prod configs diverge over time
  • No versioning — hard to track what’s deployed or roll back atomically
  • No dependency management — ordering and relationships are manual
  • Parameterization — sed and envsubst don’t scale
With 50 microservices, you might have 400+ YAML files. Changing a common pattern means editing every single one. Helm addresses all of these concerns by treating a set of manifests as a single versioned, parameterized package.

Helm Core Concepts

📦 Chart

A package of pre-configured Kubernetes resources — templates + default values + metadata

🚀 Release

A running instance of a chart in a cluster — each install creates a new release with its own history

📚 Repository

A server hosting packaged charts — similar to Docker Hub for images or npm for packages

Key relationship: A chart is installed into a cluster as a release. The same chart can be installed multiple times, each producing a distinct release.
Think of charts like Docker images and releases like running containers. A chart is the blueprint; a release is a deployed instance. You can have multiple releases of the same chart, each with different configuration values.

🏗️ Helm Architecture

Chart structure, values, and the rendering pipeline

Now let's look at how Helm charts are organized internally. Understanding the file layout and value hierarchy is essential for both consuming and authoring charts.

Chart Directory Structure

mychart/
├── Chart.yaml          # Chart metadata (name, version, dependencies)
├── Chart.lock          # Locked dependency versions
├── values.yaml         # Default configuration values
├── values.schema.json  # Optional JSON Schema for values
├── templates/          # Kubernetes manifest templates
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── _helpers.tpl    # Reusable named templates
│   ├── NOTES.txt       # Post-install usage instructions
│   └── tests/
│       └── test-connection.yaml
├── charts/             # Packaged dependency charts
└── crds/               # Custom Resource Definitions
Chart.yaml is the only required file — it defines the chart’s API version, name, and version.
The templates directory is where all Kubernetes manifests live as Go templates. The _helpers.tpl file contains reusable template snippets. NOTES.txt is rendered after install to show the user helpful information like how to access their application.

Values and Override Hierarchy

Helm merges values from multiple sources with a clear precedence order:

Override Precedence (lowest to highest)

  1. Parent chart’s values.yaml — defaults from dependency charts
  2. Chart’s values.yaml — the chart’s own defaults
  3. -f custom-values.yaml — user-supplied values files
  4. --set key=value — command-line overrides (highest priority)
# Override with a file and individual values
helm install my-app ./mychart \
  -f production-values.yaml \
  --set image.tag=v2.1.0 \
  --set replicas=3
The precedence order matters. Command-line --set flags always win. In practice, teams keep a base values.yaml in the chart and then layer environment-specific files on top. Multiple -f flags are supported and merge left to right.

📄 Templating

Go templates, functions, and control flow

Templating is the heart of Helm. Understanding Go templates will let you build flexible, reusable charts. Let's start with the fundamentals and build up to advanced patterns.

Go Template Basics

apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-app
  labels:
    app.kubernetes.io/name: {{ .Chart.Name }}
    app.kubernetes.io/instance: {{ .Release.Name }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels: { app: {{ .Release.Name }} }
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          ports:
            - containerPort: {{ .Values.service.port }}
Double curly braces {{ }} delimit actions. The dot (.) is the root context object.
Everything inside double curly braces is evaluated by the Go template engine. The dot-notation accesses the hierarchical data passed into the template. Values come from values.yaml, Release metadata comes from the Helm runtime, and Chart metadata comes from Chart.yaml.

Built-in Objects

.Values

Contents of values.yaml merged with overrides

.Release

.Name, .Namespace, .IsInstall, .IsUpgrade, .Revision

.Chart

Contents of Chart.yaml — .Name, .Version, .AppVersion

.Capabilities

.KubeVersion, .APIVersions — detect cluster features at render time

Tip: Use .Capabilities.APIVersions.Has to conditionally include resources based on what the target cluster supports.
These built-in objects are always available in every template. .Values is the most commonly used, but .Release is critical for naming and namespace scoping. .Capabilities is powerful for writing charts that work across different Kubernetes versions.

Template Functions

Helm includes 60+ functions from Go templates and the Sprig library:

# default - provide fallback values
image: {{ .Values.image.tag | default "latest" }}

# quote - wrap in double quotes
env:
  - name: APP_ENV
    value: {{ .Values.env | quote }}

# required - fail if value is missing
namespace: {{ required "namespace is required" .Values.namespace }}

# toYaml + nindent - output structured YAML
resources:
{{ toYaml .Values.resources | nindent 10 }}

# upper, lower, trim, replace
name: {{ .Values.name | lower | trunc 63 | trimSuffix "-" }}
Pipeline syntax (|) chains functions left to right, passing output as the last argument to the next function.
The pipe operator works like Unix pipes. Functions are applied in sequence. The toYaml plus nindent combination is one you'll use constantly - it converts a values block to YAML and indents it correctly. The required function is excellent for catching missing configuration early.

Control Structures

# if/else - conditional blocks
{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ .Release.Name }}-ingress
{{- end }}

# range - iterate over lists and maps
{{- range .Values.extraEnvVars }}
  - name: {{ .name }}
    value: {{ .value | quote }}
{{- end }}

# with - change scope
{{- with .Values.nodeSelector }}
nodeSelector:
  {{- toYaml . | nindent 2 }}
{{- end }}
Tip: Use {{- and -}} to trim whitespace before/after template actions. Incorrect whitespace is the #1 source of templating bugs.
The dash inside the curly braces trims whitespace on that side. Without it, you get blank lines in your rendered output that can cause YAML parsing issues. The 'with' block changes the dot context, so inside a 'with .Values.nodeSelector' block, dot refers to nodeSelector itself.

Named Templates & _helpers.tpl

Define in _helpers.tpl

{{- define "mychart.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion }}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version }}
{{- end }}

Use in templates

apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-app
  labels:
    {{- include "mychart.labels" . | nindent 4 }}
Use include over template — include captures output as a string so you can pipe it through functions like nindent.
The _helpers.tpl file is conventionally where all named templates live. The underscore prefix tells Helm not to render it as a standalone manifest. Always use 'include' instead of 'template' because include returns a string you can pipe through nindent or other functions.

🔧 Helm Operations

Install, upgrade, rollback, and release management

Now that we understand chart structure and templating, let's look at the operational commands you'll use daily to manage releases in your clusters.

Release Lifecycle Commands

# Install a chart as a new release
helm install my-release ./mychart -n production

# Upgrade an existing release (creates new revision)
helm upgrade my-release ./mychart -f prod-values.yaml

# Install or upgrade in one command (idempotent)
helm upgrade --install my-release ./mychart

# Rollback to a previous revision
helm rollback my-release 2

# Uninstall a release (removes all resources)
helm uninstall my-release -n production

# Dry run - render templates without applying
helm install my-release ./mychart --dry-run --debug
Best practice: Always use helm upgrade --install in CI/CD pipelines for idempotent deployments.
Each upgrade creates a new revision number. Rollback reverts to a previous revision's configuration and templates. The --dry-run flag is invaluable for testing changes before applying them. In production pipelines, upgrade --install is the standard pattern.

Repository Operations

# Add a chart repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx

# Update local cache of repository indexes
helm repo update

# Search for charts in added repos
helm search repo nginx
helm search repo bitnami/postgresql --versions

# Search Artifact Hub (public registry)
helm search hub prometheus

# Download a chart without installing
helm pull bitnami/postgresql --version 12.5.0
helm pull bitnami/postgresql --untar  # extract to directory
helm pull is useful for vendoring charts into your repository or inspecting their templates before deployment.
Repos are just HTTP servers hosting an index.yaml and packaged chart tarballs. Artifact Hub at artifacthub.io aggregates charts from many repositories. Always pin chart versions in production to avoid unexpected upgrades.

Debugging: helm template

# Render templates locally
helm template my-release ./mychart

# Render with overrides
helm template my-release ./mychart \
  -f staging-values.yaml \
  --set image.tag=v2.0

# Show only specific template
helm template my-release ./mychart \
  -s templates/deployment.yaml

# Validate against K8s API
helm template my-release ./mychart \
  --validate
Best friend during development: Renders everything locally without touching the cluster. Use it in CI to catch errors early.
helm template is your best friend during development. It renders everything locally without touching the cluster. Use it in CI to catch errors early.

Validation: helm lint

# Check chart for issues
helm lint ./mychart

# Lint with specific values
helm lint ./mychart \
  -f production-values.yaml

# Strict mode (warnings = errors)
helm lint ./mychart --strict

# Example output:
# ==> Linting ./mychart
# [INFO] Chart.yaml: icon is recommended
# [WARNING] templates/: directory not found
# [ERROR] values.yaml: required field missing
Use in CI pipelines: helm lint checks for structural issues and best practice violations. Together with helm template, these form the first line of defense in a chart testing pipeline.
helm lint checks for structural issues and best practice violations. Together with helm template, these form the first line of defense in a chart testing pipeline.

Release Management & History

# List all releases across namespaces
helm list --all-namespaces

# Show release history (all revisions)
helm history my-release
# REVISION  STATUS      CHART           APP VERSION  DESCRIPTION
# 1         superseded  mychart-1.0.0   2.0.0        Install complete
# 2         superseded  mychart-1.1.0   2.1.0        Upgrade complete
# 3         deployed    mychart-1.1.0   2.1.0        Rollback to 2

# Get release details
helm get values my-release        # user-supplied values
helm get values my-release --all  # all computed values
helm get manifest my-release      # rendered Kubernetes manifests
helm get notes my-release         # NOTES.txt output
Release state is stored as Secrets in the release namespace. Each revision creates a new Secret named sh.helm.release.v1.<name>.v<revision>.
Helm stores all release state as base64-encoded, gzip-compressed Secrets in the cluster. This means release data is namespace-scoped and subject to RBAC. The helm get commands are essential for debugging - 'get manifest' shows you exactly what was applied to the cluster.

🚀 Advanced Helm

Hooks, dependencies, library charts, and testing

Let's move into advanced Helm features that you'll encounter in production charts and complex deployment scenarios.

Helm Hooks

Hooks run at specific points in the release lifecycle:

apiVersion: batch/v1
kind: Job
metadata:
  name: {{ .Release.Name }}-db-migrate
  annotations:
    "helm.sh/hook": pre-upgrade
    "helm.sh/hook-weight": "1"
    "helm.sh/hook-delete-policy": hook-succeeded
spec:
  template:
    spec:
      containers:
        - name: migrate
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          command: ["./migrate", "--apply"]
      restartPolicy: Never
Common use cases: Database migrations before upgrade, cache warming after install, cleanup jobs before delete.
Hooks are regular Kubernetes resources with special annotations. Hook-weight controls execution order when multiple hooks exist. Delete policies control whether hook resources are cleaned up automatically.

Helm Hook Types

HookWhen It Runs
pre-installBefore resources are created
post-installAfter all resources are created
pre-upgrade / post-upgradeBefore/after upgrade
pre-rollback / post-rollbackBefore/after rollback
pre-delete / post-deleteBefore/after uninstall
hook-weight controls execution order when multiple hooks exist. hook-delete-policy controls whether hook resources are cleaned up automatically.
Hook-weight is an integer string that determines ordering. Lower weights run first. The delete policy options are: hook-succeeded (delete after success), hook-failed (delete after failure), and before-hook-creation (delete any existing hook resource before creating a new one).

Declaring Dependencies

Chart.yaml

apiVersion: v2
name: my-application
version: 1.0.0
dependencies:
  - name: postgresql
    version: "12.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled
  - name: redis
    version: "17.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled
    alias: cache
condition and tags let users enable/disable dependencies. Use alias to install the same chart multiple times with different configs.
Dependencies are declared in Chart.yaml. The Chart.lock file pins exact versions, similar to package-lock.json. The alias field is useful when you need two instances of the same chart, like two Redis instances.

Managing Dependencies

# Download dependencies
helm dependency update ./mychart

# List dependencies
helm dependency list ./mychart

Override subchart values in values.yaml:

postgresql:
  enabled: true
  auth:
    username: myapp
    database: myapp_db
cache:  # uses alias
  enabled: true
  architecture: standalone
Subchart values are scoped under the dependency name (or alias) in the parent's values.yaml.
Dependencies are downloaded into the charts/ directory as .tgz files. Subchart values are scoped under the dependency name in the parent's values.yaml.

Creating Library Charts

What are Library Charts?

Charts that provide only named templates — no rendered manifests. They define common patterns shared across multiple application charts.

# Chart.yaml — type: library
apiVersion: v2
name: common-lib
version: 1.0.0
type: library  # no rendered manifests

# templates/_deployment.tpl
{{- define "common.deployment" -}}
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}
  labels:
    {{- include "common.labels" . | nindent 4 }}
{{- end -}}
Library charts are powerful for platform teams. You define standard deployment patterns once, and all application teams consume them. The type: library field in Chart.yaml is what distinguishes it from a regular application chart.

Consuming Library Charts

# Chart.yaml dependency
dependencies:
  - name: common-lib
    version: "1.x"
    repository: "@internal"

# templates/deployment.yaml
{{- include "common.deployment" . }}
Use case: Enforce org-wide standards for labels, security contexts, and resource structures across hundreds of microservices.
Example: Bitnami's common chart is a well-known library chart that provides reusable templates for labels, names, and common Kubernetes patterns.
All application teams consume the library chart as a dependency. This ensures consistency across hundreds of microservices. Bitnami's common chart is a well-known example.

Helm Testing: Test Definition

# templates/tests/test-connection.yaml
apiVersion: v1
kind: Pod
metadata:
  name: {{ .Release.Name }}-test-connection
  annotations:
    "helm.sh/hook": test
spec:
  containers:
    - name: wget
      image: busybox
      command: ['wget']
      args: ['{{ .Release.Name }}-svc:{{ .Values.service.port }}']
  restartPolicy: Never
Helm tests are Pods with the helm.sh/hook: test annotation. They verify your release is functioning correctly after install or upgrade.
Helm tests are pods with the helm.sh/hook: test annotation. Tests can check connectivity, validate data, or run integration test suites.

Helm Testing: Running Tests

# Run tests after install/upgrade
helm test my-release

# Run tests with logs
helm test my-release --logs

# Example output:
# NAME: my-release
# LAST DEPLOYED: Mon Mar 13 10:00:00 2026
# STATUS: deployed
# TEST SUITE:     my-release-test-connection
# Last Started:   Mon Mar 13 10:00:05 2026

# Phase:          Succeeded
CI/CD integration: Include helm test in your pipeline after every deployment to catch issues before they reach users.
Include helm test in your CI/CD pipeline after every deployment. Tests run as pods in the cluster, so they can verify actual connectivity and functionality.

OCI Registry Support

Since Helm 3.8+, charts can be stored in OCI-compliant container registries alongside your container images. 
# Login to an OCI registry
helm registry login registry.example.com

# Package and push a chart
helm package ./mychart
helm push mychart-1.0.0.tgz oci://registry.example.com/charts

# Pull from OCI registry
helm pull oci://registry.example.com/charts/mychart --version 1.0.0

# Install directly from OCI
helm install my-release oci://registry.example.com/charts/mychart \
  --version 1.0.0

# Show chart info from OCI
helm show chart oci://registry.example.com/charts/mychart \
  --version 1.0.0
Advantages: Unified artifact storage, immutable chart versions, standard authentication, and no separate chart repository server needed.
OCI support is a major shift. Instead of running a separate chart museum or using GitHub Pages, you store charts in the same ECR, ACR, or GCR registries you already use for images. AWS ECR, Azure ACR, Google Artifact Registry, and GitHub Container Registry all support Helm OCI. This simplifies infrastructure and leverages existing auth/RBAC.

Helm vs Kustomize

FeatureHelmKustomize
ApproachTemplating (generate YAML)Patching (overlay on base YAML)
ParameterizationGo templates + valuesStrategic merge patches
PackagingCharts with versioningNo packaging concept
Release trackingBuilt-in (Secrets)None (relies on GitOps)
RollbackBuilt-in (helm rollback)Via Git revert
Learning curveSteeper (Go templates)Moderate (YAML patches)
Built into kubectlNo (separate binary)Yes (kubectl -k)
They complement each other. Many teams use Helm for packaging third-party charts and Kustomize for environment-specific overlays on top of helm template output.
This is not an either/or decision. A common production pattern: render Helm charts with helm template, then apply Kustomize overlays for environment-specific patches. ArgoCD and Flux both support this combined workflow natively. Choose based on your use case - Helm for distributable packages, Kustomize for environment configuration.

✅ Best Practices

Chart development guidelines and production patterns

Let's wrap up the teaching content with best practices for chart development and production usage that the Helm community has established.

Chart Development Guidelines

Naming & Structure

  • Follow Helm best practices guide
  • Use helm create to scaffold new charts
  • Prefix template names with chart name
  • Keep templates focused — one resource per file

Values & Defaults

  • Provide sensible defaults for all values
  • Use flat keys where possible (avoid deep nesting)
  • Document every value with comments
  • Never put secrets in values.yaml
Version your charts independently from your application. Bump the chart version for template changes, and appVersion for the application itself.
The helm create command generates a well-structured starting point. Template names should be prefixed with the chart name to avoid collisions when used as subcharts. Version separation between the chart and the app is critical - a chart template fix shouldn't require an app version bump.

Values Schema: JSON Schema

values.schema.json

{
  "$schema": "https://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["image", "service"],
  "properties": {
    "replicaCount": {
      "type": "integer", "minimum": 1, "maximum": 20
    },
    "image": {
      "type": "object",
      "required": ["repository"],
      "properties": {
        "repository": { "type": "string" },
        "tag": { "type": "string" },
        "pullPolicy": {
          "enum": ["Always", "IfNotPresent", "Never"]
        }
      }
    }
  }
}
Tip: Place values.schema.json at the chart root alongside values.yaml. Helm validates all supplied values against this schema before rendering.
The values.schema.json file is placed at the chart root alongside values.yaml. Helm validates all supplied values against this schema before rendering templates.

Values Schema: Validation Benefits

  • Type checking — catch string vs number errors
  • Required fields — fail fast on missing values
  • Range validation — enforce min/max constraints
  • Enum validation — restrict to allowed values
  • Pattern matching — validate format (e.g., semver)
Validated automatically during helm install, helm upgrade, helm lint, and helm template. Catches configuration errors before they become runtime failures.
This catches configuration errors before they become runtime failures. It's especially valuable for charts consumed by other teams who may not know all the valid options.
