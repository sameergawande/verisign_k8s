🔍 Why Helm?

The problem with raw YAML at scale

Why Helm exists and what problems it solves.

The YAML Sprawl Problem

A typical microservice requires 5–10 YAML manifests — Deployment, Service, ConfigMap, Secret, Ingress, HPA, PDB, ServiceAccount, NetworkPolicy...
  • Duplication — same patterns repeated across dozens of services
  • Environment drift — dev, staging, and prod configs diverge over time
  • No versioning — hard to track what’s deployed or roll back atomically
  • No dependency management — ordering and relationships are manual
  • Parameterization — sed and envsubst don’t scale
Helm treats a set of manifests as a single versioned, parameterized package.

Helm Core Concepts

📦 Chart

A package of pre-configured Kubernetes resources — templates + default values + metadata

🚀 Release

A running instance of a chart in a cluster — each install creates a new release with its own history

📚 Repository

A server hosting packaged charts — similar to Docker Hub for images or npm for packages

Key relationship: A chart is installed into a cluster as a release. The same chart can be installed multiple times, each producing a distinct release.
Chart = blueprint, release = deployed instance. Multiple releases of the same chart can coexist with different values.

🏗️ Helm Architecture

Chart structure, values, and the rendering pipeline

Chart file layout and value hierarchy.

Chart Directory Structure

mychart/
├── Chart.yaml          # Chart metadata (name, version, dependencies)
├── Chart.lock          # Locked dependency versions
├── values.yaml         # Default configuration values
├── values.schema.json  # Optional JSON Schema for values
├── templates/          # Kubernetes manifest templates
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── _helpers.tpl    # Reusable named templates
│   ├── NOTES.txt       # Post-install usage instructions
│   └── tests/
│       └── test-connection.yaml
├── charts/             # Packaged dependency charts
└── crds/               # Custom Resource Definitions
Chart.yaml is the only required file — it defines the chart’s API version, name, and version.
Templates/ holds Go-templated manifests. _helpers.tpl has reusable snippets. NOTES.txt renders post-install instructions.

Values and Override Hierarchy

Helm merges values from multiple sources in precedence order:

Override Precedence (lowest to highest)

  1. Parent chart’s values.yaml — defaults from dependency charts
  2. Chart’s values.yaml — the chart’s own defaults
  3. -f custom-values.yaml — user-supplied values files
  4. --set key=value — command-line overrides (highest priority)
# Override with a file and individual values
helm install my-app ./mychart \
  -f env-values.yaml \
  --set image.tag=v2.1.0
--set flags always win. Teams layer environment-specific files on top of base values.yaml. Multiple -f flags merge left to right.

📄 Templating

Go templates, functions, and control flow

Go template fundamentals through advanced patterns.

Go Template Basics

metadata:
  name: {{ .Release.Name }}-app
  labels:
    app.kubernetes.io/name: {{ .Chart.Name }}
spec:
  replicas: {{ .Values.replicaCount }}
  # ...
  containers:
    - image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
Double curly braces {{ }} delimit actions. The dot (.) is the root context object.
Dot-notation accesses hierarchical data: .Values from values.yaml, .Release from Helm runtime, .Chart from Chart.yaml.

Built-in Objects

.Values

Contents of values.yaml merged with overrides

.Release

.Name, .Namespace, .IsInstall, .IsUpgrade, .Revision

.Chart

Contents of Chart.yaml — .Name, .Version, .AppVersion

.Capabilities

.KubeVersion, .APIVersions — detect cluster features at render time

Tip: Use .Capabilities.APIVersions.Has to conditionally include resources based on what the target cluster supports.
.Values is most common. .Release is critical for naming. .Capabilities helps write charts that work across K8s versions.

Template Functions

Helm includes 60+ functions from Go templates and the Sprig library:

# default - provide fallback values
image: {{ .Values.image.tag | default "latest" }}

# quote - wrap in double quotes
env:
  - name: APP_ENV
    value: {{ .Values.env | quote }}

# required - fail if value is missing
namespace: {{ required "namespace is required" .Values.namespace }}

# toYaml + nindent - output structured YAML
resources:
{{ toYaml .Values.resources | nindent 10 }}

# upper, lower, trim, replace
name: {{ .Values.name | lower | trunc 63 | trimSuffix "-" }}
Pipeline syntax (|) chains functions left to right, passing output as the last argument to the next function.
Pipe operator works like Unix pipes. toYaml + nindent is the most common combo. required catches missing config early.

Control Structures

# if/else - conditional blocks
{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ .Release.Name }}-ingress
{{- end }}

# range - iterate over lists and maps
{{- range .Values.extraEnvVars }}
  - name: {{ .name }}
    value: {{ .value | quote }}
{{- end }}

# with - change scope
{{- with .Values.nodeSelector }}
nodeSelector:
  {{- toYaml . | nindent 2 }}
{{- end }}
Use {{- and -}} to trim whitespace. The 'with' block changes dot context to the scoped value.

Named Templates & _helpers.tpl

Define in _helpers.tpl

{{- define "mychart.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion }}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version }}
{{- end }}

Use in templates

apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-app
  labels:
    {{- include "mychart.labels" . | nindent 4 }}
Use include over template — include captures output as a string so you can pipe it through functions like nindent.
Underscore prefix tells Helm not to render it as a manifest. Use 'include' over 'template' because it returns a pipeable string.

🔧 Helm Operations

Install, upgrade, rollback, and release management

Day-to-day commands for managing releases.

Release Lifecycle Commands

# Install a chart as a new release
helm install my-release ./mychart -n myapp

# Upgrade (creates new revision)
helm upgrade my-release ./mychart -f env-values.yaml

# Install or upgrade in one command (idempotent)
helm upgrade --install my-release ./mychart

# Rollback to a previous revision
helm rollback my-release 2

# Uninstall a release
helm uninstall my-release -n myapp
Best practice: Always use helm upgrade --install in CI/CD pipelines for idempotent deployments.
Each upgrade creates a new revision. Rollback reverts to a previous revision's config and templates. upgrade --install is the standard CI/CD pattern.

Repository Operations

# Add a chart repository and update cache
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Search for charts
helm search repo nginx
helm search repo bitnami/postgresql --versions

# Download a chart without installing
helm pull bitnami/postgresql --version 12.5.0 --untar
helm pull is useful for vendoring charts into your repository or inspecting their templates before deployment.
Repos are HTTP servers hosting an index.yaml and chart tarballs. Always pin chart versions to avoid unexpected upgrades.

Debugging: helm template

# Render templates locally
helm template my-release ./mychart

# Render with overrides
helm template my-release ./mychart \
  -f staging-values.yaml --set image.tag=v2.0

# Show only a specific template
helm template my-release ./mychart -s templates/deployment.yaml
Key development tool: Renders locally without touching the cluster. Use in CI to catch errors early.
Renders locally without cluster access. Essential for CI pipelines.

Validation: helm lint

# Check chart for issues
helm lint ./mychart

# Lint with specific values; strict mode treats warnings as errors
helm lint ./mychart -f env-values.yaml --strict
Use in CI: helm lint + helm template form the first line of defense in chart validation.
lint checks structure and best practices. Pair with helm template for full validation.

Release Management & History

# List releases; show history
helm list --all-namespaces
helm history my-release

# Inspect a release
helm get values my-release        # user-supplied values
helm get manifest my-release      # rendered manifests
Release state is stored as Secrets in the release namespace. Each revision creates a new Secret named sh.helm.release.v1.<name>.v<revision>.
Release state is stored as Secrets in the cluster, namespace-scoped and subject to RBAC.

🚀 Advanced Helm

Hooks, dependencies, library charts, and testing

Advanced Helm features for real-world charts.

Helm Hooks

Hooks run at specific points in the release lifecycle:

metadata:
  name: {{ .Release.Name }}-db-migrate
  annotations:
    "helm.sh/hook": pre-upgrade
    "helm.sh/hook-weight": "1"
    "helm.sh/hook-delete-policy": hook-succeeded
# ... Job spec runs migration before upgrade
Common use cases: Database migrations before upgrade, cache warming after install, cleanup jobs before delete.
Hooks are regular K8s resources with special annotations. Hook-weight controls order; delete policies handle cleanup.

Helm Hook Types

HookWhen It Runs
pre-installBefore resources are created
post-installAfter all resources are created
pre-upgrade / post-upgradeBefore/after upgrade
pre-rollback / post-rollbackBefore/after rollback
pre-delete / post-deleteBefore/after uninstall
hook-weight controls execution order when multiple hooks exist. hook-delete-policy controls whether hook resources are cleaned up automatically.
Lower hook-weight runs first. Delete policy options: hook-succeeded, hook-failed, before-hook-creation.

Declaring Dependencies

Chart.yaml

apiVersion: v2
name: my-application
version: 1.0.0
dependencies:
  - name: postgresql
    version: "12.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled
  - name: redis
    version: "17.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled
    alias: cache
condition and tags let users enable/disable dependencies. Use alias to install the same chart multiple times with different configs.
Chart.lock pins exact versions. Alias lets you install the same chart multiple times with different configs.

Managing Dependencies

# Download dependencies
helm dependency update ./mychart

# List dependencies
helm dependency list ./mychart

Override subchart values in values.yaml:

postgresql:
  enabled: true
  auth:
    username: myapp
    database: myapp_db
cache:  # uses alias
  enabled: true
  architecture: standalone
Subchart values are scoped under the dependency name (or alias) in the parent's values.yaml.
Dependencies download into charts/ as .tgz files. Subchart values are scoped under the dependency name.

Creating Library Charts

What are Library Charts?

Charts that provide only named templates — no rendered manifests. They define common patterns shared across multiple application charts.

# Chart.yaml — type: library (no rendered manifests)
apiVersion: v2
name: common-lib
type: library
# ...

# templates/_deployment.tpl — reusable named template
{{- define "common.deployment" -}}
# ... standard Deployment with shared labels, security context, etc.
{{- end -}}
type: library distinguishes it from application charts. Define patterns once, consume everywhere.

Consuming Library Charts

# Chart.yaml dependency
dependencies:
  - name: common-lib
    version: "1.x"
    repository: "@internal"

# templates/deployment.yaml
{{- include "common.deployment" . }}
Use case: Enforce org-wide standards for labels, security contexts, and resource structures across services.
Example: Bitnami's common chart is a well-known library chart that provides reusable templates for labels, names, and common Kubernetes patterns.
Bitnami's common chart is a well-known library chart example.

Helm Testing: Test Definition

# templates/tests/test-connection.yaml
metadata:
  name: {{ .Release.Name }}-test-connection
  annotations:
    "helm.sh/hook": test
# ... Pod spec that verifies connectivity
Helm tests are Pods with the helm.sh/hook: test annotation. They verify your release is functioning correctly after install or upgrade.
Tests are Pods with helm.sh/hook: test. They verify connectivity, data, or run integration checks.

Helm Testing: Running Tests

# Run tests after install/upgrade
helm test my-release
helm test my-release --logs  # include pod logs in output
CI/CD integration: Include helm test in your pipeline after every deployment to catch issues before they reach users.
Tests run as pods in the cluster, verifying actual connectivity and functionality.

OCI Registry Support

Since Helm 3.8+, charts can be stored in OCI-compliant container registries alongside your container images. 
# Package and push to OCI registry
helm package ./mychart
helm push mychart-1.0.0.tgz oci://registry.example.com/charts

# Install directly from OCI
helm install my-release oci://registry.example.com/charts/mychart \
  --version 1.0.0
Advantages: Unified artifact storage, immutable versions, standard auth, no separate chart server needed.
Store charts in the same ECR/ACR/GCR registries used for images. Simplifies infrastructure and leverages existing auth.

Helm vs Kustomize

FeatureHelmKustomize
ApproachTemplating (generate YAML)Patching (overlay on base YAML)
ParameterizationGo templates + valuesStrategic merge patches
PackagingCharts with versioningNo packaging concept
Release trackingBuilt-in (Secrets)None (relies on GitOps)
RollbackBuilt-in (helm rollback)Via Git revert
Learning curveSteeper (Go templates)Moderate (YAML patches)
Built into kubectlNo (separate binary)Yes (kubectl -k)
They complement each other. Many teams use Helm for packaging third-party charts and Kustomize for environment-specific overlays on top of helm template output.
Not either/or. A common pattern: render with helm template, then apply Kustomize overlays. ArgoCD and Flux support this natively.

✅ Best Practices

Chart development guidelines and recommended patterns

Best practices for chart development established by the Helm community.

Chart Development Guidelines

Naming & Structure

  • Follow Helm best practices guide
  • Use helm create to scaffold new charts
  • Prefix template names with chart name
  • Keep templates focused — one resource per file

Values & Defaults

  • Provide sensible defaults for all values
  • Use flat keys where possible (avoid deep nesting)
  • Document every value with comments
  • Never put secrets in values.yaml
Version your charts independently from your application. Bump the chart version for template changes, and appVersion for the application itself.
Prefix template names with chart name to avoid subchart collisions. Separate chart version from app version.

Values Schema: JSON Schema

values.schema.json

{
  "required": ["image", "service"],
  "properties": {
    "replicaCount": { "type": "integer", "minimum": 1 },
    "image": {
      "required": ["repository"],
      "properties": {
        "repository": { "type": "string" },
        "pullPolicy": { "enum": ["Always", "IfNotPresent", "Never"] }
      }
    }
  }
}
Place values.schema.json at chart root. Helm validates values against this schema before rendering.

Values Schema: Validation Benefits

  • Type checking — catch string vs number errors
  • Required fields — fail fast on missing values
  • Range validation — enforce min/max constraints
  • Enum validation — restrict to allowed values
  • Pattern matching — validate format (e.g., semver)
Validated automatically during helm install, helm upgrade, helm lint, and helm template.
Catches config errors before runtime. Valuable for charts consumed by other teams.
