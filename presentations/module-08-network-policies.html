🌐 Kubernetes Networking Fundamentals

Understanding the Flat Network Model

Before we write a single policy, we need to understand the networking model that Kubernetes imposes. This is what makes network policies both necessary and powerful.

🌐 The Kubernetes Flat Network

Core Networking Requirement

Every pod gets its own IP address, and every pod can reach every other pod without NAT — across nodes, across namespaces.

  • Pod-to-Pod: Direct communication via cluster-internal IPs
  • Cross-Namespace: No isolation boundary by default
  • Cross-Node: CNI plugin handles overlay or native routing
  • DNS Resolution: Every service discoverable via CoreDNS
Kubernetes mandates a flat network where every pod can talk to every other pod. This simplifies application networking but creates a massive attack surface. If an attacker compromises one pod, they can reach every other pod in the cluster by default.

⚠️ The Security Problem: Default Allow

Default Behavior: Every pod can communicate with every other pod on every port. There are no firewalls between workloads.

🔴 Without Network Policies

  • Compromised frontend reaches database directly
  • Dev namespace can access production
  • Lateral movement and data exfiltration are trivial

🟢 With Network Policies

  • Frontend only reaches its API server
  • Namespaces isolated; egress restricted
  • Blast radius is contained
This is the core motivation. The flat network is great for development convenience but terrible for security. In a multi-tenant cluster like Verisign's, you absolutely need network policies. Without them, a vulnerability in one microservice compromises everything.

📜 Network Policy Basics

Anatomy, Selectors, and Rules

Now let's look at the actual NetworkPolicy resource. We'll break down every field so you understand exactly how to construct policies.

📜 NetworkPolicy Anatomy

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: api-server-policy, namespace: production }
spec:
  podSelector:                  # Which pods this policy applies to
    matchLabels: { app: api-server }
  policyTypes: [Ingress, Egress]
  ingress:                      # Allowed inbound traffic
    - from:
        - podSelector: { matchLabels: { app: frontend } }
      ports: [{ protocol: TCP, port: 8080 }]
  egress:                       # Allowed outbound traffic
    - to:
        - podSelector: { matchLabels: { app: database } }
      ports: [{ protocol: TCP, port: 5432 }]
This is the complete structure. The podSelector targets which pods are governed. policyTypes declares whether ingress, egress, or both are controlled. Once a policy selects a pod, all traffic not explicitly allowed by a rule is denied — this is the key to understand. Policies are additive, meaning multiple policies combine their allow rules.

🏷️ Pod Selectors and Label Matching

matchLabels (AND logic)

podSelector:
  matchLabels:
    app: api-server
    tier: backend
# Must match ALL labels

matchExpressions (flexible)

podSelector:
  matchExpressions:
    - key: tier
      operator: In
      values: [backend, middleware]
    - key: env
      operator: NotIn
      values: [test]
Empty podSelector {} — Selects all pods in the namespace. This is how you create default deny policies.
Label selectors are the targeting mechanism. matchLabels uses AND logic — all labels must match. matchExpressions gives you In, NotIn, Exists, and DoesNotExist operators for more flexible selection. The empty selector is critical — it selects every pod in the namespace.

📥 Ingress Rules: Controlling Inbound Traffic

ingress:
  # Rule 1: Allow from pods with label app=frontend
  - from:
      - podSelector: { matchLabels: { app: frontend } }
  # Rule 2: Allow from monitoring namespace
  - from:
      - namespaceSelector: { matchLabels: { purpose: monitoring } }
  # Rule 3: Allow from corporate VPN CIDR
  - from:
      - ipBlock:
          cidr: 10.0.0.0/8
          except: [10.0.5.0/24]
OR vs AND: Separate list items under from: are OR’d. Multiple selectors within a single list item are AND’d.
Three source types for ingress: podSelector for pods in the same namespace, namespaceSelector for pods in other namespaces, and ipBlock for CIDR ranges. The OR vs AND distinction is critical and a common source of misconfiguration. Separate dash items are OR — any match allows traffic. Selectors in the same item are AND — all must match.

📤 Egress Rules: Controlling Outbound Traffic

egress:
  # Allow DNS resolution (critical!)
  - to: [{ namespaceSelector: {} }]
    ports: [{ protocol: UDP, port: 53 }, { protocol: TCP, port: 53 }]
  # Allow traffic to database pods
  - to:
      - podSelector: { matchLabels: { app: postgres } }
    ports: [{ protocol: TCP, port: 5432 }]
  # Allow external API calls
  - to:
      - ipBlock: { cidr: 203.0.113.0/24 }
    ports: [{ protocol: TCP, port: 443 }]
Never forget DNS! If you restrict egress, you must explicitly allow UDP/TCP port 53 or your pods cannot resolve service names.
Egress rules follow the same structure as ingress but control outbound traffic. The number one mistake teams make is forgetting to allow DNS when implementing egress policies. Your application will fail silently because it can't resolve hostnames. Always include DNS as the first egress rule.

🔒 Port-Level Controls

Specific Ports

ports:
  - protocol: TCP
    port: 8080
  - protocol: TCP
    port: 8443
  - protocol: TCP
    port: 9090

Port Ranges (K8s 1.25+)

ports:
  - protocol: TCP
    port: 8000
    endPort: 9000

Covers all ports from 8000 to 9000 inclusive.

Port controls add the final layer of precision. You can specify individual ports or port ranges since Kubernetes 1.25.

🔒 Port-Level Controls — Egress Example

Named Ports

ports:
  - protocol: TCP
    port: http    # Matches the named port in the target pod spec

Named ports reference the containerPort name, decoupling policy from hard-coded numbers.

Named ports are a best practice because they decouple the policy from the actual port number — if the application changes its port, only the pod spec needs updating.

📑 Common Policy Patterns

Production-Ready Templates

Now that we understand the syntax, let's look at the most common patterns you'll implement. These are battle-tested templates that cover the majority of use cases.

🚫 Default Deny All Ingress

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}       # Selects ALL pods in namespace
  policyTypes:
    - Ingress
  # No ingress rules = deny all inbound traffic
Best Practice: Apply default-deny-ingress to every namespace first, then layer on allow rules. This ensures no pod is accidentally exposed.
This is the foundational policy. Empty podSelector means all pods. Declaring Ingress in policyTypes with no ingress rules means all inbound traffic is denied. Apply this first to every namespace, then write targeted allow rules. This is the cornerstone of zero-trust networking in Kubernetes.

🚫 Default Deny All Egress

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-egress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
    - Egress
  # No egress rules = deny all outbound traffic
Caution: Denying all egress breaks DNS resolution. Always pair with a DNS allow policy (next slide).
Egress deny is equally important but requires more care. You must pair it with a DNS allow policy or nothing in the namespace will work.

🚫 Companion: Allow DNS with Egress Deny

# Always deploy alongside default-deny-egress
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: allow-dns, namespace: production }
spec:
  podSelector: {}
  policyTypes: [Egress]
  egress:
    - to: [{ namespaceSelector: {} }]
      ports:
        - { protocol: UDP, port: 53 }
        - { protocol: TCP, port: 53 }
Best Practice: Some teams scope DNS to kube-system only, but allowing any namespace is safer for custom DNS setups.
This DNS allow policy must accompany a default-deny-egress policy. Without it, pods cannot resolve service names and will fail silently. Some teams scope the DNS rule to the kube-system namespace where CoreDNS runs, but allowing any namespace is safer for custom DNS setups.

📁 Allow Specific Namespace Communication

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: allow-from-monitoring, namespace: production }
spec:
  podSelector: {}
  policyTypes: [Ingress]
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: monitoring
      ports:
        - { protocol: TCP, port: 9090 }   # Prometheus metrics
        - { protocol: TCP, port: 8080 }   # Health checks
Tip: Use the built-in label kubernetes.io/metadata.name (added automatically since K8s 1.21) to select namespaces by name without custom labels. This pattern allows monitoring tools like Prometheus to scrape metrics from production pods. The namespace selector targets the monitoring namespace specifically.

🔁 Allow Specific Pod-to-Pod Communication

Three-Tier Application Pattern

Frontend → API Server → Database — each tier only talks to its neighbors.

# Allow API server to receive from frontend only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: api-allow-frontend, namespace: production }
spec:
  podSelector: { matchLabels: { tier: api } }
  policyTypes: [Ingress]
  ingress:
    - from:
        - podSelector: { matchLabels: { tier: frontend } }
      ports: [{ protocol: TCP, port: 8080 }]
This is the classic three-tier pattern. The frontend can reach the API server on port 8080. The API server can reach the database on port 5432. The frontend cannot reach the database directly. This limits lateral movement — if the frontend is compromised, the attacker cannot jump straight to the database.

🗃️ Database Isolation Pattern

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: database-isolation, namespace: production }
spec:
  podSelector: { matchLabels: { app: postgres, tier: database } }
  policyTypes: [Ingress, Egress]
  ingress:
    - from:
        - podSelector: { matchLabels: { tier: api } }
      ports: [{ protocol: TCP, port: 5432 }]
  egress:
    # DNS only — database should not initiate connections
    - to: [{ namespaceSelector: {} }]
      ports: [{ protocol: UDP, port: 53 }, { protocol: TCP, port: 53 }]
Tip: Databases should have both ingress AND egress restrictions. A compromised database that can make outbound connections enables data exfiltration. Database isolation is one of the most important patterns. The database only accepts connections from the API tier on the PostgreSQL port. Egress is locked down to DNS only — the database has no reason to initiate outbound connections.

🔌 CNI & Network Policy Providers

Implementation Behind the API

NetworkPolicy is just an API resource. The actual enforcement depends on your CNI plugin. Let's compare the major providers and their capabilities beyond the standard API.

Not all CNI plugins are equal. The standard Kubernetes NetworkPolicy API covers L3/L4 rules. Calico and Cilium both extend this significantly with their own CRDs. AWS VPC CNI added native network policy support in EKS 1.25, which is relevant for Verisign's platform.

🐅 Calico Network Policies

apiVersion: projectcalico.org/v3
kind: GlobalNetworkPolicy
metadata: { name: deny-external-egress }
spec:
  selector: env == 'production'
  types: [Egress]
  egress:
    - action: Allow          # Cluster-internal traffic
      destination: { nets: [10.0.0.0/8] }
    - action: Allow          # DNS
      protocol: UDP
      destination: { ports: [53] }
    - action: Deny           # Everything else
Tip: GlobalNetworkPolicy applies across all namespaces without per-namespace duplication. Calico adds explicit Allow/Deny/Log actions with ordered rule evaluation. Calico extends the standard API with GlobalNetworkPolicy, which applies cluster-wide. It also adds explicit deny actions and ordered rule evaluation, making complex policies more readable. The selector syntax uses Calico's own expression language rather than standard label selectors.

☁️ AWS VPC CNI & Network Policy on EKS

EKS Network Policy Support

Since EKS 1.25, the AWS VPC CNI plugin natively enforces Kubernetes NetworkPolicy resources using eBPF.

  • Enable via add-on: Update vpc-cni add-on with enableNetworkPolicy: true
  • eBPF-based: Enforcement happens in the kernel for minimal latency impact
  • Standard API: Uses native networking.k8s.io/v1 resources
  • VPC Flow Logs: Network flow visibility through AWS-native tooling
  • Limitation: No extended CRDs like Calico/Cilium — L3/L4 only
Verisign Context: If running VPC CNI natively, standard NetworkPolicy is fully supported. For L7 or global policies, consider Calico overlay alongside VPC CNI.
For Verisign's EKS clusters, AWS VPC CNI is likely the default. The good news is that it now supports standard NetworkPolicy without needing a third-party plugin. The trade-off is that you only get L3/L4 capabilities. If you need L7 filtering or global policies, you can run Calico as a policy-only add-on alongside VPC CNI.

🐝 Cilium: L7-Aware Network Policies

apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata: { name: api-l7-policy, namespace: production }
spec:
  endpointSelector: { matchLabels: { app: api-server } }
  ingress:
    - fromEndpoints:
        - matchLabels: { app: frontend }
      toPorts:
        - ports: [{ port: "8080", protocol: TCP }]
          rules:
            http:
              - { method: GET, path: "/api/v1/.*" }
              - { method: POST, path: "/api/v1/orders" }
L7 Filtering: Cilium can restrict traffic by HTTP method, path, headers, and even gRPC method — far beyond what standard NetworkPolicy offers.
Cilium takes network policies to the application layer. Instead of just allowing TCP port 8080, you can specify which HTTP methods and URL paths are permitted. This is extremely powerful for API security — a compromised frontend can only call the specific endpoints it needs, not arbitrary paths on the API server.

🛡️ Real-World Security Strategies

From Policy to Practice

Writing policies is one thing. Implementing a comprehensive network security strategy across a production cluster is another. Let's talk about the strategies that make network policies work at scale.

🛡️ Zero-Trust Networking in Kubernetes

Zero-Trust Principle

Never trust, always verify. Every connection is denied unless explicitly permitted, regardless of source location.

1. Default Deny

  • Apply to every namespace
  • Both ingress and egress
  • Automate via admission controller

2. Least Privilege

  • Allow only required paths
  • Restrict ports precisely
  • Scope to exact labels

3. Verify & Audit

  • Log denied connections
  • Review policies in CI/CD
  • Test with network probes
Zero-trust isn't a product, it's a strategy. In Kubernetes, it starts with default deny everywhere, then allows only the minimum required communication paths. The third pillar — verification — is often overlooked. You need to continuously validate that policies are working as intended.

🧩 Microsegmentation Strategies

Namespace-Level Segmentation

  • Isolate by environment (dev/staging/prod)
  • Isolate by team or business unit
  • Broad but effective first step
# Block all cross-namespace traffic
ingress:
  - from:
      - podSelector: {} # Same ns only

Application-Level Segmentation

  • Isolate by microservice tier
  • Per-service egress restrictions
  • Granular but higher maintenance
# Only specific pod-to-pod
ingress:
  - from:
      - podSelector:
          matchLabels:
            app: checkout
Microsegmentation is about shrinking the blast radius. Start with namespace-level isolation — it's straightforward and covers multi-tenancy. Then layer on application-level segmentation for sensitive workloads like databases, payment processing, or PII stores. You don't need to segment everything equally — focus on your crown jewels first.

🔍 Debugging Network Policies

  • Forgotten DNS: Egress policies that don't allow port 53 break all name resolution silently
  • OR vs AND confusion: Two selectors in one from: item are AND'd; separate from: items are OR'd
  • CNI doesn't enforce: Some CNI plugins (e.g., Flannel) ignore NetworkPolicy entirely
  • Namespace-scoped: A policy in namespace A cannot affect pods in namespace B
Tip: DNS breakage is the number one problem. Always allow port 53 in egress policies. The OR/AND confusion causes subtle security gaps. Verify your CNI supports enforcement with a connectivity test.

🛠️ Debugging — kubectl Commands

Policy Inspection

# List policies affecting a pod
kubectl get networkpolicy -n prod

# Describe policy details
kubectl describe netpol \
  api-server-policy -n prod

# Check pod labels match
kubectl get pods -n prod \
  --show-labels | grep api

# Validate YAML before apply
kubectl apply --dry-run=server \
  -f policy.yaml
These are your essential debugging commands. Always test connectivity after applying a policy — don't assume it works.

🛠️ Debugging — Network Testing

Connectivity Testing

# Test connectivity from a pod
kubectl run test-conn --rm -it \
  --image=busybox -- wget -qO- \
  --timeout=3 http://api-server:8080

# Test DNS resolution
kubectl run test-dns --rm -it \
  --image=busybox -- nslookup \
  api-server.production.svc

# Netcat port test
kubectl exec -it debug-pod -- \
  nc -zv postgres-svc 5432
The busybox image is your best friend for quick connectivity tests. For persistent debugging, keep a debug pod running in each namespace with network tools installed.

The recommended approach is observe-first: enable logging, map actual traffic, then write policies based on reality rather than assumptions.

🚀 Network Policy Lifecycle

Recommended Workflow

Treat network policies as code — version, review, test, and automate them through your CI/CD pipeline.

  1. Observe: Enable flow logging and map actual traffic patterns
  2. Draft: Write policies based on observed flows, store in Git
  3. Validate: Lint with kubeval or kubectl --dry-run=server
  4. Test: Apply in a staging namespace with connectivity tests
  5. Enforce: Apply to production with monitoring enabled
  6. Audit: Periodically review denied flows and policy coverage
Network policies should go through the same rigor as application code. Version control them, review changes in pull requests, validate with dry runs, test in staging, and monitor after enforcement. This lifecycle approach prevents outages caused by misconfigured policies.

Two sides of network policies: the technical fundamentals of writing correct YAML, and the operational strategy of deploying them safely. Both matter equally. A perfect policy that breaks DNS is useless, and a well-deployed policy with the wrong selectors is a false sense of security.

🎯 Key Takeaways

1. Default deny is non-negotiable. Apply default-deny ingress and egress to every namespace. No exceptions for production workloads.
2. Always allow DNS. Every egress policy must include UDP/TCP port 53. Forgetting this is the most common cause of policy-related outages.
3. Know your CNI. Standard NetworkPolicy is L3/L4 only. For L7 filtering, DNS-based rules, or global policies, use Calico or Cilium extensions.
4. Observe before enforcing. Enable flow logging, map real traffic, then write policies. Never guess at communication patterns.
Four takeaways to remember. Default deny is the foundation. DNS is the most common pitfall. Your CNI determines what's enforceable. And observing before enforcing prevents outages. These principles will serve you well as you implement network policies on the Verisign platform.
