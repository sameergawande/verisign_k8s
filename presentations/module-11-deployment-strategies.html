🔄 Rolling Updates

The Default Deployment Strategy

Rolling updates are the default Deployment strategy. They incrementally replace old pods with new ones while maintaining availability.

🔄 How Rolling Updates Work

Incremental Pod Replacement

Kubernetes gradually replaces old ReplicaSet pods with new ones, controlled by two parameters: maxSurge and maxUnavailable.

maxSurge

  • Extra pods above desired count
  • Absolute number or percentage
  • Default: 25%
  • Higher = faster rollout, more resources

maxUnavailable

  • Pods that can be down during update
  • Absolute number or percentage
  • Default: 25%
  • Lower = safer but slower rollout
maxSurge controls extra pods beyond desired count; maxUnavailable controls how many can be down simultaneously. Both default to 25%. For 4 replicas, that means 1 extra pod and 1 unavailable at a time.

📝 Rolling Update Configuration

spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # At most 5 pods during update
      maxUnavailable: 0    # Zero downtime
  revisionHistoryLimit: 10
  minReadySeconds: 10      # Wait 10s before Ready
  # ... template, container spec, readinessProbe ...
maxUnavailable 0 means we never drop below desired count; maxSurge 1 creates one new pod at a time. minReadySeconds adds a buffer -- the pod must stay healthy for 10s before the rollout proceeds. revisionHistoryLimit controls how many old ReplicaSets are retained for rollback.

📈 Monitoring Rollout Progress

# Watch rollout status in real time
kubectl rollout status deployment/api-server -n production

# View rollout history
kubectl rollout history deployment/api-server -n production

# Inspect a specific revision
kubectl rollout history deployment/api-server \
  --revision=3 -n production

# Pause a rollout mid-way (investigate issues)
kubectl rollout pause deployment/api-server -n production

# Resume after investigation
kubectl rollout resume deployment/api-server -n production
Tip: Use --watch with kubectl get pods to see individual pod transitions during a rollout. Combine with -l app=api-server to filter.
Pause is powerful: Pausing a rollout mid-way creates a partial canary — some pods run the old version, some the new. Use this for manual verification before resuming.
kubectl rollout status blocks until completion or failure -- useful in CI/CD scripts. Pause and resume let you verify the new version with partial traffic before proceeding, giving basic canary functionality without extra tooling.

⏪️ Rollback and Revision History

# Undo to previous revision
kubectl rollout undo deployment/api-server -n production

# Undo to a specific revision
kubectl rollout undo deployment/api-server \
  --to-revision=2 -n production

# Annotate deployments for better history
kubectl annotate deployment/api-server \
  kubernetes.io/change-cause="Upgrade to v2.1.0 - JIRA-4521" \
  -n production
Fast Recovery: Rollback reuses the old ReplicaSet — no image pull needed if cached. Recovery takes seconds, not minutes.
Database Caveat: Rollback reverts pods but not database migrations. Always ensure backward-compatible schema changes.
Rollback reuses old ReplicaSets, so it scales the old one up and new one down. The change-cause annotation is important -- without it, rollout history shows blank descriptions. Always annotate in CI/CD. Note: rolling back the deployment does not undo database migrations.

🔵🟢 Blue-Green Deployments

Instant Cutover with Full Rollback

Blue-green maintains two complete environments. Traffic switches atomically from blue to green, eliminating mixed versions but requiring double resources during transition.

🔵🟢 Blue-Green Architecture

Core Principle

Run two identical environments in parallel. Route all traffic to one (blue). Deploy changes to the other (green). Switch traffic atomically when ready.

✅ Advantages

  • Instant rollback (switch back)
  • No mixed versions serving
  • Full pre-flight testing

⚠️ Trade-offs

  • 2x resource cost during deploy
  • Database compatibility required
  • Session state must be external
Best for: Critical services, compliance-heavy environments, APIs with strict SLAs, and when rollback speed is paramount.
Blue-green is the simplest strategy to reason about. The trade-off is resource cost -- two full copies running simultaneously. Both blue and green must work with the same database schema.

🔘 Blue-Green: Deployment Setup

Blue Deployment (current)

# Deployment: api-blue (current version)
spec:
  replicas: 4
  selector:
    matchLabels: { app: api-server, slot: blue }
  template:
    metadata:
      labels: { app: api-server, slot: blue, version: v2.0.0 }
    # ... container spec ...
Pattern: Both Deployments share app: api-server. The slot label differentiates them for the Service selector.
Two identical Deployments with different slot labels (blue or green). The green Deployment is deployed and validated before switching traffic.

🔘 Blue-Green: Service Label Switching

Service (the traffic switch)

apiVersion: v1
kind: Service
metadata:
  name: api-server
spec:
  selector:
    app: api-server
    slot: blue    # Switch to "green"
  ports:
    - port: 80
      targetPort: 8080
# Atomic cutover
kubectl patch svc api-server \
  -p '{"spec":{"selector":{"slot":"green"}}}' \
  -n production
Atomic switch: Changing the Service selector instantly redirects all new connections to green pods. Existing connections to blue pods drain naturally.
The Service is the traffic switch. Patch its selector from blue to green to redirect traffic. To roll back, patch back to blue.

🔀 Blue-Green with Ingress / Gateway API

# HTTPRoute (Gateway API)
spec:
  rules:
    - backendRefs:
        - name: api-blue
          port: 80
          weight: 100       # Flip weights for cutover
        - name: api-green
          port: 80
          weight: 0
Gateway API advantage: Weight-based routing enables gradual cutover (90/10 → 50/50 → 0/100) rather than all-or-nothing. Blurs the line between blue-green and canary.
The Gateway API lets you control traffic weights instead of binary switching. Send 10% to green, verify metrics, then increase to 100%. On EKS, the AWS Gateway API Controller works with ALB for this.

🐦 Canary Deployments

Controlled Risk with Gradual Traffic Shifting

Canary deployments route a small percentage of traffic to the new version. If metrics are healthy, traffic increases. If problems appear, the canary is killed with minimal blast radius.

🐦 Canary Concept

Progressive Traffic Shifting

Deploy the new version alongside the current version. Route a small percentage of traffic to it. Observe metrics. Gradually increase traffic or roll back.

Canary Progression

  • Step 1: 5% traffic to canary
  • Step 2: Observe error rates, latency
  • Step 3: Promote to 25%, then 50%
  • Step 4: Full promotion to 100%

Key Metrics to Watch

  • HTTP error rate (5xx responses)
  • Request latency (p50, p95, p99)
  • Resource consumption (CPU, memory)
  • Business metrics (conversion, throughput)
Start with a small percentage of traffic and watch closely. Good observability is essential -- without metrics, canary deployments are just slow rolling updates.

🔧 Manual Canary: Stable Deployment

Stable (4 replicas)

# Deployment: api-stable
spec:
  replicas: 4
  selector:
    matchLabels: { app: api-server, track: stable }
  template:
    metadata:
      labels: { app: api-server, track: stable }
    # ... container spec ...
The Service selects on app: api-server, matching both stable and canary pods. Traffic splits proportionally by pod count.
Two Deployments behind one Service. To promote, scale up canary and scale down stable. To abort, delete the canary Deployment.

🔧 Manual Canary: Canary Deployment

Canary (1 replica)

# Deployment: api-canary
spec:
  replicas: 1              # ~20% traffic (1 of 5 total pods)
  selector:
    matchLabels: { app: api-server, track: canary }
  template:
    metadata:
      labels: { app: api-server, track: canary }
    # ... same container spec, different image tag ...
Limitation: Traffic split is tied to pod ratio (1/5 = 20%). For finer-grained control (e.g., 5%), you need ingress-level traffic splitting.
With 4 stable and 1 canary, about 20% of traffic goes to the canary. Crude but works without extra tooling.

🛠️ Canary with NGINX Ingress Annotations

# Canary Ingress — references same host as primary
metadata:
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"
    # Also: canary-by-header, canary-by-cookie
spec:
  rules:
    - host: api.example.com
      # ... paths route to api-canary Service
NGINX Ingress supports canary annotations natively. canary-weight sends a precise percentage to the canary. canary-by-header routes specific users; canary-by-cookie enables session-based routing. Much finer control than pod-ratio canaries.

🚀 Progressive Delivery: Argo Rollouts

apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: api-server
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5
        - pause: { duration: 5m }
        - setWeight: 25
        - pause: { duration: 5m }
        - setWeight: 50
        - pause: { duration: 10m }
        - setWeight: 100
      canaryService: api-canary
      stableService: api-stable
Argo Rollouts replaces Deployments with a custom Rollout CRD. It provides built-in canary steps, analysis templates, and service mesh integration.
Argo Rollouts replaces Deployments with a Rollout CRD providing built-in canary steps and Prometheus integration for metric-driven promotions.

🚀 Progressive Delivery: Flagger

apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: api-server
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  service:
    port: 80
  analysis:
    interval: 1m
    threshold: 5
    maxWeight: 50
    stepWeight: 10
    metrics:
      - name: request-success-rate
        threshold: 99
Key difference: Flagger wraps existing Deployments and automates canary analysis based on metrics. No CRD replacement needed.
Flagger watches standard Deployments and automates canary progression based on metrics. No CRD replacement needed -- it wraps existing resources.

💡 Advanced Strategies

A/B Testing, Shadow Launches & Feature Flags

Advanced patterns for more control over how users experience new code. Often combined with the core strategies covered earlier.

🔬 A/B Testing with Header-Based Routing

# NGINX Ingress: header-based routing
metadata:
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-by-header: "X-Test-Group"
    nginx.ingress.kubernetes.io/canary-by-header-value: "experiment-b"
---
# Gateway API: header match
spec:
  rules:
    - matches:
        - headers:
            - { name: X-Test-Group, value: experiment-b }
      backendRefs:
        - { name: api-experiment-b, port: 80 }
    - backendRefs:
        - { name: api-control, port: 80 }
A/B testing differs from canary in intent. Canary validates operational health (error rates, latency). A/B validates business outcomes (conversion, engagement). Header-based routing deterministically routes specific users to a version.

👁️ Shadow / Dark Launches

Traffic Mirroring

Duplicate live traffic to the new version. The shadow processes requests but its responses are discarded.

# Istio VirtualService — traffic mirroring
spec:
  http:
    - route:
        - destination: { host: api-stable }
      mirror:
        host: api-shadow
      mirrorPercentage:
        value: 100.0
Zero risk to users: Shadow receives real traffic patterns but responses are discarded. Validates performance and correctness before any user exposure.
The new version sees real request patterns and load. Compare response times and error rates between stable and shadow. Valuable for migration testing or API refactoring without user impact.

🏷️ Feature Flags vs Deployment Strategies

Deployment Strategies

  • Control at infrastructure level
  • Entire application version changes
  • Operated by platform team
  • Rollback = redeploy old version
  • Scope: all-or-nothing per pod

Feature Flags

  • Control at application level
  • Toggle individual features in code
  • Operated by development team
  • Rollback = flip the flag off
  • Scope: per-feature, per-user
Recommended pattern: Combine both. Use deployment strategies for infrastructure changes and feature flags for feature activation. Deploy dark (flag off), then enable progressively.
Not either/or. Deployment strategies handle infrastructure risk; feature flags handle product risk. This separation decouples deployment from release.

🛡️ Deployment Safety

Guardrails for Safe Releases

Regardless of strategy, you need safety mechanisms to prevent bad deployments from causing outages.

🛡️ Pod Disruption Budgets

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-server-pdb
  namespace: production
spec:
  minAvailable: 3          # Always keep at least 3 pods
  # OR: maxUnavailable: 1  # Allow at most 1 pod down
  selector:
    matchLabels:
      app: api-server
Voluntary disruptions: PDBs protect against node drains, cluster autoscaler, and maintenance operations.
Not for crashes: PDBs do not protect against involuntary disruptions like node failures or OOM kills.
During deployments: PDBs interact with rolling updates. A PDB of minAvailable: 3 on a 4-replica Deployment effectively forces maxUnavailable: 1, regardless of the Deployment's own strategy configuration.
The Deployment controller respects PDBs during rolling updates. kubectl drain refuses to evict pods if it would violate the PDB. On EKS, PDBs prevent the cluster autoscaler from being too aggressive. Define PDBs for critical services.

🚨 Deployment Conditions & Failure Detection

# Set a deadline for rollout completion
spec:
  progressDeadlineSeconds: 300  # 5-minute deadline
  minReadySeconds: 15           # Pod must be Ready for 15s
# Check deployment conditions
kubectl get deployment api-server -o json | \
  jq '.status.conditions[] | select(.type=="Progressing")'

# Example output for a stuck rollout:
# {
#   "type": "Progressing",
#   "status": "False",
#   "reason": "ProgressDeadlineExceeded",
#   "message": "has timed out progressing"
# }
Stuck rollouts: If a new pod fails readiness probes repeatedly, the rollout stalls. progressDeadlineSeconds marks it as failed after the deadline, enabling automated alerting and rollback.
Without progressDeadlineSeconds, a broken deployment can stall indefinitely. The Progressing condition switches to False after the deadline, triggering automated rollback in CI/CD. minReadySeconds prevents false positives from pods that crash shortly after starting.

🚀 CI/CD Pipeline Integration Patterns

Deployment Pipeline Flow

Build → Test → Image Push → Deploy to Staging → Integration Tests → Deploy → Verify → Promote or Rollback

# GitOps deployment pattern (ArgoCD / Flux)
# 1. Update image tag in Git repo
git commit -m "Release api-server v2.1.0"
git push origin main

# 2. ArgoCD detects change and applies
argocd app sync api-server --prune

# 3. Wait for rollout to complete
kubectl rollout status deployment/api-server \
  -n production --timeout=300s

# 4. Run smoke tests
curl -sf https://api.example.com/healthz || \
  kubectl rollout undo deployment/api-server -n production
GitOps principle: Git is the single source of truth. All deployment changes go through Git, enabling audit trails, approvals, and automated rollback to any previous commit.
In GitOps, update manifests in Git and let ArgoCD or Flux reconcile. The pipeline includes rollout status checks, smoke tests, and metric validation. If any step fails, trigger a rollback.

☁️ EKS Deployment Best Practices

Node Management

  • Use managed node groups for automatic AMI updates
  • Configure PDBs before node group updates
  • Use Karpenter for rapid scaling during rollouts

Image Management

  • Use ECR with image scanning enabled
  • Immutable tags (never reuse :latest)
  • Pre-pull images on nodes before rollout

Observability

  • CloudWatch Container Insights for metrics
  • AWS X-Ray for tracing during canary
  • ALB access logs for traffic analysis
EKS tip: When using ALB Ingress Controller for canary deployments, configure alb.ingress.kubernetes.io/actions.weighted-routing for precise traffic splitting at the load balancer level.
Managed node groups handle node updates but need PDBs for safe draining. ECR scanning should gate CI/CD. Karpenter helps with temporary capacity needs during canary and blue-green deployments.

📊 Strategy Comparison Matrix

Strategy Downtime Resource Cost Rollback Speed Complexity
Rolling Update None Low (+25%) Medium (new rollout) Low (built-in)
Blue-Green None High (2x) Instant (switch back) Low-Medium
Canary None Low-Medium Fast (kill canary) Medium-High
A/B Testing None Medium Fast (routing change) High
Shadow None High (2x) N/A (no user impact) High (mesh required)
Choosing a strategy: Start with rolling updates. Add canary for critical services. Use blue-green when instant rollback is non-negotiable. Layer in feature flags for granular control.
Not every service needs canary or blue-green. Internal services work fine with rolling updates. Customer-facing APIs with strict SLAs benefit from blue-green or canary.

🎯 Key Takeaways

1. Rolling updates are the baseline. Tune maxSurge, maxUnavailable, and minReadySeconds. Add canary for critical services.
2. Blue-green provides instant rollback. Worth 2x resource cost for critical services. Canary reduces blast radius with metric-driven promotions.
3. Safety mechanisms are not optional. PDBs, progressDeadlineSeconds, readiness probes, and revisionHistoryLimit should be on every Deployment. Combine with feature flags to decouple deployment from release.
Start simple with rolling updates. Add complexity only when the risk profile demands it. Safety mechanisms belong on every Deployment regardless of strategy.
