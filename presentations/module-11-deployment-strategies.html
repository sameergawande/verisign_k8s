🔄 Rolling Updates

The Default Deployment Strategy

Rolling updates are the default strategy for Kubernetes Deployments. They incrementally replace old pods with new ones, maintaining availability throughout the process. Let's understand the mechanics and configuration options.

🔄 How Rolling Updates Work

Incremental Pod Replacement

Kubernetes gradually replaces old ReplicaSet pods with new ones, controlled by two parameters: maxSurge and maxUnavailable.

maxSurge

  • Extra pods above desired count
  • Absolute number or percentage
  • Default: 25%
  • Higher = faster rollout, more resources

maxUnavailable

  • Pods that can be down during update
  • Absolute number or percentage
  • Default: 25%
  • Lower = safer but slower rollout
The two knobs controlling rolling updates are maxSurge and maxUnavailable. maxSurge controls how many extra pods can exist beyond the desired count — it determines how fast you scale up the new version. maxUnavailable controls how many pods can be down simultaneously. Both default to 25%. For a 4-replica deployment, that means 1 extra pod and 1 pod allowed to be unavailable at a time.

📝 Rolling Update Configuration

apiVersion: apps/v1
kind: Deployment
metadata: { name: api-server, namespace: production }
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # At most 5 pods during update
      maxUnavailable: 0    # Zero downtime
  revisionHistoryLimit: 10
  minReadySeconds: 10      # Wait 10s before Ready
  template:
    metadata:
      labels: { app: api-server, version: v2.1.0 }
    spec:
      containers:
      - name: api
        image: registry.verisign.com/api:2.1.0
        readinessProbe:
          httpGet: { path: /healthz, port: 8080 }
          initialDelaySeconds: 5
          periodSeconds: 5
This configuration is conservative: maxUnavailable 0 means we never drop below the desired count, and maxSurge 1 means we create one new pod at a time. The minReadySeconds setting adds a buffer — even after the readiness probe passes, the pod must stay healthy for 10 seconds before the rollout proceeds. This catches pods that crash shortly after startup. revisionHistoryLimit controls how many old ReplicaSets are retained for rollback.

📈 Monitoring Rollout Progress

# Watch rollout status in real time
kubectl rollout status deployment/api-server -n production

# View rollout history
kubectl rollout history deployment/api-server -n production

# Inspect a specific revision
kubectl rollout history deployment/api-server \
  --revision=3 -n production

# Pause a rollout mid-way (investigate issues)
kubectl rollout pause deployment/api-server -n production

# Resume after investigation
kubectl rollout resume deployment/api-server -n production
Tip: Use --watch with kubectl get pods to see individual pod transitions during a rollout. Combine with -l app=api-server to filter.
Pause is powerful: Pausing a rollout mid-way creates a partial canary — some pods run the old version, some the new. Use this for manual verification before resuming.
kubectl rollout status blocks until the rollout completes or fails — useful in CI/CD scripts. The pause and resume commands are underused but powerful. You can pause mid-rollout, verify the new version is behaving correctly with partial traffic, then resume. This gives you basic canary functionality without additional tooling.

⏪️ Rollback and Revision History

# Undo to previous revision
kubectl rollout undo deployment/api-server -n production

# Undo to a specific revision
kubectl rollout undo deployment/api-server \
  --to-revision=2 -n production

# Annotate deployments for better history
kubectl annotate deployment/api-server \
  kubernetes.io/change-cause="Upgrade to v2.1.0 - JIRA-4521" \
  -n production
Fast Recovery: Rollback reuses the old ReplicaSet — no image pull needed if cached. Recovery takes seconds, not minutes.
Database Caveat: Rollback reverts pods but not database migrations. Always ensure backward-compatible schema changes.
Rollback is one of Kubernetes' strongest features. Because it retains old ReplicaSets, reverting is essentially just scaling the old one up and the new one down. The change-cause annotation is critical for production — without it, rollout history shows blank descriptions, making it impossible to know which revision to roll back to. Always annotate in your CI/CD pipeline. The database caveat is important: if your new version ran a schema migration, rolling back the deployment doesn't undo the migration.

🔵🟢 Blue-Green Deployments

Instant Cutover with Full Rollback

Blue-green deployments maintain two complete environments. Traffic switches atomically from the old version (blue) to the new version (green). This eliminates the mixed-version state of rolling updates but requires double the resources during transition.

🔵🟢 Blue-Green Architecture

Core Principle

Run two identical environments in parallel. Route all traffic to one (blue). Deploy changes to the other (green). Switch traffic atomically when ready.

✅ Advantages

  • Instant rollback (switch back)
  • No mixed versions serving
  • Full pre-flight testing

⚠️ Trade-offs

  • 2x resource cost during deploy
  • Database compatibility required
  • Session state must be external
Best for: Critical production services, compliance-heavy environments, APIs with strict SLAs, and when rollback speed is paramount.
Blue-green is the simplest deployment strategy to reason about. The trade-off is resource cost — you need capacity for two full copies of the application simultaneously. For Verisign's DNS-critical services, the instant rollback capability can justify the extra cost. The database challenge applies here too: both blue and green must work with the same database schema.

🔘 Blue-Green: Deployment Setup

Blue Deployment (current)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-blue
spec:
  replicas: 4
  selector:
    matchLabels:
      app: api-server
      slot: blue
  template:
    metadata:
      labels:
        app: api-server
        slot: blue
        version: v2.0.0
Pattern: Both blue and green Deployments share the app: api-server label. The slot label differentiates them for the Service selector.
This is the simplest blue-green implementation. You create two identical Deployments, each with a different slot label (blue or green). The green Deployment is deployed and validated beforehand.

🔘 Blue-Green: Service Label Switching

Service (the traffic switch)

apiVersion: v1
kind: Service
metadata:
  name: api-server
spec:
  selector:
    app: api-server
    slot: blue    # Switch to "green"
  ports:
    - port: 80
      targetPort: 8080
# Atomic cutover
kubectl patch svc api-server \
  -p '{"spec":{"selector":{"slot":"green"}}}' \
  -n production
Atomic switch: Changing the Service selector instantly redirects all new connections to green pods. Existing connections to blue pods drain naturally.
The Service acts as the traffic switch — patching its selector from blue to green redirects all traffic. To roll back, patch the selector back to blue. Both Deployments remain running until you're confident in the new version.

🔀 Blue-Green with Ingress / Gateway API

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: api-route, namespace: production }
spec:
  parentRefs: [{ name: production-gateway }]
  hostnames: ["api.verisign.com"]
  rules:
    - backendRefs:
        # Blue-green: 100% to one backend
        - name: api-blue
          port: 80
          weight: 100
        - name: api-green
          port: 80
          weight: 0       # Flip to 100 for cutover
Gateway API advantage: Weight-based routing enables gradual cutover (90/10 → 50/50 → 0/100) rather than all-or-nothing. Blurs the line between blue-green and canary.
The Gateway API provides a more sophisticated blue-green implementation. Instead of binary switching via Service selectors, you control traffic weights. This lets you do a gradual cutover — send 10% to green, verify metrics, then increase to 100%. The Gateway API is the future of Kubernetes ingress routing and supports this natively. On EKS, the AWS Gateway API Controller works with ALB to provide this functionality.

🐦 Canary Deployments

Test in Production with Controlled Risk

Canary deployments route a small percentage of production traffic to the new version. If metrics are healthy, traffic gradually increases. If problems appear, the canary is killed with minimal blast radius. Named after the canary in the coal mine — an early warning system.

🐦 Canary Concept

Progressive Traffic Shifting

Deploy the new version alongside the current version. Route a small percentage of traffic to it. Observe metrics. Gradually increase traffic or roll back.

Canary Progression

  • Step 1: 5% traffic to canary
  • Step 2: Observe error rates, latency
  • Step 3: Promote to 25%, then 50%
  • Step 4: Full promotion to 100%

Key Metrics to Watch

  • HTTP error rate (5xx responses)
  • Request latency (p50, p95, p99)
  • Resource consumption (CPU, memory)
  • Business metrics (conversion, throughput)
The canary pattern is about controlled risk. Instead of exposing all users to a new version at once, you start with a small percentage and watch closely. The key is having good observability — you need real-time metrics to make promotion or rollback decisions. Without metrics, canary deployments are just slow rolling updates.

🔧 Manual Canary: Stable Deployment

Stable (4 replicas)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-stable
spec:
  replicas: 4
  selector:
    matchLabels:
      app: api-server
      track: stable
  template:
    metadata:
      labels:
        app: api-server
        track: stable
The Service selects on app: api-server, which matches both stable and canary pods. Traffic is split proportionally by pod count.
The simplest canary approach uses two Deployments behind one Service. The Service selects on app: api-server, which matches both stable and canary pods. To promote, scale up canary and scale down stable. To abort, delete the canary Deployment.

🔧 Manual Canary: Canary Deployment

Canary (1 replica)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-canary
spec:
  replicas: 1
  selector:
    matchLabels:
      app: api-server
      track: canary
  template:
    metadata:
      labels:
        app: api-server
        track: canary
Limitation: Traffic split is tied to pod ratio (1/5 = 20%). For finer-grained control (e.g., 5%), you need ingress-level traffic splitting.
Traffic split is proportional to pod count — with 4 stable and 1 canary, about 20% goes to the canary. This is crude but works without any extra tooling.

🛠️ Canary with NGINX Ingress Annotations

# Canary Ingress — references same host as primary Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress-canary
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"
    # Also available: canary-by-header, canary-by-cookie
spec:
  rules:
    - host: api.verisign.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api-canary
                port:
                  number: 80
NGINX Ingress Controller supports canary annotations natively. The canary-weight annotation sends a precise percentage of traffic to the canary backend. You can also use canary-by-header to route specific users or canary-by-cookie for session-based routing. This gives much finer control than pod-ratio canaries — you can send just 1% of traffic to the canary regardless of replica count.

🚀 Progressive Delivery: Argo Rollouts

apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: api-server
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5
        - pause: { duration: 5m }
        - setWeight: 25
        - pause: { duration: 5m }
        - setWeight: 50
        - pause: { duration: 10m }
        - setWeight: 100
      canaryService: api-canary
      stableService: api-stable
Argo Rollouts replaces Deployments with a custom Rollout CRD. It provides built-in canary steps, analysis templates, and service mesh integration.
Argo Rollouts provides a custom Rollout resource with built-in canary steps, analysis templates, and integration with various service meshes. It integrates with Prometheus for metric-driven promotion decisions.

🚀 Progressive Delivery: Flagger

apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: api-server
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  service:
    port: 80
  analysis:
    interval: 1m
    threshold: 5
    maxWeight: 50
    stepWeight: 10
    metrics:
      - name: request-success-rate
        threshold: 99
Key difference: Flagger wraps existing Deployments and automates canary analysis based on metrics. No CRD replacement needed.
Flagger takes a different approach from Argo Rollouts — it watches standard Deployments and automatically manages canary progression based on metrics. Both tools integrate with Prometheus for metric-driven promotion decisions.

💡 Advanced Strategies

A/B Testing, Shadow Launches & Feature Flags

Beyond the core deployment strategies, there are several advanced patterns that give you even more control over how users experience new code. These techniques are often used together with the deployment strategies we've already covered.

🔬 A/B Testing with Header-Based Routing

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ab-test
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-by-header: "X-Test-Group"
    nginx.ingress.kubernetes.io/canary-by-header-value: "experiment-b"
---
# Gateway API equivalent
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: api-ab-route }
spec:
  rules:
    - matches:
        - headers:
            - { name: X-Test-Group, value: experiment-b }
      backendRefs:
        - { name: api-experiment-b, port: 80 }
    - backendRefs:
        - { name: api-control, port: 80 }
Tip: Route testers or beta users to the experimental version using custom headers set by a load balancer upstream. A/B testing differs from canary in intent. Canary validates operational health — error rates and latency. A/B testing validates business outcomes — conversion rates and user engagement. Header-based routing lets you deterministically route specific users to a specific version, which is essential for statistically valid experiments. The upstream application or CDN sets the header based on user segmentation.

👁️ Shadow / Dark Launches

Traffic Mirroring

Duplicate production traffic to the new version without affecting responses. The shadow version processes requests but its responses are discarded.

# Istio traffic mirroring example
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: api-server
spec:
  hosts:
    - api-server
  http:
    - route:
        - destination:
            host: api-stable
      mirror:
        host: api-shadow
      mirrorPercentage:
        value: 100.0
Zero risk to users: Shadow receives real traffic patterns but responses go nowhere. Ideal for validating performance, correctness, and resource consumption before any user exposure.
Shadow launches are the safest way to test with production traffic. The new version sees real request patterns, real data shapes, and real load. You can compare response times, error rates, and even response bodies between the stable and shadow versions. This is particularly valuable for database migration testing or API refactoring where you want to verify correctness without user impact. Istio makes this straightforward with its mirror configuration.

🏷️ Feature Flags vs Deployment Strategies

Deployment Strategies

  • Control at infrastructure level
  • Entire application version changes
  • Operated by platform team
  • Rollback = redeploy old version
  • Scope: all-or-nothing per pod

Feature Flags

  • Control at application level
  • Toggle individual features in code
  • Operated by development team
  • Rollback = flip the flag off
  • Scope: per-feature, per-user
Best practice: Combine both. Use deployment strategies for safe infrastructure changes and feature flags for safe feature activation. Deploy dark (flag off), then enable progressively.
This is not an either/or choice. The most resilient systems use both. Deploy the new code with features behind flags. The deployment strategy handles the infrastructure risk — will the new binary run correctly? Feature flags handle the product risk — will users respond well to the new feature? This separation lets you deploy frequently with low risk and decouple deployment from release.

🛡️ Deployment Safety

Guardrails for Production Releases

No matter which deployment strategy you choose, you need safety mechanisms to prevent bad deployments from causing outages. Let's look at the tools Kubernetes provides to make deployments safer.

🛡️ Pod Disruption Budgets

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-server-pdb
  namespace: production
spec:
  minAvailable: 3          # Always keep at least 3 pods
  # OR: maxUnavailable: 1  # Allow at most 1 pod down
  selector:
    matchLabels:
      app: api-server
Voluntary disruptions: PDBs protect against node drains, cluster autoscaler, and maintenance operations.
Not for crashes: PDBs do not protect against involuntary disruptions like node failures or OOM kills.
During deployments: PDBs interact with rolling updates. A PDB of minAvailable: 3 on a 4-replica Deployment effectively forces maxUnavailable: 1, regardless of the Deployment's own strategy configuration.
PDBs are essential for production workloads. During a rolling update, the Deployment controller respects PDBs. But PDBs really shine during node maintenance — kubectl drain will refuse to evict pods if doing so would violate the PDB. For EKS managed node groups, PDBs prevent the cluster autoscaler from being too aggressive. Always define PDBs for your critical services.

🚨 Deployment Conditions & Failure Detection

# Set a deadline for rollout completion
spec:
  progressDeadlineSeconds: 300  # 5-minute deadline
  minReadySeconds: 15           # Pod must be Ready for 15s
# Check deployment conditions
kubectl get deployment api-server -o json | \
  jq '.status.conditions[] | select(.type=="Progressing")'

# Example output for a stuck rollout:
# {
#   "type": "Progressing",
#   "status": "False",
#   "reason": "ProgressDeadlineExceeded",
#   "message": "has timed out progressing"
# }
Stuck rollouts: If a new pod fails readiness probes repeatedly, the rollout stalls. progressDeadlineSeconds marks it as failed after the deadline, enabling automated alerting and rollback.
progressDeadlineSeconds is your safety net for stuck rollouts. Without it, a broken deployment can stall indefinitely with some pods running the old version and some pods crash-looping on the new version. The Progressing condition switches to False after the deadline, which you can use as a trigger for automated rollback in your CI/CD pipeline. minReadySeconds prevents false positives — a pod that starts and immediately crashes won't count as progress.

🚀 CI/CD Pipeline Integration Patterns

Deployment Pipeline Flow

Build → Test → Image Push → Deploy to Staging → Integration Tests → Deploy to Production → Verify → Promote or Rollback

# GitOps deployment pattern (ArgoCD / Flux)
# 1. Update image tag in Git repo
git commit -m "Release api-server v2.1.0"
git push origin main

# 2. ArgoCD detects change and applies
argocd app sync api-server --prune

# 3. Wait for rollout to complete
kubectl rollout status deployment/api-server \
  -n production --timeout=300s

# 4. Run smoke tests
curl -sf https://api.verisign.com/healthz || \
  kubectl rollout undo deployment/api-server -n production
GitOps principle: Git is the single source of truth. All deployment changes go through Git, enabling audit trails, approvals, and automated rollback to any previous commit.
CI/CD integration is where deployment strategies come to life. In a GitOps model, you never kubectl apply directly — you update manifests in Git and let ArgoCD or Flux reconcile. The pipeline includes automated verification: rollout status checks, smoke tests, and metric validation. If any step fails, the pipeline triggers a rollback. For canary deployments with Argo Rollouts, the analysis step is built into the Rollout resource itself.

☁️ EKS Deployment Best Practices

Node Management

  • Use managed node groups for automatic AMI updates
  • Configure PDBs before node group updates
  • Use Karpenter for rapid scaling during rollouts

Image Management

  • Use ECR with image scanning enabled
  • Immutable tags (never reuse :latest)
  • Pre-pull images on nodes before rollout

Observability

  • CloudWatch Container Insights for metrics
  • AWS X-Ray for tracing during canary
  • ALB access logs for traffic analysis
EKS tip: When using ALB Ingress Controller for canary deployments, configure alb.ingress.kubernetes.io/actions.weighted-routing for precise traffic splitting at the load balancer level.
EKS adds some specific considerations. Managed node groups handle node updates, but you need PDBs to ensure pods are drained safely. ECR image scanning should be part of your CI/CD gate — block deployments if critical vulnerabilities are found. Karpenter is valuable during deployments because canary and blue-green strategies temporarily need more capacity. For observability during canary, CloudWatch Container Insights and X-Ray give you the metrics needed for promotion decisions.

📊 Strategy Comparison Matrix

Strategy Downtime Resource Cost Rollback Speed Complexity
Rolling Update None Low (+25%) Medium (new rollout) Low (built-in)
Blue-Green None High (2x) Instant (switch back) Low-Medium
Canary None Low-Medium Fast (kill canary) Medium-High
A/B Testing None Medium Fast (routing change) High
Shadow None High (2x) N/A (no user impact) High (mesh required)
Choosing a strategy: Start with rolling updates. Add canary for critical services. Use blue-green when instant rollback is non-negotiable. Layer in feature flags for granular control.
This comparison helps you choose the right strategy for each service. Not every service needs canary or blue-green. Low-risk internal services are fine with rolling updates. Customer-facing APIs with strict SLAs benefit from blue-green or canary. Reserve shadow launches for major refactoring or migration projects where you need to validate with real traffic before any user exposure.

🎯 Key Takeaways

1. Rolling updates are your baseline. Tune maxSurge, maxUnavailable, and minReadySeconds for your availability needs. Add canary for critical services.
2. Blue-green provides instant rollback. Worth the 2x resource cost for critical services. Canary reduces blast radius with metric-driven promotions via Argo Rollouts or Flagger.
3. Safety mechanisms are not optional. PDBs, progressDeadlineSeconds, readiness probes, and revisionHistoryLimit are essential. Combine strategies with feature flags to decouple deployment from release risk.
Five key takeaways. Start simple with rolling updates and add complexity only when the risk profile demands it. Safety mechanisms should be on every Deployment regardless of strategy. And the combination of deployment strategies with feature flags gives you the most resilient release process possible.
