⚙️ CI/CD Fundamentals for Kubernetes

From code commit to running containers

Let's start by understanding how CI/CD pipelines differ when targeting Kubernetes versus traditional infrastructure. The containerized workflow introduces new stages and concerns.

Traditional CI/CD vs Cloud-Native CI/CD

Traditional CI/CD

  • Deploy artifacts (JARs, binaries) to VMs
  • Configuration management tools (Ansible, Chef)
  • Mutable infrastructure — update in place
  • Snowflake servers over time
  • Rollback = redeploy previous version

Cloud-Native CI/CD

  • Deploy container images to orchestrators
  • Declarative manifests (YAML, Helm, Kustomize)
  • Immutable infrastructure — replace, never patch
  • Identical environments from images
  • Rollback = point to previous image tag
Key shift: The artifact is a container image, and the desired state is a versioned manifest in Git.
The fundamental shift is from mutable to immutable infrastructure. Instead of SSH-ing into servers to deploy, we build an immutable image once and declare where it should run. This makes deployments reproducible and auditable. The manifest in Git becomes the source of truth for cluster state.

Pipeline Stages for Kubernetes

End-to-End Pipeline Flow

Code Commit → Build → Test → Scan → Push → Deploy → Verify

  • Build — compile code, build container image
  • Test — unit tests, integration tests, contract tests
  • Scan — vulnerability scanning, policy compliance
  • Push — tag and push image to registry (ECR, GHCR)
  • Deploy — update manifests, sync to cluster
  • Verify — smoke tests, health checks, monitoring
Separation of concerns: CI builds and tests the artifact. CD deploys the artifact. GitOps manages the desired state.
Each stage serves a distinct purpose and should be independently auditable. The scan stage is critical in enterprise environments for CVE detection and policy enforcement. Notice the clear separation: CI is responsible for producing a trusted artifact, while CD handles getting that artifact running in the cluster. This separation is key to the GitOps model.

Container Image Lifecycle

🔨 Build

  • Multi-stage Dockerfiles
  • Minimal base images (distroless, Alpine)
  • Build cache optimization

🏷️ Tag & Push

  • Immutable tags (Git SHA, semver)
  • Never use :latest in production
  • Push to ECR / private registry

🔒 Scan & Sign

  • Trivy / Snyk vulnerability scan
  • Cosign for image signing
  • Admission control for verified images
# Example: Build, scan, tag, push
docker build -t myapp:${GIT_SHA} .
trivy image --severity HIGH,CRITICAL myapp:${GIT_SHA}
cosign sign --key cosign.key myapp:${GIT_SHA}
ECR=123456789.dkr.ecr.us-east-1.amazonaws.com
docker tag myapp:${GIT_SHA} ${ECR}/myapp:${GIT_SHA}
docker push ${ECR}/myapp:${GIT_SHA}
The image lifecycle is the backbone of cloud-native CI. Multi-stage builds keep images lean, immutable tags ensure reproducibility, and image signing creates a chain of trust. Using Git SHA as the tag creates a direct link between the code commit and the running image. ECR supports image scanning natively, and Cosign from the Sigstore project provides keyless or key-based signing.

📚 GitOps Principles

Git as the single source of truth for infrastructure

GitOps is more than a deployment tool - it's an operational framework. Let's understand the principles that define GitOps and why they matter for Kubernetes operations at scale.

GitOps: Definition & Four Principles

What is GitOps?

An operational framework that applies DevOps practices for infrastructure automation — using Git as the single source of truth for declarative infrastructure and applications.

1. Declarative

The entire system is described declaratively (YAML manifests, Helm charts, Kustomize overlays)

2. Versioned & Immutable

The canonical desired state is versioned in Git, providing a complete audit trail

3. Pulled Automatically

Agents automatically pull the desired state and apply it to the cluster

4. Continuously Reconciled

Agents observe actual state, detect drift, and correct it automatically

These four principles were defined by the OpenGitOps project under the CNCF. Declarative means you describe what, not how. Versioned means Git provides the audit trail and rollback capability. Pull-based means agents in the cluster fetch state rather than CI pushing to the cluster. Continuous reconciliation means the system self-heals when drift is detected.

Push vs Pull-Based Deployment Models

📤 Push-Based (Traditional)

  • CI/CD tool runs kubectl apply
  • Requires cluster credentials in CI
  • No drift detection after deploy
  • Pipeline is the only deployment path
  • Examples: Jenkins, GitHub Actions (direct)

📥 Pull-Based (GitOps)

  • Agent in cluster polls Git for changes
  • No cluster credentials outside cluster
  • Continuous drift detection and repair
  • Git commit is the deployment trigger
  • Examples: ArgoCD, FluxCD
Security advantage: Pull-based models eliminate the need to store cluster credentials in external CI systems — the agent only needs read access to Git.
The security model is the primary differentiator. In push-based, your CI system needs a kubeconfig with cluster admin privileges, which is a significant attack surface. In pull-based, the agent runs inside the cluster with the necessary RBAC and only needs read access to the Git repository. The continuous reconciliation loop also ensures that manual kubectl changes are automatically reverted.

Jenkins CI/CD Pipeline Integration

⚙️ Jenkinsfile Pipeline Stages

  • Build — compile code, build container image, push to ECR
  • Test — unit tests, integration tests, code coverage
  • Scan — image vulnerability scanning (Trivy, ECR scan)
  • Deploy — kubectl apply or helm upgrade to EKS

⚡ Kubernetes Plugin

  • Dynamic build agents launched as Pods in EKS
  • Each build gets an isolated, ephemeral environment
  • Pod templates define containers (maven, docker, kubectl)
  • Scales to zero when idle — no persistent agent VMs
EKS integration: Jenkins authenticates to EKS via IRSA (IAM Roles for Service Accounts) or kubeconfig with a service account token — avoid storing long-lived cluster admin credentials.
Jenkins remains the most common CI/CD tool in enterprise environments. The Kubernetes plugin transforms Jenkins from a static VM-based system into a cloud-native build platform by scheduling build agents as Pods. For Verisign's EKS clusters, IRSA is the recommended authentication method since it ties IAM roles directly to Kubernetes service accounts without managing static credentials. However, as we discussed in the push vs pull comparison, Jenkins represents the push-based model where the CI system requires direct cluster access. Many organizations keep Jenkins for CI (build, test, scan) but hand off the CD (deploy) step to a GitOps tool like ArgoCD.

From Jenkins Push to GitOps Pull

Hybrid Pattern: Jenkins CI + GitOps CD

Developer → Git Push → Jenkins Pipeline
                              ├── Build → Container Image → ECR
                              ├── Test → Unit + Integration
                              ├── Scan → Trivy / ECR Scanning
                              └── Update → Image tag in GitOps repo
                                            ↓
                              ArgoCD detects change → Deploys to EKS
Best of both worlds: Jenkins handles CI (build, test, scan) while ArgoCD owns CD (deploy, reconcile, rollback) — no cluster credentials leave the cluster.
This hybrid pattern is rapidly becoming the industry standard. Jenkins excels at running complex build and test pipelines, but giving Jenkins direct kubectl access to production clusters is a security concern. In the hybrid model, the final Jenkins stage simply updates an image tag in the GitOps repository via a Git commit. ArgoCD, running inside the cluster, detects the change and performs the actual deployment. This gives you Jenkins's mature CI ecosystem plus GitOps security and drift detection. Many Verisign teams already running Jenkins can adopt this pattern incrementally without replacing their existing CI investment.

Git as Single Source of Truth

Repository Structure Patterns

gitops-repo/
├── apps/
│   ├── frontend/
│   │   ├── base/          # deployment.yaml, service.yaml, kustomization.yaml
│   │   └── overlays/      # dev/, staging/, production/
│   └── backend/
│       ├── base/
│       └── overlays/
├── infrastructure/        # cert-manager/, ingress-nginx/, monitoring/
└── clusters/              # dev-cluster/, staging-cluster/, prod-cluster/
Every change goes through a pull request — code review, approval, and audit trail are built into the workflow.
This mono-repo structure is one common pattern. Applications live under apps/ with Kustomize base/overlay structure. Shared infrastructure components get their own directory. Cluster-level configuration defines which apps and infra components are deployed where. Some teams prefer separate repos for apps and infrastructure. The key principle: if it's not in Git, it shouldn't be in the cluster.

🛡️ ArgoCD

Declarative, GitOps continuous delivery for Kubernetes

ArgoCD is the most widely adopted GitOps tool in the CNCF ecosystem. It's a graduated CNCF project with a rich UI, CLI, and API. Let's dive into its architecture and capabilities.

ArgoCD Architecture

API Server

  • Web UI dashboard
  • gRPC/REST API
  • RBAC enforcement
  • SSO integration (OIDC, SAML)

Repo Server

  • Clones Git repositories
  • Renders manifests (Helm, Kustomize)
  • Caches rendered output
  • Stateless & scalable

Application Controller

  • Watches live cluster state
  • Compares desired vs actual
  • Executes sync operations
  • Reports health status
Tip: Additional components include Redis (caching), Dex (SSO), ApplicationSet Controller (multi-cluster), and Notifications Controller (alerts).
The three core components work together in a loop. The repo server clones and renders manifests from Git. The application controller compares the rendered manifests against what's actually running in the cluster. When differences are detected, the controller can automatically sync or wait for manual approval depending on the sync policy. The API server provides the UI and access control layer.

ArgoCD Application CRD

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: frontend
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/verisign/gitops-repo.git
    targetRevision: main
    path: apps/frontend/overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: frontend
  syncPolicy:
    automated: { selfHeal: true, prune: true }
    syncOptions: [CreateNamespace=true]
    retry:
      limit: 5
      backoff: { duration: 5s, maxDuration: 3m0s, factor: 2 }
Source + Destination + SyncPolicy = the three pillars of every ArgoCD Application definition.
The Application CRD is the core abstraction. Source defines where manifests live in Git - the repo, branch, and path. Destination defines where to deploy - which cluster and namespace. SyncPolicy controls automation behavior. The retry block handles transient failures during sync. CreateNamespace ensures the target namespace exists. This single resource encapsulates an entire application's deployment configuration.

Sync Policies

🔄 Automated Sync

  • Auto-deploy on Git commit
  • No manual approval needed
  • Ideal for dev/staging
  • Optional for production

🚑 Self-Heal

  • Revert manual kubectl changes
  • Detect and fix drift
  • Ensures Git remains truth
  • Prevents configuration drift

🗑️ Prune

  • Delete resources removed from Git
  • Prevents orphaned resources
  • Use with caution in prod
  • Combine with Foreground deletion
Tip: Production recommendation — Enable selfHeal always. Enable automated sync only after establishing confidence. Consider manual sync approval for critical workloads.
These three policies control how aggressively ArgoCD reconciles state. Automated sync means any Git commit triggers deployment. Self-heal reverts manual changes made directly to the cluster. Prune removes resources that no longer exist in Git. In production, many teams start with self-heal enabled but automated sync disabled, requiring manual sync approval through the UI or CLI for an additional safety gate.

Multi-Environment Management

Kustomize Overlays

# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources: [../../base]
patchesStrategicMerge:
  - replica-patch.yaml
  - resource-patch.yaml
images:
  - name: myapp
    newTag: v2.1.0

ApplicationSet

apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata: { name: frontend }
spec:
  generators:
    - list:
        elements:
          - { cluster: dev, url: https://dev.k8s.local }
          - { cluster: staging, url: https://staging.k8s.local }
          - { cluster: prod, url: https://prod.k8s.local }
  template:
    spec:
      source:
        path: apps/frontend/overlays/{{cluster}}
Kustomize overlays handle per-environment differences like replica counts, resource limits, and image tags. ApplicationSets take this further by generating ArgoCD Application resources from a template. The list generator is the simplest, but you can also use Git directory generators, cluster generators, and pull request generators. This is how large organizations manage hundreds of applications across dozens of clusters.

🌸 FluxCD Overview

A toolkit approach to GitOps on Kubernetes

FluxCD takes a different architectural approach than ArgoCD. It's composed of specialized controllers rather than a monolithic application. Let's understand its design and when to choose it.

FluxCD Architecture

Toolkit of Specialized Controllers

Source Controller

  • Fetches from Git, Helm repos, S3
  • Produces versioned artifacts
  • Handles authentication

Kustomize Controller

  • Applies Kustomize overlays
  • Handles plain YAML too
  • Dependency ordering

Helm Controller

  • Manages HelmRelease CRDs
  • Automated upgrades/rollbacks
  • Values from ConfigMaps/Secrets
Additional controllers: Notification Controller (alerts & webhooks), Image Automation Controller (auto-update image tags in Git)
Flux's modular architecture means you only install what you need. The source controller is the foundation, fetching artifacts from various sources. The kustomize controller handles applying manifests. The helm controller manages Helm releases declaratively. Each controller watches its own CRDs and operates independently. The image automation controller is unique to Flux - it can automatically update image tags in your Git repo when new images are pushed to a registry.

FluxCD GitRepository & Kustomization

GitRepository

apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata: { name: app-repo, namespace: flux-system }
spec:
  interval: 1m
  url: https://github.com/verisign/gitops-repo
  ref: { branch: main }
  secretRef: { name: git-credentials }

Kustomization

apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata: { name: frontend, namespace: flux-system }
spec:
  interval: 5m
  sourceRef: { kind: GitRepository, name: app-repo }
  path: ./apps/frontend/overlays/prod
  prune: true
  healthChecks:
    - kind: Deployment
      name: frontend
      namespace: frontend
The GitRepository CRD tells the source controller where to fetch manifests and how often to check for changes. The Kustomization CRD (not to be confused with native Kustomize) tells the kustomize controller which path to apply and includes health checks to verify the deployment succeeded. The interval fields control polling frequency. The prune flag works like ArgoCD's prune - removing resources deleted from Git.

FluxCD vs ArgoCD Comparison

Feature ArgoCD FluxCD
Architecture Monolithic application Toolkit of controllers
UI Rich web UI built-in No built-in UI (Weave GitOps optional)
Multi-tenancy Projects & RBAC Namespace-scoped CRDs
Multi-cluster ApplicationSets Kustomization per cluster
Image automation Argo Image Updater (separate) Built-in image automation
CNCF Status Graduated Graduated
Both are excellent choices. ArgoCD excels with its UI and multi-cluster management. FluxCD excels with its lightweight, composable design and native image automation.
This is not a winner-takes-all comparison. ArgoCD is often preferred when teams want a visual dashboard and centralized multi-cluster management. FluxCD is preferred when teams want a lighter footprint and CLI-first workflow. Both are CNCF graduated projects with strong communities. Some organizations even use both - ArgoCD for application deployments and Flux for cluster infrastructure bootstrapping.

🛠️ FluxCD in Practice

Bootstrap, HelmReleases, Image Automation & Multi-Tenancy

Now let's go deeper into FluxCD with practical patterns you'll use day-to-day. We'll cover bootstrapping a cluster, managing Helm releases declaratively, automating image updates, and isolating teams with multi-tenancy.

🚀 Flux Bootstrap & Repo Structure

# Bootstrap Flux on a new cluster
flux bootstrap github \
  --owner=verisign \
  --repository=fleet-infra \
  --branch=main \
  --path=clusters/production \
  --personal=false

Recommended Repository Structure

fleet-infra/
├── clusters/
│   ├── production/
│   │   ├── flux-system/     # Auto-generated
│   │   ├── infrastructure.yaml
│   │   └── applications.yaml
│   └── staging/
├── infrastructure/    # Shared infra components
└── applications/      # Application manifests
Bootstrap is a one-time operation that installs Flux controllers and commits their manifests to Git, making Flux self-managing. The recommended repo structure separates cluster-specific config from shared components. Each cluster directory contains Kustomization resources referencing shared infrastructure and application directories. This pattern scales to dozens of clusters.

📜 HelmRelease CRD

apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata: { name: nginx-ingress, namespace: infrastructure }
spec:
  interval: 15m
  chart:
    spec:
      chart: ingress-nginx
      version: "4.x"  # Semver range
      sourceRef:
        kind: HelmRepository
        name: ingress-nginx
  values:
    controller:
      replicaCount: 3
      service: { type: LoadBalancer }
  upgrade: { remediation: { retries: 3 } }
  rollback: { cleanupOnFail: true }
  • version — semver ranges for auto minor/patch upgrades
  • upgrade.remediation — auto-retry or rollback on failure
  • valuesFrom — inject secrets from ConfigMaps/Secrets without storing in Git
The HelmRelease CRD replaces imperative helm install/upgrade commands with a declarative spec. Semver ranges let you automatically pick up patch releases while staying within a major version. The remediation settings define what happens when an upgrade fails, including automatic rollback after configurable retries. Use valuesFrom to reference Secrets managed by external-secrets or sealed-secrets for sensitive values.

📷 Image Automation Pipeline

# 1. Scan registry for new tags
kind: ImageRepository
spec:
  image: ghcr.io/verisign/my-app
  interval: 5m
---
# 2. Select latest matching tag
kind: ImagePolicy
spec:
  imageRepositoryRef: { name: app-image }
  policy: { semver: { range: ">=1.0.0 <2.0.0" } }
---
# 3. Write selected tag back to Git
kind: ImageUpdateAutomation
spec:
  sourceRef: { kind: GitRepository, name: app-manifests }
  update: { path: ./clusters/production, strategy: Setters }
Closed loop: CI builds image → Flux detects new tag → updates manifest in Git → deploys automatically.
Image automation is a three-step pipeline: ImageRepository scans a container registry, ImagePolicy selects the latest tag matching a semver range, and ImageUpdateAutomation writes the selected tag back to Git using setter markers. This closes the CI/CD loop entirely -- when CI builds a new image, Flux detects it and updates the deployment manifest automatically without any manual intervention.

🏢 Multi-Tenancy & Alerts

Tenant Isolation

kind: Kustomization
spec:
  serviceAccountName: team-alpha-sa
  sourceRef:
    kind: GitRepository
    name: team-alpha-repo
  path: ./deploy
  targetNamespace: team-alpha
  prune: true
  • serviceAccountName — RBAC-scoped SA per team
  • targetNamespace — force resources into one NS

Notification Alerts

kind: Provider
spec:
  type: slack
  channel: k8s-deployments
  secretRef: { name: slack-url }
---
kind: Alert
spec:
  providerRef: { name: slack-alerts }
  eventSeverity: error
  eventSources:
    - { kind: HelmRelease, name: "*" }
Monitor: Alert when gotk_reconcile_condition{status="False"} persists > 10 min. Grafana dashboard ID: 16714.
Multi-tenancy in Flux scopes each team's Kustomization to a service account with limited RBAC. The targetNamespace override prevents cross-namespace deployments. For observability, the notification controller routes Flux events to Slack, PagerDuty, or Teams. Combine with Prometheus metrics -- gotk_reconcile_condition tells you whether each resource is Ready, Stalled, or Suspended. The official Grafana dashboard 16714 provides a complete overview of GitOps health.

🔧 Pipeline Tools

CI platforms, image scanning, and policy enforcement

Now let's look at the CI side of the equation. These are the tools that build, test, scan, and prepare artifacts before they enter the GitOps workflow.

GitHub Actions for Kubernetes

name: CI Pipeline
on: { push: { branches: [main] } }
jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/github-actions
          aws-region: us-east-1
      - name: Login to ECR
        uses: aws-actions/amazon-ecr-login@v2
      - name: Build, scan, and push
        run: |
          docker build -t $ECR_REGISTRY/myapp:${{ github.sha }} .
          trivy image --exit-code 1 --severity HIGH,CRITICAL \
            $ECR_REGISTRY/myapp:${{ github.sha }}
          docker push $ECR_REGISTRY/myapp:${{ github.sha }}
      # ... Update GitOps repo: kustomize edit set image, git commit & push
This workflow demonstrates the CI portion of a GitOps pipeline. Notice the final step updates the GitOps repository with the new image tag rather than deploying directly to the cluster. OIDC-based authentication with AWS eliminates long-lived credentials. The trivy scan fails the pipeline if critical vulnerabilities are found. ArgoCD or FluxCD then picks up the Git change and deploys it.

Tekton Pipelines

  • Kubernetes-native CI/CD framework
  • Pipelines run as Pods in the cluster
  • CRDs: Task, Pipeline, PipelineRun
  • Reusable task catalog (Tekton Hub)

kind: Task

Jenkins X

  • Opinionated CI/CD for Kubernetes
  • Built on Tekton underneath
  • Automated environment promotion
  • Preview environments per PR
  • GitOps-native from the start
Tip: Kaniko builds container images inside Kubernetes without requiring Docker daemon access.
Tekton is a powerful option when you want your CI pipelines to run inside Kubernetes itself. Each step runs as a container in a Pod, which means you get all the benefits of Kubernetes scheduling, resource limits, and RBAC. Kaniko is often paired with Tekton since it can build Docker images without the Docker daemon - essential in unprivileged Pod environments. Jenkins X wraps Tekton with higher-level automation and environment promotion workflows.

Image Scanning in CI

Trivy

  • Open-source (Aqua Security)
  • OS packages + language deps
  • IaC scanning + SBOM generation
# Fail pipeline on critical CVEs
trivy image --exit-code 1 \
  --severity CRITICAL \
  --ignore-unfixed myapp:v1.2.3

# Generate SBOM
trivy image --format spdx-json \
  -o sbom.json myapp:v1.2.3

Snyk

  • Commercial with free tier
  • Deep language dependency analysis
  • Fix suggestions and PR automation
# Scan container image
snyk container test myapp:v1.2.3

# Monitor for new vulnerabilities
snyk container monitor \
  myapp:v1.2.3 --org=verisign
Tip: Never deploy unscanned images to production. Integrate scanning as a mandatory pipeline gate, not an optional step.
Image scanning should be a non-negotiable gate in your CI pipeline. Trivy is the most popular open-source option and covers OS packages, language dependencies, and even infrastructure-as-code misconfigurations. Snyk offers deeper language analysis and automated fix PRs. SBOM generation is increasingly required for compliance. The --ignore-unfixed flag in Trivy is useful to avoid blocking on CVEs that don't have patches yet.

Policy as Code

OPA Gatekeeper

  • Admission controller (Rego language)
  • ConstraintTemplates + Constraints
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata: { name: require-team-label }
spec:
  match:
    kinds: [{ apiGroups: ["apps"], kinds: ["Deployment"] }]
  parameters:
    labels: ["team", "cost-center"]

Kyverno

  • Kubernetes-native (YAML, no Rego)
  • Validate, mutate, generate resources
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata: { name: disallow-latest-tag }
spec:
  validationFailureAction: Enforce
  rules:
    - name: require-image-tag
      match: { resources: { kinds: ["Pod"] } }
      validate:
        message: "Must use a specific tag"
        pattern:
          spec:
            containers:
              - image: "!*:latest"
Policy as code completes the security story. These tools act as admission controllers, intercepting API requests before resources are created. Gatekeeper uses Rego, which is powerful but has a learning curve. Kyverno uses native YAML, making it more accessible. Common policies include requiring labels, disallowing privileged containers, enforcing resource limits, and blocking the latest tag. In a GitOps workflow, these policies ensure that even if something wrong gets into Git, the cluster rejects it.

🎯 Production Patterns

Strategies for reliable, auditable delivery at scale

Let's bring everything together with production patterns. These are the strategies teams use to manage environment promotion, secrets, and rollbacks in real-world GitOps workflows.

Environment Promotion Strategies

Promotion Flow

Dev ⟶ Staging ⟶ Pre-prod ⟶ Production

Branch-Based Promotion

  • Each environment maps to a branch
  • PR from dev → staging → main
  • Simple to understand
  • Merge conflicts can be challenging

Directory-Based Promotion

  • Single branch, overlay directories
  • PR updates overlay path (e.g., image tag)
  • No branch divergence
  • Promoted via automated PRs
Recommended: Directory-based with Kustomize overlays on a single branch. Use automated PRs to promote image tags between environments.
Branch-based promotion is intuitive but leads to merge conflicts and branch divergence over time. Directory-based promotion using Kustomize overlays on a single branch is the recommended approach. Promotion happens by updating the image tag in the next environment's overlay, typically via an automated PR created by the CI pipeline or by a tool like Flux's image automation controller. This keeps the audit trail clean and avoids long-lived branch management.

Secrets Management in CI/CD

Sealed Secrets

  • Encrypt with public key
  • Decrypt only in-cluster
  • Safe to commit to Git
kubeseal --format yaml \
  < secret.yaml \
  > sealed-secret.yaml

External Secrets Operator

  • Sync from AWS SM / Vault
  • Auto-rotation support
kind: ExternalSecret
spec:
  refreshInterval: 1h
  secretStoreRef: { name: aws-secrets }
  target: { name: db-creds }

SOPS

  • Encrypts values, not keys
  • KMS / PGP / age encryption
sops --encrypt \
  --kms arn:aws:kms:... \
  secrets.yaml > \
  secrets.enc.yaml
Tip: Never store plain-text secrets in Git. Even in private repositories, secrets must be encrypted or externally referenced.
Secrets are the hardest problem in GitOps. Sealed Secrets encrypts the entire secret with a cluster-specific key. External Secrets Operator syncs from external secret stores like AWS Secrets Manager or Vault, keeping secrets completely out of Git. SOPS encrypts individual values within a YAML file while leaving keys readable. For AWS environments, External Secrets Operator with AWS Secrets Manager is typically the best choice because it supports rotation and centralized management.

Rollback Strategies

Git Revert (Recommended)

  • git revert <commit> creates a new commit
  • Preserves complete history
  • ArgoCD/Flux auto-syncs the revert
# Revert a bad deployment
git revert abc123
git push origin main

ArgoCD Manual Rollback

# ArgoCD CLI rollback
argocd app history frontend
argocd app rollback frontend 3
GitOps rollback = Git revert. Forward-only commit history provides a clear, auditable record.
In GitOps, the preferred rollback mechanism is git revert because it maintains a forward-only history. Never use git reset or force push since that destroys audit history. ArgoCD's manual rollback is useful for emergencies but it's temporary - the next sync cycle will revert to whatever is in Git. This reinforces the principle that Git is always the source of truth. The kubectl rollout undo command still works but creates drift that ArgoCD will detect and correct if self-heal is enabled.

CI/CD Security Best Practices

Pipeline Security

  • Use OIDC for cloud auth (no long-lived keys)
  • Pin action versions by SHA, not tag
  • Restrict self-hosted runner permissions
  • Require signed commits in GitOps repos
  • Separate CI and CD credentials

Supply Chain Security

  • Sign images with Cosign / Sigstore
  • Generate and verify SBOMs
  • Use admission policies to verify signatures
  • Implement SLSA provenance
  • Audit base image dependencies
Defense in depth: Scan in CI, enforce policies at admission, monitor at runtime. No single gate is sufficient.
Supply chain security has become critical after high-profile attacks. OIDC eliminates static credentials in CI systems. Pinning actions by SHA prevents supply chain attacks through compromised GitHub Actions. Cosign and SLSA provenance create a verifiable chain from source code to running container. Defense in depth means layering security at every stage - scanning in CI, admission policies at the API server, and runtime monitoring with tools like Falco.

Putting It All Together

Complete GitOps Pipeline

1. Developer pushes code to app repo
2. CI pipeline: build → test → scan → sign → push image
3. CI updates image tag in GitOps repo (automated PR)
4. PR reviewed and merged
5. ArgoCD/Flux detects change, syncs to cluster
6. Health checks verify deployment
7. Notification sent (Slack, PagerDuty)

Benefits achieved: Declarative, auditable, reproducible, self-healing, secure
Key repos: Application repo (source code) + GitOps repo (manifests) = clean separation of concerns
This is the complete picture. Two repositories - one for application source code, one for deployment manifests. CI handles the artifact lifecycle, and the only output is an updated image tag in the GitOps repo. The GitOps agent handles deployment, drift detection, and self-healing. Every change is auditable through Git history. This pattern scales from a single application to hundreds across multiple clusters.

📋 Module Summary

  • ArgoCD and FluxCD reconcile cluster state with desired state from Git
  • Directory-based promotion with Kustomize overlays avoids branch divergence
  • Secrets management in GitOps requires Sealed Secrets, ESO, or SOPS
  • Git revert is the preferred rollback mechanism preserving audit history
  • Supply chain security layers scanning, signing, and admission policies

🎯 Key Takeaways

Remember These Points
  1. Separate application repos from GitOps manifest repos for clean separation of concerns
  2. Enable drift detection and self-healing so the cluster always matches Git
  3. Use OIDC for CI authentication and pin action versions by SHA to prevent supply chain attacks
