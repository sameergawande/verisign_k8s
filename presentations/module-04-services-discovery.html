🔌 Why Services Matter

Stable networking over ephemeral Pods

Let's start with the fundamental problem. Pods are ephemeral — they get created, destroyed, and rescheduled. Each time, they receive a new IP address. Services provide a stable abstraction layer on top of this dynamic infrastructure.

The Pod Networking Problem

Pod IPs are ephemeral. Every time a Pod restarts, it receives a new IP address from the CNI plugin.
  • Pods are mortal — Deployments replace them on failure or scaling events
  • ReplicaSets create multiple Pods for the same workload — which IP do clients use?
  • Rolling updates replace Pods one by one with new IPs
  • Nodes can fail, causing Pods to reschedule on different nodes
  • Hard-coding IPs is fragile and defeats the purpose of orchestration
Key insight: You need a stable endpoint that abstracts over a dynamic set of Pods.
Every time a Pod is recreated, it gets a brand new IP. If you have 3 replicas, that's 3 different IPs. During a rolling update, those IPs change. You can't hard-code them. This is the exact problem that Services solve.

The Service Abstraction

What a Service Provides

  • Stable virtual IP (ClusterIP) that never changes
  • Stable DNS name that resolves to the virtual IP
  • Load balancing across healthy backend Pods
  • Automatic endpoint updates as Pods come and go
  • Decoupling between producers and consumers
Result: Clients connect to a Service name or IP — Kubernetes handles routing to the right Pod.
A Service gives you a virtual IP and DNS name that persist for the lifetime of the Service object. Behind the scenes, Kubernetes watches for Pod changes and keeps the routing rules updated. Clients never need to know about individual Pod IPs.

Service Anatomy

apiVersion: v1
kind: Service
metadata:
  name: my-app-service
  namespace: production
  labels: { app: my-app, tier: frontend }
spec:
  type: ClusterIP             # Default type
  selector:
    app: my-app               # Matches Pod labels
    version: v2
  ports:
    - name: http
      port: 80                # Service port (what clients use)
      targetPort: 8080        # Pod port (where app listens)
    - name: metrics
      port: 9090
      targetPort: 9090
Here's the anatomy of a Service YAML. The selector field is critical — it defines which Pods this Service routes to by matching labels. The port is what clients connect to, and targetPort is where your application actually listens inside the Pod. You can also use named ports for targetPort to decouple from specific port numbers.

ClusterIP Service

Default Service Type

  • Allocates a virtual IP from the Service CIDR
  • Accessible only within the cluster
  • kube-proxy programs iptables/IPVS rules
  • Round-robin load balancing by default
  • Most common type for internal communication
apiVersion: v1
kind: Service
metadata:
  name: backend-api
spec:
  type: ClusterIP
  selector:
    app: backend
    tier: api
  ports:
    - port: 80
      targetPort: 3000
Tip: ClusterIP is the default — use for microservice-to-microservice communication inside the cluster. It allocates a virtual IP from the cluster's Service CIDR range, reachable only within the cluster. kube-proxy sets up iptables or IPVS rules to distribute traffic.

NodePort Service

Extends ClusterIP

  • Opens a static port (30000–32767) on every node
  • Still creates a ClusterIP internally
  • External traffic: <NodeIP>:<NodePort>
  • kube-proxy routes to backend Pods
  • Port range configurable via API server flag
apiVersion: v1
kind: Service
metadata:
  name: web-frontend
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080  # Optional
Tip: NodePort requires managing node IPs and firewall rules — rarely used directly in production. It extends ClusterIP by opening a port on every node. Useful for development and testing, but in production you'd typically put a load balancer in front.

LoadBalancer Service

Cloud-Integrated Load Balancing

  • Extends NodePort — provisions an external load balancer via cloud provider
  • On AWS EKS: creates a Classic LB, NLB, or ALB depending on annotations
  • Assigns a public or internal DNS name to the load balancer
  • Each LoadBalancer Service gets its own cloud LB (cost consideration)
apiVersion: v1
kind: Service
metadata:
  name: public-api
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
spec:
  type: LoadBalancer
  selector:
    app: public-api
  ports:
    - port: 443
      targetPort: 8443
LoadBalancer extends NodePort by asking the cloud provider to create an actual load balancer. On AWS, by default you get a Classic Load Balancer, but with the right annotations you can get an NLB or ALB. Be aware that each LoadBalancer Service provisions a separate cloud resource — this can get expensive if you have many services. That's where Ingress comes in, which we cover in Module 6.

ExternalName Service

DNS Alias (CNAME)

  • Maps a Service name to an external DNS name
  • Returns a CNAME record, not a ClusterIP
  • No selectors or endpoints
  • No proxying — DNS resolution only
  • Useful for referencing external databases or APIs
apiVersion: v1
kind: Service
metadata:
  name: external-db
  namespace: production
spec:
  type: ExternalName
  externalName: mydb.us-east-1.rds.amazonaws.com
Tip: Applications use external-db.production.svc.cluster.local — migration from external to in-cluster becomes a Service type change. ExternalName simply returns a CNAME record pointing to the external DNS name. No proxying involved.

📈 Service Type Comparison

Type Scope Access Method Use Case
ClusterIP Internal only Virtual IP / DNS Inter-service communication
NodePort Internal + external NodeIP:Port Development, bare-metal
LoadBalancer Internal + external Cloud LB DNS Production external access
ExternalName DNS alias CNAME record External service abstraction
Remember: Each type builds on the previous — LoadBalancer ⊃ NodePort ⊃ ClusterIP.
This table summarizes the four Service types. The key insight is that they build on each other: LoadBalancer creates a NodePort, which creates a ClusterIP. ExternalName is the exception — it's a pure DNS alias with no proxying. In production EKS environments, you'll use ClusterIP for internal services and Ingress with an ALB for external services.

Selectors & Label Matching

# Service selector
spec:
  selector:
    app: payment-service
    tier: backend

# Pod labels (matches)
metadata:
  labels:
    app: payment-service
    tier: backend
    version: v3
    team: payments

Matching Rules

  • Selector uses equality-based matching
  • Pod must have all selector labels
  • Pod can have additional labels
  • Labels must match in the same namespace
  • Services are namespace-scoped
Tip: Typos in labels silently cause zero endpoints — always verify with kubectl get endpoints. Service selectors use equality-based matching; every label in the selector must exist on the Pod.

Endpoints & EndpointSlices

Endpoints (Legacy)

  • Single object per Service listing all Pod IPs
  • Scales poorly beyond ~1000 endpoints
  • Any Pod change rewrites the entire object
kubectl get endpoints my-svc
# NAME     ENDPOINTS
# my-svc   10.0.1.5:8080,10.0.2.3:8080

EndpointSlices (Modern)

  • Splits endpoints into chunks (max 100 per slice)
  • More efficient updates at scale
  • Default since Kubernetes 1.21
  • Supports dual-stack (IPv4 + IPv6)
kubectl get endpointslices -l \
  kubernetes.io/service-name=my-svc
When a Service is created, Kubernetes automatically creates Endpoints or EndpointSlice objects that track which Pods match the selector. Legacy Endpoints store everything in a single object, which becomes a scalability bottleneck. EndpointSlices chunk the data into pieces of up to 100 endpoints each, making updates far more efficient. EndpointSlices are the default in modern Kubernetes.

🔍 Service Discovery

How applications find each other in the cluster

Now let's look at how applications actually discover and connect to Services. Kubernetes provides two primary mechanisms: DNS-based discovery via CoreDNS, and environment variable injection. DNS is the recommended approach for almost all use cases.

DNS-Based Discovery (CoreDNS)

How It Works

  • CoreDNS runs as a Deployment in kube-system namespace
  • Every Service gets a DNS record automatically
  • Pods are configured to use CoreDNS via /etc/resolv.conf
  • CoreDNS watches the Kubernetes API for Service/Endpoint changes
  • Supports A, AAAA, SRV, and CNAME records
In practice: Applications simply connect to the Service name — DNS handles the rest.
CoreDNS is the standard DNS server in Kubernetes clusters. It runs as a Deployment in kube-system and watches the API server for changes. When a Service is created, CoreDNS automatically creates DNS records for it. Every Pod in the cluster has its resolv.conf configured to use CoreDNS, so applications can resolve Service names without any additional setup.

DNS Record Formats

Fully Qualified Domain Names

# Service A record
<service-name>.<namespace>.svc.cluster.local

# Examples
backend-api.production.svc.cluster.local   → 10.96.0.15
redis.cache.svc.cluster.local              → 10.96.1.42

# SRV records (for port discovery)
_http._tcp.backend-api.production.svc.cluster.local

# Pod DNS (when hostname/subdomain set)
<pod-hostname>.<service-name>.<namespace>.svc.cluster.local
Shorthand works within the same namespace: Just use backend-api instead of the full FQDN. Cross-namespace: use backend-api.production.
Every Service gets an A record in the format service-name.namespace.svc.cluster.local. SRV records encode port information as well. The search domains in resolv.conf allow you to use short names within the same namespace. For cross-namespace access, you need at least service.namespace. Using the full FQDN with a trailing dot avoids unnecessary DNS search iterations.

Environment Variable Discovery

Automatic Injection

  • kubelet injects env vars for each active Service
  • Format: {SVC_NAME}_SERVICE_HOST and {SVC_NAME}_SERVICE_PORT
  • Only Services that exist before the Pod starts
  • Same namespace only
Limitation: Order-dependent — Service must exist before the Pod. DNS has no such constraint. 
# For a Service named "backend-api"
# on port 80:
BACKEND_API_SERVICE_HOST=10.96.0.15
BACKEND_API_SERVICE_PORT=80
BACKEND_API_PORT=tcp://10.96.0.15:80
BACKEND_API_PORT_80_TCP=tcp://10.96.0.15:80
BACKEND_API_PORT_80_TCP_PROTO=tcp
BACKEND_API_PORT_80_TCP_PORT=80
BACKEND_API_PORT_80_TCP_ADDR=10.96.0.15
The environment variable approach is the older mechanism inherited from Docker links. kubelet automatically injects environment variables for every Service that exists at the time the Pod starts. The major limitation is ordering — if the Service doesn't exist yet, the Pod won't get those variables. DNS-based discovery doesn't have this problem, which is why it's the recommended approach.

Headless Services

clusterIP: None

  • No virtual IP allocated
  • DNS returns all Pod IPs directly
  • Client-side service discovery
  • Essential for StatefulSets
  • Used by databases, message queues, ZooKeeper
apiVersion: v1
kind: Service
metadata: { name: cassandra }
spec:
  clusterIP: None      # Makes it headless
  selector: { app: cassandra }
  ports:
    - port: 9042
# DNS returns all Pod IPs directly
$ nslookup cassandra.default.svc
# 10.244.1.5, 10.244.2.3, 10.244.3.8
Headless Services set clusterIP to None. Instead of getting a virtual IP, DNS queries return the individual Pod IPs. This is critical for stateful applications where clients need to connect to specific instances — like Cassandra nodes, Kafka brokers, or ZooKeeper members. StatefulSets rely on headless Services to create stable DNS names for each Pod.

📦 kube-proxy & Traffic Flow

Understanding the data plane

Now let's look under the hood at how traffic actually gets from a client to a backend Pod. kube-proxy is the component responsible for implementing the Service abstraction at the network level. Understanding its modes helps with debugging and performance tuning.

kube-proxy Modes

iptables Mode (Default)

  • Programs iptables NAT rules
  • Random backend selection
  • O(n) rule evaluation
  • Proven, widely deployed
  • Performance degrades at >5000 Services

IPVS Mode

  • Uses Linux IPVS (kernel-level L4 LB)
  • O(1) connection routing with hash tables
  • Multiple algorithms: rr, lc, sh, dh
  • Better performance at scale
  • Requires IPVS kernel modules
EKS default: iptables mode. Consider IPVS for clusters with thousands of Services.
kube-proxy has two primary modes. iptables mode is the default — it programs NAT rules in the Linux kernel's netfilter framework. It works well for most clusters but performance degrades with thousands of Services because every packet traverses a chain of rules. IPVS mode uses kernel-level load balancing with hash tables for O(1) lookups, making it better at scale. EKS uses iptables by default, but you can switch to IPVS if needed.

Traffic Policies

Cluster (Default)

spec:
  internalTrafficPolicy: Cluster
  externalTrafficPolicy: Cluster
  • Traffic can route to any Pod in the cluster
  • Even load distribution
  • Extra network hop if Pod is on another node
  • Source IP is SNATed (masked)

Local

spec:
  internalTrafficPolicy: Local
  externalTrafficPolicy: Local
  • Traffic only routes to Pods on the same node
  • Preserves source IP
  • Avoids extra hop (lower latency)
  • Risk of uneven distribution
Traffic policy controls where kube-proxy sends traffic. Cluster policy distributes evenly across all Pods but adds an extra network hop and loses the client source IP via SNAT. Local policy keeps traffic on the same node, preserving source IP and reducing latency — but if Pods aren't evenly spread across nodes, some Pods get more traffic than others. Choose based on your requirements for source IP preservation and even load distribution.

Session Affinity

Sticky Sessions

  • Routes requests from the same client IP to the same Pod
  • Configured via sessionAffinity: ClientIP
  • Default timeout: 10800s (3 hours)
  • Only supports ClientIP — no cookie-based affinity at L4
Best practice: Design stateless applications. Use session affinity only when legacy requirements demand it. 
apiVersion: v1
kind: Service
metadata:
  name: legacy-app
spec:
  type: ClusterIP
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 1800
  selector:
    app: legacy-app
  ports:
    - port: 80
      targetPort: 8080
Session affinity lets you pin a client IP to a specific backend Pod. This is a Layer 4 feature only — it uses client IP, not cookies. The default timeout is 3 hours. While it exists for legacy compatibility, modern applications should be stateless and use external session stores like Redis or DynamoDB. For cookie-based session affinity, you'd use an Ingress controller, which we cover in Module 6.

🚀 Advanced Service Patterns

Production-ready configurations

Now that we understand the fundamentals, let's explore advanced patterns you'll encounter in production EKS environments. These cover multi-port services, external traffic policy implications, topology-aware routing, and service mesh concepts.

Multi-Port Services

apiVersion: v1
kind: Service
metadata: { name: full-stack-app }
spec:
  selector: { app: full-stack }
  ports:
    - { name: http,    port: 80,    targetPort: 8080 }
    - { name: https,   port: 443,   targetPort: 8443 }
    - { name: metrics, port: 9090,  targetPort: metrics }
    - { name: grpc,    port: 50051, targetPort: grpc-port }

Rules for Multi-Port

  • Each port must have a unique name
  • targetPort can reference a named port on the Pod
  • SRV records created for each named port
Named ports: Using named targetPort values decouples the Service from specific port numbers — Pods can change ports independently.
Multi-port Services let you expose several ports through a single Service. When you define multiple ports, each one must have a name. Using named targetPorts is a best practice — it means the Pod's container port can change without updating the Service definition. DNS SRV records are created for each named port, allowing clients to discover both the address and port.

ExternalTrafficPolicy Deep Dive

Cluster vs Local — Trade-offs in Practice

Aspect Cluster Local
Load Distribution Even across all Pods Uneven if Pods ≠ nodes
Source IP SNATed (lost) Preserved
Health Checks N/A LB health checks node:healthCheckNodePort
Network Hops Possible extra hop No extra hop
Failure Mode Any Pod can serve Nodes without Pods get no traffic
Verisign recommendation: Use externalTrafficPolicy: Local when source IP preservation is required for security logging or geo-routing.
This is an important production consideration. With Cluster policy, the load balancer distributes evenly but source IPs get masked by SNAT. With Local policy, source IPs are preserved and there's no extra network hop, but nodes without Pods for that Service will fail health checks and receive no traffic. This means uneven load distribution if your Pods aren't evenly spread. Choose Local when you need source IP for audit logging, security, or geographic routing.

Topology-Aware Routing

Keep Traffic Local

  • Prefers routing to Pods in the same AZ
  • Reduces cross-AZ data transfer costs
  • Lower latency for east-west traffic
  • Falls back to other zones if no local Pods
apiVersion: v1
kind: Service
metadata:
  name: data-service
  annotations:
    service.kubernetes.io/topology-mode: Auto
spec:
  selector:
    app: data-service
  ports:
    - port: 80
      targetPort: 8080
Tip: Cross-AZ traffic costs $0.01/GB — topology-aware routing can significantly reduce data transfer bills. Nodes must have topology.kubernetes.io/zone labels (auto-set on EKS). Be aware that uneven Pod distribution across zones can cause uneven load.

Service Mesh Concepts

What It Adds

  • mTLS encryption between services
  • Traffic management, canaries, and circuit breaking

Istio

  • Feature-rich Envoy sidecar (or ambient mode)
  • VirtualService / DestinationRule CRDs

Linkerd

  • Lightweight Rust-based proxy
  • Lower overhead, graduated CNCF project
When to adopt: Consider a service mesh when you need mTLS between all services, advanced traffic splitting, or deep observability across microservices.
A service mesh adds a proxy sidecar to every Pod, giving you features beyond what standard Kubernetes Services provide. Istio is the most feature-rich option, using Envoy proxies, while Linkerd is lighter weight. Both add mTLS, traffic management, and observability. The trade-off is complexity and resource overhead. Don't adopt a mesh unless you have a specific need — standard Services and Ingress cover most use cases.

☁️ AWS Load Balancer Controller

Production-Grade AWS Integration

  • Replaces the in-tree cloud provider for LB provisioning
  • Creates NLB for LoadBalancer Services, ALB for Ingress resources
  • Supports IP-mode targets (bypasses NodePort, routes directly to Pods)
  • Integrates with AWS WAF, Shield, and ACM for TLS
apiVersion: v1
kind: Service
metadata:
  name: production-api
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "external"
    service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
    service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
spec:
  type: LoadBalancer
  loadBalancerClass: service.k8s.aws/nlb
  selector: { app: production-api }
  ports:
    - port: 443
      targetPort: 8443
The AWS Load Balancer Controller is the recommended way to manage load balancers on EKS. It provisions NLBs for LoadBalancer Services and ALBs for Ingress resources. The key advantage is IP-mode targeting, which routes traffic directly to Pod IPs instead of going through NodePort — reducing hops and improving performance. It also integrates with AWS security services like WAF and Shield, and uses ACM for TLS certificate management.

NLB Target Modes on EKS

Instance Mode

  • NLB targets = EC2 instances
  • Traffic: NLB → NodePort → kube-proxy → Pod
  • Extra network hop through NodePort
  • Works with any CNI
annotations:
  service.beta.kubernetes.io/
    aws-load-balancer-nlb-target-type: "instance"

IP Mode (Recommended)

  • NLB targets = Pod IPs directly
  • Traffic: NLB → Pod (one hop)
  • Lower latency, better performance
  • Requires VPC CNI plugin
annotations:
  service.beta.kubernetes.io/
    aws-load-balancer-nlb-target-type: "ip"
EKS best practice: Use IP mode with the VPC CNI plugin for optimal performance and source IP preservation.
On EKS, you have two NLB target modes. Instance mode registers EC2 nodes as targets, so traffic goes through NodePort and kube-proxy before reaching your Pod — that's extra hops. IP mode registers Pod IPs directly as NLB targets, so traffic goes straight from the NLB to the Pod in a single hop. IP mode requires the VPC CNI plugin, which EKS uses by default. Always prefer IP mode for production workloads.

🔧 Debugging Services

Common Issues and Commands

# 1. Check Service exists and has endpoints (most common issue!)
kubectl get svc my-service -o wide
kubectl get endpoints my-service

# 2. Confirm Pod labels match the Service selector
kubectl get pods --show-labels -l app=my-app

# 3. Test DNS resolution from inside the cluster
kubectl run dns-test --rm -it --image=busybox -- nslookup my-service

# 4. Test connectivity from inside the cluster
kubectl run curl-test --rm -it --image=curlimages/curl -- \
  curl -v http://my-service:80

# 5. Check kube-proxy logs for errors
kubectl logs -n kube-system -l k8s-app=kube-proxy --tail=50
Debugging Services follows a systematic approach. Start by checking the Service exists and has a ClusterIP. Then verify endpoints are populated — empty endpoints is the number one issue and usually means a label mismatch. Check Pod labels against the Service selector. Test DNS from inside the cluster with a temporary Pod. Finally, check kube-proxy logs for any errors in programming iptables or IPVS rules.

🎯 Service Best Practices

✅ Do

  • Use DNS for service discovery
  • Use named ports in targetPort
  • Set externalTrafficPolicy: Local when source IP matters
  • Enable topology-aware routing in multi-AZ clusters
  • Use IP-mode NLB on EKS
  • Always define readiness probes

❌ Avoid

  • Hard-coding Pod IPs anywhere
  • Relying on env vars for discovery
  • Creating one LoadBalancer per service
  • Ignoring empty endpoints warnings
  • Using session affinity for new applications
  • Skipping readiness probes (sends traffic to unready Pods)
Here are the key best practices. Always use DNS over environment variables. Named targetPorts decouple Services from Pods. Set external traffic policy to Local when you need source IP. Enable topology-aware routing to save on cross-AZ costs. On EKS, always use IP-mode NLB targets. And critically, always define readiness probes — without them, Services will send traffic to Pods that aren't ready to handle requests.

🎯 Key Takeaways

1. Services are the networking primitive — They provide stable endpoints over dynamic Pod infrastructure. Never hard-code Pod IPs.
2. DNS is the standard discovery mechanism — Use service.namespace.svc.cluster.local or short names within the same namespace.
3. Choose the right traffic policy — Use Local for source IP preservation; use Cluster for even distribution.
4. Use AWS Load Balancer Controller in production — IP-mode NLB targets give you direct Pod routing with lower latency.
Four key takeaways. Services are the fundamental networking abstraction — always use them instead of direct Pod IPs. DNS via CoreDNS is the standard discovery mechanism. Choose your traffic policy based on whether you need source IP preservation or even distribution. And on EKS, always use the AWS Load Balancer Controller with IP-mode targets for the best performance.
