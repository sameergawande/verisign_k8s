⚙️ NGINX Ingress Controller

Architecture and IngressClass

NGINX Ingress Controller is the most widely deployed ingress controller. Let's understand its architecture before creating Ingress resources.

🏗️ NGINX Ingress Architecture

How It Works

  1. NGINX Ingress Controller pod runs NGINX reverse proxy
  2. Exposed via a LoadBalancer Service (single AWS NLB/ALB)
  3. Watches for Ingress resources across the cluster
  4. Dynamically generates NGINX configuration from Ingress rules
  5. Reloads NGINX when Ingress resources change
Tip: IngressClass specifies which controller handles an Ingress. Multiple controllers can coexist (e.g., internal vs external). Critical for Verisign where you might have separate internal and external ingress controllers.

🔧 IngressClass Configuration

apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx-external
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/ingress-nginx
---
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata: { name: nginx-internal }
spec:
  controller: k8s.io/ingress-nginx-internal
Production Pattern: Verisign runs separate IngressClasses for external (internet-facing) and internal (VPC-only) traffic, each backed by its own controller deployment and load balancer.
IngressClass is mandatory in modern Kubernetes. It decouples the Ingress resource from the controller. By setting is-default-class, any Ingress without an explicit ingressClassName will be picked up by that controller. Running separate internal and external controllers is a common production pattern.

📐 Ingress Resources

Path-Based and Host-Based Routing

Now let's create Ingress resources to define routing rules for incoming HTTP traffic.

📐 Path-Based Routing

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations: { nginx.ingress.kubernetes.io/rewrite-target: /$2 }
spec:
  ingressClassName: nginx-external
  rules:
  - host: app.verisign.internal
    http:
      paths:
      - path: /api(/|$)(.*)
        pathType: ImplementationSpecific
        backend: { service: { name: api-service, port: { number: 8080 } } }
      - path: /
        pathType: Prefix
        backend: { service: { name: frontend-service, port: { number: 80 } } }
Path-based routing directs traffic to different services based on the URL path. The rewrite-target annotation strips the path prefix before forwarding to the backend. pathType can be Exact, Prefix, or ImplementationSpecific. Rules are evaluated in order of specificity — more specific paths match first.

🌐 Host-Based Routing

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata: { name: multi-host-ingress, namespace: verisign-apps }
spec:
  ingressClassName: nginx-external
  rules:
  - host: api.verisign.internal
    http:
      paths:
      - path: /
        pathType: Prefix
        backend: { service: { name: api-service, port: { number: 8080 } } }
  - host: dashboard.verisign.internal
    http:
      paths:
      - path: /
        pathType: Prefix
        backend: { service: { name: dashboard-service, port: { number: 3000 } } }
  defaultBackend:
    service: { name: default-404-service, port: { number: 80 } }
Host-based routing uses the Host header to direct traffic. Each hostname maps to a different backend service. The defaultBackend handles requests that don't match any host rule — useful for returning a custom 404 page. In production, combine this with External DNS to automatically create DNS records.

📌 Path Types Explained

pathTypeMatching BehaviorExample
Exact Matches URL path exactly /api matches /api only, not /api/v1
Prefix Matches path prefix by / segments /api matches /api, /api/v1, /api/v1/users
ImplementationSpecific Depends on IngressClass controller NGINX supports regex: /api(/|$)(.*)
Tip: Prefix matching splits on / boundaries. Path /api does NOT match /apiversion — only paths with /api as a complete segment prefix. ImplementationSpecific lets you use NGINX-specific features like regex paths.

🔧 Ingress Annotations

Customizing NGINX Behavior

Annotations are the primary extension mechanism for NGINX Ingress. They control everything from redirects to rate limiting.

🔧 Essential NGINX Ingress Annotations

metadata:
  name: production-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "120"
    nginx.ingress.kubernetes.io/limit-rps: "50"
    nginx.ingress.kubernetes.io/limit-burst-multiplier: "5"
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://app.verisign.com"
These are the most commonly used annotations in production. SSL redirect ensures all HTTP traffic is upgraded to HTTPS. proxy-body-size controls upload limits. Rate limiting protects your backends from overload. CORS headers are essential for APIs consumed by web frontends. Each annotation maps to an NGINX directive in the generated configuration.

🔧 Advanced Annotations

Session Affinity

annotations:
  nginx.ingress.kubernetes.io/
    affinity: "cookie"
  nginx.ingress.kubernetes.io/
    session-cookie-name: "VSRVSID"
  nginx.ingress.kubernetes.io/
    session-cookie-max-age: "172800"

Custom Headers & Snippets

annotations:
  nginx.ingress.kubernetes.io/
    configuration-snippet: |
      more_set_headers
        "X-Request-ID: $req_id";
      more_set_headers
        "Strict-Transport-Security:
         max-age=31536000";
Tip: configuration-snippet and server-snippet annotations can inject arbitrary NGINX config. In multi-tenant clusters, disable them via controller flags to prevent abuse. An attacker could inject malicious config to steal traffic.

🔒 TLS Termination

Securing Ingress with HTTPS

TLS termination at the Ingress controller is the standard pattern for HTTPS. Let's see how to configure it with both manual certificates and automated cert-manager.

🔒 TLS with Manual Certificates

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: secure-ingress
  annotations: { nginx.ingress.kubernetes.io/ssl-redirect: "true" }
spec:
  ingressClassName: nginx-external
  tls:
  - hosts: [api.verisign.internal, dashboard.verisign.internal]
    secretName: app-tls
  rules:
  - host: api.verisign.internal
    http:
      paths:
      - path: /
        pathType: Prefix
        backend: { service: { name: api-service, port: { number: 8080 } } }
The tls block references a kubernetes.io/tls type Secret containing the certificate chain and private key. The hosts list must match the certificate's Subject Alternative Names. Multiple hosts can share the same certificate if it's a wildcard cert or has multiple SANs.

⚡ Automated TLS with cert-manager

ClusterIssuer

apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata: { name: letsencrypt-prod }
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: certs@verisign.com
    privateKeySecretRef: { name: letsencrypt-prod-key }
    solvers:
    - http01:
        ingress: { class: nginx-external }

Ingress with auto-TLS

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: auto-tls-ingress
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx-external
  tls:
  - hosts: [api.verisign.internal]
    secretName: api-tls-auto
  rules:
  - host: api.verisign.internal
    http:
      paths:
      - path: /
        pathType: Prefix
        backend: { service: { name: api-service, port: { number: 8080 } } }
cert-manager watches for Ingress resources with its annotation, automatically requests certificates, stores them in the referenced Secret, and handles renewals before expiry.

🔶 Gateway API

The Evolution of Kubernetes Ingress

Gateway API is the successor to the Ingress resource, designed to address its many limitations. It's now GA for HTTP routing and is the recommended path forward.

📐 Ingress vs Gateway API

AspectIngressGateway API
ExtensibilityAnnotations (untyped strings)Typed, structured API fields
Role modelSingle resource, one ownerRole-oriented (infra, cluster, app)
ProtocolsHTTP/HTTPS onlyHTTP, TCP, UDP, gRPC, TLS
Traffic splittingNot nativeBuilt-in weight-based routing
Header matchingNot supportedNative header/query matching
PortabilityVendor-specific annotationsPortable across implementations
StatusStable, feature-frozenGA (HTTPRoute), evolving
Gateway API addresses every major pain point of Ingress. No more untyped annotations, explicit role separation, multi-protocol support, and native traffic management features. The Ingress API is feature-frozen — all innovation is happening in Gateway API.

🏗️ Gateway API Resource Model

🛠️ GatewayClass

Role: Infrastructure Provider

  • Defines controller implementation (cluster-scoped)

🔶 Gateway

Role: Cluster Operator

  • Configures listeners, TLS, and provisions LB

📐 HTTPRoute

Role: Application Developer

  • Defines routing rules; attaches to Gateway
Role Separation: Infrastructure teams manage GatewayClass and Gateway. Application teams create HTTPRoutes. This maps cleanly to Verisign's organizational model.
The three-tier resource model is Gateway API's biggest architectural advantage. Infrastructure providers define GatewayClasses. Platform teams create Gateways. Application developers attach routes. Each role has clear boundaries and RBAC can be applied at each level. This is much cleaner than Ingress where one resource mixes all concerns.

🔶 GatewayClass and Gateway

# GatewayClass is typically pre-installed by the platform
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata: { name: verisign-gateway, namespace: gateway-system }
spec:
  gatewayClassName: envoy-gateway
  listeners:
  - name: http
    protocol: HTTP
    port: 80
    allowedRoutes: { namespaces: { from: All } }
  - name: https
    protocol: HTTPS
    port: 443
    tls:
      mode: Terminate
      certificateRefs:
      - { kind: Secret, name: wildcard-tls }
    allowedRoutes:
      namespaces:
        from: Selector
        selector: { matchLabels: { gateway-access: "true" } }
The Gateway resource defines listeners with protocols, ports, and TLS configuration. The allowedRoutes field controls which namespaces can attach routes — this is the RBAC model. Using a namespace selector with labels gives platform teams fine-grained control over which application namespaces can use the gateway.

📐 HTTPRoute Resource

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: api-routes, namespace: verisign-apps }
spec:
  parentRefs:
  - { name: verisign-gateway, namespace: gateway-system, sectionName: https }
  hostnames: ["api.verisign.internal"]
  rules:
  - matches:
    - path: { type: PathPrefix, value: /v1/dns }
      headers: [{ name: X-API-Version, value: "v1" }]
    backendRefs: [{ name: dns-service-v1, port: 8080 }]
  - matches:
    - path: { type: PathPrefix, value: /v2/dns }
    backendRefs: [{ name: dns-service-v2, port: 8080 }]
HTTPRoute is where application developers define their routing rules. Notice how matches can include both path and header conditions natively — no annotations needed. The parentRefs field attaches this route to a specific Gateway listener. Each rule can have multiple matches and multiple backends.

⚖️ Traffic Splitting

Canary Deployments with HTTPRoute Weights

One of Gateway API's killer features is native traffic splitting for canary deployments.

⚖️ Canary Deployments with HTTPRoute

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: dns-service-canary, namespace: verisign-apps }
spec:
  parentRefs:
  - { name: verisign-gateway, namespace: gateway-system }
  hostnames: ["dns.verisign.internal"]
  rules:
  - matches:
    - path: { type: PathPrefix, value: / }
    backendRefs:
    - { name: dns-service-stable, port: 8080, weight: 90 }
    - { name: dns-service-canary, port: 8080, weight: 10 }

Progressive Rollout Strategy

Start at 5% → Monitor metrics → 25% → Verify → 50% → Validate → 100%

Weight-based traffic splitting is built into the HTTPRoute spec — no custom annotations or service meshes needed. This enables canary deployments where you gradually shift traffic from the stable to the canary version. Combined with metrics monitoring, this is a powerful deployment strategy. You simply update the weights to roll forward or back.

📊 Header-Based Canary Routing

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: dns-service-header-canary, namespace: verisign-apps }
spec:
  parentRefs:
  - { name: verisign-gateway, namespace: gateway-system }
  hostnames: ["dns.verisign.internal"]
  rules:
  - matches:
    - headers: [{ name: X-Canary, value: "true" }]
    backendRefs:
    - { name: dns-service-canary, port: 8080 }
  - matches:
    - path: { type: PathPrefix, value: / }
    backendRefs:
    - { name: dns-service-stable, port: 8080 }
Tip: QA team sets X-Canary: true header to test new versions in production before enabling weight-based rollout. This lets you verify the canary in the real production environment before exposing it to all users.

🏙️ Verisign Integration

Envoy Gateway + External DNS + cert-manager

Let's look at how these components come together in Verisign's EKS platform.

🏙️ Verisign EKS Ingress Stack

Production Architecture

🔶 Envoy Gateway

  • Gateway API with Envoy data plane
  • Advanced L7: gRPC + HTTP routing

🌐 External DNS

  • Watches Gateway/Ingress resources
  • Auto-creates Route53 records

🔒 cert-manager

  • Automated certificate lifecycle with auto-renewal
  • Internal CA, ACME, or Gateway API integration
Workflow: Developer creates HTTPRoute → Envoy Gateway configures proxy → External DNS creates Route53 record → cert-manager provisions TLS certificate → Application is live with HTTPS.
This is the golden path at Verisign. An application developer creates an HTTPRoute, and the platform automation handles everything else — DNS records, TLS certificates, and proxy configuration. This reduces the time from code to production and eliminates manual certificate and DNS management.

🔚 Egress Controls

Managing Outbound Traffic

Ingress handles traffic coming in. Egress controls manage what your applications can reach on the way out — equally important for security.

🔚 Why Egress Controls Matter

🛡️ Security Concerns

  • Compromised pods phoning home or exfiltrating data
  • Supply chain attacks reaching external registries
  • Compliance requirements (PCI-DSS, SOC2)

🔧 Control Mechanisms

  • NetworkPolicy egress rules (Module 7)
  • Egress gateways (service mesh) or NAT gateways
  • Proxy servers for HTTP traffic
Egress controls are often overlooked but are critical for security. If an attacker compromises a pod, unrestricted egress lets them exfiltrate data or communicate with command-and-control servers. Network policies for egress are covered in depth in Module 7. Here we'll focus on the egress gateway and NAT patterns.

🔌 Egress Patterns on EKS

Common Egress Architectures

NAT Gateway (Default EKS)

Pod → Node → NAT GW → Internet
                                 ↓
                          Fixed Elastic IP
  • All egress shares NAT GW IP
  • Useful for IP allowlisting
  • No application-level control

Certificates for External Endpoints

# Mount CA bundle for external APIs
volumes:
- name: ca-certs
  secret: { secretName: external-ca-bundle }
containers:
- name: app
  volumeMounts:
  - { name: ca-certs, mountPath: /etc/ssl/custom, readOnly: true }
  env:
  - { name: SSL_CERT_DIR, value: /etc/ssl/custom }
On EKS, pods egress through the VPC's NAT gateway by default, providing a fixed IP for external services that require IP allowlisting. For mTLS with external endpoints, mount custom CA certificates into your pods. The certificate management patterns we covered in the TLS section apply to egress connections as well.

🎯 Key Takeaways

1. Consolidate with Ingress/Gateway — Avoid one LoadBalancer per service. Route through a shared entry point.
2. Gateway API is the future — Typed resources, role separation, and native traffic management replace annotation-driven Ingress.
3. Automate TLS — Use cert-manager for certificate lifecycle. Never manage certificates manually in production.
4. Control egress too — Ingress is only half the story. Egress controls are critical for security and compliance.
Four takeaways: consolidate external access, adopt Gateway API, automate certificate management, and don't forget egress controls.
