HPA, VPA, and Cluster Autoscaler in Practice

💡 Autoscaling in Kubernetes

↔ HPA

Scales the number of pods based on CPU, memory, or custom metrics

↕ VPA

Adjusts resource requests/limits for individual containers

📦 Cluster Autoscaler

Scales the number of nodes when pods cannot be scheduled

🔑 Key Point: These three autoscalers work at different layers and can be used together. HPA adjusts pod count, VPA adjusts pod size, and Cluster Autoscaler adjusts infrastructure capacity.
Draw the relationship: HPA triggers more pods, which may trigger Cluster Autoscaler if nodes are full. VPA resizes pods vertically. Note that HPA and VPA should not target the same metric on the same deployment.

Step 1

Verify Metrics Server

Metrics-server is a hard dependency for HPA. If it is not running, HPA will not function. Verify this before proceeding.

📊 Verify Metrics Server

HPA requires metrics-server to collect resource usage data:


# Check if metrics-server is running
kubectl get pods -n kube-system | grep metrics-server

# Verify the metrics API is available
kubectl get apiservices | grep metrics

# Test that metrics are being collected
kubectl top nodes
kubectl top pods -A
✅ Expected Output: The metrics-server pod should be Running with 1/1 ready. The apiservice v1beta1.metrics.k8s.io should show True in the AVAILABLE column. kubectl top should return CPU and memory data.
⚠ If metrics-server is not installed: 

# Install metrics-server (if you have permissions)
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

# Wait for it to be ready
kubectl wait --for=condition=ready pod -l k8s-app=metrics-server -n kube-system --timeout=120s
On EKS, metrics-server is typically pre-installed. If not, the apply command will install it. It may take 1-2 minutes for metrics to start populating after installation.

Step 2

Deploy a CPU-Intensive Application

We use the php-apache image from the Kubernetes documentation. It is designed to consume CPU under load, making it ideal for demonstrating HPA.

🚀 Deploy the php-apache Application

Create a namespace and deploy an application with resource requests:


# Create lab namespace
kubectl create namespace lab02
kubectl config set-context --current --namespace=lab02

Create the deployment manifest:


# Save as php-apache-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: php-apache
  namespace: lab02
spec:
  replicas: 1
  selector:
    matchLabels:
      app: php-apache
  template:
    metadata:
      labels:
        app: php-apache
    spec:
      containers:
      - name: php-apache
        image: registry.k8s.io/hpa-example
        ports:
        - containerPort: 80
        resources:
          requests: { cpu: 200m, memory: 128Mi }
          limits:   { cpu: 500m, memory: 256Mi }
Emphasize that resource requests are REQUIRED for HPA to work with CPU metrics. Without requests, HPA cannot calculate utilization percentage.

🚀 Apply and Expose the Application


# Apply the deployment
kubectl apply -f php-apache-deployment.yaml

# Expose it as a service
kubectl expose deployment php-apache --port=80 --target-port=80

# Verify everything is running
kubectl get all -n lab02
✅ Expected Output: One pod in Running state, one deployment with 1/1 ready, and a ClusterIP service on port 80.
📝 Important: The resources.requests.cpu: 200m field is critical. HPA calculates utilization as current_usage / request. Without a CPU request, HPA cannot determine the scaling target percentage.
Wait for the pod to be fully running before proceeding. Have students verify with kubectl get pods -w.

Step 3

Create a Horizontal Pod Autoscaler

This is the core of the lab. Students will create their first HPA and observe it in action.

↔ Create an HPA

Create an HPA targeting 50% average CPU utilization:


# Create HPA using imperative command
kubectl autoscale deployment php-apache \
  --cpu-percent=50 \
  --min=1 \
  --max=10 \
  -n lab02

# Examine the HPA
kubectl get hpa -n lab02

# View detailed HPA information
kubectl describe hpa php-apache -n lab02
✅ Expected Output: The HPA should show TARGETS: <current>/50%, MINPODS: 1, MAXPODS: 10, and REPLICAS: 1. The current value may show <unknown>/50% for the first 15-30 seconds while metrics are collected.
⚠ Common Mistake: If TARGETS shows <unknown> for more than 60 seconds, check that metrics-server is running and that the deployment has CPU resource requests defined.
The unknown target is normal for the first metrics collection interval (typically 15 seconds). If it persists, troubleshoot metrics-server connectivity.

Step 4

Generate Load

Now we generate artificial load to trigger the HPA. Students should run this in one terminal and monitor in another.

🔥 Generate Load on the Application

Run a load generator in a separate pod:


# Start a load generator pod (run in Terminal 1)
kubectl run load-generator \
  --image=busybox:1.36 \
  --restart=Never \
  -n lab02 \
  -- /bin/sh -c "while sleep 0.01; do wget -q -O- http://php-apache; done"
📝 Important: Open a second terminal for the next step. Keep the load generator running in this terminal. You will need to monitor the HPA from the second terminal.

💡 How This Works

The busybox pod sends continuous HTTP requests to the php-apache service. Each request triggers PHP computation, driving CPU usage above the 50% target threshold, which signals the HPA to scale up.

The load generator runs a tight loop hitting the php-apache endpoint. It takes about 1-2 minutes for CPU utilization to rise enough to trigger scaling. Have students keep this running.

Step 5

Observe Horizontal Scaling

This is the exciting part. Students will see the HPA react to load in real-time. Give it 2-3 minutes for the full scaling effect.

📈 Observe Horizontal Scaling

In your second terminal, watch the HPA and pods respond:


# Watch HPA status (updates every 15 seconds)
kubectl get hpa php-apache -n lab02 --watch

# In yet another terminal, watch pods scale up
kubectl get pods -n lab02 --watch
✅ Expected Output: Within 1-3 minutes you should see:
  • TARGETS increasing above 50% (e.g., 250%/50%)
  • REPLICAS increasing from 1 to multiple pods (up to 10)
  • New pods appearing in Pending then Running state
🔑 Key Point: The HPA evaluates metrics every 15 seconds by default. It uses the formula: desiredReplicas = ceil(currentReplicas * (currentMetric / targetMetric)). Scaling up happens quickly, but scale-down has a default stabilization window of 5 minutes to prevent flapping.
Walk around and verify that students see replicas increasing. The exact number depends on how much CPU the load generates. Discuss the scaling formula on the slide.

Step 6

Examine HPA Events and Status

Understanding HPA events is key to troubleshooting autoscaling issues in production.

🔎 Examine HPA Events and Status

Inspect the HPA decision-making process:


# Detailed HPA description with events
kubectl describe hpa php-apache -n lab02

# View HPA in YAML format for full status
kubectl get hpa php-apache -n lab02 -o yaml

💡 Key Fields in HPA Status

  • Events — shows each scaling decision with reason and timestamp
  • currentMetrics — actual resource utilization observed
  • desiredReplicas — what HPA wants to scale to
  • conditions — AbleToScale, ScalingActive, ScalingLimited
✅ Checkpoint: In the Events section, you should see messages like: New size: X; reason: cpu resource utilization (percentage of request) above target. This confirms HPA is actively scaling based on CPU metrics.
Have students look at the Events section specifically. Each scaling event shows the old and new replica counts and the reason. Also point out the conditions section which shows if scaling is limited by min/max bounds.

Step 7

Stop Load and Observe Scale-Down

Scale-down is slower than scale-up by design. This prevents oscillation when load fluctuates. Students should observe this behavior.

📉 Stop Load and Observe Scale-Down

Stop the load generator and watch the HPA scale down:


# Delete the load generator pod
kubectl delete pod load-generator -n lab02

# Watch the HPA scale down (this takes ~5 minutes)
kubectl get hpa php-apache -n lab02 --watch

# In another terminal, watch pods terminate
kubectl get pods -n lab02 --watch
✅ Expected Output: After approximately 5 minutes, you should see:
  • TARGETS dropping below 50% (eventually to 0%/50%)
  • REPLICAS gradually decreasing back to 1
  • Pods entering Terminating state
📝 Why So Slow? The default --horizontal-pod-autoscaler-downscale-stabilization window is 5 minutes. This prevents rapid scale-down/scale-up oscillation (thrashing) during variable load patterns.
While waiting for scale-down, use this time to discuss stabilization windows and why asymmetric scaling behavior (fast up, slow down) is desirable in production.

Step 8

HPA with Custom Metrics (v2 API)

The v2 HPA API supports multiple metrics, custom metrics, and more granular scaling behavior. This is what production deployments should use.

📝 HPA v2 API with Multiple Metrics

Delete the existing HPA and create a more advanced one:


# Remove the existing HPA
kubectl delete hpa php-apache -n lab02

Create the v2 HPA manifest:


# Save as hpa-v2.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: php-apache-v2, namespace: lab02 }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: php-apache }
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource: { name: cpu, target: { type: Utilization, averageUtilization: 50 } }
  - type: Resource
    resource: { name: memory, target: { type: Utilization, averageUtilization: 80 } }
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies: [{ type: Percent, value: 50, periodSeconds: 60 }]
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - { type: Percent, value: 100, periodSeconds: 15 }
      - { type: Pods, value: 4, periodSeconds: 15 }
      selectPolicy: Max
Walk through each section of the YAML. Highlight the behavior section, which gives fine-grained control over scale-up and scale-down rates. The selectPolicy: Max means it picks whichever policy allows more scaling.

🚀 Apply and Examine the v2 HPA


# Apply the v2 HPA
kubectl apply -f hpa-v2.yaml

# Examine the HPA
kubectl get hpa php-apache-v2 -n lab02

# View the full spec
kubectl describe hpa php-apache-v2 -n lab02
✅ Expected Output: The HPA should show two metrics targets: cpu: <current>/50% and memory: <current>/80%. The REFERENCE should point to Deployment/php-apache.

💡 v2 API Advantages

  • Multiple metrics — scale on CPU and memory simultaneously
  • Scaling behavior — control scale-up/down speed and stabilization
  • Custom metrics — scale on application-specific metrics (requests/sec, queue depth)
  • External metrics — scale on metrics from outside the cluster (SQS queue length, etc.)
Point out that the behavior section allows different policies for scale-up vs scale-down. In the manifest, scale-up is aggressive (100% increase every 15s) while scale-down is conservative (50% decrease every 60s with 5-min stabilization).

Step 9

Vertical Pod Autoscaler (VPA)

VPA may not be installed in all clusters. If it is not available, walk through the manifest as a review exercise. Students can still learn the concepts.

↕ Configure Vertical Pod Autoscaler

First, check if VPA is available in your cluster:


# Check for VPA CRDs
kubectl get crd | grep verticalpodautoscaler

# Check for VPA components
kubectl get pods -n kube-system | grep vpa
📝 Note: VPA is an optional component. If the CRDs are not installed, review the manifest below as a reference but skip the apply step.

Create a VPA resource in Off mode (recommendation only):


# Save as vpa.yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: php-apache-vpa
  namespace: lab02
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: php-apache
  updatePolicy:
    updateMode: "Off"
  resourcePolicy:
    containerPolicies:
    - containerName: php-apache
      minAllowed:
        cpu: 100m
        memory: 64Mi
      maxAllowed:
        cpu: 1000m
        memory: 512Mi
Off mode is the safest way to start with VPA. It provides recommendations without making changes. In production, you can switch to Auto mode once you trust the recommendations.

🔎 Examine VPA Recommendations


# Apply the VPA (only if CRDs are installed)
kubectl apply -f vpa.yaml

# Check VPA recommendations (may take a few minutes to populate)
kubectl describe vpa php-apache-vpa -n lab02

# View recommendations in JSON
kubectl get vpa php-apache-vpa -n lab02 -o jsonpath='{.status.recommendation}' | python3 -m json.tool
✅ Expected Output: The VPA status should show three recommendation levels:
  • lowerBound — minimum recommended resources
  • target — optimal recommended resources
  • upperBound — maximum recommended resources
⚠ Warning: Do not use VPA in Auto mode with HPA on the same CPU/memory metrics. VPA will resize pods while HPA tries to change pod count, creating conflicts. Use VPA for right-sizing resource requests, and HPA for scaling replicas.
If VPA is not installed, discuss the recommendations concept and how teams can use Off mode VPA to gather data before setting resource requests. This is a common pattern at Verisign for right-sizing workloads.

Step 10

Cluster Autoscaler Behavior

Cluster Autoscaler adds or removes nodes. On EKS this integrates with Auto Scaling Groups. This step may not trigger actual node scaling depending on cluster capacity, but the concepts are essential.

📦 Test Cluster Autoscaler Behavior

Deploy a workload that requests more resources than currently available:


# Save as inflate-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: inflate
  namespace: lab02
spec:
  replicas: 20
  selector:
    matchLabels:
      app: inflate
  template:
    metadata:
      labels:
        app: inflate
    spec:
      containers:
      - name: inflate
        image: public.ecr.aws/eks-distro/kubernetes/pause:3.7
        resources:
          requests:
            cpu: 250m
            memory: 256Mi
                

kubectl apply -f inflate-deployment.yaml
The pause container consumes almost no resources but reserves them via requests. 20 replicas requesting 250m CPU each means 5 full CPU cores worth of requests. This will likely exceed available capacity on a small cluster.

🔍 Monitor Cluster Autoscaler Activity


# Watch pods - some should be Pending if cluster is at capacity
kubectl get pods -n lab02 -l app=inflate --watch

# Check for Pending pods
kubectl get pods -n lab02 -l app=inflate --field-selector=status.phase=Pending

# Examine why pods are pending
kubectl describe pod <PENDING_POD_NAME> -n lab02 | grep -A 5 "Events:"

# Check Cluster Autoscaler logs (if available)
kubectl logs -n kube-system -l app=cluster-autoscaler --tail=50
✅ Expected Output: Some pods will show Pending status with events showing FailedScheduling: 0/X nodes are available: insufficient cpu. If Cluster Autoscaler is active, you may see new nodes joining within 2-5 minutes.

💡 Cluster Autoscaler Logic

  • Scale Up — triggered when pods are Pending due to insufficient resources
  • Scale Down — triggered when nodes are underutilized for 10+ minutes
  • EKS Integration — modifies the ASG desired count for the managed node group
If nodes do not scale, explain that the Cluster Autoscaler may not be configured or the ASG max may already be reached. Use this as a discussion point about capacity planning and ASG configuration.

Step 11

Clean Up All Resources

Important to clean up, especially the inflate deployment which reserves significant cluster resources.

🧹 Clean Up All Resources

Remove everything created during this lab:


# Reset namespace context
kubectl config set-context --current --namespace=default

# Delete the entire lab namespace
kubectl delete namespace lab02

# Verify cleanup
kubectl get namespaces
kubectl get hpa -A | grep lab02
✅ Checkpoint: The lab02 namespace should be gone. No HPA resources should reference lab02. If Cluster Autoscaler added nodes, they will scale down automatically after the workloads are removed (this may take 10+ minutes).
📝 Note: Also delete any YAML files you created locally if you no longer need them, or keep them as reference material.

📝 Autoscaling Best Practices

Practice Recommendation
Always set resource requests HPA cannot calculate utilization without them
Use v2 HPA API Supports multiple metrics and scaling behavior control
Start with conservative targets 50-70% CPU utilization is a good starting point
Set realistic min/max replicas Min handles baseline load; max prevents runaway scaling
Use VPA in Off mode first Gather recommendations before enabling auto-adjustment
Do not mix HPA + VPA on same metric They will conflict; use HPA for scaling, VPA for right-sizing
Configure scaling behavior Fast scale-up, slow scale-down prevents thrashing
Monitor HPA events Use kubectl describe hpa to debug scaling decisions
Review these best practices. They represent production lessons learned. Emphasize that autoscaling is not set-and-forget; it requires tuning based on application behavior.

🎯 Key Takeaways

  • ✅ HPA scales pod count based on CPU, memory, or custom metrics
  • ✅ Resource requests are mandatory for HPA to function correctly
  • ✅ The v2 API provides fine-grained control over scaling behavior
  • ✅ Scale-up is fast; scale-down is deliberately slow to prevent thrashing
  • ✅ VPA right-sizes resource requests; use Off mode to gather data first
  • ✅ Cluster Autoscaler adds nodes when pods cannot be scheduled
  • ✅ All three autoscalers work at different layers and complement each other
Reinforce the layered autoscaling model. In production at Verisign, teams typically use HPA for most workloads, VPA in Off mode for recommendations, and rely on Cluster Autoscaler for infrastructure elasticity.
