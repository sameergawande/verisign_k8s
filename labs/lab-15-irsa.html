Lab 15: Configuring IRSA

IAM Roles for Service Accounts

Estimated Duration: 40–50 minutes

This lab walks students through the full IRSA lifecycle: verifying the OIDC provider, examining existing IRSA configurations, creating a new IRSA role, deploying a pod that uses it, and debugging intentional misconfigurations. Ensure the lab namespace and pre-created IAM policy are ready before students begin.

🎯 Lab Overview

Objectives

  • Verify the EKS OIDC provider configuration
  • Examine existing IRSA setups (cert-manager, external-dns)
  • Create an IAM policy, IAM role, and trust policy for IRSA
  • Deploy a pod with an IRSA-annotated ServiceAccount
  • Validate AWS credentials and test permission boundaries
  • Debug a deliberately broken IRSA configuration
Prerequisites: AWS CLI configured with appropriate permissions, kubectl connected to the EKS cluster, cluster OIDC provider already provisioned by Terraform. Review the objectives and prerequisites with the class. Confirm everyone has AWS CLI access and kubectl connectivity before starting. The OIDC provider and base IAM resources should already exist from the Terraform provisioning.

Step 1

Verify the OIDC Provider

First, we verify the foundational infrastructure — the OIDC provider. Without this, IRSA cannot function.

🔍 Step 1: Verify OIDC Provider

Confirm the EKS cluster has an OIDC issuer and that the corresponding IAM OIDC Identity Provider exists.

1a. Get the OIDC issuer URL

# Retrieve the OIDC issuer URL for the cluster
aws eks describe-cluster \
  --name platform-lab \
  --query "cluster.identity.oidc.issuer" \
  --output text

# Expected output (example):
# https://oidc.eks.us-east-1.amazonaws.com/id/EXAMPLED539D4633E53DE1B71EXAMPLE

1b. Extract the OIDC ID and verify in IAM

# Extract just the OIDC ID
OIDC_ID=$(aws eks describe-cluster --name platform-lab \
  --query "cluster.identity.oidc.issuer" --output text | cut -d'/' -f5)
echo "OIDC ID: $OIDC_ID"

# List OIDC providers and check for our cluster's ID
aws iam list-open-id-connect-providers | grep $OIDC_ID

# Expected: ARN of the OIDC provider matching our cluster
If the grep returns nothing, the OIDC provider has not been registered in IAM. This must be fixed (usually via Terraform) before IRSA will work. Students should see the ARN matching their cluster's OIDC ID.

🔍 Step 1: Verify OIDC Provider (continued)

1c. Get full details of the OIDC provider

# Get the full OIDC provider ARN
OIDC_ARN=$(aws iam list-open-id-connect-providers \
  --query "OpenIDConnectProviderList[?ends_with(Arn, '$OIDC_ID')].Arn" \
  --output text)
echo "OIDC Provider ARN: $OIDC_ARN"

# Describe the provider to verify configuration
aws iam get-open-id-connect-provider --open-id-connect-provider-arn $OIDC_ARN
Checkpoint: You should see the OIDC provider with ClientIDList containing sts.amazonaws.com. If this is missing, IRSA token exchange will fail. The get-open-id-connect-provider output shows the full configuration including thumbprints, client IDs, and creation date. The critical check is that sts.amazonaws.com is in the client ID list. Save the OIDC_ARN and OIDC_ID values — we will use them in later steps.

Step 2

Examine Existing IRSA Configurations

Before creating our own IRSA configuration, let's examine the ones already running in the cluster — cert-manager and external-dns.

🔎 Step 2: Examine Existing IRSA

Look at how cert-manager and external-dns are configured with IRSA in the Verisign platform.

2a. Examine the cert-manager ServiceAccount

# View the cert-manager ServiceAccount and its IRSA annotation
kubectl get sa cert-manager -n cert-manager -o yaml

# Look for this annotation:
#   annotations:
#     eks.amazonaws.com/role-arn: \
#       arn:aws:iam::ACCOUNT:role/platform-lab-cert-manager

2b. Examine the cert-manager pod

# Describe a cert-manager pod to see injected volumes and env vars
kubectl get pods -n cert-manager -l app=cert-manager -o name | head -1 | \
  xargs kubectl describe -n cert-manager

# Look for:
#   Environment:
#     AWS_ROLE_ARN: arn:aws:iam::...
#     AWS_WEB_IDENTITY_TOKEN_FILE: /var/run/secrets/eks.amazonaws.com/...
#   Volumes:
#     aws-iam-token (projected)
Have students identify the three key IRSA indicators in the pod description: the IRSA annotation on the ServiceAccount, the AWS environment variables injected into the containers, and the projected aws-iam-token volume. These three elements confirm IRSA is active.

🔎 Step 2: Examine external-dns

2c. Check external-dns ServiceAccount

# View external-dns ServiceAccount
kubectl get sa external-dns -n external-dns -o yaml

# Verify it also has the eks.amazonaws.com/role-arn annotation

2d. Compare the two IRSA roles in IAM

# List roles matching our cluster name
aws iam list-roles --query "Roles[?contains(RoleName, \
  'platform-lab')].[RoleName]" \
  --output table

# You should see:
# platform-lab-cert-manager
# platform-lab-external-dns
Observation: Each workload has its own IAM Role with its own trust policy scoped to a specific namespace and ServiceAccount. This is the IRSA best practice in action.
Point out the pattern: separate roles for separate workloads, each scoped to their own namespace and ServiceAccount. This is the foundation of least-privilege in IRSA. Students should see that both roles exist and follow the same naming convention.

Step 3

Create an IAM Policy

Now we create our own IRSA configuration from scratch. We will start with the IAM policy that defines what the role can do.

📜 Step 3: Create an IAM Policy

Create a policy that grants read-only access to a specific S3 bucket. This follows the least-privilege principle.

3a. Create the policy document

# Create the policy JSON file
cat <<'EOF' > /tmp/s3-read-policy.json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::platform-lab-irsa-demo",
        "arn:aws:s3:::platform-lab-irsa-demo/*"
      ]
    }
  ]
}
EOF

cat /tmp/s3-read-policy.json
We use a very scoped policy: only GetObject and ListBucket on a single bucket. This is the minimum needed for our test. In production, you would scope even further — perhaps to a specific prefix within the bucket.

📜 Step 3: Create the Policy in IAM

3b. Create the IAM policy

# Create the policy in IAM
aws iam create-policy \
  --policy-name platform-lab-irsa-demo-s3-read \
  --policy-document file:///tmp/s3-read-policy.json \
  --description "Read-only S3 access for IRSA demo lab"

# Save the policy ARN for the next step
POLICY_ARN=$(aws iam list-policies \
  --query "Policies[?PolicyName=='platform-lab-irsa-demo-s3-read'].Arn" \
  --output text)
echo "Policy ARN: $POLICY_ARN"
Checkpoint: You should see the policy ARN printed. Save this value you will need it when attaching the policy to the role in Step 4. If the policy already exists from a previous lab run, students will get an EntityAlreadyExists error. They can safely skip creation and just retrieve the ARN. The policy name follows the same convention as the platform team's policies.

Step 4

Create an IAM Role with Trust Policy

The trust policy is the most critical part of IRSA configuration. It defines which Kubernetes ServiceAccount can assume this role. Get this wrong and IRSA silently fails.

📜 Step 4: Create the Trust Policy

The trust policy tells IAM: "Allow the EKS OIDC provider to vouch for a specific ServiceAccount to assume this role."

4a. Build the trust policy JSON

# Get the OIDC issuer URL and account ID
OIDC_ISSUER=$(aws eks describe-cluster --name platform-lab \
  --query "cluster.identity.oidc.issuer" --output text | sed 's|https://||')
ACCOUNT_ID=$(aws sts get-caller-identity --query "Account" --output text)

cat <<EOF > /tmp/trust-policy.json
{ "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::${ACCOUNT_ID}:oidc-provider/${OIDC_ISSUER}"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": { "StringEquals": {
      "${OIDC_ISSUER}:aud": "sts.amazonaws.com",
      "${OIDC_ISSUER}:sub": "system:serviceaccount:irsa-lab:s3-reader"
    }}
  }]
}
EOF
cat /tmp/trust-policy.json
Walk through the trust policy carefully. The Federated principal is the OIDC provider ARN. The Condition uses StringEquals to lock this role to a specific ServiceAccount (s3-reader) in a specific namespace (irsa-lab). Note how the OIDC issuer URL (without https://) appears in the condition keys. Have students verify the JSON looks correct before proceeding.

📜 Step 4: Create the Role and Attach Policy

4b. Create the IAM role

# Create the role with the trust policy
aws iam create-role \
  --role-name platform-lab-irsa-demo-s3-reader \
  --assume-role-policy-document file:///tmp/trust-policy.json \
  --description "IRSA demo role for S3 read access"

# Save the role ARN
ROLE_ARN=$(aws iam get-role \
  --role-name platform-lab-irsa-demo-s3-reader \
  --query "Role.Arn" --output text)
echo "Role ARN: $ROLE_ARN"

4c. Attach the S3 read policy to the role

# Attach the policy we created in Step 3
aws iam attach-role-policy \
  --role-name platform-lab-irsa-demo-s3-reader \
  --policy-arn $POLICY_ARN

# Verify the attachment
aws iam list-attached-role-policies \
  --role-name platform-lab-irsa-demo-s3-reader
Checkpoint: You should see the S3 read policy attached to the role. Save the ROLE_ARN value for the next step. The role and policy are now ready on the AWS side. The trust policy ensures only the ServiceAccount named s3-reader in the irsa-lab namespace can assume this role. Next we create that ServiceAccount on the Kubernetes side to complete the bidirectional link.

Step 5

Create an Annotated ServiceAccount

Now we create the Kubernetes side of the IRSA configuration — the namespace and annotated ServiceAccount.

📄 Step 5: Create Annotated ServiceAccount

5a. Create the lab namespace

# Create the namespace for the lab
kubectl create namespace irsa-lab

# Verify
kubectl get namespace irsa-lab

5b. Create the ServiceAccount with IRSA annotation

# Save as /tmp/irsa-sa.yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: s3-reader
  namespace: irsa-lab
  annotations:
    eks.amazonaws.com/role-arn: "${ROLE_ARN}"
# Create the file with the actual ROLE_ARN substituted
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: ServiceAccount
metadata:
  name: s3-reader
  namespace: irsa-lab
  annotations:
    eks.amazonaws.com/role-arn: "${ROLE_ARN}"
EOF

5c. Verify the ServiceAccount

kubectl get sa s3-reader -n irsa-lab -o yaml
The annotation is the critical piece that tells the EKS mutating webhook to inject the IRSA token and environment variables. Make sure students verify the annotation is present and the role ARN is correct. A typo here is one of the most common IRSA failures.

Step 6

Deploy a Test Pod

Now we deploy a pod that uses the annotated ServiceAccount. We will use the AWS CLI container image so we can run AWS commands from inside the pod.

📦 Step 6: Deploy a Test Pod

6a. Create the pod manifest

cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: irsa-test
  namespace: irsa-lab
spec:
  serviceAccountName: s3-reader
  containers:
  - name: aws-cli
    image: amazon/aws-cli:latest
    command: ["sleep", "3600"]
  restartPolicy: Never
EOF

6b. Wait for the pod to be running

# Wait for the pod to be ready
kubectl wait --for=condition=Ready pod/irsa-test -n irsa-lab --timeout=60s

# Verify pod status
kubectl get pod irsa-test -n irsa-lab
Note: The serviceAccountName: s3-reader field is what triggers the EKS mutating webhook to inject the IRSA token and environment variables. The pod uses the amazon/aws-cli image which has the AWS CLI pre-installed. The sleep command keeps the pod running so we can exec into it. The serviceAccountName field is the link between the pod and the IRSA-annotated ServiceAccount.

Step 7

Verify IRSA from Inside the Pod

This is the moment of truth. We will exec into the pod and verify that IRSA is working: environment variables, token file, and AWS identity.

✅ Step 7: Verify IRSA Inside the Pod

7a. Check environment variables

# Check for AWS IRSA environment variables
kubectl exec irsa-test -n irsa-lab -- env | grep AWS

# Expected output:
# AWS_ROLE_ARN=arn:aws:iam::ACCOUNT:role/platform-lab-irsa-demo-s3-reader
# AWS_WEB_IDENTITY_TOKEN_FILE=\
#   /var/run/secrets/eks.amazonaws.com/serviceaccount/token

7b. Examine the projected token

# View the token (it's a JWT)
kubectl exec irsa-test -n irsa-lab -- \
  cat /var/run/secrets/eks.amazonaws.com/serviceaccount/token

# The token is a base64-encoded JWT with claims including:
# iss: OIDC issuer URL
# sub: system:serviceaccount:irsa-lab:s3-reader
# aud: sts.amazonaws.com
Both environment variables must be present. If they are missing, the mutating webhook did not fire — check the ServiceAccount annotation. The token is a standard JWT that can be decoded (e.g., at jwt.io) to inspect the claims. The sub claim matches what the trust policy expects.

✅ Step 7: Verify AWS Identity

7c. Verify the assumed role identity

# Check which AWS identity the pod is using
kubectl exec irsa-test -n irsa-lab -- aws sts get-caller-identity

# Expected output:
# {
#     "UserId": "AROA...:botocore-session-...",
#     "Account": "123456789012",
#     "Arn": "arn:aws:sts::123456789012:\
#       assumed-role/platform-lab-irsa-demo-s3-reader/..."
# }

7d. Test S3 access

# List the S3 bucket contents
kubectl exec irsa-test -n irsa-lab -- aws s3 ls s3://platform-lab-irsa-demo/

# Try to read a specific object
kubectl exec irsa-test -n irsa-lab -- \
  aws s3 cp s3://platform-lab-irsa-demo/test-file.txt -
Checkpoint: The get-caller-identity output should show the IRSA role ARN, not the EC2 instance role. S3 list and read commands should succeed. The critical check is that get-caller-identity shows the IRSA role, not the node's instance profile role. If it shows the instance role, IRSA is not working — the SDK is falling back to the instance metadata service. If S3 access works, the full IRSA chain is validated end-to-end.

Step 8

Test Permission Boundaries

Now we verify that the principle of least privilege is actually enforced. The pod should only be able to do what the IAM policy allows.

🛡️ Step 8: Test Permission Boundaries

Verify that the IRSA role only grants the permissions defined in the IAM policy. Actions outside the policy should be denied.

8a. Try an unauthorized action

# Try to list EC2 instances (NOT in our policy)
kubectl exec irsa-test -n irsa-lab -- aws ec2 describe-instances \
  --region us-east-1

# Expected: AccessDenied error
# An error occurred (UnauthorizedOperation) when calling the
# DescribeInstances operation: You are not authorized to perform
# this operation.

8b. Try to write to S3 (read-only policy)

# Try to upload a file (our policy only allows read)
kubectl exec irsa-test -n irsa-lab -- \
  bash -c 'echo "test" | aws s3 cp - \
    s3://platform-lab-irsa-demo/unauthorized.txt'

# Expected: AccessDenied error
Least Privilege Confirmed: The pod can read from S3 but cannot write, and cannot access any other AWS service. This is IRSA enforcing the scoped IAM policy.
This step demonstrates that IRSA is not just providing AWS access — it is providing precisely scoped access. The AccessDenied errors prove that the pod cannot exceed its granted permissions. This is the fundamental security advantage over node-level IAM roles.

Step 9

Verify Without IRSA

To truly appreciate IRSA, let's see what happens when a pod does NOT have an IRSA-annotated ServiceAccount.

🚫 Step 9: Pod Without IRSA

9a. Deploy a pod with the default ServiceAccount

# Deploy a pod WITHOUT an IRSA-annotated ServiceAccount
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: no-irsa-test
  namespace: irsa-lab
spec:
  containers:
  - name: aws-cli
    image: amazon/aws-cli:latest
    command: ["sleep", "3600"]
  restartPolicy: Never
EOF

kubectl wait --for=condition=Ready pod/no-irsa-test -n irsa-lab --timeout=60s

9b. Check environment variables

# Check for AWS environment variables
kubectl exec no-irsa-test -n irsa-lab -- env | grep AWS

# Expected: NO AWS_ROLE_ARN or AWS_WEB_IDENTITY_TOKEN_FILE
Without the IRSA annotation on the ServiceAccount, the mutating webhook does not inject anything. There is no projected token, no AWS environment variables. The pod either gets the node's instance profile (if IMDS is accessible) or has no AWS identity at all.

🚫 Step 9: The Security Difference

9c. Try to get AWS identity

# Try to get caller identity without IRSA
kubectl exec no-irsa-test -n irsa-lab -- aws sts get-caller-identity

# Possible outcomes:
# 1. Shows the EC2 instance role (if IMDS accessible) — BAD: overly broad
# 2. "Unable to locate credentials" — GOOD: no unintended access

Without IRSA

  • Falls back to node role or no access
  • All pods on node share same permissions
  • No audit trail per workload

With IRSA

  • Pod-specific IAM role
  • Scoped to exact permissions needed
  • Full CloudTrail audit per workload
The comparison makes the value of IRSA tangible. If the pod gets the node role, point out that this role likely has broad permissions for kubelet, container runtime, and ECR access. If it gets no credentials, that is actually the more secure default but means the pod cannot access AWS at all. IRSA gives you the middle ground: specific access for specific workloads.

Step 10

Debugging Common Issues

Let's deliberately break the IRSA configuration to see what the error messages look like. This builds debugging muscle memory.

🔧 Step 10: Debugging IRSA

We will deliberately introduce a misconfiguration, observe the failure, diagnose it, and fix it.

10a. Create a ServiceAccount with a wrong role ARN

# Create a SA with a typo in the role ARN + a test pod
cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: ServiceAccount
metadata:
  name: broken-sa
  namespace: irsa-lab
  annotations:
    eks.amazonaws.com/role-arn: "arn:aws:iam::123456789012:role/DOES-NOT-EXIST"
---
apiVersion: v1
kind: Pod
metadata: { name: broken-irsa-test, namespace: irsa-lab }
spec:
  serviceAccountName: broken-sa
  containers:
  - name: aws-cli
    image: amazon/aws-cli:latest
    command: ["sleep", "3600"]
  restartPolicy: Never
EOF
kubectl wait --for=condition=Ready pod/broken-irsa-test -n \
  irsa-lab --timeout=60s
This simulates a common mistake: a typo in the role ARN annotation. The pod will start normally — IRSA failures are not pod startup failures. The error only appears when the AWS SDK tries to assume the role.

🔧 Step 10: Observe and Fix the Failure

10b. Observe the STS error

# Try to get caller identity — this will fail
kubectl exec broken-irsa-test -n irsa-lab -- aws sts get-caller-identity
# Error: AccessDenied calling AssumeRoleWithWebIdentity

10c. Diagnose: check the SA annotation

kubectl get sa broken-sa -n irsa-lab -o \
  jsonpath='{.metadata.annotations}' | python3 -m json.tool
echo "Expected: $ROLE_ARN"  # annotation has a non-existent ARN

10d. Fix the annotation and redeploy

# Fix annotation, delete pod, and recreate
kubectl annotate sa broken-sa -n irsa-lab \
  eks.amazonaws.com/role-arn=$ROLE_ARN --overwrite
kubectl delete pod broken-irsa-test -n irsa-lab
# Re-apply the pod spec from Step 10a
Key learning: the pod starts fine even with a wrong IRSA annotation. The failure only shows up at runtime when the AWS SDK tries to exchange the token. Also reinforce that fixing the SA annotation is not enough — the pod must be recreated for the mutating webhook to re-inject the correct token. For the redeploy, students should re-apply the same pod YAML from step 10a.

🔧 Step 10: Verify the Fix

10e. Confirm IRSA works after the fix

# Now the identity should resolve correctly
kubectl exec broken-irsa-test -n irsa-lab -- aws sts get-caller-identity

# Expected: Shows the correct IRSA role ARN
Checkpoint: After fixing the annotation and recreating the pod, get-caller-identity shows the correct IRSA role. The debugging exercise is complete.
Debugging Takeaway: IRSA failures are silent at pod startup. Always verify with aws sts get-caller-identity from inside the pod. Fix → re-annotate SA → recreate the pod.
This debugging workflow is the most important skill from this lab. In production, IRSA misconfigurations can cause subtle application failures that only show up when specific AWS API calls are made. Building the habit of verifying identity from inside the pod prevents hours of debugging.

Step 11

Clean Up

Time to clean up all the resources we created during this lab.

🗑️ Step 11: Clean Up

11a. Delete Kubernetes resources

# Delete all pods and ServiceAccounts in the lab namespace
kubectl delete pod irsa-test no-irsa-test broken-irsa-test -n \
  irsa-lab --ignore-not-found
kubectl delete sa s3-reader broken-sa -n irsa-lab --ignore-not-found

# Delete the namespace
kubectl delete namespace irsa-lab

11b. Delete IAM resources

# Detach the policy from the role
aws iam detach-role-policy \
  --role-name platform-lab-irsa-demo-s3-reader \
  --policy-arn $POLICY_ARN

# Delete the role
aws iam delete-role --role-name platform-lab-irsa-demo-s3-reader

# Delete the policy
aws iam delete-policy --policy-arn $POLICY_ARN

# Verify clean up
aws iam list-roles --query "Roles[?contains(RoleName, \
  'irsa-demo')]" --output table
Checkpoint: No lab resources remain. The irsa-lab namespace, pods, ServiceAccounts, IAM role, and IAM policy are all deleted. Order matters: detach the policy before deleting the role, as IAM will not allow deleting a role with attached policies. The namespace deletion will cascade-delete any remaining pods and ServiceAccounts within it. Verify that the IAM resources are gone with the list-roles query.

IRSA Flow You Implemented

Key Commands

  • aws eks describe-cluster
  • aws iam create-role

  • aws sts get-caller-identity

Security Checklist

  • Scoped IAM policy (specific actions + resources)
  • StringEquals trust condition
  • One role per workload
  • Verify identity from inside pods
