Step 1

Explore StorageClasses

StorageClasses are the foundation of dynamic storage provisioning in Kubernetes. Students will discover what is available on their EKS cluster before creating any storage resources.

🔍 Step 1: Explore StorageClasses

List all StorageClasses available on the cluster:


# Create a namespace for this lab
$ kubectl create namespace lab-storage

# List all StorageClasses
$ kubectl get storageclass

Expected Output:


NAME            PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE
gp2 (default)  kubernetes.io/aws-ebs   Delete          WaitForFirstConsumer
gp3-encrypted  ebs.csi.aws.com         Delete          WaitForFirstConsumer
The output may vary depending on the EKS cluster configuration. Most EKS clusters have a default gp2 StorageClass. Some may also have gp3 classes configured. The key fields to explain are provisioner (who creates the volume), reclaim policy (what happens when the PVC is deleted), and volume binding mode (when the volume gets created).

🔍 Step 1: Describe the Default StorageClass

Examine the default StorageClass in detail:


# Describe the default StorageClass
$ kubectl describe storageclass gp2

Expected Output (key fields):


Name:            gp2
IsDefaultClass:  Yes
Annotations:     storageclass.kubernetes.io/is-default-class=true
Provisioner:     kubernetes.io/aws-ebs
Parameters:      fsType=ext4,type=gp2
ReclaimPolicy:   Delete
VolumeBindingMode:  WaitForFirstConsumer

Key Concepts: The Provisioner tells Kubernetes which driver creates the actual storage. WaitForFirstConsumer means the volume is not created until a pod actually needs it, ensuring it is provisioned in the correct availability zone.

Walk through each field in the describe output. WaitForFirstConsumer is particularly important in multi-AZ EKS clusters because EBS volumes are AZ-specific. If the volume were created immediately, it might end up in a different AZ than the pod that needs it. The Delete reclaim policy means the underlying EBS volume is destroyed when the PVC is deleted, which is important for cost management.

Step 2

Create a PersistentVolumeClaim

Now students will create their first PVC. This is the standard way applications request storage in Kubernetes. The PVC acts as a request for storage that the StorageClass fulfills dynamically.

📦 Step 2: Create a PersistentVolumeClaim

Create a PVC requesting 1Gi of storage:


# Save as lab-pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: lab-data-pvc
  namespace: lab-storage
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: gp2
  resources:
    requests:
      storage: 1Gi
                

# Apply the PVC
$ kubectl apply -f lab-pvc.yaml

# Check the PVC status
$ kubectl get pvc -n lab-storage

Note: The PVC status will show Pending because the binding mode is WaitForFirstConsumer. The volume will not be provisioned until a pod references this PVC. This is expected behavior.

Students may be confused by the Pending status. Emphasize that this is expected with WaitForFirstConsumer. The volume does not get created until a pod actually mounts it. This prevents creating volumes in the wrong availability zone. ReadWriteOnce means only one node can mount the volume at a time, which is the standard mode for EBS volumes.

Step 3

Deploy a Pod with the PVC Mounted

This step demonstrates how to mount a PVC into a pod. The busybox container will write data to the mounted volume so students can verify persistence in the next step.

💻 Step 3: Deploy a Pod with the PVC

Create a pod that mounts the PVC and writes data:


# Save as lab-writer-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: data-writer
  namespace: lab-storage
spec:
  containers:
    - name: writer
      image: busybox:1.36
      command: ["/bin/sh", "-c"]
      args:
        - |
          echo "Written at $(date)" > /data/testfile.txt
          echo "Hostname: $(hostname)" >> /data/testfile.txt
          cat /data/testfile.txt
          sleep 3600
      volumeMounts:
        - name: lab-volume
          mountPath: /data
  volumes:
    - name: lab-volume
      persistentVolumeClaim:
        claimName: lab-data-pvc
                

# Apply the pod
$ kubectl apply -f lab-writer-pod.yaml

# Wait for the pod to be running
$ kubectl get pod data-writer -n lab-storage -w
Once the pod is created, Kubernetes will bind the PVC and provision the EBS volume. The busybox container writes a timestamp and hostname to a file in the mounted volume, then sleeps. Students should wait until the pod status shows Running before proceeding.

💻 Step 3: Verify the Volume is Bound

Check that the PVC is now bound and the data was written:


# Check PVC status - should now be Bound
$ kubectl get pvc -n lab-storage

# Verify the data was written
$ kubectl exec data-writer -n lab-storage -- cat /data/testfile.txt

# Check the pod logs
$ kubectl logs data-writer -n lab-storage

Expected Output:


NAME           STATUS   VOLUME                                     CAPACITY
lab-data-pvc   Bound    pvc-a1b2c3d4-e5f6-7890-abcd-ef1234567890   1Gi

Written at Thu Mar 13 14:30:00 UTC 2026
Hostname: data-writer
Lab 3 persistence test
The PVC should now show Bound status. The VOLUME column shows the dynamically provisioned PersistentVolume name. The data written by the container confirms the volume is mounted and writable. Note the timestamp as students will verify this same data survives pod deletion in the next step.

Step 4

Verify Data Persistence

This is the critical step that demonstrates why persistent volumes matter. Students will delete the pod and create a new one that reads the same volume to prove data survives pod lifecycle events.

🔄 Step 4: Verify Data Persistence

Delete the pod and create a new one to verify data survives:


# Delete the writer pod
$ kubectl delete pod data-writer -n lab-storage

# Verify the PVC is still Bound
$ kubectl get pvc -n lab-storage
                

# Save as lab-reader-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: data-reader
  namespace: lab-storage
spec:
  containers:
    - name: reader
      image: busybox:1.36
      command: ["/bin/sh", "-c"]
      args:
        - echo "=== Data from previous pod ===" && cat /data/testfile.txt && sleep 3600
      volumeMounts:
        - name: lab-volume
          mountPath: /data
  volumes:
    - name: lab-volume
      persistentVolumeClaim:
        claimName: lab-data-pvc
                

# Apply the reader pod
$ kubectl apply -f lab-reader-pod.yaml

# Verify data from the previous pod is still there
$ kubectl logs data-reader -n lab-storage

Success: The file written by data-writer is still present in the volume, even though that pod no longer exists. The original timestamp proves this is the same data.

This is the key insight of the lab. The PVC and its underlying PV persist independently of any pod. The data written by the first pod is still available to the second pod. This is exactly how databases, message queues, and other stateful services maintain data across pod restarts and rescheduling events. Point out that the PVC remained Bound even after the pod was deleted.

Step 5

Explore the Dynamically Provisioned PV

Students will now examine the PersistentVolume that was automatically created by the StorageClass provisioner. This connects the Kubernetes abstraction to the actual AWS EBS volume.

🔎 Step 5: Explore the PersistentVolume

Examine the dynamically created PV:


# List all PersistentVolumes
$ kubectl get pv

# Describe the PV (use the name from the output above)
$ kubectl describe pv $(kubectl get pvc lab-data-pvc -n lab-storage -o jsonpath='{.spec.volumeName}')

Key Fields in the Output:


Source:
  Type:       AWSElasticBlockStore (or CSI)
  VolumeID:   vol-0abc123def456789
  FSType:     ext4
Claim:        lab-storage/lab-data-pvc
Node Affinity:
  Required Terms:
    topology.kubernetes.io/zone: us-east-1a

Key Insight: The PV is bound to a specific AWS availability zone. This is why WaitForFirstConsumer is essential — it ensures the volume is created in the same AZ as the pod.

Walk students through the describe output. The VolumeID corresponds to an actual EBS volume in AWS. The node affinity constraint ensures the volume can only be attached to nodes in the same AZ. This is a fundamental constraint of EBS that students need to understand when designing stateful workloads. The FSType shows the filesystem format used on the volume.

Step 6

Deploy a StatefulSet

Now students transition from manual PVC management to StatefulSets, which automate the creation of per-replica persistent storage. This is the standard pattern for databases and other stateful services in Kubernetes.

🚀 Step 6: Deploy a StatefulSet

Create a StatefulSet with volumeClaimTemplates:


# Save as lab-statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata: { name: web, namespace: lab-storage }
spec:
  serviceName: "web-headless"
  replicas: 3
  selector: { matchLabels: { app: web } }
  template:
    metadata: { labels: { app: web } }
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports: [{ containerPort: 80 }]
          volumeMounts:
            - { name: web-data, mountPath: /usr/share/nginx/html }
  volumeClaimTemplates:
    - metadata: { name: web-data }
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: gp2
        resources: { requests: { storage: 1Gi } }
Point out the key differences from a Deployment: the serviceName field links to a headless Service, and volumeClaimTemplates automatically creates a unique PVC for each replica. Each pod gets its own dedicated volume rather than sharing one. The naming convention is predictable: web-data-web-0, web-data-web-1, web-data-web-2.

🚀 Step 6: Create the Headless Service and Apply

A StatefulSet requires a headless Service for DNS-based pod discovery:


# Save as lab-headless-svc.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-headless
  namespace: lab-storage
spec:
  clusterIP: None
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 80
                

# Apply both resources
$ kubectl apply -f lab-headless-svc.yaml
$ kubectl apply -f lab-statefulset.yaml

# Watch pods come up in order
$ kubectl get pods -n lab-storage -l app=web -w

Observe: Pods are created sequentially: web-0 must be Running and Ready before web-1 starts, and so on. This ordered startup is a key StatefulSet guarantee.

The headless Service (clusterIP: None) is required for StatefulSets. It does not load-balance traffic but instead creates DNS records for each individual pod. Students should observe the ordered creation: web-0 starts first, then web-1, then web-2. Press Ctrl+C to exit the watch once all three pods are Running.

Step 7

Verify Stable Identities

StatefulSets provide three guarantees: stable network identity, stable storage, and ordered operations. Students will now verify the first two by examining pod names, DNS records, and PVCs.

🎯 Step 7: Verify Stable Identities

Examine the predictable naming and DNS records:


# List pods - note the sequential naming
$ kubectl get pods -n lab-storage -l app=web -o wide

# List the auto-created PVCs
$ kubectl get pvc -n lab-storage

# Test DNS resolution for each pod
$ kubectl run dns-test --image=busybox:1.36 -n lab-storage --rm -it \
  --restart=Never -- nslookup web-0.web-headless.lab-storage.svc.cluster.local

Expected Output:


NAME    READY   STATUS    IP
web-0   1/1     Running   10.0.1.15
web-1   1/1     Running   10.0.2.22
web-2   1/1     Running   10.0.3.18

NAME                STATUS   VOLUME        CAPACITY
web-data-web-0      Bound    pvc-...       1Gi
web-data-web-1      Bound    pvc-...       1Gi
web-data-web-2      Bound    pvc-...       1Gi
Key Insight: Each pod has a predictable name (web-0, web-1, web-2) and its own dedicated PVC (web-data-web-0, etc.). Even if a pod is deleted and rescheduled, it reconnects to the same PVC and gets the same DNS name.
Highlight the naming convention: the PVC name combines the volumeClaimTemplate name with the pod name. Each pod can be addressed individually via DNS using the pattern pod-name.service-name.namespace.svc.cluster.local. This is how database replicas discover each other for replication. If the DNS test pod fails, ensure the headless service was created correctly.

Step 8

Test Data Persistence Per Replica

Students will write unique data to each pod's volume, demonstrating that each StatefulSet replica has its own independent storage.

📝 Step 8: Write Unique Data to Each Replica

Write unique content to each pod's volume:


# Write unique data to each pod
$ kubectl exec web-0 -n lab-storage -- \
  sh -c 'echo "<h1>Hello from web-0</h1>" > /usr/share/nginx/html/index.html'

$ kubectl exec web-1 -n lab-storage -- \
  sh -c 'echo "<h1>Hello from web-1</h1>" > /usr/share/nginx/html/index.html'

$ kubectl exec web-2 -n lab-storage -- \
  sh -c 'echo "<h1>Hello from web-2</h1>" > /usr/share/nginx/html/index.html'

# Verify each pod serves its own content
$ for i in 0 1 2; do
    echo "--- web-$i ---"
    kubectl exec web-$i -n lab-storage -- curl -s localhost
  done

Expected: Each pod returns its own unique content, confirming that volumes are not shared between replicas.

This demonstrates that each StatefulSet replica has its own isolated storage. Unlike a Deployment with a shared PVC (which is not even possible with ReadWriteOnce), each pod in a StatefulSet gets a dedicated volume. This is essential for databases where each replica has its own data directory.

🔄 Step 8: Verify Persistence After Pod Deletion

Delete a pod and verify its data survives:


# Delete web-1
$ kubectl delete pod web-1 -n lab-storage

# Watch it get recreated with the same name
$ kubectl get pods -n lab-storage -l app=web -w

# Once web-1 is Running again, verify data persisted
$ kubectl exec web-1 -n lab-storage -- curl -s localhost

Success: The recreated web-1 pod still serves "Hello from web-1" because it reattached to the same PVC (web-data-web-1).

Common Mistake: Do not confuse this with a Deployment. In a Deployment, the replacement pod gets a random name and may get a different volume. StatefulSets guarantee identity and storage affinity.

This is the most important demonstration in the lab. When web-1 is deleted, the StatefulSet controller creates a new pod with the exact same name (web-1) and mounts the exact same PVC (web-data-web-1). The data written earlier is still present. This guarantee is what makes StatefulSets suitable for databases, where losing a replica's data would be catastrophic.

Step 9

Scale the StatefulSet

Scaling StatefulSets follows ordered semantics. Students will observe that scale-up adds pods from the highest ordinal and scale-down removes from the highest ordinal, in sequence.

📈 Step 9: Scale Up the StatefulSet

Scale from 3 to 5 replicas and observe ordered creation:


# Scale up to 5 replicas
$ kubectl scale statefulset web -n lab-storage --replicas=5

# Watch the ordered scale-up
$ kubectl get pods -n lab-storage -l app=web -w

Expected: web-3 is created first, then web-4 after web-3 is Running. New PVCs web-data-web-3 and web-data-web-4 are also created.


# Verify new PVCs were created
$ kubectl get pvc -n lab-storage
Scale-up is ordered: web-3 must be Running and Ready before web-4 starts. Each new replica gets its own PVC automatically from the volumeClaimTemplate. This ordered startup is critical for distributed databases where new replicas need to sync from existing ones before the next replica can join.

📈 Step 9: Scale Down the StatefulSet

Scale down from 5 to 2 replicas and observe ordered termination:


# Scale down to 2 replicas
$ kubectl scale statefulset web -n lab-storage --replicas=2

# Watch the ordered scale-down
$ kubectl get pods -n lab-storage -l app=web -w

Expected: Pods are removed in reverse order: web-4 terminates first, then web-3, then web-2.


# Check PVCs - they persist even after scale-down!
$ kubectl get pvc -n lab-storage

Important: PVCs are not deleted when a StatefulSet scales down. This is by design — it protects data. The PVCs for web-2, web-3, and web-4 still exist and will be reattached if you scale back up.

Two critical observations: First, scale-down is reverse-ordered (highest ordinal first). This protects databases where the lowest-ordinal pod is typically the primary. Second, PVCs are intentionally retained after scale-down. This is a safety feature. If you scale back up, the pods will reattach to their existing PVCs with all data intact. Manual PVC deletion is required if you want to reclaim the storage.

Step 10

Test Volume Expansion

Volume expansion is a common operational task. Students will learn how to increase PVC size without downtime, provided the StorageClass supports it.

🔼 Step 10: Test Volume Expansion

Check if the StorageClass supports volume expansion:


# Check if expansion is allowed
$ kubectl get storageclass gp2 -o jsonpath='{.allowVolumeExpansion}'

Note: If the output is empty or false, you need to patch the StorageClass first:


# Enable volume expansion if needed
$ kubectl patch storageclass gp2 -p '{"allowVolumeExpansion": true}'

# Expand the PVC for web-0 from 1Gi to 2Gi
$ kubectl patch pvc web-data-web-0 -n lab-storage \
  -p '{"spec":{"resources":{"requests":{"storage":"2Gi"}}}}'

# Monitor the expansion
$ kubectl get pvc web-data-web-0 -n lab-storage -w

Expected: The PVC capacity will update to 2Gi. The underlying EBS volume is resized by the CSI driver. Filesystem expansion happens automatically when the pod is restarted or in-place on newer drivers.

Warning: Volume expansion is a one-way operation. You cannot shrink a PVC after expanding it. Always verify the target size before applying.

Volume expansion is supported by most modern CSI drivers including the EBS CSI driver. The process involves two steps: the API server updates the PVC spec, and the CSI driver resizes the underlying volume. Filesystem expansion may require a pod restart depending on the driver version. Emphasize that expansion cannot be reversed. In production, always coordinate volume expansion with the application team.

Step 11

Clean Up

Clean up all resources created during the lab. Deleting the namespace will remove all resources within it, including PVCs and their underlying PVs (because the reclaim policy is Delete).

🗑️ Step 11: Clean Up Lab Resources

Remove all resources created during this lab:


# Delete the StatefulSet first
$ kubectl delete statefulset web -n lab-storage

# Delete remaining pods
$ kubectl delete pod data-reader -n lab-storage --ignore-not-found

# Delete all PVCs in the namespace
$ kubectl delete pvc --all -n lab-storage

# Delete the namespace (cleans up services and any remaining resources)
$ kubectl delete namespace lab-storage

# Verify PVs are also deleted (due to Delete reclaim policy)
$ kubectl get pv

Note: Deleting the PVCs triggers deletion of the underlying PersistentVolumes and their AWS EBS volumes because the reclaim policy is Delete. Always verify PV cleanup to avoid orphaned EBS volumes that incur cost.

Order matters during cleanup. Delete the StatefulSet first so it does not try to recreate pods while you are deleting PVCs. Then delete PVCs, which triggers PV deletion. Finally delete the namespace. Verify that no PVs remain. In production, you might use a Retain reclaim policy for critical data, which requires manual PV cleanup.

📊 StatefulSet vs Deployment

StatefulSet Guarantees

  • Stable pod names — web-0, web-1, web-2
  • Stable DNS — pod-name.service.namespace
  • Ordered operations — create/delete in sequence
  • Per-replica storage — each pod gets its own PVC
  • PVC retention — data survives scale-down

When to Use StatefulSets

  • Databases (PostgreSQL, MySQL, MongoDB)
  • Message queues (Kafka, RabbitMQ)
  • Distributed caches (Redis Cluster, Memcached)
  • Search engines (Elasticsearch)
  • Any workload needing stable identity
Rule of Thumb: If your application needs to know "who it is" or needs data to survive pod restarts, use a StatefulSet. If it is stateless and interchangeable, use a Deployment.
This comparison slide helps students decide when to use each workload type. Most microservices are stateless and should use Deployments. Databases and similar infrastructure components should use StatefulSets. At Verisign, the platform team manages StatefulSets for shared services while application teams typically deploy stateless services via Deployments.
