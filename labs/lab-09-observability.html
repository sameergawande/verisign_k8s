Step 1

Explore Container Logs with kubectl

We start with the most fundamental observability tool: kubectl logs. Students should understand all the important flags.

📝 Set Up the Lab Namespace

Create a dedicated namespace for this lab:


# Create the lab namespace
kubectl create namespace observability-lab

# Set as the current context
kubectl config set-context --current --namespace=observability-lab
Expected Output: namespace/observability-lab created Context modified. Starting with a clean namespace keeps the lab isolated and makes cleanup straightforward at the end.

📝 Deploy a Simple Pod for Log Exploration


# Deploy a pod that generates log output
kubectl run log-generator --image=busybox \
  --restart=Never -- \
  sh -c 'i=0; while true; do
    echo "$(date +%Y-%m-%dT%H:%M:%S) INFO  request_id=$i \
      status=200 path=/api/health duration=12ms";
    echo "$(date +%Y-%m-%dT%H:%M:%S) DEBUG request_id=$i cache_hit=true";
    if [ $((i % 10)) -eq 0 ]; then
      echo "$(date +%Y-%m-%dT%H:%M:%S) WARN  request_id=$i \
        high_latency=true duration=1500ms";
    fi;
    i=$((i+1)); sleep 2;
  done'

Wait for the pod to be running:


kubectl wait --for=condition=Ready pod/log-generator --timeout=30s
This creates a pod that produces structured log output at different levels. Every 10th entry includes a WARN line to simulate latency spikes.

🔍 Explore kubectl logs Flags


# View last 20 lines of logs
kubectl logs log-generator --tail=20

# Follow logs in real time (Ctrl+C to stop)
kubectl logs log-generator -f

# View logs from the last 5 minutes
kubectl logs log-generator --since=5m

# View logs with timestamps
kubectl logs log-generator --timestamps
Key Flags: -f -- Follow (stream) logs in real time --previous -- Show logs from a previous container instance -c -- Select a specific container in multi-container pods --tail=N -- Show only the last N lines --since=duration -- Show logs newer than a relative duration The --previous flag is extremely valuable for debugging CrashLoopBackOff situations where the container has already restarted and the current logs are empty. The -c flag is essential for multi-container pods like those with sidecars.

📦 Multi-Container Pod Logs


cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: multi-container-app
  namespace: observability-lab
spec:
  containers:
  - name: app
    image: busybox
    command: ['sh', '-c', 'while true; do echo "$(date) APP: \
      Processing request"; sleep 3; done']
  - name: sidecar
    image: busybox
    command: ['sh', '-c', 'while true; do echo "$(date) SIDECAR: \
      Collecting metrics"; sleep 5; done']
EOF
                

# View logs from a specific container
kubectl logs multi-container-app -c app
kubectl logs multi-container-app -c sidecar

# View logs from ALL containers
kubectl logs multi-container-app --all-containers=true
Multi-container pods are common with sidecar patterns. Being able to target specific container logs is essential.

Step 2

Structured JSON Logging

Structured logs are the standard in production environments because they can be parsed and queried by log aggregation systems.

📊 Deploy an App with JSON Logs


cat <<'EOF' | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata: { name: json-logger, namespace: observability-lab }
spec:
  replicas: 2
  selector:
    matchLabels: { app: json-logger }
  template:
    metadata: { labels: { app: json-logger } }
    spec:
      containers:
      - name: logger
        image: busybox
        command: ['sh', '-c', 'i=0; while true; do
          printf "{\"timestamp\":\"%s\",\"level\":\"info\",
          \"service\":\"order-api\",\"request_id\":\"%d\",
          \"status\":200}\n"
          "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "$i";
          sleep 1; i=$((i+1)); done']
EOF
Expected: Deployment creates 2 replicas producing structured JSON log lines. JSON logging is the gold standard for production because it enables automated parsing, filtering, and aggregation. Tools like Fluentd, Fluent Bit, and Loki can ingest these directly.

🔎 View and Filter JSON Logs


# View logs from all pods in the deployment
kubectl logs deployment/json-logger --all-containers=true --tail=5

# Follow logs from the deployment
kubectl logs deployment/json-logger -f --max-log-requests=5

# Use label selector to get logs from specific pods
kubectl logs -l app=json-logger --tail=10
Tip: You can pipe JSON logs through jq for local filtering: kubectl logs -l app=json-logger --tail=20 | jq 'select(.status != 200)' The label selector approach is powerful because it works regardless of how many pod replicas exist. The jq tool is invaluable for local log analysis.

Step 3

Log Patterns and Best Practices

Review logging best practices before moving on to metrics-based observability.

📋 Log Patterns and Best Practices

✅ Good Practices

  • Structured JSON format
  • Log to stdout/stderr
  • Use consistent log levels
  • Include correlation IDs and timestamps

❌ Anti-Patterns

  • Writing logs to files inside containers
  • Logging sensitive data (tokens, passwords)
  • Excessive DEBUG logging in production
Kubernetes captures stdout and stderr from containers. Writing to files inside the container bypasses the kubelet log collection pipeline entirely.

🎯 Deploy a Resource-Intensive Workload


cat <<'EOF' | kubectl apply -f -

  replicas: 2

This deployment creates CPU load so students can see real resource consumption in kubectl top and later in Prometheus/Grafana.

🚀 Install kube-prometheus-stack

# Install the kube-prometheus-stack

Step 6

Query Prometheus Metrics

PromQL is the query language for Prometheus. Understanding its fundamentals is essential for effective monitoring and alerting.

🌐 Access the Prometheus UI


# Port-forward to Prometheus
kubectl port-forward -n monitoring \
  svc/monitoring-kube-prometheus-prometheus 9090:9090 &

echo "Prometheus UI: http://localhost:9090"
Tip: Open http://localhost:9090 in your browser. Navigate to Status → Targets to see all scrape targets. The Targets page shows all endpoints Prometheus is scraping. Green means healthy; red indicates a scrape failure that needs investigation.

📊 Run PromQL Queries

Navigate to the Graph tab and run these queries:


# Total CPU usage per node
sum(rate(node_cpu_seconds_total{mode!="idle"}[5m])) by (instance)

# Memory usage percentage per node
100 * (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes))

# Pod CPU usage in the observability-lab namespace
sum(rate(container_cpu_usage_seconds_total{
  namespace="observability-lab"
}[5m])) by (pod)

# Container restart count
sum(kube_pod_container_status_restarts_total) by (namespace, pod)
Key PromQL Concepts: rate() -- Per-second rate of increase for counters sum() by (label) -- Aggregate and group results [5m] -- Range vector selector (lookback window) Encourage students to switch between the Console and Graph views to see both raw numbers and time-series visualizations. The rate function is essential because CPU counters are cumulative.

🔎 Useful Cluster Queries


count(kube_pod_info) by (namespace)

kube_pod_container_status_waiting_reason{reason="CrashLoopBackOff"}

kube_deployment_status_replicas_available
  / kube_deployment_spec_replicas

topk(5, container_memory_working_set_bytes{
  container!="", container!="POD"
})
🔑 Key Point: PromQL queries form the foundation of both dashboards and alerting rules. Master these patterns and you can monitor any aspect of your cluster.
These queries represent the most common production monitoring scenarios. The CrashLoopBackOff query is particularly valuable for on-call engineers.

Step 8

Create a Custom PrometheusRule Alert

PrometheusRule CRDs allow you to define alerting rules that Prometheus evaluates continuously. This is how production alerting is configured in Kubernetes.

🔔 Define a High Restart Count Alert


cat <<'EOF' | kubectl apply -f -
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: pod-restart-alert
  namespace: monitoring
  labels: { release: monitoring }
spec:
  groups:
  - name: pod-restarts
    rules:
    - alert: HighPodRestartCount
      expr: increase(kube_pod_container_status_restarts_total[1h]) > 3
      for: 5m
      labels: { severity: warning }
      annotations:
        summary: >-
          Pod {{ $labels.pod }} restarted
          {{ $value }} times
    - alert: PodCrashLooping
      expr: rate(kube_pod_container_status_restarts_total[15m]) * 60 * 15 > 0
      labels: { severity: critical }
EOF
Tip: The labels.release must match the Helm release name for the Prometheus operator to pick up this rule. The 'for' clause prevents transient spikes from firing alerts.

✅ Verify the Alert Rule


# Check the PrometheusRule resource
kubectl get prometheusrules -n monitoring

# Verify in Prometheus UI: Status -> Rules
# Navigate to http://localhost:9090/rules
# Look for the "pod-restarts" group
Expected: The pod-restart-alert rule appears in the Prometheus Rules page with state inactive (no pods are currently restarting excessively). Troubleshooting: If the rule does not appear, verify the release: monitoring label matches your Helm release name: helm list -n monitoring Rules may take up to 60 seconds to be loaded by Prometheus after creation. The inactive state means the condition is not currently true — that is expected.

Step 10

Debug a Failing Application

This is the culmination of the lab: using all the observability tools together in a systematic debugging workflow.

🐛 Deploy a Broken Application


cat <<'EOF' | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata: { name: buggy-app, namespace: observability-lab }
spec:
  replicas: 3
  selector:
    matchLabels: { app: buggy-app }
  template:
    metadata: { labels: { app: buggy-app } }
    spec:
      containers:
      - name: app
        image: busybox
        command: ['sh', '-c', 'echo "Starting app..."; sleep \
          $((RANDOM % 10 + 5)); echo "ERROR: Out of memory"; exit 137']
EOF
Expected: This app will crash and enter CrashLoopBackOff after a few iterations. That is intentional! The app simulates an OOM-like crash. Students will use the systematic debugging approach to diagnose it.

🔍 Systematic Debugging: Step 1 — Events


# Check pod status
kubectl get pods -l app=buggy-app

# Check events for the namespace
kubectl get events --sort-by='.lastTimestamp' \
  -n observability-lab | tail -20

# Describe a specific crashing pod
kubectl describe pod -l app=buggy-app | grep -A 10 "State:"
What to Look For: Pod status: CrashLoopBackOff Last State: Terminated with exit code 137 Restart count increasing Exit code 137 means the process received SIGKILL, which typically indicates an OOM kill. The describe command shows both current and last state.

🔍 Systematic Debugging: Step 2 — Logs


# View current container logs
kubectl logs -l app=buggy-app --tail=20

# View PREVIOUS container logs (from before the crash)
kubectl logs -l app=buggy-app --previous --tail=20
Expected Output: Starting app... ERROR: Out of memory - cannot allocate buffer
🔑 Key Insight: The --previous flag is essential when a container has crashed and restarted. Without it, you only see the current (possibly empty) container's logs.
This is the most common debugging mistake: forgetting --previous. When a pod is in CrashLoopBackOff, the current container may have just started and have no useful logs yet.

🔍 Systematic Debugging: Step 3 — Metrics


# Check resource consumption
kubectl top pods -l app=buggy-app

# In Prometheus, query restart count:
# increase(kube_pod_container_status_restarts_total{
#   namespace="observability-lab", pod=~"buggy-app.*"
# }[30m])

# Check OOMKilled status in Prometheus:
# kube_pod_container_status_last_terminated_reason{
#   namespace="observability-lab",
#   reason="OOMKilled"
# }

🎯 Debugging Flowchart

  1. Events → What happened? When?
  2. Logs → What did the app report?
  3. Metrics → What was the resource state?
  4. Describe → What are the pod's current conditions?
This systematic approach — events, logs, metrics, describe — should become second nature. It covers 90% of production debugging scenarios.
