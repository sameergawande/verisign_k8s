Step 1

Explore the CI/CD Pipeline Structure

We start by examining a sample GitHub Actions workflow to understand the structure of a Kubernetes CI/CD pipeline. This sets the context for the hands-on steps that follow.

📄 Sample GitHub Actions Workflow

Examine a typical CI/CD pipeline for Kubernetes:


# .github/workflows/deploy.yaml
name: Build and Deploy
on: { push: { branches: [main] } }
env:
  ECR_REGISTRY: ${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.us-east-1.amazonaws.com
  ECR_REPOSITORY: my-app
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Configure AWS credentials
      uses: aws-actions/configure-aws-credentials@v4
      with:
        aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
        aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        aws-region: us-east-1
    - name: Build and push image
      run: |
        IMAGE_TAG=$(git rev-parse --short HEAD)
        docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
        docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
Walk through each section of the workflow. Explain how secrets are used for credentials, how the git SHA is used for image tagging, and why ECR is a common choice for AWS-based Kubernetes deployments.

📄 Pipeline Stages

A Kubernetes CI/CD pipeline typically has these stages:

🛠 Build

  • Run tests
  • Build container image
  • Tag with git SHA
  • Scan for vulnerabilities

📦 Push

  • Authenticate to registry
  • Push image to ECR
  • Tag as latest (optional)
  • Sign image (optional)

🚀 Deploy

  • Update manifest/values
  • Apply via kubectl or GitOps
  • Verify rollout
  • Run smoke tests
Emphasize that the Build and Push stages are the CI part, while the Deploy stage is the CD part. GitOps tools like ArgoCD and Flux handle the CD part by watching a git repository for changes.

Step 2

Build and Tag a Container Image

Now we build a container image and tag it with the git SHA. Using the git SHA as the image tag provides traceability from the running container back to the exact source code commit.

🛠 Create a Sample Application

Set up a simple application to build and deploy:


# Create working directory, init git, create app
mkdir -p ~/cicd-lab && cd ~/cicd-lab && git init

cat <<'EOF' > index.html
<html><body>
  <h1>CI/CD Demo Application</h1>
  <p>Build: __BUILD_SHA__</p>
  <p>Deployed: __DEPLOY_TIME__</p>
</body></html>
EOF

cat <<'EOF' > Dockerfile
FROM nginx:1.25-alpine
COPY index.html /usr/share/nginx/html/index.html
EXPOSE 80
EOF

git add . && git commit -m "Initial application"
This creates a minimal application that displays its build SHA and deploy time. The placeholders will be replaced during the build process.

🛠 Build and Tag with Git SHA

Build the image and tag it with the current git commit SHA:


# Get the short git SHA
GIT_SHA=$(git rev-parse --short HEAD)
echo "Building with SHA: $GIT_SHA"

# Replace the build placeholder
sed -i.bak "s/__BUILD_SHA__/$GIT_SHA/g" index.html
sed -i.bak "s/__DEPLOY_TIME__/$(date -u +%Y-%m-%dT%H:%M:%SZ)/g" index.html

# Build the Docker image with the git SHA tag
docker build -t my-app:$GIT_SHA .
docker build -t my-app:latest .

# Verify the images
docker images my-app

# Restore the template file
mv index.html.bak index.html
✅ Expected Output: Two tagged images: my-app:<sha> and my-app:latest. The SHA tag provides immutable, traceable references.
⚠ Best Practice: Never rely solely on :latest for production deployments. Always use a specific, immutable tag like the git SHA.
Explain why git SHA tagging is critical. It creates a direct link between the running container and the exact source code. The latest tag is mutable and can lead to inconsistent deployments.

Step 3

Push to Container Registry (ECR)

Pushing to ECR is a standard step in AWS-based Kubernetes deployments. The ECR login token is valid for 12 hours. If ECR is not available, students can skip this step and use a local image for deployment.

📤 Authenticate and Push to ECR

Login to ECR and push the built image:


# Set your ECR registry URL
export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
export ECR_REGISTRY="${AWS_ACCOUNT_ID}.dkr.ecr.us-east-1.amazonaws.com"
export ECR_REPO="cicd-lab-app"

# Create the ECR repository (if it doesn't exist)
aws ecr create-repository --repository-name $ECR_REPO \
  --region us-east-1 || echo "Repository already exists"

# Authenticate Docker to ECR
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin $ECR_REGISTRY

# Tag and push the image
docker tag my-app:$GIT_SHA $ECR_REGISTRY/$ECR_REPO:$GIT_SHA
docker push $ECR_REGISTRY/$ECR_REPO:$GIT_SHA

# Verify the push
aws ecr describe-images --repository-name $ECR_REPO --region us-east-1
✅ Expected Output: The image should appear in the ECR repository with the git SHA as the image tag. The describe-images output shows the image digest and tags.
⚠ Troubleshooting: If authentication fails, verify your AWS credentials: aws sts get-caller-identity. Ensure your IAM role has ecr:GetAuthorizationToken and ecr:BatchCheckLayerAvailability permissions.
ECR login tokens expire after 12 hours. In CI/CD pipelines, the login step runs every time. For local development, you may need to re-authenticate periodically.

Step 4

Create a Deployment Manifest

The deployment manifest references the image we just pushed to ECR. In a GitOps workflow, this manifest lives in a git repository and changes trigger deployments.

📝 Create the Deployment Manifest

Write a Kubernetes Deployment manifest referencing the ECR image:


# Save as k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cicd-demo
  namespace: cicd-lab
  annotations: { git-commit: "${GIT_SHA}" }
spec:
  replicas: 3
  selector:
    matchLabels: { app: cicd-demo }
  template:
    metadata:
      labels: { app: cicd-demo }
    spec:
      containers:
      - name: app
        image: ${ECR_REGISTRY}/${ECR_REPO}:${GIT_SHA}
        ports: [{containerPort: 80}]
        # ... resource requests/limits omitted for brevity
---
# ... Service: cicd-demo-svc (port 80 -> 80)
Note the git-commit annotation. This embeds the source commit reference directly in the Kubernetes resource, making it easy to trace back from a running deployment to the exact code version.

Step 5

Deploy Using kubectl apply

This is the imperative deployment approach. It works for learning but has limitations in production. GitOps tools provide a more robust and auditable deployment process.

🚀 Deploy with kubectl

Substitute variables and apply the manifest:


# Create the namespace
kubectl create namespace cicd-lab

# Create the manifests directory
mkdir -p ~/cicd-lab/k8s

# Substitute environment variables and apply
envsubst < k8s/deployment.yaml | kubectl apply -f -

# Watch the rollout
kubectl rollout status deployment/cicd-demo -n cicd-lab

# Verify the deployment
kubectl get pods -n cicd-lab -l app=cicd-demo
kubectl get svc -n cicd-lab

# Test the application
kubectl run curl-test --image=curlimages/curl --rm -it --restart=Never \
  -n cicd-lab -- curl -s cicd-demo-svc
✅ Expected Output: 3 pods running with the ECR image. The curl test should return the HTML page showing the build SHA.
⚠ Why Not kubectl in Production? Imperative deployments lack auditability, drift detection, and automated reconciliation. GitOps tools address all of these concerns.
If envsubst is not available, students can manually replace the variables in the YAML. This step demonstrates the simplest deployment method before introducing GitOps patterns.

Step 6

Set Up ArgoCD Application

ArgoCD is a GitOps continuous delivery tool for Kubernetes. It watches a git repository and ensures the cluster state matches the desired state defined in git. If ArgoCD is not installed, demonstrate the concepts using the Application CRD definition.

📦 Verify ArgoCD Installation

Check if ArgoCD is available in the cluster:


# Check if ArgoCD is installed
kubectl get namespace argocd

# List ArgoCD components
kubectl get pods -n argocd

# Check the ArgoCD CLI
argocd version
✅ If ArgoCD is available: You should see pods for argocd-server, argocd-repo-server, argocd-application-controller, and argocd-dex-server all in Running state.
⚠ If ArgoCD is not installed: Review the Application CRD definition below to understand the pattern. You can install ArgoCD with: kubectl create namespace argocd && kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
If ArgoCD is not installed, walk through the concepts using the CRD definition as a reference. The important thing is understanding the pattern, not necessarily having a running ArgoCD instance.

📦 Create an ArgoCD Application

Define an ArgoCD Application CRD:


# Save as argocd-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: cicd-demo
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/your-org/cicd-demo.git
    targetRevision: main
    path: k8s
  destination:
    server: https://kubernetes.default.svc
    namespace: cicd-lab
  syncPolicy:
    automated:
      prune: true            # Remove resources not in git
      selfHeal: true         # Revert manual changes
    syncOptions:
    - CreateNamespace=true

💡 Key ArgoCD Concepts

  • Source — the git repository and path containing manifests
  • Destination — the target cluster and namespace
  • Sync Policy — automated means ArgoCD syncs on git changes; prune removes deleted resources; selfHeal reverts manual drift
The automated sync policy is what makes this GitOps. Any change pushed to the main branch in the specified path will automatically be applied to the cluster. The selfHeal feature reverts any manual kubectl changes.

Step 7

Observe ArgoCD Sync

ArgoCD provides both a CLI and a web UI for monitoring application sync status. Students will learn both interfaces for observing deployment progress.

🔍 Monitor ArgoCD Sync Status

Use the ArgoCD CLI and UI to observe the sync:


# Apply the ArgoCD Application (if ArgoCD is available)
kubectl apply -f argocd-app.yaml

# Check application status via CLI

# Manually trigger a sync

# Watch the sync progress

# View the application in the web UI
# Get the ArgoCD server URL
kubectl get svc argocd-server -n argocd
# Get the initial admin password
kubectl -n argocd get secret argocd-initial-admin-secret \

✅ Expected Output: The application status should show Health: Healthy and Sync: Synced. All resources should show a green checkmark in the UI.
If using the web UI, port-forward the ArgoCD server: kubectl port-forward svc/argocd-server -n argocd 8080:443. The UI provides a visual dependency graph of all resources in the application.

Step 8

Trigger Pipeline with Code Change

In a real CI/CD pipeline, pushing a code change triggers the build. Here we simulate the process by updating the image tag in the deployment manifest and observing the automated sync.

🔄 Simulate a Pipeline Trigger

Make a code change, rebuild, and update the manifest:


# Make a code change and commit
cd ~/cicd-lab
sed -i 's/CI\/CD Demo Application/CI\/CD Demo v2/' index.html
git add . && git commit -m "Update application to v2"

# Get the new SHA and build
NEW_SHA=$(git rev-parse --short HEAD)
echo "New SHA: $NEW_SHA"
docker build -t my-app:$NEW_SHA .

# Update the deployment manifest with new image tag
sed -i "s|image: .*|image: ${ECR_REGISTRY}/${ECR_REPO}:${NEW_SHA}|" k8s/deployment.yaml
sed -i "s|git-commit: .*|git-commit: \"${NEW_SHA}\"|" k8s/deployment.yaml

# In a GitOps workflow, this commit triggers ArgoCD sync
git add k8s/deployment.yaml
git commit -m "Deploy image $NEW_SHA"
In a real pipeline, the CI system would build the image and a separate step or tool would update the manifest. Some teams use image updater tools that automatically detect new images in the registry.

🔄 Apply and Verify the Update

Apply the updated manifest and observe the rollout:


# Push the new image to ECR (if ECR is configured)
docker tag my-app:$NEW_SHA $ECR_REGISTRY/$ECR_REPO:$NEW_SHA
docker push $ECR_REGISTRY/$ECR_REPO:$NEW_SHA

# Apply the updated manifest (or let ArgoCD auto-sync)
envsubst < k8s/deployment.yaml | kubectl apply -f -

# Watch the rollout
kubectl rollout status deployment/cicd-demo -n cicd-lab

# If using ArgoCD, observe the sync
# argocd app get cicd-demo

# Verify the new version is running
kubectl run curl-test2 --image=curlimages/curl --rm -it --restart=Never \
  -n cicd-lab -- curl -s cicd-demo-svc
✅ Expected Output: The curl response should show CI/CD Demo v2 with the new build SHA. The deployment annotation should show the new git commit.
With ArgoCD automated sync, pushing the manifest change to git would trigger the deployment automatically. Without ArgoCD, we use kubectl apply manually. The key point is that the manifest in git is the source of truth.

Step 9

Rollback via ArgoCD

GitOps rollback is different from kubectl rollback. In GitOps, you rollback by reverting the git commit, which triggers a new sync. ArgoCD also supports direct rollback to a previous sync revision.

⏪ Rollback Strategies

📜 Git Revert (Preferred)


# Revert the last commit
git revert HEAD --no-edit

# Push to trigger ArgoCD sync
git push origin main

Maintains full audit trail in git history. ArgoCD detects the change and syncs automatically.

⚡ ArgoCD Rollback (Quick)


# View sync history
argocd app history cicd-demo

# Rollback to previous version
argocd app rollback cicd-demo 1

Faster but creates drift between git and cluster. ArgoCD will eventually re-sync from git.

⚠ Important: ArgoCD rollback creates temporary drift between the cluster and git. If automated sync is enabled, ArgoCD will re-sync from git within the poll interval (default 3 minutes). Use git revert for a permanent rollback.
Emphasize that the git revert approach is the true GitOps way. It maintains the audit trail and ensures git remains the source of truth. ArgoCD rollback is useful for emergency situations where speed is critical.

Step 10

Explore FluxCD Reconciliation

FluxCD is another popular GitOps tool. Unlike ArgoCD which uses a pull-based approach with a UI, Flux uses Kubernetes-native CRDs and a reconciliation loop. If Flux is not installed, review the concepts.

🌀 FluxCD Overview

If FluxCD is available, explore its reconciliation model:


# Check if Flux is installed
flux check

# View git sources
flux get sources git

# View kustomizations
flux get kustomizations

# Trigger a manual reconciliation
flux reconcile kustomization flux-system --with-source

FluxCD CRDs

  • GitRepository — watches a git repo
  • Kustomization — reconciles manifests from source
  • HelmRelease — manages Helm releases
  • HelmRepository — watches a chart repo

ArgoCD vs FluxCD

  • ArgoCD: web UI, app-of-apps pattern
  • FluxCD: CLI-first, Kubernetes-native CRDs
  • Both: GitOps, pull-based, reconciliation
  • Choose based on team preferences
Flux is CNCF graduated and deeply integrated with Kubernetes. It uses the same reconciliation pattern as Kubernetes controllers. ArgoCD provides a richer UI experience. Both are excellent choices for GitOps.

Step 11

Clean Up

Clean up all resources. If ArgoCD was used, delete the Application resource first to let ArgoCD clean up its managed resources.

🧹 Clean Up Resources

Remove all resources created during this lab:


# If ArgoCD Application was created, delete it first
kubectl delete application cicd-demo -n argocd --ignore-not-found

# Delete the namespace
kubectl delete namespace cicd-lab

# Delete the ECR repository (optional)
aws ecr delete-repository --repository-name cicd-lab-app \
  --region us-east-1 --force || echo "Skipping ECR cleanup"

# Clean up local files
rm -rf ~/cicd-lab

# Remove local Docker images
docker rmi $(docker images my-app -q) 2>/dev/null || true

# Verify cleanup
kubectl get namespace cicd-lab
✅ Expected Output: namespace "cicd-lab" deleted
Error from server (NotFound): namespaces "cicd-lab" not found
⚠ Note: If you installed ArgoCD specifically for this lab and no longer need it: kubectl delete namespace argocd. Be careful not to remove a shared ArgoCD installation.
When ArgoCD manages resources, deleting the Application CRD triggers ArgoCD to prune the managed resources. Always delete ArgoCD Applications before deleting the namespace to ensure clean teardown.

Step 12

Summary & CI/CD Patterns Reference

Wrap up with a summary of CI/CD patterns and best practices for Kubernetes deployments.

📊 CI/CD Patterns Comparison

Pattern Push vs Pull Audit Trail Drift Detection Complexity
kubectl apply Push Limited None Low
ArgoCD Pull Full (git) Automatic Medium
FluxCD Pull Full (git) Automatic Medium
Helm + CI Push Release history Manual Medium
Key Takeaway: GitOps (ArgoCD/FluxCD) provides the strongest guarantees: git as single source of truth, automatic drift detection, and full auditability. Start with kubectl for learning, adopt GitOps for production.
Discuss how organizations typically evolve from kubectl apply to GitOps. The journey often goes: manual kubectl, CI pipeline with kubectl, Helm, then GitOps with ArgoCD or Flux.
