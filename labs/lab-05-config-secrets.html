Lab 5: ConfigMaps and Secrets

Externalizing Configuration and Managing Sensitive Data

⏰ Estimated Time: 45 minutes

Welcome to Lab 5. In this lab, students will learn how to externalize application configuration using ConfigMaps and manage sensitive data with Secrets. These are foundational Kubernetes patterns that decouple configuration from container images, enabling the same image to run across dev, staging, and production environments on the Verisign EKS platform. Students will create ConfigMaps from literals and files, consume them as environment variables and volume mounts, create and consume Secrets, explore immutable resources, projected volumes, and observe ConfigMap update propagation behavior.

📚 Lab Overview

What You Will Do

  • Create ConfigMaps from literals and files
  • Consume ConfigMaps and Secrets as env vars and volume mounts
  • Create Secrets (Opaque, TLS) and use immutable resources
  • Combine resources with projected volumes
  • Observe ConfigMap update propagation

Prerequisites

  • Completion of Labs 1–4 with kubectl access
  • Familiarity with pods, deployments, and services
⚠️ Note: Secrets in Kubernetes are base64-encoded, not encrypted at rest by default. In production, use envelope encryption or an external secrets manager.
Review the objectives with students. Emphasize that ConfigMaps and Secrets are the standard way to inject configuration into pods. Highlight the security caveat about Secrets being base64-encoded, not encrypted. On the Verisign platform, AWS KMS envelope encryption should be enabled for etcd-level encryption of Secrets.

Step 1

Create ConfigMaps from Literals

Students begin by creating ConfigMaps imperatively using literal key-value pairs. This is the simplest way to create configuration data and is useful for quick testing and prototyping.

🚀 Step 1: Create ConfigMaps from Literals

Create a namespace and a ConfigMap using literal key-value pairs:


# Create the lab namespace
$ kubectl create namespace lab-config

# Create a ConfigMap from literal values
$ kubectl create configmap app-config \
    --from-literal=APP_ENV=production \
    --from-literal=APP_LOG_LEVEL=info \
    --from-literal=APP_MAX_CONNECTIONS=100 \
    --from-literal=APP_CACHE_TTL=3600 \
    -n lab-config
                

# Inspect the ConfigMap
$ kubectl get configmap app-config -n lab-config -o yaml
✅ Expected Output: The ConfigMap shows a data section with four key-value pairs. Each literal becomes a separate key in the map.
Walk through the output structure. Point out the apiVersion (v1), kind (ConfigMap), and the data section. Each --from-literal flag creates one key-value pair. The keys are case-sensitive and follow standard naming conventions.

🚀 Step 1: Verify and Describe

Use describe for a human-readable view:


$ kubectl describe configmap app-config -n lab-config
✅ Expected Output: 
Name:         app-config
Namespace:    lab-config
Data
====
APP_CACHE_TTL:
----
3600
APP_ENV:
----
production
APP_LOG_LEVEL:
----
info
APP_MAX_CONNECTIONS:
----
100
The describe output is easier to read than raw YAML for quick inspection. Point out that the data keys are sorted alphabetically in the describe output. Students should note that all values are stored as strings, even numeric values like 3600.

Step 2

Create ConfigMaps from Files

Now students will create ConfigMaps from files. This is the most common approach for injecting configuration files such as nginx.conf, application.properties, or any structured configuration file into pods.

🚀 Step 2: Create Configuration Files

Create local configuration files, then load them into a ConfigMap:


# Create an nginx configuration file
$ cat <<'EOF' > nginx.conf
server {
    listen 80;
    server_name localhost;
    location / { root /usr/share/nginx/html; index index.html; }
    location /health { access_log off; return 200 "OK\n"; }
    location /api {
        proxy_pass http://backend-svc:8080;
        proxy_set_header Host $host;
    }
}
EOF

# Create an application properties file
$ cat <<'EOF' > app.properties
database.host=postgres-svc.lab-config.svc.cluster.local
database.port=5432
database.name=appdb
database.pool.size=10
cache.enabled=true
cache.ttl=3600
EOF
Students create two files: an nginx configuration and an application properties file. The nginx config includes a health check endpoint and a proxy pass to a backend service using Kubernetes DNS. The app.properties file uses the full FQDN format for the database host.

🚀 Step 2: Load Files into a ConfigMap

Create a ConfigMap from the files:


# Create ConfigMap from files
$ kubectl create configmap app-files \
    --from-file=nginx.conf \
    --from-file=app.properties \
    -n lab-config

# Verify the ConfigMap
$ kubectl get configmap app-files -n lab-config -o yaml
✅ Expected Output: The ConfigMap data section contains two keys (nginx.conf and app.properties), each with the full file content as the value.
💡 Tip: Use --from-file=custom-key=filename to override the default key name. For example: --from-file=my-nginx=nginx.conf
When using --from-file, the filename becomes the key and the entire file content becomes the value. Students should understand the difference between --from-literal (single values) and --from-file (entire file contents). The custom key syntax is useful when the filename does not match the expected mount name.

Step 3

Consume ConfigMap as Environment Variables

Students will now consume ConfigMap data as environment variables in pods. There are two approaches: envFrom (inject all keys) and valueFrom (inject specific keys). Both patterns are common in production workloads.

🚀 Step 3: Using envFrom

Inject all ConfigMap keys as environment variables:


# Save as pod-envfrom.yaml
apiVersion: v1
kind: Pod
metadata:
  name: env-from-demo
  namespace: lab-config
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "env | sort && sleep 3600"]
      envFrom:
        - configMapRef:
            name: app-config
  restartPolicy: Never
                

$ kubectl apply -f pod-envfrom.yaml

# Wait for pod to be running, then check environment
$ kubectl logs env-from-demo -n lab-config | grep APP_
✅ Expected Output: 
APP_CACHE_TTL=3600
APP_ENV=production
APP_LOG_LEVEL=info
APP_MAX_CONNECTIONS=100
envFrom injects all keys from the ConfigMap as environment variables. This is convenient but can pollute the environment namespace. Point out that keys with invalid environment variable characters (like dots or hyphens) will be skipped with a warning event on the pod.

🚀 Step 3: Using valueFrom for Selective Injection

Inject specific keys with optional renaming:


# Save as pod-valuefrom.yaml
apiVersion: v1
kind: Pod
metadata:
  name: value-from-demo
  namespace: lab-config
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "echo Env=$ENVIRONMENT Log=$LOG_LEVEL && sleep 3600"]
      env:
        - name: ENVIRONMENT
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: APP_ENV
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: APP_LOG_LEVEL
  restartPolicy: Never
                

$ kubectl apply -f pod-valuefrom.yaml
$ kubectl logs value-from-demo -n lab-config
✅ Expected Output: Env=production Log=info
valueFrom allows selective injection and renaming of keys. This is the preferred approach when you only need specific values or when ConfigMap keys do not match the expected environment variable names. Point out the configMapKeyRef structure with name and key fields.

Step 4

Consume ConfigMap as Volume Mounts

Volume mounts are the other major way to consume ConfigMaps. This approach is essential for injecting configuration files that applications read from disk, such as nginx.conf or application.yaml files.

🚀 Step 4: Mount ConfigMap as a Volume

Mount the configuration files into an nginx container:


# Save as pod-volume-mount.yaml
apiVersion: v1
kind: Pod
metadata:
  name: volume-mount-demo
  namespace: lab-config
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports: [{ containerPort: 80 }]
      volumeMounts:
        - { name: nginx-config, mountPath: /etc/nginx/conf.d }
        - { name: app-props, mountPath: /etc/app, readOnly: true }
  volumes:
    - name: nginx-config
      configMap:
        name: app-files
        items: [{ key: nginx.conf, path: default.conf }]
    - name: app-props
      configMap:
        name: app-files
        items: [{ key: app.properties, path: app.properties }]
The items field allows mapping specific keys to specific file paths within the mount. Without items, all keys are mounted as files. The nginx.conf is mounted as default.conf in the conf.d directory so nginx picks it up. The readOnly flag is a best practice for configuration volumes.

🚀 Step 4: Verify the Mounted Files


$ kubectl apply -f pod-volume-mount.yaml

# Wait for the pod to be running
$ kubectl wait --for=condition=Ready pod/volume-mount-demo \
    -n lab-config --timeout=60s

# Verify the nginx config is mounted
$ kubectl exec volume-mount-demo -n lab-config -- \
    cat /etc/nginx/conf.d/default.conf

# Verify the app properties are mounted
$ kubectl exec volume-mount-demo -n lab-config -- \
    cat /etc/app/app.properties

# Test the health endpoint
$ kubectl exec volume-mount-demo -n lab-config -- \
    curl -s http://localhost/health
✅ Expected Output: The nginx.conf and app.properties contents are displayed. The health endpoint returns OK.
Walk through the verification steps. The nginx config should be rendered at /etc/nginx/conf.d/default.conf with the exact contents of the file we created earlier. The curl to /health should return OK, confirming nginx loaded the custom configuration. This pattern is used extensively in the Verisign platform for sidecar proxy configuration.

Step 5

Create Secrets

Students now move to Secrets, which are structurally similar to ConfigMaps but designed for sensitive data. Kubernetes provides several Secret types. Students will create Opaque and TLS secrets.

🚀 Step 5: Create an Opaque Secret

Create a generic (Opaque) Secret for database credentials:


# Create a Secret from literal values
$ kubectl create secret generic db-credentials \
    --from-literal=DB_USERNAME=app_user \
    --from-literal=DB_PASSWORD='S3cur3P@ssw0rd!' \
    --from-literal=DB_HOST=postgres-svc.lab-config.svc.cluster.local \
    -n lab-config

# Inspect the Secret
$ kubectl get secret db-credentials -n lab-config -o yaml
                

# Decode a secret value
$ kubectl get secret db-credentials -n lab-config \
    -o jsonpath='{.data.DB_PASSWORD}' | base64 -d
⚠️ Security Note: Base64 is encoding, not encryption. Always use RBAC to restrict Secret access and enable encryption at rest via AWS KMS.
Expected Output: Values are base64-encoded in the data section. The type is Opaque. Emphasize the security model for Secrets. Base64 encoding is NOT encryption. On the Verisign EKS platform, ensure that envelope encryption with AWS KMS is enabled. Also point out that Secrets are stored in etcd and should be protected with proper RBAC policies. The jsonpath query demonstrates how easily values can be decoded.

🚀 Step 5: Create a TLS Secret

Generate a self-signed certificate and create a TLS Secret:


# Generate a self-signed certificate (for lab purposes only)
$ openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
    -keyout tls.key -out tls.crt \
    -subj "/CN=app.lab-config.local/O=Verisign Lab"

# Create a TLS Secret
$ kubectl create secret tls app-tls \
    --cert=tls.crt \
    --key=tls.key \
    -n lab-config

# Verify the TLS Secret
$ kubectl get secret app-tls -n lab-config -o yaml
✅ Expected Output: The Secret type is kubernetes.io/tls with keys tls.crt and tls.key in the data section.
TLS Secrets are a specific Secret type that Kubernetes validates for proper certificate and key format. The keys must be named tls.crt and tls.key. This type is commonly used with Ingress controllers for TLS termination, which students will explore in Lab 6. In production, certificates should come from a trusted CA or cert-manager.

Step 6

Consume Secrets as Env Vars and Volume Mounts

The consumption patterns for Secrets are identical to ConfigMaps. Students will inject Secret data both as environment variables and as mounted files, using the same envFrom, valueFrom, and volume mount patterns.

🚀 Step 6: Secret as Environment Variables


# Save as pod-secret-env.yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-env-demo
  namespace: lab-config
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "echo Host=$DB_HOST User=$DB_USERNAME && sleep 3600"]
      env:
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef: { name: db-credentials, key: DB_USERNAME }
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef: { name: db-credentials, key: DB_PASSWORD }
        - name: DB_HOST
          valueFrom:
            secretKeyRef: { name: db-credentials, key: DB_HOST }
  restartPolicy: Never
                

$ kubectl apply -f pod-secret-env.yaml
$ kubectl logs secret-env-demo -n lab-config
✅ Expected Output: Host=postgres-svc.lab-config.svc.cluster.local User=app_user
The secretKeyRef works identically to configMapKeyRef. Note that we intentionally do not echo the password in the command to avoid exposing it in logs. In production, avoid logging sensitive environment variables. Kubernetes also masks Secret values in kubectl describe output.

🚀 Step 6: Secret as Volume Mount


# Save as pod-secret-volume.yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-vol-demo
  namespace: lab-config
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "ls -la /etc/db-creds && sleep 3600"]
      volumeMounts:
        - name: db-creds
          mountPath: /etc/db-creds
          readOnly: true
  volumes:
    - name: db-creds
      secret:
        secretName: db-credentials
        defaultMode: 0400
  restartPolicy: Never
                

$ kubectl apply -f pod-secret-volume.yaml
$ kubectl logs secret-vol-demo -n lab-config

# Read a specific secret file
$ kubectl exec secret-vol-demo -n lab-config -- cat /etc/db-creds/DB_USERNAME
Expected Output: Files are mounted with 0400 permissions (read-only by owner). Each Secret key becomes a file. Best Practice: Use defaultMode: 0400 for Secrets to restrict file permissions. Volume-mounted Secrets are stored in tmpfs (memory-backed), never written to disk on the node. Highlight the security advantages of volume-mounted Secrets: they are stored in tmpfs (in-memory filesystem), never touch the node's disk, and can have restrictive file permissions via defaultMode. The 0400 mode means only the file owner can read the file. This is more secure than environment variables, which can be exposed via /proc or process listing.

Step 7

Immutable ConfigMaps and Secrets

Immutable ConfigMaps and Secrets were introduced as a stable feature in Kubernetes 1.21. They protect against accidental updates, significantly reduce apiserver load (no watches needed), and improve cluster performance at scale.

🚀 Step 7: Create an Immutable ConfigMap


# Save as immutable-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: immutable-app-config
  namespace: lab-config
data:
  APP_VERSION: "2.1.0"
  FEATURE_FLAGS: "dark-mode=true,beta-api=false"
  RELEASE_DATE: "2026-03-13"
immutable: true
                

$ kubectl apply -f immutable-config.yaml

# Try to update it (this will fail)
$ kubectl patch configmap immutable-app-config \
    -n lab-config \
    --type merge \
    -p '{"data":{"APP_VERSION":"2.2.0"}}'
❌ Expected Error: 
Error: configmaps "immutable-app-config" is immutable
Key Point: To update an immutable ConfigMap, you must delete and recreate it. Use a naming convention with version suffixes (e.g., app-config-v2) for safe rollouts.
Immutable resources cannot be modified after creation. This is a deliberate design choice to prevent accidental configuration drift. The recommended pattern is to create new ConfigMaps with version suffixes and update the pod spec to reference the new version. This also enables easy rollbacks by reverting the pod spec. At scale, immutable ConfigMaps significantly reduce etcd watch overhead.

Step 8

Projected Volumes

Projected volumes allow combining multiple volume sources into a single mount point. This is useful for aggregating configuration from ConfigMaps, Secrets, and the downward API into one directory that the application reads from.

🚀 Step 8: Combine ConfigMap, Secret, and Downward API


# Save as pod-projected.yaml
apiVersion: v1
kind: Pod
metadata:
  name: projected-demo
  namespace: lab-config
  labels: { app: projected-demo, version: v1 }
spec:
  containers:
    - name: app
      image: busybox:1.36
      command: ["sh", "-c", "ls /etc/projected && cat /etc/projected/APP_ENV && sleep 3600"]
      volumeMounts:
        - { name: all-config, mountPath: /etc/projected, readOnly: true }
  volumes:
    - name: all-config
      projected:
        sources:
          - configMap: { name: app-config, items: [{ key: APP_ENV, path: APP_ENV }] }
          - secret: { name: db-credentials, items: [{ key: DB_USERNAME, path: DB_USERNAME }] }
          - downwardAPI:
              items:
                - { path: labels, fieldRef: { fieldPath: metadata.labels } }
  restartPolicy: Never
Projected volumes merge multiple sources into a single directory. This is particularly powerful when applications expect all configuration in one location. The downward API injects pod metadata like labels, annotations, namespace, and resource limits as files. This enables applications to be self-aware of their deployment context without hardcoding values.

🚀 Step 8: Verify Projected Volume


$ kubectl apply -f pod-projected.yaml

# Wait for the pod to be running
$ kubectl wait --for=condition=Ready pod/projected-demo \
    -n lab-config --timeout=60s

# Check all projected files
$ kubectl exec projected-demo -n lab-config -- ls -la /etc/projected

# View the downward API labels
$ kubectl exec projected-demo -n lab-config -- cat /etc/projected/labels

# View the namespace
$ kubectl exec projected-demo -n lab-config -- cat /etc/projected/namespace
Expected Output: The /etc/projected directory contains files from all three sources: APP_ENV, APP_LOG_LEVEL (ConfigMap), DB_USERNAME (Secret), labels, and namespace (downward API). Labels output: app="projected-demo" version="v1". Verify that all five files are present in the projected directory. The labels file contains the pod's labels in key=value format. The namespace file contains "lab-config". This demonstrates how a single volume mount can aggregate configuration from multiple sources, reducing the number of volumeMounts needed in the pod spec.

Step 9

ConfigMap Update Propagation

This step demonstrates a critical behavioral difference between environment variables and volume mounts when a ConfigMap is updated. Understanding this difference is essential for designing reliable configuration update strategies.

🚀 Step 9: Deploy a Pod with Both Consumption Methods


# Save as pod-update-test.yaml
apiVersion: v1
kind: Pod
metadata:
  name: update-test
  namespace: lab-config
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c", "while true; do
      echo \"ENV: $APP_LOG_LEVEL | FILE: $(cat /etc/config/APP_LOG_LEVEL)\";
      sleep 10; done"]
    envFrom:
    - configMapRef: { name: app-config }
    volumeMounts:
    - { name: config-vol, mountPath: /etc/config, readOnly: true }
  volumes:
  - name: config-vol
    configMap: { name: app-config }
  restartPolicy: Never
                

$ kubectl apply -f pod-update-test.yaml
$ kubectl wait --for=condition=Ready pod/update-test \
    -n lab-config --timeout=60s

# Check initial values
$ kubectl logs update-test -n lab-config --tail=1
✅ Initial Output: ENV: info | FILE: info
This pod consumes the same ConfigMap in two ways simultaneously: as environment variables (envFrom) and as a volume mount. The container runs a loop printing both values every 10 seconds. Initially both show "info" since they come from the same source.

🚀 Step 9: Update the ConfigMap and Observe


# Update the ConfigMap
$ kubectl patch configmap app-config -n lab-config \
    --type merge \
    -p '{"data":{"APP_LOG_LEVEL":"debug"}}'

# Wait 30-60 seconds for kubelet sync, then check logs
$ sleep 60
$ kubectl logs update-test -n lab-config --tail=3
Key Point: Volume mounts update automatically (~60s); environment variables do NOT update until pod restart. Use volume mounts with a file watcher for live config reload.
Expected Output (after ~60 seconds): ENV: info | FILE: debug. Critical Insight: Volume mount is updated automatically (kubelet sync period, up to 60s + cache TTL). Environment variable is NOT updated -- remains "info" until pod restart. For env var changes, a rolling restart is required: kubectl rollout restart deployment/name. This is one of the most important lessons in this lab. The volume mount updates because the kubelet periodically syncs mounted ConfigMaps. The environment variable does NOT update because env vars are set at container start time and are immutable for the lifetime of the process. The kubelet sync period is configurable but defaults to about 60 seconds plus cache propagation delay.

🚑 Step 10: Clean Up Resources

Remove all resources created during this lab:


# Delete the entire namespace (removes all resources within it)
$ kubectl delete namespace lab-config

Deleting the namespace is the cleanest way to remove all lab resources. It cascades to all pods, ConfigMaps, Secrets, and other namespaced resources. Remind students to also clean up any local files they created. If the namespace delete hangs, it usually means a resource has a finalizer that cannot be resolved.

🎓 Lab Summary

ConfigMaps

📊 ConfigMap vs Secret Comparison

Feature ConfigMap Secret
Data encoding Plain text Base64-encoded
Size limit 1 MiB 1 MiB
Volume storage Node filesystem tmpfs (in-memory)
RBAC Standard Restricted recommended
Use case Non-sensitive config Passwords, tokens, certs
Immutable support Yes Yes
This comparison table is a handy reference. The key distinction is that Secrets are stored in tmpfs when volume-mounted and should have restricted RBAC access. Both have the same 1 MiB size limit. For data larger than 1 MiB, consider using init containers to fetch configuration from an external store like AWS Systems Manager Parameter Store or HashiCorp Vault.
