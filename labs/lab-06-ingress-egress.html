Lab 6: Ingress and Gateway API

HTTP Routing, TLS Termination, and Egress Controls

Step 2

Deploy Two Sample Applications

Students deploy two versions of a sample application. These represent different microservices or application versions that will be routed to via the Ingress resource using host-based and path-based rules.

🚀 Step 2: Deploy app-v1


# Save as app-v1.yaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: app-v1, namespace: lab-ingress }
spec:
  replicas: 2
  selector: { matchLabels: { app: web, version: v1 } }
  template:
    metadata: { labels: { app: web, version: v1 } }
    spec:
      containers:
        - name: app
          image: hashicorp/http-echo:0.2.3
          args: ["-text=Hello from App V1", "-listen=:8080"]
          ports: [{ containerPort: 8080 }]
          readinessProbe: { httpGet: { path: /, port: 8080 }, initialDelaySeconds: 3 }
---
apiVersion: v1
kind: Service
metadata: { name: app-v1-svc, namespace: lab-ingress }
spec:
  type: ClusterIP
  selector: { app: web, version: v1 }
  ports: [{ port: 80, targetPort: 8080 }]
app-v1 uses hashicorp/http-echo which returns a static text response. The ClusterIP service is intentional: the Ingress controller handles external access. Each app has a readiness probe to ensure traffic is only sent to healthy pods. The service maps port 80 to the container port 8080.

Step 3

Create a Basic Ingress (Host-Based Routing)

Host-based routing directs traffic based on the HTTP Host header. This is the most common Ingress pattern, allowing multiple applications to share a single load balancer IP by using different domain names.

🚀 Step 3: Create Host-Based Ingress


# Save as ingress-host.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata: { name: app-ingress-host, namespace: lab-ingress }
annotations: { nginx.ingress.kubernetes.io/rewrite-target: / }
spec:
  ingressClassName: nginx
  rules:
    - host: v1.lab.local
      http:
        paths:
          - { path: /, pathType: Prefix,
              backend: { service: { name: app-v1-svc, port: { number: 80 } } } }
    - host: v2.lab.local
      http:
        paths:
          - { path: /, pathType: Prefix,
              backend: { service: { name: app-v2-svc, port: { number: 80 } } } }
                

$ kubectl apply -f ingress-host.yaml
$ kubectl get ingress -n lab-ingress
This Ingress defines two host rules. Requests to v1.lab.local are routed to app-v1-svc, and requests to v2.lab.local are routed to app-v2-svc. The ingressClassName must match an available IngressClass. If using AWS ALB Controller, change ingressClassName to alb and add the appropriate annotations. The rewrite-target annotation is specific to NGINX Ingress Controller.

🚀 Step 3: Verify the Ingress


# Check Ingress details
$ kubectl describe ingress app-ingress-host -n lab-ingress
✅ Expected Output: 
Name:             app-ingress-host
Namespace:        lab-ingress
Rules:
  Host           Path  Backends
  ----           ----  --------
  v1.lab.local   /     app-v1-svc:80 (10.0.x.x:8080,10.0.x.x:8080)
  v2.lab.local   /     app-v2-svc:80 (10.0.x.x:8080,10.0.x.x:8080)
💡 Note: The Backends column shows the pod IP addresses. If it shows <error: endpoints ... not found>, the service selector does not match any pods.
The describe output shows the routing table created by the Ingress controller. Each host maps to a backend service and its endpoints (pod IPs). If the backends show error messages, check that the service selectors match the pod labels. The ADDRESS field may take a minute to populate on AWS as the load balancer is provisioned.

Step 5

Add Path-Based Routing

Path-based routing directs traffic based on the URL path. This allows a single hostname to serve multiple backends based on the request path. It is commonly used for API versioning and microservice routing.

🚀 Step 5: Create Path-Based Ingress


# Save as ingress-path.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata: { name: app-ingress-path, namespace: lab-ingress }
annotations: { nginx.ingress.kubernetes.io/rewrite-target: / }
spec:
  ingressClassName: nginx
  rules:
    - host: app.lab.local
      http:
        paths:
          - { path: /v1, pathType: Prefix,
              backend: { service: { name: app-v1-svc, port: { number: 80 } } } }
          - { path: /v2, pathType: Prefix,
              backend: { service: { name: app-v2-svc, port: { number: 80 } } } }
          - { path: /, pathType: Prefix,
              backend: { service: { name: app-v1-svc, port: { number: 80 } } } }
                

$ kubectl apply -f ingress-path.yaml
This Ingress uses a single hostname with multiple path rules. The /v1 prefix routes to app-v1, /v2 routes to app-v2, and the catch-all / routes to app-v1 as the default. Path matching uses longest-prefix-first ordering. The rewrite-target annotation strips the path prefix before forwarding to the backend.

🚀 Step 5: Test Path-Based Routing


# Test path /v1
$ curl -s -H "Host: app.lab.local" http://$INGRESS_IP/v1

# Test path /v2
$ curl -s -H "Host: app.lab.local" http://$INGRESS_IP/v2

# Test default path
$ curl -s -H "Host: app.lab.local" http://$INGRESS_IP/
✅ Expected Output: 
$ curl -H "Host: app.lab.local" http://$INGRESS_IP/v1
Hello from App V1

$ curl -H "Host: app.lab.local" http://$INGRESS_IP/v2
Hello from App V2

$ curl -H "Host: app.lab.local" http://$INGRESS_IP/
Hello from App V1

pathType Values

  • Prefix: Matches URL paths where the path element is a prefix (split by /)
  • Exact: Matches the URL path exactly, case-sensitive
  • ImplementationSpecific: Matching depends on the IngressClass
All three requests use the same Host header but different paths. The Ingress controller matches the longest prefix first, so /v2 is matched before / for paths starting with /v2. Discuss the difference between Prefix and Exact pathType. Prefix matching splits on / boundaries, so /v1 matches /v1, /v1/, and /v1/anything but not /v1extra.

Step 6

Configure TLS Termination

TLS termination at the Ingress controller is the standard pattern for HTTPS. The Ingress controller handles TLS decryption, forwarding unencrypted traffic to the backend services within the cluster network. This centralizes certificate management and offloads TLS processing from application pods.

🚀 Step 6: Create TLS Secret and Ingress


# Generate a self-signed certificate for the lab domain
$ openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
    -keyout tls-ingress.key -out tls-ingress.crt \
    -subj "/CN=*.lab.local/O=Verisign Lab"

# Create the TLS Secret in the lab namespace
$ kubectl create secret tls lab-tls-secret \
    --cert=tls-ingress.crt \
    --key=tls-ingress.key \
    -n lab-ingress
                

# Save as ingress-tls.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress-tls
  namespace: lab-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
    - hosts: [secure.lab.local]
      secretName: lab-tls-secret
  rules:
    - host: secure.lab.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service: { name: app-v1-svc, port: { number: 80 } }
The wildcard certificate covers all subdomains of lab.local. The TLS block references the Secret by name and associates it with specific hosts. The ssl-redirect annotation forces HTTP to HTTPS redirection. In production on the Verisign platform, certificates should be managed by cert-manager with Let's Encrypt or an internal CA, or use AWS ACM certificates with the ALB Controller.

🚀 Step 6: Test TLS Termination


$ kubectl apply -f ingress-tls.yaml

# Test HTTPS (use -k to accept self-signed cert)
$ curl -sk -H "Host: secure.lab.local" https://$INGRESS_IP

# Verify the certificate details
$ curl -skv -H "Host: secure.lab.local" https://$INGRESS_IP 2>&1 | \
    grep -E "subject|issuer|expire"

# Test HTTP redirect to HTTPS
$ curl -sI -H "Host: secure.lab.local" http://$INGRESS_IP
✅ Expected Output: 
Hello from App V1

* subject: CN=*.lab.local; O=Verisign Lab

HTTP/1.1 308 Permanent Redirect
Location: https://secure.lab.local/
⚠️ Note: The -k flag skips certificate verification (required for self-signed certs). Never use -k in production scripts. The 308 redirect confirms that HTTP traffic is automatically redirected to HTTPS.
Walk through the three tests: HTTPS access with -k for the self-signed cert, certificate inspection using -v, and HTTP redirect test. The 308 Permanent Redirect confirms the ssl-redirect annotation is working. Discuss the TLS termination flow: client connects to the Ingress controller over TLS, the controller decrypts and forwards plain HTTP to the backend ClusterIP service.

Step 7

Explore Ingress Annotations

Annotations are the primary mechanism for configuring Ingress controller behavior beyond basic routing. The available annotations depend on the Ingress controller in use. This step covers commonly used NGINX Ingress annotations for rewrite, rate limiting, and CORS.

🚀 Step 7: Advanced Annotations


# Save as ingress-annotations.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress-advanced
  namespace: lab-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
    nginx.ingress.kubernetes.io/limit-rps: "10"
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://app.verisign.com"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    nginx.ingress.kubernetes.io/configuration-snippet: |
      add_header X-Served-By "verisign-eks" always;
spec:
  ingressClassName: nginx
  rules:
    - host: api.lab.local
      http:
        paths:
          - { path: "/api(/|$)(.*)", pathType: ImplementationSpecific,
              backend: { service: { name: app-v1-svc, port: { number: 80 } } } }
Walk through each annotation group. The rewrite-target with regex captures enables stripping path prefixes. Rate limiting protects backends from traffic spikes. CORS headers are essential for browser-based API access. Proxy timeouts prevent hung connections. The configuration-snippet allows injecting raw NGINX directives. Emphasize that annotations are controller-specific and not portable between NGINX and ALB controllers.

🚀 Step 7: Test Annotations


$ kubectl apply -f ingress-annotations.yaml

# Test URL rewriting: /api/anything routes to /anything on the backend
$ curl -s -H "Host: api.lab.local" http://$INGRESS_IP/api/

# Check response headers for CORS and custom headers
$ curl -sI -H "Host: api.lab.local" \
    -H "Origin: https://app.verisign.com" \
    http://$INGRESS_IP/api/ 2>&1 | grep -E "cors|X-Served"

# Test rate limiting (rapid requests)
$ for i in $(seq 1 15); do
    curl -s -o /dev/null -w "%{http_code} " \
        -H "Host: api.lab.local" http://$INGRESS_IP/api/
done
echo ""
✅ Expected Output: 
Access-Control-Allow-Origin: https://app.verisign.com
X-Served-By: verisign-eks

200 200 200 200 200 200 200 200 200 200 503 503 503 503 503
Key Point: After 10 requests per second (with burst), excess requests receive 503 Service Temporarily Unavailable. Rate limiting protects backend services from overload.
The rate limiting test sends 15 rapid requests. The first 10 (plus burst) succeed, and the remainder get 503 responses. The CORS headers appear only when an Origin header is sent. The X-Served-By custom header confirms the configuration-snippet is working. Discuss how these annotations translate to NGINX configuration directives under the hood.

Step 8

Gateway API: GatewayClass and Gateway

The Gateway API is the successor to Ingress, offering a more expressive, role-oriented, and portable API for traffic routing. It separates infrastructure concerns (GatewayClass, Gateway) from application routing (HTTPRoute). This step deploys the infrastructure layer.

🚀 Step 8: Check Gateway API CRDs


# Verify Gateway API CRDs are installed
$ kubectl get crd | grep gateway

# Expected CRDs:
# gatewayclasses.gateway.networking.k8s.io
# gateways.gateway.networking.k8s.io
# httproutes.gateway.networking.k8s.io
⚠️ If CRDs are not installed: Install them with: 
$ kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.2.0/standard-install.yaml
If installation is not permitted, read through steps 8–9 as reference material.
The Gateway API requires CRDs to be installed separately from Kubernetes. Check if they exist before proceeding. If the cluster does not have them and students cannot install CRDs, these steps can be instructor-led walkthroughs. The Gateway API is GA as of Kubernetes 1.29 for the core HTTP routing features.

🚀 Step 8: Deploy GatewayClass and Gateway


# Save as gateway.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata: { name: lab-gateway-class }
spec: { controllerName: gateway.nginx.org/nginx-gateway-controller }
---
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata: { name: lab-gateway, namespace: lab-ingress }
spec:
  gatewayClassName: lab-gateway-class
  listeners:
    - { name: http, protocol: HTTP, port: 80,
        allowedRoutes: { namespaces: { from: Same } } }
    - { name: https, protocol: HTTPS, port: 443,
        tls: { mode: Terminate, certificateRefs: [{ name: lab-tls-secret }] },
        allowedRoutes: { namespaces: { from: Same } } }
                

$ kubectl apply -f gateway.yaml
$ kubectl get gateway -n lab-ingress
$ kubectl describe gateway lab-gateway -n lab-ingress
The GatewayClass is a cluster-scoped resource that defines the controller implementation (similar to IngressClass). The Gateway is namespace-scoped and defines listeners on specific ports and protocols. The allowedRoutes field controls which namespaces can attach routes to this Gateway. The TLS listener references the same TLS Secret we created in Step 6. This separation of concerns allows platform teams to manage Gateways while application teams manage Routes.

🚀 Step 8: Verify Gateway Status


$ kubectl get gateway lab-gateway -n lab-ingress -o yaml | \
    kubectl neat 2>/dev/null || \
    kubectl get gateway lab-gateway -n lab-ingress -o yaml
✅ Expected Status: The Gateway should show Accepted: True and Programmed: True conditions. If the controller is not running, the status will show Accepted: False with a reason.

Gateway API Role Separation

Infrastructure Provider
  • Manages GatewayClass
  • Deploys the controller
Cluster Operator
  • Manages Gateway
  • Configures listeners, TLS
Application Developer — Manages HTTPRoute, attaches to Gateway
The role-based model is a key advantage of Gateway API over Ingress. Platform teams at Verisign manage the GatewayClass and Gateway resources, while application teams only need to create HTTPRoute resources to define their routing rules. This provides better separation of concerns and RBAC enforcement. The status conditions indicate whether the Gateway controller has accepted and programmed the configuration.

Step 9

HTTPRoute for Traffic Splitting

HTTPRoute is where application developers define routing rules. A powerful feature of Gateway API is native traffic splitting (weighted routing), which enables canary deployments and A/B testing without additional tooling.

🚀 Step 9: Create HTTPRoute with Traffic Splitting


# Save as httproute.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: app-route, namespace: lab-ingress }
spec:
  parentRefs:
    - { name: lab-gateway, sectionName: http }
  hostnames: ["app.lab.local"]
  rules:
    - matches:
        - path: { type: PathPrefix, value: / }
      backendRefs:
        - { name: app-v1-svc, port: 80, weight: 80 }
        - { name: app-v2-svc, port: 80, weight: 20 }
                

$ kubectl apply -f httproute.yaml
$ kubectl get httproute -n lab-ingress
$ kubectl describe httproute app-route -n lab-ingress
The HTTPRoute attaches to the Gateway via parentRefs. The weight field on backendRefs enables traffic splitting: 80% to v1 and 20% to v2. This is a native feature of Gateway API that requires custom annotations or service mesh with Ingress. The sectionName references the specific listener on the Gateway. This pattern is ideal for canary deployments where you gradually shift traffic from an old version to a new one.

🚀 Step 9: Test Traffic Splitting


# Get the Gateway address
$ GATEWAY_IP=$(kubectl get gateway lab-gateway -n lab-ingress \
    -o jsonpath='{.status.addresses[0].value}')

# Send 20 requests and count the distribution
$ for i in $(seq 1 20); do
    curl -s -H "Host: app.lab.local" http://$GATEWAY_IP
done | sort | uniq -c | sort -rn
✅ Expected Output (approximately): 
  16 Hello from App V1
   4 Hello from App V2
Key Point: Traffic splitting is probabilistic. With 20 requests, expect roughly 80/20 distribution. In production canary deployments, start with 5% to the new version and gradually increase as confidence grows.
⚠️ Note: If the Gateway controller is not running or CRDs are missing, this test will not work. Use the Ingress-based routing from earlier steps as the fallback.
The distribution will not be exactly 80/20 with only 20 requests due to statistical variance. With larger sample sizes, the distribution will converge to the configured weights. This is a powerful primitive for progressive delivery strategies. Discuss how this compares to Istio VirtualService weight-based routing, which provides similar functionality but requires a full service mesh.

Step 10

Configure Egress Controls with NetworkPolicy

While Ingress controls inbound traffic routing, egress NetworkPolicies control outbound traffic from pods. This is critical for security: restricting which external services pods can communicate with reduces the blast radius of a compromise.

🚀 Step 10: Test Default Egress (No Restrictions)


# Deploy a test pod
$ kubectl run egress-test --image=busybox:1.36 \
    -n lab-ingress --restart=Never \
    --command -- sleep 3600

# Test outbound connectivity (DNS and HTTP)
$ kubectl exec egress-test -n lab-ingress -- \
    nslookup kubernetes.default.svc.cluster.local

$ kubectl exec egress-test -n lab-ingress -- \
    wget -qO- --timeout=5 http://app-v1-svc.lab-ingress.svc.cluster.local

$ kubectl exec egress-test -n lab-ingress -- \
    wget -qO- --timeout=5 http://example.com
✅ Expected Output: All three requests succeed. By default, pods have unrestricted egress to both cluster-internal and external destinations.
Before applying any NetworkPolicy, demonstrate that pods have unrestricted egress by default. The test pod can resolve DNS, reach cluster-internal services, and access external websites. This is the baseline we will restrict with NetworkPolicies. Note that NetworkPolicies require a CNI plugin that supports them (Calico, Cilium, etc.). The default AWS VPC CNI does not enforce NetworkPolicies without an additional policy engine.

🚀 Step 10: Apply Egress NetworkPolicy


# Save as egress-policy.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: restrict-egress, namespace: lab-ingress }
spec:
  podSelector: { matchLabels: { run: egress-test } }
  policyTypes: [Egress]
  egress:
    # Allow DNS resolution (required for service discovery)
    - to:
        - namespaceSelector: {}
          podSelector: { matchLabels: { k8s-app: kube-dns } }
      ports: [{ protocol: UDP, port: 53 }, { protocol: TCP, port: 53 }]
    # Allow traffic to cluster-internal services only
    - to:
        - namespaceSelector:
            matchLabels: { kubernetes.io/metadata.name: lab-ingress }
      ports: [{ protocol: TCP, port: 80 }]
                

$ kubectl apply -f egress-policy.yaml
This NetworkPolicy restricts egress from the test pod to only DNS (port 53 to kube-dns) and HTTP (port 80) to services within the lab-ingress namespace. All other egress traffic is denied, including access to external websites. The DNS exception is critical: without it, pods cannot resolve any service names. The namespaceSelector with the kubernetes.io/metadata.name label is a built-in label available on all namespaces.

🚀 Step 10: Verify Egress Restrictions


# DNS should still work
$ kubectl exec egress-test -n lab-ingress -- \
    nslookup kubernetes.default.svc.cluster.local

# Internal service access should work
$ kubectl exec egress-test -n lab-ingress -- \
    wget -qO- --timeout=5 http://app-v1-svc.lab-ingress.svc.cluster.local

# External access should be BLOCKED
$ kubectl exec egress-test -n lab-ingress -- \
    wget -qO- --timeout=5 http://example.com
✅ Expected Output: 
DNS:       Server: 10.100.0.10  Address: 10.100.0.10:53  Name: kubernetes...
Internal:  Hello from App V1
External:  wget: download timed out
⚠️ CNI Requirement: NetworkPolicy enforcement requires a compatible CNI plugin (Calico, Cilium). If using the default AWS VPC CNI without a policy engine, NetworkPolicies are accepted but not enforced. Verify with your cluster administrator.
The key result is that external access is now blocked while internal service discovery and communication remain functional. This is a defense-in-depth measure that limits lateral movement and data exfiltration in case of a pod compromise. Emphasize the CNI requirement: if the external access still works, the cluster may not have a NetworkPolicy-capable CNI. On EKS, this typically means Calico or Cilium must be installed alongside the VPC CNI.

📊 Ingress vs Gateway API Comparison

Feature Ingress Gateway API
API maturity Stable (v1 since 1.19) GA core (v1 since 1.29)
Role separation Single resource GatewayClass / Gateway / Route
Traffic splitting Annotation-dependent Native (weight field)
Header matching Annotation-dependent Native (matches block)
Portability Annotations not portable Spec is portable
Protocol support HTTP/HTTPS only HTTP, gRPC, TCP, TLS, UDP
This comparison table highlights why Gateway API is considered the future of Kubernetes traffic management. The key advantages are role separation, native traffic splitting, and portability across controller implementations. However, Ingress remains widely used and well-supported, so both should be understood. On EKS, the AWS Gateway API Controller provides native integration with AWS networking services.
