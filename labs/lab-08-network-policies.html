Lab 8: Network Policies

Controlling Pod-to-Pod Communication with NetworkPolicy Resources

Intermediate Kubernetes — Module 8 of 16

Lab 8 covers NetworkPolicies for controlling pod-to-pod communication. Students will deploy a three-tier application and progressively apply network policies. Estimated time: 45 minutes. Requires a CNI that supports NetworkPolicies (Calico, Cilium, Weave Net).

📋 Lab Overview

Objectives

  • Deploy a three-tier application
  • Verify default open connectivity
  • Apply a default-deny NetworkPolicy
  • Create targeted ingress policies per tier
  • Test and verify allowed/blocked traffic paths
  • Understand egress policy controls

Prerequisites

  • Completed Labs 1–7
  • kubectl configured with cluster access
  • CNI with NetworkPolicy support (Calico or Cilium)
⏱ Duration: ~45 minutes
NetworkPolicy enforcement requires a CNI that supports it. The default AWS VPC CNI does not enforce NetworkPolicies without Calico or Cilium. Verify the CNI before starting. If policies are not enforced, the lab can still be completed as a conceptual exercise.

Step 1

Deploy a Three-Tier Application

We start by deploying a frontend, backend, and database tier. Each tier is labeled so we can target them individually with NetworkPolicies. We use simple nginx and busybox containers to simulate the tiers.

🏗 Deploy the Three-Tier Application

Create a namespace and deploy frontend, backend, and database tiers:


# Create the lab namespace
kubectl create namespace lab08

Deploy the database tier:


# Save as database.yaml
apiVersion: v1
kind: Pod
metadata:
  name: database
  namespace: lab08
  labels: { app: bookstore, tier: database }
spec:
  containers:
  - name: db
    image: nginx:1.25
    ports: [{ containerPort: 80 }]
---
apiVersion: v1
kind: Service
metadata:
  name: database
  namespace: lab08
spec:
  selector: { tier: database }
  ports: [{ port: 3306, targetPort: 80 }]
We use nginx as a stand-in for actual services. The labels (tier: database, tier: backend, tier: frontend) are critical — NetworkPolicies use these selectors to target pods.

🏗 Deploy Backend and Frontend Tiers

Deploy the backend tier:


# Save as backend.yaml
apiVersion: v1
kind: Pod
metadata:
  name: backend
  namespace: lab08
  labels: { app: bookstore, tier: backend }
spec:
  containers:
  - name: api
    image: nginx:1.25
    ports: [{ containerPort: 80 }]
---
apiVersion: v1
kind: Service
metadata:
  name: backend
  namespace: lab08
spec:
  selector: { tier: backend }
  ports: [{ port: 8080, targetPort: 80 }]

Deploy the frontend tier:


# Save as frontend.yaml
apiVersion: v1
kind: Pod
metadata:
  name: frontend
  namespace: lab08
  labels: { app: bookstore, tier: frontend }
spec:
  containers:
  - name: web
    image: nginx:1.25
    ports: [{ containerPort: 80 }]
---
apiVersion: v1
kind: Service
metadata:
  name: frontend
  namespace: lab08
spec:
  selector: { tier: frontend }
  ports: [{ port: 80, targetPort: 80 }]
Each tier has a pod and a service. The services provide stable DNS names (frontend.lab08.svc.cluster.local, etc.) that we will use to test connectivity.

🏗 Apply and Verify the Deployment


# Apply all three tiers
kubectl apply -f database.yaml
kubectl apply -f backend.yaml
kubectl apply -f frontend.yaml

# Wait for all pods to be ready
kubectl wait --for=condition=Ready pod --all -n lab08 --timeout=60s

# Verify everything is running
kubectl get pods -n lab08 -o wide --show-labels

# Verify services
kubectl get services -n lab08
✅ Expected Output: Three pods (frontend, backend, database) all in Running state. Three corresponding services with ClusterIP addresses. Each pod has app=bookstore and its respective tier label.
🔑 Key Point: Labels are the foundation of NetworkPolicies. The tier labels on each pod will be used as selectors in our policies. Well-designed labeling is essential for effective network segmentation.
Verify all three pods are running before proceeding. If any pod is stuck in Pending, check node resources. The -o wide flag shows the pod IPs which will be useful for understanding connectivity later.

Step 2

Verify Default Connectivity

By default, Kubernetes allows all pod-to-pod communication. This step establishes the baseline — all pods can talk to all other pods. We will then restrict this with NetworkPolicies.

🔌 Verify Default Connectivity

Confirm that all pods can communicate with each other (default behavior):


# Frontend → Backend (should SUCCEED)
kubectl exec frontend -n lab08 -- \
  curl -s --max-time 3 http://backend.lab08.svc.cluster.local:8080

# Frontend → Database (should SUCCEED)
kubectl exec frontend -n lab08 -- \
  curl -s --max-time 3 http://database.lab08.svc.cluster.local:3306

# Backend → Database (should SUCCEED)
kubectl exec backend -n lab08 -- \
  curl -s --max-time 3 http://database.lab08.svc.cluster.local:3306

# Database → Frontend (should SUCCEED)
kubectl exec database -n lab08 -- \
  curl -s --max-time 3 http://frontend.lab08.svc.cluster.local:80
✅ Expected Output: All four curl commands return the nginx welcome page HTML. Every pod can reach every other pod — this is the Kubernetes default.
⚠ Important: By default, Kubernetes has a flat network with no restrictions. Any pod can communicate with any other pod in any namespace. This is why NetworkPolicies are critical for security.
This baseline is important. Students need to see the "before" state so the "after" state (with policies) is meaningful. In a real three-tier architecture, the database should never be directly accessible from the frontend.

Step 3

Apply Default Deny-All Ingress Policy

The first and most important step in network policy design is to apply a default deny-all policy. This follows the zero-trust principle — deny everything, then explicitly allow what is needed.

🛑 Apply Default Deny-All Ingress Policy

Create a policy that blocks all incoming traffic to all pods in the namespace:


# Save as deny-all-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all-ingress
  namespace: lab08
spec:
  podSelector: {}
  policyTypes:
  - Ingress
                

# Apply the deny-all policy
kubectl apply -f deny-all-ingress.yaml

# Verify the policy
kubectl get networkpolicies -n lab08

# Describe the policy
kubectl describe networkpolicy default-deny-all-ingress -n lab08
✅ Expected Output: The NetworkPolicy is created. The podSelector: {} matches all pods in the namespace. The empty Ingress policyType with no ingress rules means all inbound traffic is denied.
🔑 Key Point: A podSelector: {} (empty selector) matches all pods in the namespace. Specifying policyTypes: [Ingress] with no ingress rules creates a deny-all for inbound traffic. This is the foundation of zero-trust networking.
Emphasize that the empty podSelector is how you target all pods. The policy says: for all pods, apply ingress policy type, but with no ingress rules — meaning deny all ingress. Egress is not affected by this policy.

Step 4

Verify Isolation

Now we verify that the deny-all policy is working. All communication between pods should be blocked.

🚫 Verify Isolation

Test that all pod-to-pod communication is now blocked:


# Frontend → Backend (should FAIL / timeout)
kubectl exec frontend -n lab08 -- \
  curl -s --max-time 3 http://backend.lab08.svc.cluster.local:8080
echo "Exit code: $?"

# Frontend → Database (should FAIL / timeout)
kubectl exec frontend -n lab08 -- \
  curl -s --max-time 3 http://database.lab08.svc.cluster.local:3306
echo "Exit code: $?"

# Backend → Database (should FAIL / timeout)
kubectl exec backend -n lab08 -- \
  curl -s --max-time 3 http://database.lab08.svc.cluster.local:3306
echo "Exit code: $?"

# Database → Frontend (should FAIL / timeout)
kubectl exec database -n lab08 -- \
  curl -s --max-time 3 http://frontend.lab08.svc.cluster.local:80
echo "Exit code: $?"
✅ Expected Output: All four curl commands timeout after 3 seconds with a non-zero exit code. No pod can receive traffic from any other pod. The deny-all policy is working.
📝 Note: The curl commands will hang for 3 seconds before timing out. This is expected — the packets are being dropped by the CNI, not rejected. A timeout (not a connection refused) is the typical behavior for NetworkPolicy denials.
The difference between timeout and connection refused is important. NetworkPolicies silently drop packets rather than sending a TCP RST. This is a security feature — it does not reveal to the sender whether the destination exists.

Step 5

Allow Frontend → Backend Communication

Now we start selectively opening communication paths. The first rule allows the frontend to talk to the backend — the first hop in our three-tier architecture.

➡ Allow Frontend → Backend Communication

Create a policy allowing the backend to receive traffic from the frontend:


# Save as allow-frontend-to-backend.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: lab08
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - protocol: TCP
      port: 80
                

# Apply the policy
kubectl apply -f allow-frontend-to-backend.yaml

# Test: Frontend → Backend (should now SUCCEED)
kubectl exec frontend -n lab08 -- \
  curl -s --max-time 3 http://backend.lab08.svc.cluster.local:8080

# Test: Database → Backend (should still FAIL)
kubectl exec database -n lab08 -- \
  curl -s --max-time 3 http://backend.lab08.svc.cluster.local:8080
✅ Expected Output: Frontend to backend returns the nginx page. Database to backend still times out. Only the explicitly allowed path works.
🔑 Key Point: NetworkPolicies are additive. The deny-all policy is still in effect, but this new policy adds an exception: backend pods accept ingress from frontend pods on TCP port 80. All other ingress to backend pods remains denied.
Point out that the podSelector targets the backend (the destination), and the ingress.from targets the frontend (the source). NetworkPolicies are written from the perspective of the target pod.

Step 6

Allow Backend → Database Communication

The second hop: backend to database. This completes the allowed communication path: frontend → backend → database. The frontend should never reach the database directly.

➡ Allow Backend → Database Communication

Create a policy allowing the database to receive traffic from the backend:


# Save as allow-backend-to-database.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-backend-to-database
  namespace: lab08
spec:
  podSelector:
    matchLabels:
      tier: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: backend
    ports:
    - protocol: TCP
      port: 80
                

# Apply the policy
kubectl apply -f allow-backend-to-database.yaml

# Test: Backend → Database (should now SUCCEED)
kubectl exec backend -n lab08 -- \
  curl -s --max-time 3 http://database.lab08.svc.cluster.local:3306

# Test: Frontend → Database (should still FAIL)
kubectl exec frontend -n lab08 -- \
  curl -s --max-time 3 http://database.lab08.svc.cluster.local:3306
✅ Expected Output: Backend to database returns the nginx page. Frontend to database still times out. The database is only accessible from the backend tier.
This completes the second leg of the three-tier architecture. The database is now protected — only the backend can reach it. The frontend must go through the backend to access data.

Step 7

Test the Complete Policy Set

Run a comprehensive test of all communication paths to verify the complete policy set is working as intended.

🧪 Comprehensive Policy Verification

Test all communication paths and verify only allowed paths work:


echo "=== ALLOWED PATHS ==="
echo -n "Frontend -> Backend: "
kubectl exec frontend -n lab08 -- curl -s -o /dev/null \
  -w "%{http_code}" --max-time 3 http://backend:8080 2>/dev/null || echo "BLOCKED"

echo -n "Backend -> Database: "
kubectl exec backend -n lab08 -- curl -s -o /dev/null \
  -w "%{http_code}" --max-time 3 http://database:3306 2>/dev/null || echo "BLOCKED"

echo ""
echo "=== BLOCKED PATHS ==="
echo -n "Frontend -> Database: "
kubectl exec frontend -n lab08 -- curl -s -o /dev/null \
  -w "%{http_code}" --max-time 3 http://database:3306 2>/dev/null || echo "BLOCKED"

echo -n "Database -> Frontend: "
kubectl exec database -n lab08 -- curl -s -o /dev/null \
  -w "%{http_code}" --max-time 3 http://frontend:80 2>/dev/null || echo "BLOCKED"

echo -n "Database -> Backend: "
kubectl exec database -n lab08 -- curl -s -o /dev/null \
  -w "%{http_code}" --max-time 3 http://backend:8080 2>/dev/null || echo "BLOCKED"
✅ Expected Output:
Frontend → Backend: 200
Backend → Database: 200
Frontend → Database: BLOCKED
Database → Frontend: BLOCKED
Database → Backend: BLOCKED
🔑 Key Point: The policies enforce a strict traffic flow: Frontend → Backend → Database. No reverse paths, no direct frontend-to-database access, and no lateral movement from the database tier. This is the principle of least privilege applied to networking.
This is the key checkpoint of the lab. The traffic matrix should show exactly two allowed paths and all others blocked. Use this to discuss micro-segmentation and defense in depth at the network layer.

Step 8

Add Egress Rules

Egress policies control outbound traffic from pods. A common requirement is allowing DNS resolution (port 53) while restricting other outbound connections.

⬆ Add Egress Rules

Apply a default deny-all egress policy, then selectively allow DNS and specific traffic:


# Save as deny-all-egress.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all-egress
  namespace: lab08
spec:
  podSelector: {}
  policyTypes:
  - Egress
                

# Apply deny-all egress
kubectl apply -f deny-all-egress.yaml

# Test: DNS resolution is now broken
kubectl exec frontend -n lab08 -- \
  nslookup backend.lab08.svc.cluster.local 2>&1 || echo "DNS FAILED"
⚠ DNS is now broken! The deny-all egress policy blocks all outbound traffic, including DNS queries to kube-dns (port 53). We need to explicitly allow DNS resolution.
This is a very common gotcha. When you deny all egress, you break DNS. Services that resolve by name will stop working. Always allow DNS egress when implementing egress policies.

⬆ Allow DNS and Tier-Specific Egress

Allow DNS resolution and backend egress to the database:


# Save as allow-dns-egress.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns-egress
  namespace: lab08
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector: {}
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
                

# Apply DNS egress policy
kubectl apply -f allow-dns-egress.yaml

# Verify DNS works again
kubectl exec frontend -n lab08 -- \
  nslookup backend.lab08.svc.cluster.local

# Verify the full path still works
kubectl exec frontend -n lab08 -- \
  curl -s --max-time 3 http://backend.lab08.svc.cluster.local:8080
✅ Expected Output: DNS resolution works again. The namespaceSelector: {} allows DNS queries to kube-dns in the kube-system namespace. The frontend-to-backend path is restored.
🔑 Key Point: When implementing egress policies, always create a DNS allow rule first. Use namespaceSelector: {} to allow DNS to any namespace (kube-dns might be in kube-system). Include both UDP and TCP port 53.
DNS uses UDP port 53 by default but can fall back to TCP port 53 for large responses. Always allow both. The namespaceSelector: {} allows traffic to pods in any namespace, which is necessary because kube-dns runs in kube-system.

Step 9

Test Namespace-Based Policies

Namespace selectors allow cross-namespace policies. A common use case is allowing a monitoring namespace to scrape metrics from application namespaces.

🌍 Namespace-Based Policies

Allow a monitoring namespace to access pods in the application namespace:


# Create a monitoring namespace
kubectl create namespace monitoring

# Label the monitoring namespace
kubectl label namespace monitoring purpose=monitoring

# Deploy a monitoring pod
kubectl run monitor --image=nginx:1.25 -n monitoring
kubectl wait --for=condition=Ready pod/monitor -n monitoring --timeout=60s
                

# Test: Monitor cannot reach backend (should FAIL due to deny-all)
kubectl exec monitor -n monitoring -- \
  curl -s --max-time 3 http://backend.lab08.svc.cluster.local:8080

Create a policy allowing ingress from the monitoring namespace:


# Save as allow-monitoring-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-monitoring-ingress
  namespace: lab08
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          purpose: monitoring
    ports:
    - protocol: TCP
      port: 80
The namespaceSelector targets the namespace label, not a pod label. This is critical — make sure students understand the difference between podSelector and namespaceSelector.

🌍 Test Monitoring Access


# Apply the monitoring ingress policy
kubectl apply -f allow-monitoring-ingress.yaml

# Test: Monitor → Backend (should now SUCCEED)
kubectl exec monitor -n monitoring -- \
  curl -s --max-time 3 http://backend.lab08.svc.cluster.local:8080

# Test: Monitor → Frontend (should also SUCCEED)
kubectl exec monitor -n monitoring -- \
  curl -s --max-time 3 http://frontend.lab08.svc.cluster.local:80

# Test: Monitor → Database (should also SUCCEED)
kubectl exec monitor -n monitoring -- \
  curl -s --max-time 3 http://database.lab08.svc.cluster.local:3306
✅ Expected Output: All three requests from the monitoring namespace succeed. The namespaceSelector with purpose: monitoring allows any pod in the monitoring namespace to reach any pod in lab08 on TCP port 80.
🔑 Key Point: namespaceSelector selects by namespace labels, not by namespace name. This is why we labeled the monitoring namespace with purpose=monitoring. Always label namespaces that you intend to use in cross-namespace NetworkPolicies.
Point out that namespaceSelector matches on namespace labels, not the namespace name. This is a common source of confusion. Also note that you can combine podSelector and namespaceSelector for more granular cross-namespace rules.

Step 10

Debug a Broken Policy

This exercise introduces an intentionally misconfigured policy. Students need to identify the bug and fix it. This builds troubleshooting skills for real-world scenarios.

🐛 Debug a Broken Policy

Apply this intentionally broken policy and find the bug:


# Save as broken-policy.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-external-to-frontend
  namespace: lab08
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - protocol: TCP
      port: 8080
                

# Apply the broken policy
kubectl apply -f broken-policy.yaml

# This should allow external traffic to frontend, but it does not work correctly
# Try accessing the frontend from outside the tier labels
kubectl run test-client --image=nginx:1.25 -n lab08 --rm -it \
  --restart=Never -- curl -s --max-time 3 http://frontend.lab08.svc.cluster.local:80
❌ The policy is broken! Can you identify the two bugs?
Give students 2-3 minutes to analyze the policy. The two bugs are: (1) the podSelector in the ingress.from matches tier=frontend, meaning the frontend can only receive traffic from itself — it should use a different label or be empty for external access, and (2) the port is 8080 but the frontend listens on port 80.

🐛 Fix the Broken Policy

The two bugs in the policy:

Bug 1: Wrong Source Selector

The from.podSelector matches tier: frontend — the frontend can only receive traffic from itself. For external access, the from rule should be empty or use a broader selector.

Bug 2: Wrong Port

The policy allows port 8080, but the frontend container listens on port 80. The port in the policy must match the container port.

The corrected policy:


# Save as fixed-policy.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-external-to-frontend
  namespace: lab08
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
  - Ingress
  ingress:
  - from: []       # Allow from all sources
    ports:
    - protocol: TCP
      port: 80     # Correct port
                    

# Apply the fixed policy
kubectl apply -f fixed-policy.yaml

# Test again - should now work
kubectl run test-client --image=nginx:1.25 -n lab08 --rm -it \
  --restart=Never -- curl -s --max-time 3 http://frontend.lab08.svc.cluster.local:80
✅ Expected Output: The test client can now reach the frontend and receives the nginx welcome page. The fixed policy uses from: [] (allow all sources) and the correct port 80.
The from: [] syntax means allow from any source. This is different from omitting the from field entirely. Walk through common debugging techniques: check labels match, verify ports, use kubectl describe to inspect the policy, and always test with curl.

Step 11

Clean Up Resources

Clean up both namespaces used in this lab. NetworkPolicies are namespace-scoped, so deleting the namespace removes them automatically.

🧹 Clean Up Resources

Remove all resources created during this lab:


# Delete both namespaces (cascades to all resources within)
kubectl delete namespace lab08
kubectl delete namespace monitoring

# Verify cleanup
kubectl get namespace lab08 2>/dev/null || echo "Namespace lab08 deleted"
kubectl get namespace monitoring 2>/dev/null || echo "Namespace monitoring deleted"
✅ Checkpoint: Both namespaces and all their resources (pods, services, NetworkPolicies) are deleted. NetworkPolicies are namespace-scoped, so they are automatically removed when the namespace is deleted.
🔑 Key Point: Unlike ClusterRoles and ClusterRoleBindings (from Lab 7), NetworkPolicies are always namespace-scoped. Deleting the namespace is sufficient to clean up all network policies within it.
Remind students that NetworkPolicies are namespace-scoped — no cluster-level cleanup needed. This is simpler than RBAC cleanup. Namespace deletion is the cleanest approach.

Step 12

Summary

Review the NetworkPolicy patterns and commands used in this lab.

📝 Summary — Network Policy Patterns Reference

Pattern Use Case
podSelector: {} + no rules Default deny all (ingress or egress)
podSelector.matchLabels Target specific pods by label
ingress.from.podSelector Allow from specific pods (same namespace)
ingress.from.namespaceSelector Allow from pods in labeled namespaces
egress.to + ports: [53] Allow DNS resolution (always include this)
from: [] Allow from all sources (open ingress)
kubectl get netpol -n <ns> List policies in a namespace
kubectl describe netpol <name> View policy details and selectors
This reference table covers the most common NetworkPolicy patterns. Encourage students to keep this handy. The deny-all + selective allow pattern is the recommended approach for production environments.

🎯 Key Takeaways

  • ✅ Kubernetes defaults to allowing all pod-to-pod communication — NetworkPolicies add restrictions
  • ✅ Start with a default deny-all policy, then selectively allow required traffic
  • ✅ NetworkPolicies are additive — multiple policies combine their allowed paths
  • ✅ Always allow DNS (UDP/TCP port 53) when implementing egress policies
  • ✅ Use namespaceSelector for cross-namespace communication (based on namespace labels)
  • ✅ Test policies thoroughly — verify both allowed and blocked paths
These takeaways represent the core NetworkPolicy concepts. The deny-all-then-allow pattern is the gold standard for production Kubernetes security. Combined with Lab 7's RBAC and Pod Security, students now have a comprehensive security toolkit.
