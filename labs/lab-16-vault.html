Lab 16: Vault and External Secrets

Hands-On Lab

Intermediate Kubernetes — Module 16 of 16

This is the final hands-on lab of the course. Students will work through the complete Vault and External Secrets Operator flow — from storing secrets in Vault to consuming them in a running pod. Allow 45 to 60 minutes.
📋 Lab Overview

What You Will Do

  • Access Vault via the UI and CLI
  • Store and retrieve secrets using the KV v2 engine
  • Create a Vault policy and Kubernetes auth role
  • Verify the ClusterSecretStore connection
  • Create an ExternalSecret and verify it syncs to a K8s Secret
  • Deploy a pod that consumes the synced secret
  • Test secret rotation and policy boundaries
⏰ Duration: 45–60 minutes
⚠️ Prerequisites: Vault running in the vault namespace, ESO installed in external-secrets namespace
Review the overview and prerequisites. Vault and ESO should already be running in the cluster from the platform bootstrap. If any student's environment is not ready, help them verify before starting.

Step 1

Access the Vault UI

First step is to get oriented with Vault — both the UI and the CLI. We will use port-forwarding to access the web UI and kubectl exec to run CLI commands.

Step 1: Access the Vault UI

Port-forward to Vault

# Start port-forwarding to the Vault service
kubectl port-forward svc/vault 8200:8200 -n vault
🌐 Open your browser: Navigate to http://localhost:8200
Login with the root token: root (dev mode default)

Verify via CLI

# Check Vault status from within the pod
kubectl exec -it vault-0 -n vault -- vault status

# List enabled secrets engines
kubectl exec -it vault-0 -n vault -- vault secrets list

# List enabled auth methods
kubectl exec -it vault-0 -n vault -- vault auth list
💡 Expected output: vault status should show Sealed: false. vault secrets list should include secret/ (KV v2). vault auth list should include kubernetes/.
Have students start the port-forward in one terminal and keep it running. They can open the Vault UI in their browser. The dev mode root token is 'root'. The CLI commands verify that Vault is unsealed and that the KV v2 engine and Kubernetes auth method are already enabled by the platform bootstrap.

Step 2

Store Secrets in Vault

Now we will store some application secrets in Vault using the KV v2 engine. We will exec into the Vault pod to use the CLI.

Step 2: Store Secrets in Vault

Write database credentials to Vault

# Exec into the Vault pod
kubectl exec -it vault-0 -n vault -- /bin/sh

# Store database credentials
vault kv put secret/lab16/database \
  username=admin \
  password=s3cureP@ss \
  host=postgres.lab16.svc.cluster.local \
  port=5432

Verify the secret was stored

# Read the secret back
vault kv get secret/lab16/database

# Read as JSON for structured output
vault kv get -format=json secret/lab16/database
✅ Checkpoint: You should see all four key-value pairs (username, password, host, port) in the output. The version should be 1.
Students exec into the Vault pod and use the vault CLI directly. The kv put command stores four key-value pairs at the path secret/lab16/database. The kv get confirms the data was written. The JSON output is useful for scripting and shows metadata like version number.

Step 3

Create a Vault Policy

Next we create a Vault policy that grants read-only access to our lab secrets path. This policy will be assigned to the Kubernetes auth role in the next step.

Step 3: Create a Vault Policy

Write a read-only policy for lab16 secrets

# Still inside the Vault pod shell
# Create the policy
vault policy write lab16-readonly - <<EOF
# Allow reading secrets under lab16/
path "secret/data/lab16/*" {
  capabilities = ["read", "list"]
}

# Allow listing secret metadata
path "secret/metadata/lab16/*" {
  capabilities = ["read", "list"]
}
EOF

Verify the policy

# Read the policy back
vault policy read lab16-readonly
💡 Note the /data/ prefix: KV v2 stores data at secret/data/... even though you write to secret/.... Policies must use the full API path including /data/.
The policy grants read and list capabilities on secret/data/lab16/* which covers all secrets under the lab16 path. We also grant read on the metadata path so ESO can check version information. Remind students that the /data/ prefix is required in policies for KV v2 — this is the most common policy mistake.

Step 4

Configure Kubernetes Auth

Now we create a Vault role that maps a Kubernetes ServiceAccount to the policy we just created. This is the bridge between K8s identity and Vault authorization.

Step 4: Configure Kubernetes Auth

Verify K8s auth is enabled, then create a role

# Still inside the Vault pod shell
# Verify Kubernetes auth is enabled
vault auth list | grep kubernetes

# Create a role mapping a K8s ServiceAccount to the policy
vault write auth/kubernetes/role/lab16-role \
  bound_service_account_names=lab16-sa \
  bound_service_account_namespaces=lab16 \
  policies=lab16-readonly \
  ttl=1h

Verify the role

# Read the role back
vault read auth/kubernetes/role/lab16-role

# Exit the Vault pod shell
exit
What this role says: "If a pod running as ServiceAccount lab16-sa in namespace lab16 authenticates, grant it the lab16-readonly policy with a 1-hour TTL."
The role binds three things together: the ServiceAccount name, the namespace, and the Vault policy. Only pods running as lab16-sa in the lab16 namespace will be able to authenticate with this role. The TTL means the Vault token expires after 1 hour. After verifying the role, students should exit the Vault pod shell.

Step 5

Verify the ClusterSecretStore

The ClusterSecretStore was created by the platform bootstrap. Let us verify it is ready and properly configured.

Step 5: Verify ClusterSecretStore

Check the platform-provisioned ClusterSecretStore

# List all ClusterSecretStores
kubectl get clustersecretstore

# Describe the vault-backend store for details
kubectl describe clustersecretstore vault-backend
✅ Expected output: The ClusterSecretStore should show STATUS: Ready. The describe output should show the Vault server URL and Kubernetes auth configuration.
❌ If not Ready: Check that Vault is running (kubectl get pods -n vault), verify the Vault address in the ClusterSecretStore spec, and check ESO controller logs (kubectl logs -l app.kubernetes.io/name=external-secrets -n external-secrets).
The ClusterSecretStore is provisioned by the platform. Students just need to verify it is in a Ready state. If it is not Ready, walk through the troubleshooting steps. The most common issue is Vault not being accessible at the configured address.

Step 6

Create Namespace and ServiceAccount

We need to create the namespace and ServiceAccount that match what we configured in the Vault role. The ServiceAccount is the identity that ESO will use when fetching secrets for this namespace.

Step 6: Create Namespace and ServiceAccount

# Create the lab16 namespace
kubectl create namespace lab16

# Create the ServiceAccount that matches the Vault role
kubectl create serviceaccount lab16-sa -n lab16

Verify

# Confirm the namespace exists
kubectl get namespace lab16

# Confirm the ServiceAccount exists
kubectl get serviceaccount lab16-sa -n lab16
💡 Why this matters: The Vault role lab16-role is bound to lab16-sa in namespace lab16. If the ServiceAccount name or namespace does not match exactly, Vault authentication will fail.
The namespace and ServiceAccount must match exactly what was configured in the Vault role. This is the identity chain: pod runs as lab16-sa in lab16 namespace, Vault validates this against the role binding, and grants the associated policy.

Step 7

Create an ExternalSecret

This is the core step — creating the ExternalSecret CRD that tells ESO to sync secrets from Vault into a Kubernetes Secret.

Step 7: Create an ExternalSecret

Define the ExternalSecret YAML

cat <<'EOF' | kubectl apply -f -
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: lab16
spec:
  refreshInterval: "1m"
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: db-credentials
    creationPolicy: Owner
  data:
    - secretKey: username
      remoteRef: { key: secret/lab16/database, property: username }
    - secretKey: password
      remoteRef: { key: secret/lab16/database, property: password }
    # ... host and port entries follow same pattern
EOF
Walk through each field: refreshInterval is set to 1 minute for lab purposes (in production you might use 5m). secretStoreRef points to the platform-provisioned ClusterSecretStore. The target defines the K8s Secret name. Each data entry maps a Vault field to a key in the K8s Secret. The remoteRef key uses the Vault CLI path (without /data/) — ESO handles the translation.

Step 8

Verify Secret Sync

Now we verify that ESO successfully synced the secret from Vault into a Kubernetes Secret.

Step 8: Verify Secret Sync

Check the ExternalSecret status

# Check ExternalSecret status
kubectl get externalsecret -n lab16

# Look for STATUS: SecretSynced
kubectl describe externalsecret db-credentials -n lab16

Inspect the synced K8s Secret

# View the Secret (base64 encoded)
kubectl get secret db-credentials -n lab16 -o yaml

# Decode individual values
kubectl get secret db-credentials -n lab16 \
  -o jsonpath='{.data.username}' | base64 -d && echo
kubectl get secret db-credentials -n lab16 \
  -o jsonpath='{.data.password}' | base64 -d && echo
kubectl get secret db-credentials -n lab16 \
  -o jsonpath='{.data.host}' | base64 -d && echo
kubectl get secret db-credentials -n lab16 \
  -o jsonpath='{.data.port}' | base64 -d && echo
✅ Checkpoint: Decoded values should match what you stored in Vault: admin, s3cureP@ss, postgres.lab16.svc.cluster.local, 5432
The ExternalSecret status should show SecretSynced. If it shows an error, use describe to see the error message. The K8s Secret contains base64-encoded values — the jsonpath and base64 decode commands let us verify the actual values match what we stored in Vault. This proves the end-to-end sync is working.

Step 9

Use the Secret in a Pod

Now we deploy a pod that consumes the synced secret as environment variables, proving the complete flow from Vault to application.

Step 9: Use the Secret in a Pod

Deploy a pod that consumes the secret

cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: secret-consumer
  namespace: lab16
spec:
  serviceAccountName: lab16-sa
  containers:
  - name: app
    image: busybox:1.36
    command: ["sleep", "3600"]
    env:
    - { name: DB_USERNAME, valueFrom: { secretKeyRef: { name: db-credentials, key: username }}}
    - { name: DB_PASSWORD, valueFrom: { secretKeyRef: { name: db-credentials, key: password }}}
    - { name: DB_HOST, valueFrom: { secretKeyRef: { name: db-credentials, key: host }}}
    - { name: DB_PORT, valueFrom: { secretKeyRef: { name: db-credentials, key: port }}}
EOF
This pod uses the lab16-sa ServiceAccount and consumes all four keys from the db-credentials Secret as environment variables. The pod runs busybox with a sleep command so we can exec in and inspect the environment.

Step 9: Verify the Secret in the Pod

# Wait for the pod to be running
kubectl wait --for=condition=Ready pod/secret-consumer -n lab16 --timeout=60s

# Exec into the pod and check environment variables
kubectl exec -it secret-consumer -n lab16 -- env | grep DB_
✅ Expected output: 
DB_USERNAME=admin
DB_PASSWORD=s3cureP@ss
DB_HOST=postgres.lab16.svc.cluster.local
DB_PORT=5432
💡 The complete chain: Vault → ESO (via ClusterSecretStore + ExternalSecret) → K8s Secret → Pod env vars. The application has no idea Vault exists — it just reads standard environment variables.
The env command filtered with grep DB_ should show all four variables with the values from Vault. This proves the complete end-to-end flow. The application code does not need any Vault SDK or configuration — it reads standard environment variables. The secret management is completely transparent to the application.

Step 10

Test Secret Rotation

One of the key benefits of the ESO pattern is automatic secret rotation. Let us update the secret in Vault and watch it propagate to Kubernetes.

Step 10: Test Secret Rotation

Update the secret in Vault

# Update the password in Vault
kubectl exec -it vault-0 -n vault -- \
  vault kv put secret/lab16/database \
    username=admin \
    password=r0tatedN3wP@ss \
    host=postgres.lab16.svc.cluster.local \
    port=5432

# Verify the update in Vault
kubectl exec -it vault-0 -n vault -- \
  vault kv get secret/lab16/database
⏰ Wait for the refreshInterval (1 minute in our ExternalSecret), then check if the K8s Secret was updated. 
# After ~1 minute, check the K8s Secret
kubectl get secret db-credentials -n lab16 \
  -o jsonpath='{.data.password}' | base64 -d && echo

# Expected: r0tatedN3wP@ss
Update the password in Vault. The version number in Vault will increment to 2. After the refreshInterval (1 minute), ESO will detect the change and update the K8s Secret. Students should wait about a minute and then check the decoded password value. It should show the new password.

Step 10: Env Vars vs Volume Mounts

⚠️ Important: The K8s Secret was updated, but our pod still has the OLD password in its environment variables. 
# Check the pod's env — still shows old password!
kubectl exec -it secret-consumer -n lab16 -- env | grep DB_PASSWORD
# Output: DB_PASSWORD=s3cureP@ss  (old value!)

# The pod needs a restart to pick up the new env var value
kubectl delete pod secret-consumer -n lab16

# Recreate the pod (or use a Deployment for rolling restart)
# Re-apply the pod spec from Step 9, then verify:
kubectl wait --for=condition=Ready pod/secret-consumer -n lab16 --timeout=60s
kubectl exec -it secret-consumer -n lab16 -- env | grep DB_PASSWORD
# Output: DB_PASSWORD=r0tatedN3wP@ss  (new value!)
💡 Key lesson: Environment variables are set at pod start and never update. Volume-mounted secrets update automatically (~60s kubelet sync). For env vars, use stakater/Reloader to trigger automatic rolling restarts when secrets change.
This is a critical lesson. The K8s Secret was updated by ESO, but environment variables are immutable after pod startup. The pod must be restarted to see the new values. In production, use a Deployment with the stakater/Reloader annotation to automate this. Volume-mounted secrets are updated automatically by the kubelet.

Step 11

Test Policy Boundaries

Now let us verify that Vault policies actually work by trying to access a path that is not covered by our policy.

Step 11: Test Policy Boundaries

Store a secret outside the policy scope

# Store a secret at a DIFFERENT path (outside lab16/)
kubectl exec -it vault-0 -n vault -- \
  vault kv put secret/lab16-other/credentials \
    api_key=super-secret-key

Try to access it via ExternalSecret

cat <<'EOF' | kubectl apply -f -
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: unauthorized-secret
  namespace: lab16
spec:
  refreshInterval: "1m"
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: unauthorized-secret
    creationPolicy: Owner
  data:
    - secretKey: api_key
      remoteRef:
        key: secret/lab16-other/credentials
        property: api_key
EOF
We store a secret at a path outside what the lab16-readonly policy covers. Then we create an ExternalSecret that tries to access it. This should fail because the policy only grants access to secret/data/lab16/*.

Step 11: Verify Policy Denial

# Check the ExternalSecret status
kubectl get externalsecret unauthorized-secret -n lab16

# Describe for the error message
kubectl describe externalsecret unauthorized-secret -n lab16
❌ Expected result: The ExternalSecret should show a SecretSyncedError status. The describe output will show a permission denied error from Vault.
💡 This confirms: Vault policies enforce least-privilege access. The lab16-readonly policy only permits access to secret/data/lab16/*. Requests to any other path are denied, regardless of what the ExternalSecret requests.
The ExternalSecret will fail with a permission denied error. This proves that Vault policies are working correctly. Even though we could create the ExternalSecret CRD in Kubernetes, Vault denies the actual secret fetch because the policy does not grant access to that path. This is the defense-in-depth that Vault provides.

Step 12

Clean Up

Time to clean up the lab resources. We will remove the Kubernetes resources and the Vault secrets and policy.

Step 12: Clean Up

Remove lab resources

# Delete the Kubernetes resources
kubectl delete namespace lab16

# Clean up Vault resources (exec into Vault pod)
kubectl exec -it vault-0 -n vault -- /bin/sh -c '
  vault kv metadata delete secret/lab16/database
  vault kv metadata delete secret/lab16-other/credentials
  vault delete auth/kubernetes/role/lab16-role
  vault policy delete lab16-readonly
'

Verify cleanup

# Confirm namespace is deleted
kubectl get namespace lab16
# Expected: Error from server (NotFound)

# Confirm Vault policy is removed
kubectl exec -it vault-0 -n vault -- vault policy list | grep lab16
# Expected: no output
✅ Cleanup complete. All lab resources have been removed from the cluster and Vault.
Deleting the namespace removes all K8s resources within it (pods, secrets, ExternalSecrets, ServiceAccounts). The Vault cleanup removes the secret data, the auth role, and the policy. Verify both are clean before moving on.
