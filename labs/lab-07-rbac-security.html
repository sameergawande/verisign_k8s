Step 1

Explore Existing ClusterRoles

Start by exploring the built-in ClusterRoles that ship with every Kubernetes cluster. These are the foundation of RBAC and understanding them helps students design their own roles.

🔍 Explore Existing ClusterRoles

First, create a namespace for this lab and examine the built-in ClusterRoles:


# Create a namespace for this lab
kubectl create namespace lab07-$STUDENT_NAME

# List all ClusterRoles (there are many built-in ones)
kubectl get clusterroles | head -20

# Count the total number of ClusterRoles
kubectl get clusterroles --no-headers | wc -l

Examine the three key built-in ClusterRoles:


# The 'admin' ClusterRole - full access within a namespace
kubectl describe clusterrole admin

# The 'edit' ClusterRole - read/write but no role management
kubectl describe clusterrole edit

# The 'view' ClusterRole - read-only access
kubectl describe clusterrole view
✅ Expected Output: You should see dozens of ClusterRoles. The admin role grants full namespace access, edit allows modifications but not role changes, and view is strictly read-only.
🔑 Key Point: Kubernetes ships with several default ClusterRoles. The admin, edit, and view roles are designed to be bound at the namespace level using RoleBindings, even though they are cluster-scoped definitions.
Walk through the output of each describe command. Point out the difference between admin (includes role management), edit (no role management), and view (read-only). These are aggregate ClusterRoles that Kubernetes auto-populates.

Step 2

Create a Namespace-Scoped Role

Now students will create their own Role. Roles are namespace-scoped and define what actions are permitted on which resources within that namespace.

📝 Create a Namespace-Scoped Role

Create a Role that allows read-only access to pods:


# Save as pod-reader-role.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: lab07-$STUDENT_NAME
  name: pod-reader
rules:
- apiGroups: [""]
# ... (remaining lines omitted for brevity)

# Apply the Role
kubectl apply -f pod-reader-role.yaml

# Verify the Role was created
kubectl get roles -n lab07-$STUDENT_NAME

# Examine the Role details
kubectl describe role pod-reader -n lab07-$STUDENT_NAME
🔑 Key Point: Roles use an additive model — there are no “deny” rules. If a permission is not explicitly granted, it is denied by default. The apiGroups: [""] refers to the core API group.
Tip: The Role pod-reader should appear in namespace lab07-$STUDENT_NAME with rules allowing get, list, watch on pods and get on pods/log. Emphasize the additive-only model. There is no way to deny a specific permission in RBAC; you can only grant. Also note that pods/log is a sub-resource, which requires a separate rule entry.

Step 3

Create a ServiceAccount and RoleBinding

ServiceAccounts provide an identity for processes running in pods. RoleBindings connect a Role to a subject (user, group, or ServiceAccount).

🔗 Create a ServiceAccount and RoleBinding

Create a ServiceAccount and bind the pod-reader Role to it:


kubectl create serviceaccount pod-viewer -n lab07-$STUDENT_NAME
kubectl get serviceaccounts -n lab07-$STUDENT_NAME
                

# Save as pod-reader-binding.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: { name: read-pods, namespace: lab07-$STUDENT_NAME }
subjects:
- kind: ServiceAccount
# ... (remaining lines omitted for brevity)

kubectl apply -f pod-reader-binding.yaml
kubectl describe rolebinding read-pods -n lab07-$STUDENT_NAME
Explain the three parts of RBAC: the Role (what permissions), the Subject (who gets them), and the RoleBinding (connecting the two). The roleRef is immutable once created — you must delete and recreate the binding to change it. | ✅ Expected Output: The RoleBinding read-pods should show the pod-viewer ServiceAccount as a subject and pod-reader as the referenced Role.

Step 4

Test Permissions with kubectl auth can-i

The kubectl auth can-i command is essential for testing RBAC configurations without needing to actually perform the action. The --as flag allows impersonation.

🧪 Test Permissions with kubectl auth can-i

Use the --as flag to impersonate the ServiceAccount and test permissions:


# Test: Can the pod-viewer list pods in lab07-$STUDENT_NAME? (should be YES)
kubectl auth can-i list pods -n lab07-$STUDENT_NAME \
  --as=system:serviceaccount:lab07-$STUDENT_NAME:pod-viewer

# Test: Can the pod-viewer get pods in lab07-$STUDENT_NAME? (should be YES)
kubectl auth can-i get pods -n lab07-$STUDENT_NAME \
  --as=system:serviceaccount:lab07-$STUDENT_NAME:pod-viewer

# Test: Can the pod-viewer delete pods? (should be NO)
kubectl auth can-i delete pods -n lab07-$STUDENT_NAME \
  --as=system:serviceaccount:lab07-$STUDENT_NAME:pod-viewer

# Test: Can the pod-viewer create deployments? (should be NO)
kubectl auth can-i create deployments -n lab07-$STUDENT_NAME \
  --as=system:serviceaccount:lab07-$STUDENT_NAME:pod-viewer
                

# List all allowed actions in the namespace
kubectl auth can-i --list -n lab07-$STUDENT_NAME \
  --as=system:serviceaccount:lab07-$STUDENT_NAME:pod-viewer
Tip: Expected output -- list pods: yes, get pods: yes, delete pods: no, create deployments: no. Always test RBAC changes with kubectl auth can-i before relying on them. The --as flag impersonates any user or ServiceAccount. The --list flag is great for auditing what a ServiceAccount can actually do. The --as flag requires impersonation privileges, which cluster-admin has by default.

Step 5

Deploy a Pod Using the ServiceAccount

Now students will deploy a pod that uses the ServiceAccount and verify from inside the pod that the RBAC permissions work as expected.

🚀 Deploy a Pod Using the ServiceAccount

Create a pod that uses the pod-viewer ServiceAccount, then test permissions from inside:


# Save as rbac-test-pod.yaml
apiVersion: v1
kind: Pod
metadata: { name: rbac-test, namespace: lab07-$STUDENT_NAME }
spec:
  serviceAccountName: pod-viewer
# ... (remaining lines omitted for brevity)

kubectl apply -f rbac-test-pod.yaml
kubectl wait --for=condition=Ready pod/rbac-test -n lab07-$STUDENT_NAME --timeout=60s

# This should SUCCEED - list pods
kubectl exec rbac-test -n lab07-$STUDENT_NAME -- kubectl get pods -n lab07-$STUDENT_NAME
# This should FAIL - try to create a deployment
kubectl exec rbac-test -n lab07-$STUDENT_NAME -- kubectl create deployment \
  test --image=nginx -n lab07-$STUDENT_NAME
This is a powerful demonstration. The pod's ServiceAccount token is automatically mounted and used by kubectl inside the container. Students can see that RBAC is enforced at the API server level, not at the client. | ✅ Expected Output: The get pods command succeeds and shows the running pods. The create deployment command fails with a Forbidden error, confirming RBAC enforcement.

Step 6

Create a ClusterRole for Cross-Namespace Access

ClusterRoles differ from Roles in that they can be used across all namespaces when bound with a ClusterRoleBinding, or within a single namespace when bound with a RoleBinding.

🌍 Create a ClusterRole for Cross-Namespace Read Access

Create a ClusterRole that grants read access to pods and deployments across all namespaces:


# Save as cluster-reader-role.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-pod-reader-$STUDENT_NAME
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log", "services"]
# ... (remaining lines omitted for brevity)

# Apply the ClusterRole
kubectl apply -f cluster-reader-role.yaml

# Create a new ServiceAccount for cluster-wide access
kubectl create serviceaccount cluster-viewer -n lab07-$STUDENT_NAME

# Verify the ClusterRole
kubectl describe clusterrole cluster-pod-reader-$STUDENT_NAME
🔑 Key Point: A ClusterRole is a cluster-scoped resource (no namespace in its metadata). It can be bound with a ClusterRoleBinding for cluster-wide access or a RoleBinding for access within a single namespace.
Tip: The ClusterRole cluster-pod-reader-$STUDENT_NAME should list rules for pods, pods/log, services (core API group) and deployments, replicasets (apps API group). ClusterRoles are reusable across namespaces. This pattern is common in production where you define permissions once and bind them per-namespace or cluster-wide as needed.

Step 7

Test ClusterRoleBinding

ClusterRoleBindings grant permissions across all namespaces. This is powerful but must be used carefully in production environments.

🔏 Test ClusterRoleBinding

Create a ClusterRoleBinding to grant cluster-wide read access:


# Save as cluster-reader-binding.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: cluster-pod-reader-binding-$STUDENT_NAME
subjects:
- kind: ServiceAccount
  name: cluster-viewer
# ... (remaining lines omitted for brevity)

# Apply the ClusterRoleBinding
kubectl apply -f cluster-reader-binding.yaml

# Test: Can cluster-viewer list pods in kube-system? (should be YES)
kubectl auth can-i list pods -n kube-system \
  --as=system:serviceaccount:lab07-$STUDENT_NAME:cluster-viewer

# Test: Can cluster-viewer list pods in default? (should be YES)
# ... (more output)
Expected Output: The cluster-viewer can list pods in any namespace (yes, yes) but cannot delete pods anywhere (no). This confirms cluster-wide read-only access. Production Warning: ClusterRoleBindings grant access across all namespaces, including system namespaces like kube-system. Use them sparingly and prefer namespace-scoped RoleBindings when possible. Contrast this with the namespace-scoped pod-viewer from earlier.

Step 8

Apply Pod Security Standards

Pod Security Standards (PSS) replaced PodSecurityPolicies in Kubernetes 1.25+. They use namespace labels to enforce security profiles: privileged, baseline, and restricted.

🛡 Apply Pod Security Standards

Create a new namespace with the restricted Pod Security Standard:


# Create a namespace with Pod Security Standards labels
kubectl create namespace lab07-restricted-$STUDENT_NAME

# Apply the restricted profile (enforce mode)
kubectl label namespace lab07-restricted-$STUDENT_NAME \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/enforce-version=latest \
  pod-security.kubernetes.io/warn=restricted \
# ... (more output)

# Verify the labels
kubectl get namespace lab07-restricted-$STUDENT_NAME --show-labels

# View the labels in YAML format
kubectl get namespace lab07-restricted-$STUDENT_NAME -o yaml

Pod Security Standard Profiles

  • Privileged: Unrestricted (no enforcement)
  • Baseline: Prevents known privilege escalations
  • Restricted: Hardened, follows pod security best practices
Explain the three enforcement modes: enforce blocks the pod from being created, warn shows a warning to the user but allows it, and audit logs the violation but allows it. In production, you might start with warn/audit and migrate to enforce once workloads are compliant. | ✅ Expected Output: The namespace should have all six pod-security.kubernetes.io labels set to restricted. Three modes are active: enforce (blocks violations), warn (shows warnings), and audit (logs violations).

Step 9

Test Pod Security

Students will now attempt to deploy pods that violate the restricted profile and observe what happens. This demonstrates the enforcement in action.

🚫 Test Pod Security — Violations

Try deploying pods that violate the restricted profile:


# Save as privileged-pod.yaml
apiVersion: v1
kind: Pod
metadata: { name: privileged-test, namespace: lab07-restricted-$STUDENT_NAME }
# ... (remaining lines omitted for brevity)

# Both should FAIL under the restricted profile
kubectl apply -f privileged-pod.yaml
kubectl apply -f root-pod.yaml
Expected Error: Blocked again. The restricted profile requires runAsNonRoot: true, allowPrivilegeEscalation: false, and dropping all capabilities. Both pods should fail. Walk through the error messages carefully.

Step 10

Configure SecurityContext on a Pod

Now students will create a properly secured pod that satisfies the restricted profile. This shows the correct security patterns for production workloads.

🔒 Configure SecurityContext on a Pod

Create a pod that complies with the restricted security profile:


# Save as secure-pod.yaml
apiVersion: v1
kind: Pod
metadata: { name: secure-app, namespace: lab07-restricted-$STUDENT_NAME }
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
# ... (remaining lines omitted for brevity)

# Deploy the secure pod (should SUCCEED)
kubectl apply -f secure-pod.yaml

# Verify it is running
kubectl get pod secure-app -n lab07-restricted-$STUDENT_NAME

# Examine the security settings
kubectl get pod secure-app -n lab07-restricted-$STUDENT_NAME -o \
# ... (more output)
Tip: The secure-app pod should be Running. It passes the restricted profile because it: runs as non-root, drops all capabilities, disallows privilege escalation, uses a read-only root filesystem, and has a RuntimeDefault seccomp profile. This is the gold standard for secure pod configuration. Walk through each security setting: runAsNonRoot prevents root, readOnlyRootFilesystem prevents filesystem writes (hence the /tmp emptyDir), capabilities drop ALL removes all Linux capabilities, and the seccomp profile restricts system calls.

🔍 Verify Security Settings Inside the Pod

Confirm the security restrictions from inside the container:


# Check who the process is running as
kubectl exec secure-app -n lab07-restricted-$STUDENT_NAME -- id

# Try to write to the root filesystem (should FAIL)
kubectl exec secure-app -n lab07-restricted-$STUDENT_NAME -- \
  touch /test-file

# Writing to /tmp should work (emptyDir mount)
kubectl exec secure-app -n lab07-restricted-$STUDENT_NAME -- \
  touch /tmp/test-file

# Verify no extra capabilities
kubectl exec secure-app -n lab07-restricted-$STUDENT_NAME -- \
  cat /proc/1/status | grep -i cap
✅ Expected Output:
id → uid=1000 gid=1000 groups=1000
touch /test-file → Read-only file system error
touch /tmp/test-file → succeeds (no output)
CapEff → 0000000000000000 (no capabilities)
🔑 Key Point: The combination of runAsNonRoot, readOnlyRootFilesystem, and dropping all capabilities creates a defense-in-depth posture. Even if an attacker compromises the application, they cannot escalate privileges, modify the container filesystem, or leverage Linux capabilities.
This is a great checkpoint to verify understanding. The CapEff value of all zeros means no effective capabilities. Compare this to a default pod which would have several capabilities set. The read-only filesystem prevents malware installation or config tampering.

Part 3

IRSA — IAM Roles for Service Accounts

Part 3 extends the security theme from Kubernetes RBAC into the AWS IAM layer. IRSA (IAM Roles for Service Accounts) lets individual pods assume specific IAM roles, replacing the older pattern of sharing the node's instance profile. This uses the EKS OIDC provider, which should already be provisioned.

Step 11

Annotate a ServiceAccount with an IAM Role

We will use a pre-created IAM role and policy to keep the exercise focused on the Kubernetes side. The IAM role grants read-only access to a specific S3 bucket and has a trust policy scoped to the ServiceAccount we create here.

🔐 Step 11: Create an IRSA-Annotated ServiceAccount

Retrieve the IAM role ARN and create a ServiceAccount linked to it:


# Capture the OIDC issuer and account ID
OIDC_ISSUER=$(aws eks describe-cluster --name platform-lab \
  --query "cluster.identity.oidc.issuer" --output text | sed 's|https://||')
ACCOUNT_ID=$(aws sts get-caller-identity --query "Account" --output text)

# Build the IRSA role ARN (pre-created by the platform team)
ROLE_ARN="arn:aws:iam::${ACCOUNT_ID}:role/irsa-s3-reader-$STUDENT_NAME"
echo "Role ARN: $ROLE_ARN"
                

# Create a namespace for the IRSA exercise
kubectl create namespace lab07-irsa-$STUDENT_NAME

# Create the annotated ServiceAccount
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: ServiceAccount
metadata:
  name: s3-reader-$STUDENT_NAME
  namespace: lab07-irsa-$STUDENT_NAME
  annotations:
    eks.amazonaws.com/role-arn: "${ROLE_ARN}"
EOF

# Verify the annotation is present
kubectl get sa s3-reader-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME -o yaml

How IRSA Works

The eks.amazonaws.com/role-arn annotation tells the EKS mutating webhook to inject a projected service-account token and two environment variables (AWS_ROLE_ARN, AWS_WEB_IDENTITY_TOKEN_FILE) into every pod that uses this ServiceAccount.

The annotation is the critical link between Kubernetes and AWS IAM. Make sure students verify the annotation appears in the YAML output and that the role ARN matches their student name. A typo here is the most common IRSA failure.

Step 12

Deploy a Pod Using IRSA to Access S3

Now we deploy a pod that references the annotated ServiceAccount and verify it can reach S3.

🚀 Step 12: Deploy an IRSA Test Pod

Create a pod that uses the IRSA-annotated ServiceAccount:


# Save as irsa-test-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: irsa-test-$STUDENT_NAME
  namespace: lab07-irsa-$STUDENT_NAME
spec:
  serviceAccountName: s3-reader-$STUDENT_NAME
  containers:
  - name: aws-cli
    image: amazon/aws-cli:latest
    command: ["sleep", "3600"]
  restartPolicy: Never

# Apply and wait for the pod
kubectl apply -f irsa-test-pod.yaml
kubectl wait --for=condition=Ready \
  pod/irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME --timeout=60s

# Confirm it is running
kubectl get pod irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME
The key field is serviceAccountName. This is what triggers the EKS mutating webhook to inject the projected token and AWS environment variables. The amazon/aws-cli image gives us the AWS CLI inside the pod for testing.

🔍 Step 12: Test S3 Access from the Pod

Verify the pod can read from the S3 bucket using its IRSA credentials:


# List the S3 bucket contents
kubectl exec irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME \
  -- aws s3 ls s3://platform-lab-irsa-demo/

# Read a specific object
kubectl exec irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME \
  -- aws s3 cp s3://platform-lab-irsa-demo/test-file.txt -
                

# Confirm the write is denied (read-only policy)
kubectl exec irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME \
  -- bash -c 'echo "test" | aws s3 cp - \
    s3://platform-lab-irsa-demo/unauthorized-$STUDENT_NAME.txt'
# Expected: AccessDenied — the policy only allows read
✅ Expected: s3 ls and s3 cp (read) succeed. The write attempt returns AccessDenied. This confirms least-privilege enforcement through IRSA.
The read succeeds because the IAM policy attached to the role allows s3:GetObject and s3:ListBucket. The write fails because s3:PutObject is not in the policy. This demonstrates that IRSA enforces the exact permissions defined in the IAM policy, not the broader node-level permissions.

Step 13

Verify the Temporary Credentials

This step proves the credential chain: projected JWT token, STS exchange, temporary AWS credentials. Understanding this flow is essential for debugging IRSA in production.

🔏 Step 13: Inspect the IRSA Credential Chain

Examine the injected environment variables and token:


# Check the injected AWS environment variables
kubectl exec irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME \
  -- env | grep AWS
# Expected:
#   AWS_ROLE_ARN=arn:aws:iam::ACCOUNT:role/irsa-s3-reader-$STUDENT_NAME
#   AWS_WEB_IDENTITY_TOKEN_FILE=/var/run/secrets/eks.amazonaws.com/serviceaccount/token

# Verify the assumed-role identity
kubectl exec irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME \
  -- aws sts get-caller-identity
# Expected: Arn contains "assumed-role/irsa-s3-reader-$STUDENT_NAME"
                

# Peek at the projected token (a JWT issued by the OIDC provider)
kubectl exec irsa-test-$STUDENT_NAME -n lab07-irsa-$STUDENT_NAME \
  -- cat /var/run/secrets/eks.amazonaws.com/serviceaccount/token
# The token contains claims: iss (OIDC URL), sub (SA identity), aud (sts.amazonaws.com)
🔑 Key Point: The credentials are temporary and automatically rotated. The pod never holds long-lived AWS keys. STS exchanges the projected JWT for short-lived credentials, and the SDK refreshes them transparently.
The get-caller-identity output must show the IRSA role ARN, not the EC2 instance role. If it shows the node role, IRSA is not working and the SDK is falling back to the instance metadata service. The two environment variables (AWS_ROLE_ARN and AWS_WEB_IDENTITY_TOKEN_FILE) are the mechanism — the AWS SDK detects them automatically and uses the web-identity token flow.

Summary

Review the RBAC command reference, security concepts, and IRSA flow from this lab.

📝 Summary — Command Reference

Command Purpose
kubectl get roles -n <ns> List namespace-scoped Roles
kubectl get clusterroles List cluster-scoped Roles
kubectl auth can-i <verb> <resource> Test current user permissions
kubectl auth can-i --as=<sa> Test another identity's permissions
kubectl create sa <name> Create a ServiceAccount
kubectl label ns <ns> pod-security... Apply Pod Security Standards
kubectl annotate sa <name> eks...role-arn=<ARN> Link a ServiceAccount to an IAM role (IRSA)

This table serves as a quick reference for RBAC and IRSA operations. Encourage students to keep this handy during subsequent labs and real-world operations.

🎯 Key Takeaways

  • ✅ RBAC uses an additive model — permissions are granted, never denied
  • ✅ Roles are namespace-scoped; ClusterRoles are cluster-scoped
  • ✅ RoleBindings and ClusterRoleBindings connect Roles to subjects
  • ✅ kubectl auth can-i is essential for testing and auditing permissions
  • ✅ Pod Security Standards (restricted, baseline, privileged) replace PodSecurityPolicies
  • ✅ SecurityContext settings provide defense-in-depth at the container level
  • ✅ IRSA provides pod-level IAM identities via projected service-account tokens
  • ✅ Always verify IRSA with aws sts get-caller-identity from inside the pod
Reinforce that RBAC, Pod Security, and IRSA are complementary security layers. RBAC controls who can do what at the Kubernetes API level. Pod Security Standards control what workloads can run. SecurityContexts provide runtime hardening. IRSA extends the security boundary into AWS, giving each workload its own scoped IAM identity instead of sharing the node's instance profile.
