Lab 7: RBAC and Security

Role-Based Access Control, Pod Security Standards, and SecurityContexts

Intermediate Kubernetes — Module 7 of 16

Lab 7 covers RBAC (Roles, RoleBindings, ClusterRoles, ClusterRoleBindings), ServiceAccounts, Pod Security Standards, and SecurityContexts. Estimated time: 45 minutes. Students should have a running cluster from previous labs.

📋 Lab Overview

Objectives

  • Explore existing ClusterRoles and bindings
  • Create Roles, RoleBindings, and ServiceAccounts
  • Apply Pod Security Standards and SecurityContexts
  • Test RBAC permission boundaries

Prerequisites

  • Completed Labs 1–6
  • kubectl with cluster-admin access on a running EKS cluster
⏱ Duration: ~45 minutes
This lab requires cluster-admin permissions to create Roles and ClusterRoles. Students will create restricted ServiceAccounts and test their permissions to understand RBAC boundaries.

Step 1

Explore Existing ClusterRoles

Start by exploring the built-in ClusterRoles that ship with every Kubernetes cluster. These are the foundation of RBAC and understanding them helps students design their own roles.

🔍 Explore Existing ClusterRoles

First, create a namespace for this lab and examine the built-in ClusterRoles:


# Create a namespace for this lab
kubectl create namespace lab07

# List all ClusterRoles (there are many built-in ones)
kubectl get clusterroles | head -20

# Count the total number of ClusterRoles
kubectl get clusterroles --no-headers | wc -l

Examine the three key built-in ClusterRoles:


# The 'admin' ClusterRole - full access within a namespace
kubectl describe clusterrole admin

# The 'edit' ClusterRole - read/write but no role management
kubectl describe clusterrole edit

# The 'view' ClusterRole - read-only access
kubectl describe clusterrole view
✅ Expected Output: You should see dozens of ClusterRoles. The admin role grants full namespace access, edit allows modifications but not role changes, and view is strictly read-only.
🔑 Key Point: Kubernetes ships with several default ClusterRoles. The admin, edit, and view roles are designed to be bound at the namespace level using RoleBindings, even though they are cluster-scoped definitions.
Walk through the output of each describe command. Point out the difference between admin (includes role management), edit (no role management), and view (read-only). These are aggregate ClusterRoles that Kubernetes auto-populates.

Step 2

Create a Namespace-Scoped Role

Now students will create their own Role. Roles are namespace-scoped and define what actions are permitted on which resources within that namespace.

📝 Create a Namespace-Scoped Role

Create a Role that allows read-only access to pods:


# Save as pod-reader-role.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: lab07
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get"]
                

# Apply the Role
kubectl apply -f pod-reader-role.yaml

# Verify the Role was created
kubectl get roles -n lab07

# Examine the Role details
kubectl describe role pod-reader -n lab07
✅ Expected Output: The Role pod-reader should appear in namespace lab07 with rules allowing get, list, watch on pods and get on pods/log.
🔑 Key Point: Roles use an additive model — there are no “deny” rules. If a permission is not explicitly granted, it is denied by default. The apiGroups: [""] refers to the core API group.
Emphasize the additive-only model. There is no way to deny a specific permission in RBAC; you can only grant. Also note that pods/log is a sub-resource, which requires a separate rule entry.

Step 3

Create a ServiceAccount and RoleBinding

ServiceAccounts provide an identity for processes running in pods. RoleBindings connect a Role to a subject (user, group, or ServiceAccount).

🔗 Create a ServiceAccount and RoleBinding

Create a ServiceAccount and bind the pod-reader Role to it:


# Create a ServiceAccount
kubectl create serviceaccount pod-viewer -n lab07

# Verify the ServiceAccount
kubectl get serviceaccounts -n lab07

Now create the RoleBinding:


# Save as pod-reader-binding.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: lab07
subjects:
- kind: ServiceAccount
  name: pod-viewer
  namespace: lab07
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
                    

# Apply the RoleBinding
kubectl apply -f pod-reader-binding.yaml

# Verify the binding
kubectl describe rolebinding read-pods -n lab07
✅ Expected Output: The RoleBinding read-pods should show the pod-viewer ServiceAccount as a subject and pod-reader as the referenced Role.
Explain the three parts of RBAC: the Role (what permissions), the Subject (who gets them), and the RoleBinding (connecting the two). The roleRef is immutable once created — you must delete and recreate the binding to change it.

Step 4

Test Permissions with kubectl auth can-i

The kubectl auth can-i command is essential for testing RBAC configurations without needing to actually perform the action. The --as flag allows impersonation.

🧪 Test Permissions with kubectl auth can-i

Use the --as flag to impersonate the ServiceAccount and test permissions:


# Test: Can the pod-viewer list pods in lab07? (should be YES)
kubectl auth can-i list pods -n lab07 \
  --as=system:serviceaccount:lab07:pod-viewer

# Test: Can the pod-viewer get pods in lab07? (should be YES)
kubectl auth can-i get pods -n lab07 \
  --as=system:serviceaccount:lab07:pod-viewer

# Test: Can the pod-viewer delete pods? (should be NO)
kubectl auth can-i delete pods -n lab07 \
  --as=system:serviceaccount:lab07:pod-viewer

# Test: Can the pod-viewer create deployments? (should be NO)
kubectl auth can-i create deployments -n lab07 \
  --as=system:serviceaccount:lab07:pod-viewer
✅ Expected Output:
list pods → yes
get pods → yes
delete pods → no
create deployments → no

List all permissions for this ServiceAccount:


# List all allowed actions in the namespace
kubectl auth can-i --list -n lab07 \
  --as=system:serviceaccount:lab07:pod-viewer
🔑 Key Point: Always test RBAC changes with kubectl auth can-i before relying on them. The --as flag impersonates any user or ServiceAccount, making it invaluable for verifying least-privilege configurations.
The --as flag requires the caller to have impersonation privileges, which cluster-admin has by default. In production, not every user will be able to use --as. The --list flag is great for auditing what a ServiceAccount can actually do.

Step 5

Deploy a Pod Using the ServiceAccount

Now students will deploy a pod that uses the ServiceAccount and verify from inside the pod that the RBAC permissions work as expected.

🚀 Deploy a Pod Using the ServiceAccount

Create a pod that uses the pod-viewer ServiceAccount:


# Save as rbac-test-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: rbac-test
  namespace: lab07
spec:
  serviceAccountName: pod-viewer
  containers:
  - name: kubectl
    image: bitnami/kubectl:latest
    command: ["sleep", "3600"]
                

# Deploy the test pod
kubectl apply -f rbac-test-pod.yaml

# Wait for it to be ready
kubectl wait --for=condition=Ready pod/rbac-test -n lab07 --timeout=60s

Test permissions from inside the pod:


# This should SUCCEED - list pods
kubectl exec rbac-test -n lab07 -- kubectl get pods -n lab07

# This should FAIL - try to create a deployment
kubectl exec rbac-test -n lab07 -- kubectl create deployment \
  test --image=nginx -n lab07
✅ Expected Output: The get pods command succeeds and shows the running pods. The create deployment command fails with a Forbidden error, confirming RBAC enforcement.
This is a powerful demonstration. The pod's ServiceAccount token is automatically mounted and used by kubectl inside the container. Students can see that RBAC is enforced at the API server level, not at the client.

Step 6

Create a ClusterRole for Cross-Namespace Access

ClusterRoles differ from Roles in that they can be used across all namespaces when bound with a ClusterRoleBinding, or within a single namespace when bound with a RoleBinding.

🌍 Create a ClusterRole for Cross-Namespace Read Access

Create a ClusterRole that grants read access to pods and deployments across all namespaces:


# Save as cluster-reader-role.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-pod-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log", "services"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch"]
                

# Apply the ClusterRole
kubectl apply -f cluster-reader-role.yaml

# Create a new ServiceAccount for cluster-wide access
kubectl create serviceaccount cluster-viewer -n lab07

# Verify the ClusterRole
kubectl describe clusterrole cluster-pod-reader
✅ Expected Output: The ClusterRole cluster-pod-reader should list rules for pods, pods/log, services (core API group) and deployments, replicasets (apps API group).
🔑 Key Point: A ClusterRole is a cluster-scoped resource (no namespace in its metadata). It can be bound with a ClusterRoleBinding for cluster-wide access or a RoleBinding for access within a single namespace.
Highlight that ClusterRoles are reusable across namespaces. This pattern is common in production where you define permissions once and bind them per-namespace or cluster-wide as needed.

Step 7

Test ClusterRoleBinding

ClusterRoleBindings grant permissions across all namespaces. This is powerful but must be used carefully in production environments.

🔏 Test ClusterRoleBinding

Create a ClusterRoleBinding to grant cluster-wide read access:


# Save as cluster-reader-binding.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: cluster-pod-reader-binding
subjects:
- kind: ServiceAccount
  name: cluster-viewer
  namespace: lab07
roleRef:
  kind: ClusterRole
  name: cluster-pod-reader
  apiGroup: rbac.authorization.k8s.io
                

# Apply the ClusterRoleBinding
kubectl apply -f cluster-reader-binding.yaml

# Test: Can cluster-viewer list pods in kube-system? (should be YES)
kubectl auth can-i list pods -n kube-system \
  --as=system:serviceaccount:lab07:cluster-viewer

# Test: Can cluster-viewer list pods in default? (should be YES)
kubectl auth can-i list pods -n default \
  --as=system:serviceaccount:lab07:cluster-viewer

# Test: Can cluster-viewer delete pods anywhere? (should be NO)
kubectl auth can-i delete pods -n kube-system \
  --as=system:serviceaccount:lab07:cluster-viewer
✅ Expected Output: The cluster-viewer can list pods in any namespace (yes, yes) but cannot delete pods anywhere (no). This confirms cluster-wide read-only access.
⚠ Production Warning: ClusterRoleBindings grant access across all namespaces, including system namespaces like kube-system. Use them sparingly and prefer namespace-scoped RoleBindings when possible.
Contrast this with the namespace-scoped pod-viewer from earlier. The cluster-viewer can now see pods in kube-system, default, and every other namespace. This is the difference between RoleBinding and ClusterRoleBinding.

Step 8

Apply Pod Security Standards

Pod Security Standards (PSS) replaced PodSecurityPolicies in Kubernetes 1.25+. They use namespace labels to enforce security profiles: privileged, baseline, and restricted.

🛡 Apply Pod Security Standards

Create a new namespace with the restricted Pod Security Standard:


# Create a namespace with Pod Security Standards labels
kubectl create namespace lab07-restricted

# Apply the restricted profile (enforce mode)
kubectl label namespace lab07-restricted \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/enforce-version=latest \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/warn-version=latest \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/audit-version=latest
                

# Verify the labels
kubectl get namespace lab07-restricted --show-labels

# View the labels in YAML format
kubectl get namespace lab07-restricted -o yaml
✅ Expected Output: The namespace should have all six pod-security.kubernetes.io labels set to restricted. Three modes are active: enforce (blocks violations), warn (shows warnings), and audit (logs violations).

Pod Security Standard Profiles

  • Privileged: Unrestricted (no enforcement)
  • Baseline: Prevents known privilege escalations
  • Restricted: Hardened, follows pod security best practices
Explain the three enforcement modes: enforce blocks the pod from being created, warn shows a warning to the user but allows it, and audit logs the violation but allows it. In production, you might start with warn/audit and migrate to enforce once workloads are compliant.

Step 9

Test Pod Security

Students will now attempt to deploy pods that violate the restricted profile and observe what happens. This demonstrates the enforcement in action.

🚫 Test Pod Security — Violations

Try deploying a pod that violates the restricted profile:


# Save as privileged-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: privileged-test
  namespace: lab07-restricted
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    securityContext:
      privileged: true
                

# Try to create the privileged pod (should FAIL)
kubectl apply -f privileged-pod.yaml
❌ Expected Error: The pod creation is blocked with a message like:
Error: pods "privileged-test" is forbidden: violates PodSecurity "restricted:latest" - privileged

Try another violation — running as root:


# Save as root-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: root-test
  namespace: lab07-restricted
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    # No securityContext - defaults to running as root
                    

# Try to create the root pod (should FAIL)
kubectl apply -f root-pod.yaml
❌ Expected Error: Blocked again. The restricted profile requires runAsNonRoot: true, allowPrivilegeEscalation: false, and dropping all capabilities.
Both of these should fail. Walk through the error messages carefully — they list every violation. This is the enforce mode at work. If you had only set warn mode, the pods would be created with a warning.

Step 10

Configure SecurityContext on a Pod

Now students will create a properly secured pod that satisfies the restricted profile. This shows the correct security patterns for production workloads.

🔒 Configure SecurityContext on a Pod

Create a pod that complies with the restricted security profile:


# Save as secure-pod.yaml
apiVersion: v1
kind: Pod
metadata: { name: secure-app, namespace: lab07-restricted }
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 1000
    fsGroup: 1000
    seccompProfile: { type: RuntimeDefault }
  containers:
  - name: app
    image: python:3.11-slim
    command: ["python", "-m", "http.server", "8080"]
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities: { drop: [ALL] }
    resources: { limits: { memory: "128Mi", cpu: "250m" },
                 requests: { memory: "64Mi", cpu: "125m" } }
    volumeMounts: [{ name: tmp, mountPath: /tmp }]
  volumes: [{ name: tmp, emptyDir: {} }]
                

# Deploy the secure pod (should SUCCEED)
kubectl apply -f secure-pod.yaml

# Verify it is running
kubectl get pod secure-app -n lab07-restricted

# Examine the security settings
kubectl get pod secure-app -n lab07-restricted -o jsonpath='{.spec.containers[0].securityContext}' | python3 -m json.tool
✅ Checkpoint: The secure-app pod should be Running. It passes the restricted profile because it: runs as non-root, drops all capabilities, disallows privilege escalation, uses a read-only root filesystem, and has a RuntimeDefault seccomp profile.
This is the gold standard for secure pod configuration. Walk through each security setting: runAsNonRoot prevents root, readOnlyRootFilesystem prevents filesystem writes (hence the /tmp emptyDir), capabilities drop ALL removes all Linux capabilities, and the seccomp profile restricts system calls.

🔍 Verify Security Settings Inside the Pod

Confirm the security restrictions from inside the container:


# Check who the process is running as
kubectl exec secure-app -n lab07-restricted -- id

# Try to write to the root filesystem (should FAIL)
kubectl exec secure-app -n lab07-restricted -- \
  touch /test-file

# Writing to /tmp should work (emptyDir mount)
kubectl exec secure-app -n lab07-restricted -- \
  touch /tmp/test-file

# Verify no extra capabilities
kubectl exec secure-app -n lab07-restricted -- \
  cat /proc/1/status | grep -i cap
✅ Expected Output:
id → uid=1000 gid=1000 groups=1000
touch /test-file → Read-only file system error
touch /tmp/test-file → succeeds (no output)
CapEff → 0000000000000000 (no capabilities)
🔑 Key Point: The combination of runAsNonRoot, readOnlyRootFilesystem, and dropping all capabilities creates a defense-in-depth posture. Even if an attacker compromises the application, they cannot escalate privileges, modify the container filesystem, or leverage Linux capabilities.
This is a great checkpoint to verify understanding. The CapEff value of all zeros means no effective capabilities. Compare this to a default pod which would have several capabilities set. The read-only filesystem prevents malware installation or config tampering.

Step 11

Clean Up Resources

Always clean up cluster-scoped resources (ClusterRoles, ClusterRoleBindings) in addition to namespace-scoped resources.

🧹 Clean Up Resources

Remove all resources created during this lab:


# Delete cluster-scoped resources first
kubectl delete clusterrolebinding cluster-pod-reader-binding
kubectl delete clusterrole cluster-pod-reader

# Delete both namespaces (cascades to all resources within)
kubectl delete namespace lab07
kubectl delete namespace lab07-restricted

# Verify cleanup
kubectl get clusterrole cluster-pod-reader 2>/dev/null || echo "ClusterRole deleted"
kubectl get namespace lab07 2>/dev/null || echo "Namespace lab07 deleted"
kubectl get namespace lab07-restricted 2>/dev/null || echo "Namespace lab07-restricted deleted"
✅ Checkpoint: All three verification commands should show the “deleted” messages. Both namespaces and the cluster-scoped resources are gone.
⚠ Important: Deleting a namespace removes Roles and RoleBindings within it, but ClusterRoles and ClusterRoleBindings are cluster-scoped — they must be deleted separately.
Emphasize the difference between namespace-scoped and cluster-scoped cleanup. This is a common gotcha in production — orphaned ClusterRoleBindings can persist long after the namespace is deleted.

Step 12

Summary

Review the RBAC command reference and key concepts from this lab.

📝 Summary — RBAC Command Reference

Command Purpose
kubectl get roles -n <ns> List namespace-scoped Roles
kubectl get clusterroles List cluster-scoped Roles
kubectl describe role <name> View Role rules and permissions
kubectl auth can-i <verb> <resource> Test current user permissions
kubectl auth can-i --as=<sa> Test another identity's permissions
kubectl auth can-i --list List all allowed actions
kubectl create sa <name> Create a ServiceAccount
kubectl label ns <ns> pod-security... Apply Pod Security Standards
This table serves as a quick reference for RBAC operations. Encourage students to keep this handy during subsequent labs and real-world operations.

🎯 Key Takeaways

  • ✅ RBAC uses an additive model — permissions are granted, never denied
  • ✅ Roles are namespace-scoped; ClusterRoles are cluster-scoped
  • ✅ RoleBindings and ClusterRoleBindings connect Roles to subjects
  • ✅ kubectl auth can-i is essential for testing and auditing permissions
  • ✅ Pod Security Standards (restricted, baseline, privileged) replace PodSecurityPolicies
  • ✅ SecurityContext settings provide defense-in-depth at the container level
Reinforce that RBAC and Pod Security are complementary layers. RBAC controls who can do what at the API level. Pod Security Standards control what workloads can run. SecurityContexts provide runtime hardening.
