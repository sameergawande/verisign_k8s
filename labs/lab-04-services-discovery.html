Lab 4: Services and Service Discovery

ClusterIP, NodePort, LoadBalancer, and DNS-Based Discovery

⏰ Estimated Time: 45 minutes

Welcome to Lab 4. In this lab, students will explore all Kubernetes Service types and learn how DNS-based service discovery enables communication between application tiers. This lab uses a multi-tier application pattern that reflects real-world microservice architectures on the Verisign EKS platform. Students will create ClusterIP, NodePort, LoadBalancer, and headless Services, then examine Endpoints, DNS records, session affinity, and multi-port configurations.

📚 Lab Overview

What You Will Do

  • Deploy a multi-tier application
  • Create ClusterIP, NodePort, and LoadBalancer Services
  • Test DNS-based service discovery and examine Endpoints
  • Configure headless Services, session affinity, and multi-port Services

Prerequisites

  • Completion of Labs 1–3 with kubectl access
  • Familiarity with pods and deployments
Review the objectives with students. This lab progresses from internal-only services to externally accessible ones, mirroring how real applications are built. Ensure all students can access their cluster before beginning. The LoadBalancer step requires the AWS Load Balancer Controller to be installed on the cluster.

Step 1

Deploy a Multi-Tier Application

Students will deploy a simple two-tier application consisting of a frontend (nginx) and a backend (httpbin). This mirrors common microservice patterns where a frontend proxies requests to backend APIs.

🚀 Step 1: Create the Namespace and Backend

Set up the lab namespace and deploy the backend tier:


# Create the lab namespace
$ kubectl create namespace lab-services
                

# Save as backend-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
  namespace: lab-services
spec:
  replicas: 3
  selector:
    matchLabels: { app: backend, tier: api }
  template:
    metadata:
      labels: { app: backend, tier: api }
    spec:
      containers:
        - name: backend
          image: kennethreitz/httpbin:latest
          ports: [{ containerPort: 80 }]
          readinessProbe:
            httpGet: { path: /health, port: 80 }
            initialDelaySeconds: 5
            periodSeconds: 10
                

$ kubectl apply -f backend-deployment.yaml
$ kubectl get pods -n lab-services -l tier=api -w
The backend uses httpbin, a well-known HTTP testing service that echoes request details. It has three replicas to demonstrate load balancing later. The readiness probe ensures only healthy pods receive traffic. Wait until all three replicas show Ready before proceeding.

🚀 Step 1: Deploy the Frontend Tier

Deploy the frontend that will communicate with the backend:


# Save as frontend-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
  namespace: lab-services
spec:
  replicas: 2
  selector:
    matchLabels: { app: frontend, tier: web }
  template:
    metadata:
      labels: { app: frontend, tier: web }
    spec:
      containers:
        - name: frontend
          image: nginx:1.25
          ports:
            - containerPort: 80
            - containerPort: 443
          readinessProbe:
            httpGet: { path: /, port: 80 }
            initialDelaySeconds: 3
            periodSeconds: 5
                

$ kubectl apply -f frontend-deployment.yaml

# Verify all pods are running
$ kubectl get pods -n lab-services -o wide

Checkpoint: You should have 5 pods running — 3 backend and 2 frontend. All should show 1/1 Ready.

The frontend exposes both port 80 and 443 to demonstrate multi-port Services later. At this point, neither tier can communicate with the other because no Services exist yet. This is the baseline state before we add networking.

Step 2

Create a ClusterIP Service for the Backend

ClusterIP is the default and most common Service type. It creates an internal virtual IP that load-balances traffic across all matching pods. This is how microservices communicate within the cluster.

🔗 Step 2: Create a ClusterIP Service

Create an internal Service for the backend:


# Save as backend-svc.yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-svc
  namespace: lab-services
spec:
  type: ClusterIP
  selector:
    app: backend
    tier: api
  ports:
    - name: http
      port: 80
      targetPort: 80
      protocol: TCP
                

$ kubectl apply -f backend-svc.yaml

# Examine the Service
$ kubectl get svc backend-svc -n lab-services
$ kubectl describe svc backend-svc -n lab-services

Expected Output:


NAME          TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
backend-svc   ClusterIP   10.100.45.120   <none>        80/TCP    5s

Key Concept: The ClusterIP (e.g., 10.100.45.120) is a virtual IP managed by kube-proxy. It is only reachable from within the cluster. Traffic sent to this IP is load-balanced across all pods matching the selector.

Point out the ClusterIP address and explain it is a virtual IP that only exists in iptables or IPVS rules on each node. There is no external-IP because ClusterIP Services are internal-only by design. The describe output shows the Endpoints, which should list the IPs of all three backend pods. If Endpoints is empty, the selector labels do not match any pods.

Step 3

Test DNS-Based Service Discovery

DNS is the primary service discovery mechanism in Kubernetes. Students will exec into a frontend pod and use DNS to find and communicate with the backend Service. This is how real applications discover their dependencies.

🔎 Step 3: DNS-Based Discovery

Exec into a frontend pod and discover the backend via DNS:


# Get a frontend pod name
$ FRONTEND_POD=$(kubectl get pod -n lab-services -l tier=web \
  -o jsonpath='{.items[0].metadata.name}')

# Install curl in the nginx container (for testing)
$ kubectl exec $FRONTEND_POD -n lab-services -- \
  apt-get update -qq && kubectl exec $FRONTEND_POD -n lab-services -- \
  apt-get install -y -qq curl dnsutils > /dev/null

# Test DNS resolution of the backend service
$ kubectl exec $FRONTEND_POD -n lab-services -- nslookup backend-svc

Expected Output:


Server:         10.100.0.10
Address:        10.100.0.10#53

Name:   backend-svc.lab-services.svc.cluster.local
Address: 10.100.45.120
                    

# Curl the backend through the Service
$ kubectl exec $FRONTEND_POD -n lab-services -- curl -s backend-svc/ip

Key Insight: Inside the same namespace, you can reach the Service using just its name (backend-svc). Kubernetes DNS automatically resolves it to the ClusterIP address.

The DNS server address (10.100.0.10) is the CoreDNS ClusterIP. Within the same namespace, pods can use the short Service name. The FQDN is also available. The curl command hits the httpbin /ip endpoint which returns the client IP as seen by the backend, demonstrating that traffic is flowing through the Service. Run curl multiple times to show that different backend pods may respond.

Step 4

Examine DNS Records in Detail

Students will explore the full DNS hierarchy that Kubernetes creates for Services. Understanding the FQDN format is essential for cross-namespace communication.

📑 Step 4: Examine DNS Records

Explore the DNS hierarchy for Services:


# Full FQDN lookup
$ kubectl exec $FRONTEND_POD -n lab-services -- \
  nslookup backend-svc.lab-services.svc.cluster.local

# Dig for detailed DNS record information
$ kubectl exec $FRONTEND_POD -n lab-services -- \
  dig backend-svc.lab-services.svc.cluster.local +short

# Verify the search domains configured in the pod
$ kubectl exec $FRONTEND_POD -n lab-services -- cat /etc/resolv.conf

Expected resolv.conf:


nameserver 10.100.0.10
search lab-services.svc.cluster.local svc.cluster.local cluster.local
options ndots:5

DNS Resolution Hierarchy

Format Scope Example
service-name Same namespace only backend-svc
service.namespace Cross-namespace backend-svc.lab-services
service.namespace.svc Explicit SVC backend-svc.lab-services.svc
service.namespace.svc.cluster.local Fully qualified backend-svc.lab-services.svc.cluster.local
The resolv.conf file shows the DNS search domains. Because of these search domains and the ndots:5 setting, short names like backend-svc are resolved by appending each search domain in order. For cross-namespace communication, you must include at least the namespace in the DNS name. The FQDN ending with a dot (cluster.local.) avoids all search domain lookups and is the most efficient form for production configurations.

Step 5

Create a NodePort Service

NodePort exposes a Service on a static port on every node in the cluster. While not commonly used in production EKS environments (LoadBalancer is preferred), understanding NodePort is important because it is the building block that LoadBalancer Services use internally.

🌐 Step 5: Create a NodePort Service

Expose the frontend externally via NodePort:


# Save as frontend-nodeport.yaml
apiVersion: v1
kind: Service
metadata:
  name: frontend-nodeport
  namespace: lab-services
spec:
  type: NodePort
  selector:
    app: frontend
    tier: web
  ports:
    - name: http
      port: 80
      targetPort: 80
      nodePort: 30080
      protocol: TCP
                

$ kubectl apply -f frontend-nodeport.yaml
$ kubectl get svc frontend-nodeport -n lab-services

Expected Output:


NAME                TYPE       CLUSTER-IP      EXTERNAL-IP   PORT(S)
frontend-nodeport   NodePort   10.100.78.200   <none>        80:30080/TCP
                    

# Get a node's external IP to test
$ kubectl get nodes -o wide

# Test via NodePort (from within the cluster or if nodes are accessible)
$ curl http://<NODE-EXTERNAL-IP>:30080

Note: In EKS, node security groups may block NodePort traffic by default. NodePort is primarily useful for development and testing. Production workloads should use LoadBalancer or Ingress instead.

The nodePort 30080 is allocated on every node in the cluster. Even nodes that do not run frontend pods will accept traffic on this port and forward it to the correct pods. The valid NodePort range is 30000-32767 by default. Point out that NodePort also creates a ClusterIP, so the Service is accessible both internally and externally. In EKS, security groups typically need to be updated to allow NodePort traffic, which is why LoadBalancer is preferred.

Step 6

Create a LoadBalancer Service

LoadBalancer is the standard way to expose Services externally in EKS. It provisions an AWS Network Load Balancer (NLB) or Application Load Balancer (ALB) that routes traffic to the Service. This is the production pattern used at Verisign.

⚡ Step 6: Create a LoadBalancer Service

Expose the frontend via an AWS Load Balancer:


# Save as frontend-lb.yaml
apiVersion: v1
kind: Service
metadata:
  name: frontend-lb
  namespace: lab-services
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "external"
    service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
    service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
spec:
  type: LoadBalancer
  selector:
    app: frontend
    tier: web
  ports:
    - name: http
      port: 80
      targetPort: 80
      protocol: TCP
                

$ kubectl apply -f frontend-lb.yaml

# Watch for the external hostname/IP to be assigned
$ kubectl get svc frontend-lb -n lab-services -w

Expected: After 1–3 minutes, the EXTERNAL-IP field will show an AWS NLB hostname like a1b2c3d4-abc123.elb.us-east-1.amazonaws.com.

Common Issue: If EXTERNAL-IP stays as <pending> for more than 5 minutes, check that the AWS Load Balancer Controller is installed: kubectl get deployment -n kube-system aws-load-balancer-controller

The annotations are specific to the AWS Load Balancer Controller. The nlb-target-type: ip routes directly to pod IPs (bypassing NodePort), which is more efficient. The scheme: internet-facing creates a public-facing NLB. For internal services, use internal instead. The provisioning takes 1-3 minutes as AWS creates the NLB, registers targets, and performs health checks. Students should wait until the EXTERNAL-IP is populated before testing.

⚡ Step 6: Test the Load Balancer

Verify the LoadBalancer is routing traffic:


# Get the LoadBalancer hostname
$ LB_HOST=$(kubectl get svc frontend-lb -n lab-services \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')

# Test the endpoint
$ curl -s http://$LB_HOST

# Verify traffic is being distributed across pods
$ for i in $(seq 1 6); do
    curl -s http://$LB_HOST -o /dev/null -w "Response from: %{remote_ip}\n"
  done

Success: You should see the nginx default page. The NLB distributes traffic across the frontend pods.


# Examine the Service details
$ kubectl describe svc frontend-lb -n lab-services
The describe output shows the LoadBalancer Ingress hostname and the events from the cloud provider. If DNS has not propagated yet, wait a minute and retry. The for loop demonstrates load balancing by making multiple requests. In production, the NLB hostname would be aliased to a friendly domain name via Route53.

Step 7

Explore Endpoints and EndpointSlices

Endpoints and EndpointSlices are the mechanism by which Services track which pods should receive traffic. Understanding these resources is essential for debugging Service connectivity issues.

📐 Step 7: Examine Endpoints

Inspect the Endpoints backing the backend Service:


# View the Endpoints resource
$ kubectl get endpoints backend-svc -n lab-services

# View EndpointSlices (the modern replacement)
$ kubectl get endpointslices -n lab-services -l kubernetes.io/service-name=backend-svc

# Describe for full details
$ kubectl describe endpointslice -n lab-services \
  -l kubernetes.io/service-name=backend-svc

Expected Output:


NAME          ENDPOINTS                                 AGE
backend-svc   10.0.1.15:80,10.0.2.22:80,10.0.3.18:80   10m

Endpoints vs EndpointSlices

  • Endpoints — Legacy API, one resource per Service containing all pod IPs
  • EndpointSlices — Modern API, sharded into smaller objects (max 100 endpoints each), more efficient for large Services
The Endpoints resource lists every pod IP that matches the Service selector and has passed its readiness probe. If a pod fails its readiness probe, its IP is removed from Endpoints and it stops receiving traffic. EndpointSlices are the newer API that scale better for Services with many pods. Both are auto-managed by the endpoints controller. If Endpoints shows fewer IPs than expected, check pod readiness.

📐 Step 7: Watch Endpoints Update in Real Time

Scale the backend and watch Endpoints change:


# In one terminal, watch endpoints
$ kubectl get endpoints backend-svc -n lab-services -w

# In another terminal, scale the backend down
$ kubectl scale deployment backend -n lab-services --replicas=1

# Observe endpoints reduce to 1 IP
# Then scale back up
$ kubectl scale deployment backend -n lab-services --replicas=3

# Observe endpoints grow back to 3 IPs

Key Insight: Endpoints are dynamic. As pods come and go, the Endpoints resource is automatically updated. This is how Services provide seamless load balancing even during rolling updates and scaling events.

This exercise demonstrates the dynamic nature of Endpoints. As pods scale down, their IPs are removed. As new pods become ready, their IPs are added. This happens automatically and is the foundation of zero-downtime deployments. During rolling updates, new pods are added to Endpoints only after passing readiness probes, while old pods are removed before termination.

Step 8

Create a Headless Service

Headless Services (clusterIP: None) do not allocate a virtual IP. Instead, DNS queries return the individual pod IPs directly. This is essential for StatefulSets and any application that needs to address specific pods.

🎯 Step 8: Create a Headless Service

Create a headless Service for direct pod access:


# Save as backend-headless.yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-headless
  namespace: lab-services
spec:
  clusterIP: None
  selector:
    app: backend
    tier: api
  ports:
    - name: http
      port: 80
      targetPort: 80
                

$ kubectl apply -f backend-headless.yaml
$ kubectl get svc backend-headless -n lab-services

Expected: The CLUSTER-IP column shows None.


NAME               TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)
backend-headless   ClusterIP   None         <none>        80/TCP
The key difference is clusterIP: None. Without a virtual IP, the Service cannot load-balance traffic. Instead, it serves as a DNS-only discovery mechanism. This is used with StatefulSets (as seen in Lab 3) and any application that implements its own load balancing or needs to enumerate all backend instances.

🎯 Step 8: Test Headless Service DNS

Compare DNS responses between normal and headless Services:


# Normal ClusterIP Service - returns the virtual IP
$ kubectl exec $FRONTEND_POD -n lab-services -- \
  dig backend-svc.lab-services.svc.cluster.local +short

# Headless Service - returns individual pod IPs
$ kubectl exec $FRONTEND_POD -n lab-services -- \
  dig backend-headless.lab-services.svc.cluster.local +short

Expected:


# ClusterIP Service returns 1 virtual IP:
10.100.45.120

# Headless Service returns all pod IPs:
10.0.1.15
10.0.2.22
10.0.3.18
Key Difference: A ClusterIP Service returns one virtual IP (load balancing happens at the network layer). A headless Service returns all pod IPs (the client decides which pod to connect to).
This is a critical comparison. With a regular ClusterIP Service, the client sees one IP and kube-proxy handles load balancing transparently. With a headless Service, the client receives all pod IPs and must implement its own selection logic. This is how database clients discover all replicas, how Kafka consumers find all brokers, and how custom service meshes enumerate backends.

Step 9

Test Session Affinity

Session affinity ensures that requests from the same client are consistently routed to the same backend pod. This is useful for applications that store session state in memory rather than in a shared store.

🔒 Step 9: Configure Session Affinity

Create a Service with session affinity enabled:


# Save as backend-sticky.yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-sticky
  namespace: lab-services
spec:
  type: ClusterIP
  selector:
    app: backend
    tier: api
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 300
  ports:
    - name: http
      port: 80
      targetPort: 80
                

$ kubectl apply -f backend-sticky.yaml
$ kubectl describe svc backend-sticky -n lab-services | grep -A 2 "Session Affinity"

Expected:


Session Affinity:  ClientIP
Session affinity is configured at the Service level using sessionAffinity: ClientIP. The timeoutSeconds field controls how long the affinity persists (default is 10800 seconds / 3 hours). Kubernetes does not support cookie-based affinity at the Service level; that requires an Ingress controller. ClientIP affinity uses the source IP to hash requests to the same backend.

🔒 Step 9: Verify Sticky Sessions

Send multiple requests and verify they all go to the same pod:


# Without session affinity - requests are distributed
$ for i in $(seq 1 6); do
    kubectl exec $FRONTEND_POD -n lab-services -- \
      curl -s backend-svc/headers | grep -o '"Host":.*'
  done

# With session affinity - all requests go to the same pod
$ for i in $(seq 1 6); do
    kubectl exec $FRONTEND_POD -n lab-services -- \
      curl -s backend-sticky/headers | grep -o '"Host":.*'
  done

Expected: Requests to backend-svc may hit different pods. Requests to backend-sticky consistently hit the same pod (same response pattern).

Caveat: Session affinity is based on the client's source IP. When traffic passes through a load balancer or proxy, all clients may appear to have the same IP, making session affinity ineffective. Use application-level session management for production workloads.

The comparison between the two Services clearly shows the effect of session affinity. Without it, kube-proxy distributes requests across backends using round-robin or random selection. With it, the same source IP always maps to the same backend. The caveat about proxies is important for production. If all requests originate from the same pod (as in this test), they all have the same source IP and will always hit the same backend regardless of session affinity settings.

Step 10

Multi-Port Service

Many applications expose multiple ports (HTTP, HTTPS, metrics, gRPC). Kubernetes Services can expose all of these on a single Service definition, simplifying service discovery.

🔀 Step 10: Create a Multi-Port Service

Expose both HTTP and HTTPS on the same Service:


# Save as frontend-multiport.yaml
apiVersion: v1
kind: Service
metadata:
  name: frontend-multiport
  namespace: lab-services
spec:
  type: ClusterIP
  selector:
    app: frontend
    tier: web
  ports:
    - name: http
      port: 80
      targetPort: 80
      protocol: TCP
    - name: https
      port: 443
      targetPort: 443
      protocol: TCP
                

$ kubectl apply -f frontend-multiport.yaml
$ kubectl get svc frontend-multiport -n lab-services
$ kubectl describe svc frontend-multiport -n lab-services

Expected Output:


NAME                 TYPE        CLUSTER-IP      PORT(S)
frontend-multiport   ClusterIP   10.100.92.50    80/TCP,443/TCP

Important: When a Service has multiple ports, each port must have a name field. Kubernetes uses port names for DNS SRV records and Endpoint identification.

Multi-port Services are common in production. Web servers expose HTTP and HTTPS. Applications expose a main port and a metrics port for Prometheus. gRPC services may expose both gRPC and a health check HTTP port. The name field is mandatory for multi-port Services and is used in DNS SRV records. Clients can also reference ports by name in Ingress and other configurations.

🔀 Step 10: DNS SRV Records for Multi-Port

Examine the SRV records created for named ports:


# Query SRV records for the http port
$ kubectl exec $FRONTEND_POD -n lab-services -- \
  dig SRV _http._tcp.frontend-multiport.lab-services.svc.cluster.local +short

# Query SRV records for the https port
$ kubectl exec $FRONTEND_POD -n lab-services -- \
  dig SRV _https._tcp.frontend-multiport.lab-services.svc.cluster.local +short

Expected: SRV records return the port number and the Service hostname, allowing clients to discover both the address and port dynamically.

Production Pattern: Multi-port Services are commonly used to expose application traffic and Prometheus metrics on separate ports (e.g., port 8080 for HTTP and port 9090 for metrics), all discoverable through a single Service name.
SRV records are an advanced DNS feature that includes port information alongside the hostname. This enables fully dynamic service discovery where clients do not need to know port numbers in advance. While most applications use A records (hostname to IP), some service mesh proxies and advanced clients use SRV records for port discovery. This is also how some database drivers discover replica ports.

Step 11

Clean Up All Resources

Clean up all resources created during the lab. Deleting the namespace removes all objects within it. The LoadBalancer Service deletion triggers AWS to decommission the NLB.

🗑️ Step 11: Clean Up Lab Resources

Remove all resources created during this lab:


# Delete the LoadBalancer Service first (triggers NLB deletion)
$ kubectl delete svc frontend-lb -n lab-services

# Wait for NLB deprovisioning (check AWS console or wait ~60s)
$ echo "Waiting for NLB deprovisioning..."
$ sleep 30

# Delete the entire namespace (removes all remaining resources)
$ kubectl delete namespace lab-services

# Verify cleanup
$ kubectl get all -n lab-services
$ kubectl get svc -n lab-services

Note: Delete the LoadBalancer Service before the namespace to ensure the AWS Load Balancer Controller has time to clean up the NLB. If you delete the namespace first, the NLB may become orphaned and require manual cleanup in AWS.

Cost Warning: Orphaned NLBs incur ongoing AWS charges. Always verify that the NLB has been deprovisioned by checking the AWS console or running aws elbv2 describe-load-balancers.

Order of deletion matters here. The LoadBalancer Service must be deleted while the AWS Load Balancer Controller is still running and the finalizer can execute. If the namespace is force-deleted, the controller may not get a chance to decommission the NLB. In production, this can lead to orphaned resources and unexpected AWS charges. Always verify cleanup in the AWS console.

📚 Lab 4 Summary

Service Type Comparison

Service Type Scope Use Case Access Method
ClusterIP Internal only Service-to-service communication Virtual IP / DNS name
NodePort External via node IP Development, testing node-ip:30000-32767
LoadBalancer External via cloud LB Production external access NLB/ALB hostname
Headless Internal, no VIP StatefulSets, direct pod access DNS returns pod IPs
Review the comparison table with students. This is the most important reference from the lab. In production at Verisign, most inter-service communication uses ClusterIP, external APIs use LoadBalancer with NLB, and databases use headless Services with StatefulSets. NodePort is rarely used in production EKS environments.

📊 Key Concepts Learned

Service Discovery

  • DNS is the primary discovery mechanism (short names and FQDN)
  • SRV records provide port discovery
  • Endpoints track ready pod IPs dynamically

Advanced Features

  • Session Affinity — ClientIP-based sticky sessions
  • Multi-Port and Headless — Multiple ports or direct pod IP discovery
  • EndpointSlices — Scalable endpoint tracking
  • AWS NLB — Production external access
Production Guidance: At Verisign, use ClusterIP for internal APIs, LoadBalancer with NLB annotations for external endpoints, and headless Services for StatefulSets. Always configure readiness probes so only healthy pods receive traffic.
This summary reinforces the practical patterns students will use daily. The production guidance is specific to the Verisign EKS platform. Emphasize the importance of readiness probes, as they control when pods are added to or removed from Service Endpoints. Without readiness probes, traffic can be sent to pods that are not yet ready to handle requests.
