Lab 14: Working with FluxCD

Hands-On GitOps with FluxCD on EKS

Intermediate Kubernetes — Module 14 of 16

This lab provides hands-on experience with FluxCD. Students will inspect the existing Flux installation, create sources and Kustomizations, observe reconciliation, test drift detection, deploy Helm releases, build dependency chains, and troubleshoot common failures. Allow 45-60 minutes for completion.

📋 Lab Overview

What You Will Do

  • Verify the FluxCD installation and inspect existing sources
  • Create a GitRepository source and a Flux Kustomization
  • Observe reconciliation and test drift detection
  • Deploy an application using a HelmRelease
  • Build a dependency chain between Kustomizations
  • Suspend, resume, and troubleshoot Flux resources

Duration: 45–60 minutes

Prerequisites: Flux CLI installed, cluster access configured, kubectl working

Before starting, verify that all students have the Flux CLI installed (flux --version) and can access the cluster (kubectl get nodes). If anyone is missing the Flux CLI, it can be installed with: curl -s https://fluxcd.io/install.sh | sudo bash

Step 1

Verify FluxCD Installation

This first step ensures that Flux is properly installed and all controllers are running. If any controller is not healthy, the rest of the lab will not work correctly.

🔎 Step 1: Verify FluxCD Installation

Run the following commands to confirm FluxCD is installed and healthy:


# Check Flux installation health
$ flux check

# Expected output:
# ✔ kubectl 1.28.0 >= 1.20.0-0
# ✔ Kubernetes 1.28.0 >= 1.25.0-0
# ✔ source-controller: deployment ready
# ✔ kustomize-controller: deployment ready
# ✔ helm-controller: deployment ready
# ✔ notification-controller: deployment ready
# ✔ All checks passed
                

# View Flux controller pods
$ kubectl get pods -n flux-system

# List Flux CRDs installed in the cluster
$ kubectl get crds | grep flux
Expected: All four controllers should show `Ready` status. You should see CRDs including `gitrepositories`, `kustomizations`, `helmreleases`, `providers`, and `alerts`. Walk around and verify that every student sees all checks passing. If source-controller is not ready, check its logs with kubectl logs -n flux-system deploy/source-controller. Common issues at this point include: Flux not installed (run the bootstrap command), or the flux-system namespace not existing. The CRD list should show approximately 10-12 Flux CRDs depending on whether image automation controllers are installed.

Step 2

Explore Existing Sources

Now that we have confirmed Flux is running, let us look at what sources are already configured. In the Verisign platform, Terraform has already set up the initial sources.

🔎 Step 2: Explore Existing Sources

Inspect the Git and Helm sources that are already configured:


# List all Git sources
$ flux get sources git

# List all Helm repository sources
$ flux get sources helm

# Get detailed YAML for a specific Git source
$ kubectl get gitrepository -n flux-system -o yaml

# Examine the source details
$ flux get sources git -o wide
Observe: Note the REVISION column showing the current Git commit SHA, the READY status, and the AGE showing when the source was last fetched. Check the spec.interval and spec.ref fields in the YAML output.
Questions to answer: What branch is each source tracking? What is the reconciliation interval? When was the last successful fetch? Have students examine the output carefully. The revision field shows the exact commit SHA that Flux has fetched. The interval shows how often Flux checks for new commits. Ask students to identify the relationship between the source interval and how quickly changes would be deployed. The wide output format shows additional columns including the URL and the last fetched time.

Step 3

Create a GitRepository Source

In this step, students create their own GitRepository source. This teaches the fundamental building block that all other Flux resources depend on.

📁 Step 3: Create a GitRepository Source

Create a new GitRepository source pointing to a sample repository:


# Save as git-source.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: lab-sample-app
  namespace: flux-system
spec:
  interval: 2m
  url: https://github.com/fluxcd/flux2-kustomize-helm-example.git
  ref:
    branch: main
                

# Apply the source
$ kubectl apply -f git-source.yaml

# Watch it become Ready
$ flux get sources git lab-sample-app --watch

# Verify the artifact was fetched
$ kubectl describe gitrepository lab-sample-app -n flux-system
Expected: The source should transition to `READY: True` within a few seconds. The REVISION column should show the latest commit SHA from the main branch. This is a public repository, so no authentication is needed. The source should become Ready almost immediately. If it does not, check the events with flux events --for GitRepository/lab-sample-app. Common issues include DNS resolution failures or network policies blocking egress to GitHub. Once the source is Ready, it has fetched and stored the repository contents as a tarball artifact that other controllers can consume.

Step 4

Create a Kustomization

Now we connect a Kustomization to the source we just created. This tells Flux to apply the manifests from a specific path within the repository.

📑 Step 4: Create a Flux Kustomization

Create a Kustomization that references the GitRepository source:


apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata: { name: lab-sample-app, namespace: flux-system }
spec:
  interval: 5m
  sourceRef: { kind: GitRepository, name: lab-sample-app }
  path: ./clusters/staging
  prune: true
  timeout: 2m
                

# Apply the Kustomization
$ kubectl apply -f lab-kustomization.yaml

# Watch the reconciliation
$ flux get kustomizations lab-sample-app --watch

# See what resources Flux created
$ flux events --for Kustomization/lab-sample-app
Tip: Watch the Kustomization transition through Progressing to Ready. Flux is running kustomize build on the path and applying the resulting manifests to the cluster. The Kustomization points to the clusters/staging path within the sample repository. Flux will build the Kustomize overlay at that path and apply all resulting resources. Students should observe the resources appearing in the cluster. Have them run kubectl get all in the relevant namespaces to see what was created. The prune: true setting means that if we later delete this Kustomization, Flux will also delete all the resources it created.

Step 5

Observe Reconciliation

This step demonstrates the core GitOps loop: make a change in the source, and watch Flux apply it automatically.

🔄 Step 5: Observe Reconciliation in Action

Force a reconciliation and observe Flux applying changes:


# Check the current state of managed resources
$ flux get kustomizations lab-sample-app
$ kubectl get deployments -A -l kustomize.toolkit.fluxcd.io/name=lab-sample-app

# Force an immediate reconciliation
$ flux reconcile kustomization lab-sample-app --with-source

# Watch the events during reconciliation
$ flux events --for Kustomization/lab-sample-app

# Check the revision that was applied
$ flux get kustomizations lab-sample-app

What Just Happened?

  1. The --with-source flag triggered the source controller to re-fetch from Git
  2. The kustomize controller detected the (potentially updated) artifact
  3. It ran kustomize build on the configured path
  4. It applied the resulting manifests using server-side apply
  5. It verified health checks (if configured) and reported Ready
Even if nothing changed in Git, the reconciliation still runs and verifies the cluster state matches the desired state. This is the continuous reconciliation loop in action. Point out the revision in the output: if it matches the previous reconciliation, it means the source had not changed. If it is different, new commits were picked up. The with-source flag is important; without it, Flux would reconcile against the cached artifact, which might be stale.

✅ Step 5: Checkpoint

Success: The Kustomization shows Ready: True and the REVISION column reflects the latest commit SHA. The flux events output confirms a successful reconciliation cycle.

Step 6

Test Drift Detection

This is one of the most impactful exercises in the lab. Students will make a manual change and watch Flux revert it. This drives home the GitOps principle that Git is the single source of truth.

🔍 Step 6: Test Drift Detection

Introduce manual drift and watch Flux correct it:


$ kubectl get deployments -A -l kustomize.toolkit.fluxcd.io/name=lab-sample-app

$ kubectl get deployment <DEPLOYMENT_NAME> -n <NAMESPACE> \
    -o jsonpath='{.spec.replicas}'

$ kubectl scale deployment <DEPLOYMENT_NAME> -n <NAMESPACE> --replicas=10

$ kubectl get deployment <DEPLOYMENT_NAME> -n <NAMESPACE>

$ flux reconcile kustomization lab-sample-app

$ kubectl get deployment <DEPLOYMENT_NAME> -n <NAMESPACE> --watch
Observe: The replica count reverts to the value defined in Git. This is Flux enforcing the desired state. In production, this would also happen automatically at the next reconciliation interval without any manual trigger. This exercise is the moment where GitOps clicks for most students. They manually scale to 10 replicas, see it take effect, then watch Flux revert it back. Emphasize that in production, they would not need to run the reconcile command; it would happen automatically at the next interval. Ask students: what would happen if you needed to truly change the replica count? The answer is to update the manifest in Git, commit, and push. Flux will pick up the change on the next reconciliation.

✅ Step 6: Checkpoint

Success: After manually scaling to 10 replicas and triggering reconciliation, Flux reverted the replica count to the Git-defined value. This confirms drift detection and correction are working.

Step 7

Create a HelmRelease

Now we switch from plain manifests to Helm charts. This step shows how Flux manages the Helm release lifecycle declaratively through the HelmRelease CRD.

⛐️ Step 7: Create a HelmRelease (Part 1 — Source)

First, create a HelmRepository source for the Bitnami chart registry:


# Save as helm-source.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: bitnami
  namespace: flux-system
spec:
  interval: 30m
  url: https://charts.bitnami.com/bitnami
                

# Apply the Helm source
$ kubectl apply -f helm-source.yaml

# Verify it becomes Ready
$ flux get sources helm bitnami
Note: The HelmRepository interval is set to 30 minutes because chart indexes change less frequently than Git repositories. Flux caches the chart index and only re-fetches at this interval. The HelmRepository source is similar to running helm repo add. It tells Flux where to find Helm charts. The interval for Helm repos can be longer since chart indexes are updated less frequently. Once this source is Ready, we can reference it from a HelmRelease.

⛐️ Step 7: Create a HelmRelease (Part 2 — Release)

Now create a HelmRelease to deploy nginx using the Bitnami chart:


apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: lab-nginx
  namespace: default
spec:
  interval: 15m
  chart:
    spec:
      chart: nginx
      version: ">=15.0.0 <16.0.0"
      sourceRef: { kind: HelmRepository, name: bitnami, namespace: flux-system }
  values:
    replicaCount: 2
    service: { type: ClusterIP }
  install:
    remediation: { retries: 3 }
  upgrade:
    remediation: { retries: 3, remediationStrategy: rollback }
The chart version uses a semver range, meaning Flux will automatically pick up the latest patch release within the 15.x range. The values section overrides the chart defaults. The remediation section ensures that if the install or upgrade fails, Flux will retry up to 3 times and rollback on failure. This makes the release self-healing.

⛐️ Step 7: Create a HelmRelease (Part 3 — Verify)


$ kubectl apply -f helm-release.yaml

$ flux get helmreleases lab-nginx --watch

$ helm list -n default

$ kubectl get all -n default -l app.kubernetes.io/instance=lab-nginx

$ flux get helmreleases lab-nginx -o wide
Key Observation: The helm list command shows the release because Flux uses the Helm SDK internally. The release is fully Helm-compatible. However, running helm upgrade manually would create drift that Flux would revert on the next reconciliation.
Point out that Flux-managed Helm releases appear in helm list just like manually installed releases. The difference is that Flux owns the lifecycle. If a student runs helm upgrade manually, Flux will detect the drift and revert to the declared state on the next reconciliation. The -o wide flag shows additional details including the chart version and the last applied values hash.

✅ Step 7: Checkpoint

Success: The HelmRelease lab-nginx shows Ready: True and appears in helm list. The nginx pods are running in the default namespace with the configured replica count.

Step 8

Dependency Chains

This step demonstrates how dependsOn ordering works. Students will create two Kustomizations where one depends on the other and observe the ordering behavior.

🔗 Step 8: Build a Dependency Chain

Create two Kustomizations where B depends on A:


apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: lab-infra-base
  namespace: flux-system
spec:
  interval: 5m
  sourceRef: { kind: GitRepository, name: lab-sample-app }
  path: ./infrastructure/controllers
  prune: true
---
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: lab-infra-configs
  namespace: flux-system
spec:
  dependsOn: [{name: lab-infra-base}]
  sourceRef: { kind: GitRepository, name: lab-sample-app }
  path: ./infrastructure/configs
  prune: true
This creates two Kustomizations from the same source but different paths. The second one depends on the first. When applied, Flux will reconcile lab-infra-base first, wait for it to be Ready, and only then begin reconciling lab-infra-configs. If the base fails, configs will remain blocked.

🔗 Step 8: Verify Dependency Ordering


$ kubectl apply -f dependency-chain.yaml

$ flux get kustomizations --watch

# lab-infra-configs ... waiting for dependency 'lab-infra-base' ...
# lab-infra-base    ... True   Applied revision: main@sha1:...
# lab-infra-configs ... True   Applied revision: main@sha1:...

$ flux events --for Kustomization/lab-infra-configs
Expected: The `lab-infra-configs` Kustomization shows "waiting for dependency" until `lab-infra-base` becomes Ready. Once the base is healthy, the dependent Kustomization begins its reconciliation. The key observation is the timing. Have students note the timestamps in the events. The configs Kustomization should have a later start time than the base. If the base fails, the configs Kustomization will remain in a waiting state indefinitely until the base recovers. This is exactly how the Verisign platform dependency tree works at a larger scale.

Step 9

Suspend and Resume

Suspend and resume are essential operational controls. They allow you to temporarily pause reconciliation during maintenance, debugging, or emergency situations.

⏸️ Step 9: Suspend and Resume Reconciliation


$ flux suspend kustomization lab-sample-app

$ flux get kustomizations lab-sample-app

# Make a manual change (will NOT be reverted while suspended)
$ kubectl scale deployment <DEPLOYMENT_NAME> -n <NAMESPACE> --replicas=5
$ kubectl get deployment <DEPLOYMENT_NAME> -n <NAMESPACE>

$ flux resume kustomization lab-sample-app
$ kubectl get deployment <DEPLOYMENT_NAME> -n <NAMESPACE> --watch
Important: Suspending a Kustomization also blocks all dependents. If you suspend a core component, everything that depends on it will not reconcile until you resume. This exercise clearly demonstrates the difference between suspended and active reconciliation. While suspended, manual changes persist because Flux is not checking. The moment you resume, Flux detects the drift and corrects it. This is useful during incident response when you need to make a temporary manual fix and do not want Flux reverting it while you work on a proper Git commit. Always remember to resume after maintenance is complete.

Step 10

Troubleshooting

In this step, students deliberately introduce an error and practice the troubleshooting workflow from the presentation. This builds muscle memory for diagnosing real issues.

🔧 Step 10: Troubleshooting (Part 1 — Introduce Error)

Create a Kustomization with a deliberate error to practice troubleshooting:


# Save as broken-kustomization.yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: lab-broken
  namespace: flux-system
spec:
  interval: 5m
  sourceRef:
    kind: GitRepository
    name: lab-sample-app
  path: ./this/path/does/not/exist   # Deliberate error!
  prune: true
                

# Apply the broken Kustomization
$ kubectl apply -f broken-kustomization.yaml

# Check the status
$ flux get kustomizations lab-broken
# READY should be "False" with an error message
Expected: The Kustomization will fail with a message like "kustomization path not found" because the specified directory does not exist in the source repository. This simulates a common real-world error: pointing a Kustomization at the wrong path. Students will now practice the troubleshooting workflow to diagnose and fix the issue. The error message from Flux is usually clear and actionable, but knowing where to look is the important skill.

🔧 Step 10: Troubleshooting (Part 2 — Diagnose and Fix)

Use the troubleshooting workflow to diagnose and resolve the issue:


$ flux get kustomizations lab-broken

$ flux events --for Kustomization/lab-broken

$ flux logs --kind=Kustomization --name=lab-broken --level=error

                

# Step 5: Re-apply the fixed Kustomization
$ kubectl apply -f broken-kustomization.yaml

# Step 6: Force reconciliation and verify
$ flux reconcile kustomization lab-broken
$ flux get kustomizations lab-broken
# READY should now be "True"
Success: After fixing the path and reconciling, the Kustomization should transition to Ready. This workflow — check status, read events, read logs, fix, reconcile — applies to all Flux troubleshooting. Walk students through each step of the diagnostic workflow. Emphasize that this is the same process for any Flux issue: status, events, logs, fix, reconcile. The error message in the status field usually tells you exactly what is wrong. Events provide a timeline. Logs provide the most detail. Once the fix is applied, force reconciliation to verify immediately rather than waiting for the interval.

✅ Step 10: Checkpoint

Success: After fixing the path and reconciling, the lab-broken Kustomization transitioned to Ready: True. You have practiced the full troubleshooting workflow: check status, read events, read logs, fix, and reconcile.

Step 11

Clean Up

Clean up all resources created during the lab. With prune enabled, deleting the Kustomizations will also clean up the resources they created.

🗑️ Step 11: Clean Up Lab Resources

Remove all resources created during this lab:


# Delete Kustomizations (prune: true will clean up managed resources)
$ kubectl delete kustomization lab-sample-app -n flux-system
$ kubectl delete kustomization lab-broken -n flux-system
$ kubectl delete kustomization lab-infra-base -n flux-system
$ kubectl delete kustomization lab-infra-configs -n flux-system

$ kubectl delete helmrelease lab-nginx -n default

$ kubectl delete gitrepository lab-sample-app -n flux-system
$ kubectl delete helmrepository bitnami -n flux-system

$ flux get all
$ kubectl get all -n default -l app.kubernetes.io/instance=lab-nginx
Success: Once flux get all shows no lab resources and kubectl get all -n default confirms the nginx pods are gone, the cleanup is complete. Because prune: true was set, deleting the Kustomization also deletes the resources it managed. The HelmRelease deletion triggers a helm uninstall automatically. The order of deletion matters: delete Kustomizations before their sources so that Flux can properly clean up managed resources.

📊 Key CRDs & Reconciliation Flow

CRDs You Created

  • GitRepository / HelmRepository — Source definitions
  • Kustomization — Applied manifests with pruning and health checks
  • HelmRelease — Managed a Helm chart lifecycle

Reconciliation Flow

  1. Source fetches artifact from external system
  2. Controller builds and applies manifests
  3. Controller verifies health checks and reports status
  4. Dependents begin their reconciliation
Remember: Git is the source of truth. Manual changes are drift. Flux corrects drift automatically. Always commit changes to Git.
This summary reinforces the two most important concepts from the lab: the CRDs that define the desired state, and the reconciliation flow that converges actual state to desired state. The key point at the bottom is the single most important takeaway for developers working with GitOps on the Verisign platform.
